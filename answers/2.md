# Oscilar - Orchestration Layer và Quản lý Workflow trong Fintech

## 1. Oscilar có phải là Orchestration Layer không?

**Câu trả lời: Có, một phần.**

Oscilar không chỉ đơn thuần là một **API Gateway** mà còn đóng vai trò là một **Orchestration Layer** (lớp điều phối) trong kiến trúc fintech. Tuy nhiên, mức độ orchestration phụ thuộc vào cách triển khai cụ thể.

### Phân biệt các vai trò:

| Vai trò | Chức năng chính | Oscilar |
|---------|----------------|---------|
| **API Gateway** | Routing, authentication, rate limiting | ✅ |
| **Orchestration Layer** | Điều phối luồng nghiệp vụ, gọi nhiều service | ✅ (tùy cấu hình) |
| **Service Mesh** | Giao tiếp service-to-service | ❌ (không phải chức năng chính) |
| **ESB (Enterprise Service Bus)** | Tích hợp legacy systems | ✅ (có adapter) |

### b. Các mẫu thiết kế (Design Patterns) được áp dụng

| Pattern | Mục đích | Ví dụ trong Oscilar |
|---------|----------|---------------------|
| **Saga Pattern** | Quản lý distributed transaction | Chuyển tiền qua nhiều ngân hàng |
| **Circuit Breaker** | Prevent cascade failure | Khi third party down |
| **Retry Pattern** | Tự động thử lại khi lỗi | Network timeout |
| **Compensating Transaction** | Rollback khi failure | Hoàn tiền khi giao dịch lỗi |
| **Choreography** | Event-based coordination | Webhook notifications |

## 4. So sánh: Orchestration vs Choreography trong Oscilar

### Orchestration (Tập trung)

```
                    ┌─────────────┐
                    │ Oscilar     │
                    │ Orchestrator│
                    └──────┬──────┘
                           │
         ┌─────────────────┼─────────────────┐
         │                 │                 │
         ▼                 ▼                 ▼
   ┌──────────┐     ┌──────────┐     ┌──────────┐
   │ Service A│     │ Service B│     │ Service C│
   └──────────┘     └──────────┘     └──────────┘
```

**Ưu điểm:**
- Dễ theo dõi luồng nghiệp vụ
- Debug và monitoring tập trung
- Phù hợp workflow phức tạp

**Nhược điểm:**
- Single point of failure
- Có thể trở thành bottleneck

### Choreography (Phân tán)

```
   ┌──────────┐       ┌──────────┐       ┌──────────┐
   │ Service A│──────▶│ Service B│──────▶│ Service C│
   └──────────┘ event └──────────┘ event └──────────┘
```

**Ưu điểm:**
- Phân tán, không có bottleneck
- Các service độc lập

**Nhược điểm:**
- Khó debug và theo dõi
- Phức tạp khi workflow có nhiều nhánh

## 5. Workflow Definition Language

Oscilar thường sử dụng các chuẩn để định nghĩa workflow:

### a. BPMN 2.0 (Business Process Model and Notation)

```xml
<process id="moneyTransfer" name="Money Transfer">
  <startEvent id="start"/>
  <serviceTask id="validateCustomer" name="Validate Customer"/>
  <serviceTask id="checkBalance" name="Check Balance"/>
  <exclusiveGateway id="fraudCheck"/>
  <serviceTask id="fraudDetection" name="Fraud Detection"/>
  <endEvent id="end"/>
</process>
```

### b. YAML/JSON DSL

```yaml
workflow:
  name: loan-application
  version: "1.0"
  steps:
    - id: credit-check
      type: service-call
      service: credit-bureau-api
      timeout: 5s
      retry:
        max: 3
        backoff: exponential
        
    - id: approval-decision
      type: decision
      rules:
        - condition: creditScore >= 700
          next: approve
        - condition: creditScore >= 500
          next: manual-review
        - default: reject
```

### c. AsyncAPI (cho event-driven workflows)

```yaml
asyncapi: 2.6.0
channels:
  payment/processed:
    subscribe:
      operationId: onPaymentProcessed
      message:
        payload:
          type: object
          properties:
            transactionId: string
            status: string
```

## 6. Quản lý State và Persistence

Oscilar cần lưu trữ trạng thái workflow:

```
┌─────────────────────────────────────────┐
│          Workflow State Store            │
├─────────────────────────────────────────┤
│  • In-memory (Redis) → Fast, volatile   │
│  • Database (PostgreSQL) → Persistent   │
│  • Event Store (Event Sourcing)         │
└─────────────────────────────────────────┘
```

### State Machine Example:

```
[Created] → [Validated] → [Approved] → [Processing] → [Completed]
     │            │             │              │
     │            │             │              └──▶ [Failed]
     │            │             └──▶ [Rejected]
     │            └──▶ [Manual Review]
     └──▶ [Cancelled]
```
