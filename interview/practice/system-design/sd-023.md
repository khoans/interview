**23.** What is cache invalidation? Why is it called one of the two hard problems in CS?

========== THIS SECTION IS THE ANSWER ==========

Cache invalidation is ensuring cached data stays in sync with the source of truth. It's hard because you need to know when data changes, which caches hold that data, and how to update them without race conditions — and in a distributed system, all of this gets exponentially harder.

In our project, we had a painful cache invalidation bug. When a dealer updated a listing price in PostgreSQL, we invalidated the Redis cache for that listing's detail page. But we forgot to invalidate the search results cache that also displayed the price. So buyers saw the old price in search results but the new price on the detail page — for up to 60 seconds until the search cache TTL expired. The fix was a cache invalidation event: any listing write published an SNS message, and the search cache subscriber invalidated all cached search results containing that listing. This worked but was expensive — one listing update could invalidate hundreds of cached search queries.

The trade-off: TTL-based expiry is simple but tolerates staleness. Event-driven invalidation is precise but complex and can cause cache stampedes (thousands of invalidated entries repopulating simultaneously). We used TTL as the safety net (60-second max staleness) plus event-driven invalidation for critical data (pricing). The lesson: perfect cache invalidation is nearly impossible — design your system to tolerate bounded staleness where the business allows it.
