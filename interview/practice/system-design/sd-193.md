**193.** What is seccomp and AppArmor in Kubernetes? How do they restrict container syscalls?

========== THIS SECTION IS THE ANSWER ==========

Seccomp (Secure Computing Mode) restricts which Linux system calls a container can make -- it uses a whitelist/blacklist of allowed syscalls. AppArmor restricts file access, network access, and capabilities using profiles loaded on the node. Both provide kernel-level defense against container breakouts by limiting what a compromised process can do even within its container.

In our marketplace, we ran containers on ECS with the default Docker seccomp profile, which blocks around 44 dangerous syscalls (like reboot, mount, ptrace). In Kubernetes, you set the seccomp profile in the Security Context: securityProfile.type=RuntimeDefault enables the container runtime default profile. For stricter control, create a custom seccomp profile that only allows the ~100 syscalls your Java application actually uses (write, read, futex, epoll for NIO). AppArmor would further restrict our Spring Boot container to only read /app/ and /tmp/, write to /tmp/ and mounted volumes, and make network connections on specific ports.

The trade-off: custom profiles require profiling your application to know which syscalls it uses -- too restrictive and the app crashes with permission denied errors. The RuntimeDefault profile is a safe starting point that blocks the most dangerous syscalls without breaking standard applications. Custom profiles provide stronger security but require maintenance as your application evolves.
