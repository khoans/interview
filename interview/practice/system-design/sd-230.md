**230.** What is dependency confusion / typosquatting attack? How do you prevent it?

Dependency confusion tricks package managers into downloading a malicious public package instead of an intended private one by publishing a package with the same name but a higher version number to a public registry. Typosquatting publishes packages with names similar to popular ones (e.g., sprig-boot-starter instead of spring-boot-starter).

On our platform with dozens of Spring Boot microservices pulling from Maven Central and npm packages for our Next.js frontend, we mitigated this by: pinning exact dependency versions in pom.xml and package-lock.json, configuring Maven to only resolve from trusted repositories (Maven Central plus our private Nexus), using Dependabot to flag unexpected version jumps, and reviewing any new dependency additions in PR code review.

The trade-off is convenience vs security. Allowing version ranges like ^1.2.0 in npm makes updates easy but opens the door to malicious versions. Pinning versions is safer but creates maintenance burden for keeping dependencies current.
