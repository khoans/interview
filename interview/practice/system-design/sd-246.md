**246.** What are HTTP methods (GET, POST, PUT, PATCH, DELETE)? Explain idempotency for each.

HTTP methods define the action on a resource. **GET** retrieves data (idempotent, safe). **POST** creates a resource (NOT idempotent -- calling twice creates two resources). **PUT** replaces an entire resource (idempotent -- same result regardless of repetition). **PATCH** partially updates a resource (not guaranteed idempotent). **DELETE** removes a resource (idempotent -- deleting twice yields same state).

On our automotive marketplace REST APIs: GET /api/vehicles/{id} fetched a vehicle listing (cached in ElastiCache Redis), POST /api/vehicles created a new listing (non-idempotent, so we added an idempotency key header to prevent duplicate listings from network retries), PUT /api/vehicles/{id} replaced the full listing, PATCH /api/vehicles/{id} updated specific fields like price, and DELETE /api/vehicles/{id} soft-deleted a listing. For our SQS message processing, idempotency was critical -- we stored processed message IDs in DynamoDB to handle SQS at-least-once delivery.

The lesson: in distributed systems with retries (SQS, network timeouts), assuming idempotency without implementing it causes data corruption. Always design POST endpoints with explicit idempotency keys.
