**6.** What is the difference between latency and throughput? How do you optimize each?

========== THIS SECTION IS THE ANSWER ==========

Latency is how long a single request takes (measured in milliseconds). Throughput is how many requests the system handles per second (measured in RPS/QPS). They're related but not the same — you can have low latency but low throughput (one fast server), or high throughput but high latency (batch processing).

In our project, our search API target was: p50 latency under 200ms, p99 under 500ms, throughput of 2000 RPS. To optimize latency, we added caching (Redis), optimized Solr queries, and used connection pooling. To optimize throughput, we scaled horizontally (more ECS instances) and used async processing for non-critical work (analytics events sent to SQS instead of processed inline).

The trade-off: optimizing for one can hurt the other. Adding caching reduces latency but doesn't directly increase throughput if your bottleneck is CPU. Adding more instances increases throughput but doesn't fix a slow database query. You need to identify which is your actual bottleneck — use APM tools (New Relic) to find where time is spent before optimizing blindly.
