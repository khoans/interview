**299.** You need to encrypt sensitive columns in an existing database with millions of rows. What is your migration strategy?

========== THIS SECTION IS THE ANSWER ==========

Use a phased dual-write approach: add encrypted columns, dual-write, backfill in batches, switch reads, then drop old columns. Never do a big-bang migration on a live production database.

On our marketplace with PostgreSQL on RDS, if we needed to encrypt dealer financial columns, the approach would be: (1) add new encrypted columns via Flyway migration, (2) deploy code that writes to both old and new columns simultaneously but reads from old, (3) run a batch job backfilling existing rows in chunks of 1,000 using KMS envelope encryption to avoid table locks and KMS throttling, (4) verify backfill completeness with a count comparison query, (5) switch application reads to encrypted columns, (6) drop plaintext columns in a follow-up Flyway migration.

The trade-off: this multi-phase approach takes weeks and temporarily doubles storage for affected columns. But a single ALTER TABLE on millions of rows locks the table, causes downtime, and if it fails midway you have a partially encrypted mess. Slow and safe beats fast and broken for production data.
