**37.** What is Apache Kafka? How does it differ from traditional message queues?

========== THIS SECTION IS THE ANSWER ==========

Kafka is a distributed event streaming platform. Unlike traditional queues (SQS, RabbitMQ) where messages are deleted after consumption, Kafka retains messages in an append-only log for a configurable retention period. Multiple consumer groups can independently read the same topic at different offsets. Kafka also provides ordering guarantees within partitions and high throughput through sequential disk I/O.

In our project, we used SQS rather than Kafka because our message volumes (~10K events/sec peak) didn't justify Kafka's operational overhead. However, the data engineering team used Kafka for the clickstream pipeline — every user action (search, page view, click, lead submission) was published to Kafka topics. The real-time analytics consumer read events at the current offset for live dashboards, while a batch consumer read the same events from an earlier offset for daily aggregation. This "replay" capability was impossible with SQS — once consumed, messages are gone. Kafka retained 7 days of events, allowing the data team to reprocess if they found a bug in their analytics logic.

The trade-off: Kafka gives you message replay, ordering, and high throughput but requires managing brokers, Zookeeper (or KRaft), partitions, and consumer group rebalancing — significant operational complexity. SQS is fully managed, zero-ops, but no replay and no ordering guarantees (standard queues). The lesson: use SQS for simple async decoupling between microservices; use Kafka when you need event replay, ordering, or multiple consumers reading the same stream independently.
