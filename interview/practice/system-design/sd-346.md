**346.** CQRS: When does separating read and write models pay off? When is it over-engineering?

========== THIS SECTION IS THE ANSWER ==========

CQRS pays off when read and write patterns are drastically different — different data shapes, different scaling needs, or different performance requirements. It is over-engineering when reads and writes operate on the same model at the same scale.

On our marketplace, we effectively had CQRS without calling it that. Writes went to PostgreSQL (normalized, ACID, dealer-centric model) while reads came from Solr (denormalized, search-optimized, consumer-centric model). The write model was optimized for dealer operations — update a listing with proper validation and relationships. The read model was optimized for 30M consumers — fast, faceted search across millions of listings with relevance scoring. These were fundamentally different data shapes serving different users at different scales (thousands of dealer writes vs millions of consumer reads).

The trade-off: CQRS gives you independent optimization and scaling of reads and writes, but adds complexity — you maintain two models, a synchronization mechanism (our SQS events), and handle eventual consistency. For a simple CRUD app where reads and writes use the same model at similar scale, CQRS doubles your code for no benefit. Only adopt it when the read/write asymmetry is genuinely painful.
