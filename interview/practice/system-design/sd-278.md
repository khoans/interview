**278.** What is impersonation / delegation in authentication? When is it legitimate?

Impersonation allows an admin to act as another user, seeing exactly what they see. Delegation allows one service or user to act on behalf of another with limited permissions (like OAuth2 -- the app acts on behalf of the user with scoped permissions). Both are legitimate when properly controlled and audited.

On our marketplace, we had legitimate impersonation for customer support: when a dealer reported issues with their listing dashboard, our support team needed to see the exact view. We implemented "act-as" functionality where a PLATFORM_ADMIN could impersonate a DEALER_ADMIN with full audit logging -- every action taken while impersonating was logged with both the real admin ID and the impersonated user ID in Kibana. The impersonation session was time-limited (30 minutes) and read-only by default. For delegation, our Lambda functions used OAuth2 on-behalf-of flow to call APIs with the original user context.

The trade-off: impersonation is a powerful tool that can be abused. Without proper audit trails and restrictions, an admin could impersonate any user and take actions without accountability. We required a support ticket ID to initiate impersonation, logged everything, and reviewed impersonation logs monthly.
