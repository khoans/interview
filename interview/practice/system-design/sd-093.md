**93.** Design a notification system. How do you handle push, email, and SMS at scale?

========== THIS SECTION IS THE ANSWER ==========
The core design: a notification service receives events, resolves user preferences and channels, then fans out to channel-specific workers (push, email, SMS) via message queues. Each channel has its own queue, retry policy, and rate limits based on the downstream provider.

**Real example:** On our automotive marketplace, when a saved search matched new listings, we sent notifications via SNS triggering SQS queues per channel. The email queue consumed messages and sent via Amazon SES (bulk-optimized). Push notifications went through Firebase Cloud Messaging. Each queue had its own DLQ for failures — SES bounces and complaints fed back into a suppression list in DynamoDB. We processed ~500K notification events daily with user preferences stored in PostgreSQL (email only, push only, both, or none).

**Trade-off:** Sending all channels from a single queue creates head-of-line blocking — slow SMS delivery blocks email sends. Separate queues per channel solve this but add operational complexity (more queues to monitor). Also, deduplication is critical: we once had a bug where a retry storm sent the same price-drop email 3 times to users. We added an idempotency key (hash of userId + eventId + channel) checked against a Redis set with 24-hour TTL.
