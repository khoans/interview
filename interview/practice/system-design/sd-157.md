**157.** What is API key vs OAuth token vs JWT? When would you use each?

========== THIS SECTION IS THE ANSWER ==========

API keys are simple static strings that identify the calling application -- good for server-to-server calls where you trust both sides. OAuth tokens are temporary credentials obtained through an authorization flow -- good for delegated access on behalf of users. JWTs are self-contained tokens carrying claims that can be verified without a database lookup -- they are often the format used for OAuth access or ID tokens.

In our marketplace: (1) API keys identified partner integrations -- each dealer management system got a unique key stored in AWS Secrets Manager, used for rate limiting and billing attribution. (2) OAuth2 access tokens were used when third-party apps acted on behalf of dealers (e.g., updating their listings). (3) JWTs were the token format for our internal auth -- after login, Spring Security issued a JWT containing userId, roles, and dealerId, which our microservices validated locally without calling the auth service. The JWT was signed with a key from KMS.

The trade-off: API keys are simple but cannot represent user context or expire gracefully. OAuth tokens add complexity but enable scoped, temporary access. JWTs avoid database lookups but cannot be revoked instantly -- we used short TTL (15 min) plus refresh tokens to mitigate this.
