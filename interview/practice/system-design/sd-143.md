**143.** What is XSS (Cross-Site Scripting)? Explain stored, reflected, and DOM-based XSS.

========== THIS SECTION IS THE ANSWER ==========
XSS injects malicious scripts into web pages viewed by other users. Stored XSS: the script is persisted in the database (e.g., in a comment field) and served to every user who views it. Reflected XSS: the script is included in a URL parameter and reflected back in the response. DOM-based XSS: the script manipulates the client-side DOM directly without server involvement (e.g., reading from document.location and injecting into innerHTML).

**Real example:** On our automotive marketplace, stored XSS was our biggest concern — dealers could enter vehicle descriptions with HTML, and buyers viewed those descriptions. We prevented it with output encoding: Next.js (React) auto-escapes all rendered content by default. For the rare cases where we needed to render HTML (formatted vehicle descriptions), we used DOMPurify to sanitize HTML before rendering with dangerouslySetInnerHTML. On the backend, Spring Boot returned Content-Type: application/json for API responses and set X-XSS-Protection and Content-Security-Policy headers via Spring Security's headers() configuration.

**Trade-off:** Aggressive input sanitization can break legitimate content (a dealer's description containing angle brackets or special characters). Output encoding (escaping at render time) is safer than input sanitization because it preserves the original data and only escapes when rendering in an HTML context. Content-Security-Policy with strict-dynamic is the strongest defense but requires careful setup — we once blocked our own analytics scripts with an overly strict CSP policy.
