**179.** What is etcd? Why is it critical for Kubernetes and how do you back it up?

========== THIS SECTION IS THE ANSWER ==========

etcd is a distributed key-value store that serves as the single source of truth for all Kubernetes cluster state -- every pod, service, secret, configmap, and RBAC rule is stored in etcd. If etcd is lost, the entire cluster state is lost. It uses the Raft consensus protocol for leader election and data replication across an odd number of members (typically 3 or 5).

In our marketplace on EKS, AWS managed etcd for us (one of the key benefits of managed Kubernetes). For a self-managed cluster, you would run etcd as a 3-node cluster across availability zones with periodic snapshots: etcdctl snapshot save /backup/etcd-snapshot.db. We backed up cluster state daily and before any major change. The backup should be stored off-cluster (S3 in our case) and tested regularly by restoring to a staging cluster. etcd performance is I/O-sensitive -- it requires low-latency SSD storage (gp3 or io2 EBS volumes).

The trade-off: etcd is a single point of failure for the entire cluster -- if all etcd members go down, the API server cannot function and no changes can be made. Running etcd on dedicated nodes with SSD storage and cross-AZ replication is essential. On managed services like EKS, this complexity is abstracted away, which is why we chose EKS over self-managed Kubernetes.
