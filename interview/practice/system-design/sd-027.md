**27.** What is a hot spot in caching? How do you handle a celebrity problem (thundering herd)?

========== THIS SECTION IS THE ANSWER ==========

A hot spot occurs when one cache key receives disproportionately high traffic, overwhelming a single cache node. The thundering herd (celebrity problem) happens when a popular cached item expires and thousands of requests simultaneously hit the database to regenerate it.

In our project, we experienced this during a viral car deal. A dealer listed a rare car at an unusually low price, it went viral on social media, and that single listing's cache key received 10x normal traffic. When the 60-second TTL expired, ~500 concurrent requests hit PostgreSQL simultaneously to reload that listing. Response time spiked to 5 seconds. We fixed it with three techniques: (1) Cache locking — when the cache misses, the first request acquires a Redis lock (SET NX with TTL), fetches from the database, and populates the cache. Other requests wait briefly and then read from the refreshed cache. (2) Stale-while-revalidate — serve the slightly stale cached value while one background thread refreshes it. (3) For predictably hot items, we used different TTLs — popular listings got 5-minute TTL, normal ones got 60 seconds.

The trade-off: cache locking adds latency for the first request and complexity to handle lock failures. Stale-while-revalidate is simpler but serves stale data briefly. The lesson: identify your hot keys proactively (monitor key access frequency in Redis) and apply thundering herd protection before it becomes a production incident.
