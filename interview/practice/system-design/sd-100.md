**100.** Design a key-value store. What are the trade-offs between consistency and availability?

========== THIS SECTION IS THE ANSWER ==========
A distributed key-value store must choose between consistency and availability during network partitions (CAP theorem). CP systems (like ZooKeeper) reject writes during partitions to maintain consistency. AP systems (like DynamoDB, Cassandra) accept writes on available nodes and reconcile later, risking stale reads.

**Real example:** On our automotive marketplace, we used two key-value stores with different trade-offs. DynamoDB (AP, eventually consistent by default) stored session data and user preferences — if a user saw a slightly stale "recently viewed" list for a few seconds, that was acceptable. ElastiCache Redis (single-primary, CP-leaning) stored rate limiting counters and cache locks where stale data could cause real problems (allowing extra API calls or cache stampedes). For DynamoDB, we used strongly consistent reads only for critical paths like payment tokens, accepting the 2x cost and higher latency for those specific operations.

**Trade-off:** Eventually consistent systems offer higher throughput and lower latency (reads from any replica), but you must design your application to handle stale data gracefully. Strongly consistent systems are simpler to reason about but have higher latency and lower availability during failures. The lesson: choose consistency level per use case, not per system — DynamoDB's per-read consistency flag was perfect for this, letting us mix both in the same table.
