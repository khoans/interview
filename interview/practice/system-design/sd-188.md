**188.** What is image scanning in Kubernetes? How do you enforce only trusted images (admission controllers)?

========== THIS SECTION IS THE ANSWER ==========

Image scanning analyzes container images for known CVEs, malware, and misconfigurations before or during deployment. Admission controllers intercept API requests to the K8s API server and can reject pods that use unscanned or untrusted images. Together, they form a supply chain security gate.

In our marketplace, we scanned Docker images in our GitHub Actions pipeline using Trivy before pushing to ECR. ECR also had native scanning enabled that ran on every push. In Kubernetes, you would add an admission controller (OPA/Gatekeeper or Kyverno) with a policy like: only allow images from our private ECR registry (no Docker Hub in production), reject images with critical CVEs, and require images to have a specific tag format (no :latest). A ValidatingWebhookConfiguration intercepts pod creation and checks these rules before the pod is scheduled.

The trade-off: strict image policies can block urgent deployments if scanning has not completed or finds a CVE in a base image you cannot immediately fix. We handled this with an exception process -- a senior engineer could temporarily whitelist a specific image digest with a time-boxed exemption. Also, scanning only catches known CVEs -- zero-days slip through. Combine with runtime security tools like Falco for defense in depth.
