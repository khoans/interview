**187.** What is Sealed Secrets or External Secrets Operator? How do they solve K8s secret management?

========== THIS SECTION IS THE ANSWER ==========

Sealed Secrets (by Bitnami) encrypts secrets client-side with a cluster-specific public key so that encrypted manifests can be safely stored in git -- only the cluster controller can decrypt them. External Secrets Operator (ESO) syncs secrets from external providers (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager) into Kubernetes Secrets, keeping the source of truth outside the cluster.

In our marketplace, we used External Secrets Operator pattern conceptually -- our ECS tasks pulled secrets from AWS Secrets Manager at startup via task definition references. Mapped to Kubernetes: ESO would have a SecretStore pointing to our AWS Secrets Manager, and ExternalSecret resources would declare which secrets to sync (database-url, api-keys). The operator creates native K8s Secrets that pods reference normally. When secrets rotate in Secrets Manager, ESO automatically updates the K8s Secret. No secrets in git, no manual kubectl create secret commands.

The trade-off: Sealed Secrets are simpler (no external dependency) but secrets are baked at deploy time -- rotation requires redeployment. ESO adds an operator dependency but enables dynamic rotation and centralizes secret management across clusters. For our multi-environment setup (dev, staging, prod), ESO was the clear winner because each environment pointed to different Secrets Manager entries.
