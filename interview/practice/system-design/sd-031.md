**31.** What is REST? What are its constraints and how does it differ from RPC?

========== THIS SECTION IS THE ANSWER ==========

REST (Representational State Transfer) is an architectural style with constraints: stateless communication, uniform interface (resources identified by URIs, manipulated via HTTP methods), client-server separation, cacheability, and layered system. RPC (Remote Procedure Call) is action-oriented — you call a remote function (getUser, createOrder). REST is resource-oriented — you operate on resources (GET /users/123, POST /orders).

In our project, all public-facing APIs were RESTful. The vehicle listing API followed REST conventions: GET /api/v2/listings/{id} to fetch a listing, POST /api/v2/listings to create one, PATCH /api/v2/listings/{id} to update, DELETE to remove. We versioned via URL path (/v2/) for simplicity. The uniform interface made it easy for our Next.js frontend and external dealer integration partners to consume the API — they just needed to know the resource structure. We used HATEOAS sparingly — only for pagination links (next, prev) in search results.

The trade-off: REST is great for CRUD operations on well-defined resources but gets awkward for action-oriented operations — "transfer money from account A to B" doesn't map cleanly to a resource. RPC (or even a POST to an action endpoint like POST /transfers) is more natural there. Also, REST over HTTP/1.1 has overhead (text headers, no multiplexing) compared to binary RPC protocols like gRPC. The lesson: REST for public APIs and CRUD-heavy services; consider RPC for internal service-to-service calls where performance matters.
