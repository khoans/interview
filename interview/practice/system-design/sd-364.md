**364.** How do you configure Nginx as a reverse proxy? Explain proxy_pass, upstream blocks, and header forwarding (X-Forwarded-For, X-Real-IP).

========== THIS SECTION IS THE ANSWER ==========

`proxy_pass` tells Nginx to forward requests to a backend server. `upstream` blocks define a pool of backend servers for load balancing. Header forwarding ensures your backend knows the real client IP — without it, every request looks like it comes from Nginx's IP.

In our project, the config was: `upstream spring_boot { server 127.0.0.1:8080; }` then `location /api/ { proxy_pass http://spring_boot; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_set_header X-Forwarded-Proto $scheme; proxy_set_header Host $host; }`. X-Real-IP passes the client's actual IP. X-Forwarded-For appends the client IP to the chain (important when there are multiple proxies). X-Forwarded-Proto tells the backend whether the original request was HTTP or HTTPS. Host preserves the original domain name so the backend can do virtual host routing if needed.

The trade-off: header forwarding is critical for security (rate limiting by real IP, audit logging) and application logic (generating correct redirect URLs). But you must trust these headers only from known proxies — a malicious client can fake X-Forwarded-For. In Spring Boot, we configured `server.forward-headers-strategy=native` and set trusted proxy IPs to ensure only Nginx-set headers were trusted.
