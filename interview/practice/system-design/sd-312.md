**312.** REST vs GraphQL: When should you use each? When does GraphQL add more problems than it solves?

========== THIS SECTION IS THE ANSWER ==========

Use GraphQL when multiple clients need significantly different views of the same data and over-fetching is a real performance problem. Use REST when your API has predictable, well-defined resource endpoints and your clients are few.

On our marketplace, REST was the right choice. Our API served a Next.js frontend and dealer integrations with well-defined endpoints — GET /listings, GET /listings/{id}, GET /dealers/{id}. The data shape was predictable. GraphQL would have added complexity (query parsing, N+1 problems, caching difficulty) without solving a real pain point. GraphQL shines when you have a mobile app needing minimal fields, a web app needing full objects, and a third-party needing a subset — all from the same API. We did not have that diversity.

The trade-off: GraphQL eliminates over-fetching and under-fetching but introduces new problems — caching is harder (no HTTP caching by URL), authorization per field is complex, and unbounded queries can DDoS your database if you do not implement query depth limiting. It also requires a steeper learning curve and specialized tooling.
