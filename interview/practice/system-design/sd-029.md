**29.** What is a bloom filter? How can it reduce unnecessary database lookups?

========== THIS SECTION IS THE ANSWER ==========

A bloom filter is a space-efficient probabilistic data structure that tells you "definitely not in set" or "probably in set." It uses multiple hash functions to set bits in a bit array. False positives are possible (says yes when the answer is no), but false negatives never happen (if it says no, the item is definitely not there).

In our project, we used a bloom filter concept for our vehicle VIN (Vehicle Identification Number) lookup service. Before querying PostgreSQL to check if a VIN existed in our inventory (15M VINs), we checked a bloom filter loaded in memory. About 70% of VIN lookups from external partners were for vehicles we didn't have — the bloom filter rejected these instantly without touching the database. With 15M VINs and a 1% false positive rate, the bloom filter was only ~18MB in memory. This reduced unnecessary database queries by 70%, significantly lowering our PostgreSQL read load during peak API traffic.

The trade-off: bloom filters can't delete elements (standard version), and false positives mean some unnecessary database queries still happen. You also need to rebuild the filter when data changes. We rebuilt ours every hour from a scheduled job. The lesson: bloom filters shine when you have a large dataset, most lookups are "not found," and you want to protect your database from useless queries. They're not a cache — they're a guard.
