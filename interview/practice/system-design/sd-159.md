**159.** What is SAST vs DAST vs SCA? How do they fit into a CI/CD pipeline?

========== THIS SECTION IS THE ANSWER ==========

SAST (Static Application Security Testing) analyzes source code for vulnerabilities without running it -- catches issues like SQL injection patterns and hardcoded secrets. DAST (Dynamic Application Security Testing) tests the running application by sending malicious requests -- finds runtime issues like XSS and misconfigurations. SCA (Software Composition Analysis) scans dependencies for known CVEs -- catches vulnerable libraries.

In our marketplace CI/CD pipeline on GitHub Actions: (1) SCA ran first via Dependabot and Snyk -- every PR was checked against the NVD database; we caught a critical Log4Shell vulnerability in a transitive dependency this way. (2) SAST ran during build using SonarQube -- it flagged potential SQL injection in a raw query and a hardcoded test credential. (3) DAST ran in our staging environment after deployment using OWASP ZAP -- it found missing security headers and an open redirect. Pipeline gates: SCA critical CVEs block merge, SAST high severity block merge, DAST runs nightly as informational.

The trade-off: SAST has false positives that require triage time, DAST needs a running environment so feedback is slower, SCA only catches known CVEs. You need all three for coverage, but prioritize SCA and SAST in PRs for fast feedback.
