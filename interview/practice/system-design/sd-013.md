**13.** What is the difference between SQL and NoSQL databases? When would you choose each?

========== THIS SECTION IS THE ANSWER ==========

SQL databases are relational — structured schema, ACID transactions, powerful joins (PostgreSQL, MySQL). NoSQL databases are non-relational — flexible schema, horizontal scaling, optimized for specific access patterns (document stores like MongoDB, key-value stores like DynamoDB, wide-column like Cassandra).

In our project, we used both. PostgreSQL for core transactional data — vehicle listings, dealer accounts, pricing — because we needed ACID guarantees (a dealer updating a price shouldn't show partial data to buyers) and complex queries (search filters with joins across make, model, trim, features tables). DynamoDB for user activity data — search history, recently viewed vehicles, saved searches — because the access pattern was simple key-value lookups by userId, the volume was massive (millions of events/day), and we needed single-digit millisecond latency regardless of scale.

The trade-off: SQL gives you data integrity and query flexibility but scaling writes is hard. NoSQL gives you scale and speed but you lose joins and must design around specific access patterns — if your query pattern changes, you may need to restructure the entire table. The lesson: use both where each fits, not one religion for everything.
