**352.** How did you prevent unauthorized access to internal microservices that should not be publicly accessible?

========== THIS SECTION IS THE ANSWER ==========

We used network-level isolation combined with service-level authentication. No internal service had a public IP or was reachable from the internet.

On our marketplace, all microservices ran in private subnets within our VPC. The only public-facing component was the ALB in the public subnet — it routed external traffic to the appropriate ECS service based on path rules. Internal services (listing ingestion, Solr indexing workers, analytics processors) had security groups that only accepted traffic from other internal service security groups on specific ports. There was no route from the internet to these services. Service-to-service communication used OAuth2 client credentials — even if an attacker somehow reached the network, they could not call internal APIs without a valid token. We also used VPC endpoints for AWS services (S3, SQS, Secrets Manager) so that traffic never traversed the public internet.

Lesson learned: network isolation is necessary but not sufficient. We once found that an internal service accidentally had a public ALB listener during a configuration change. The OAuth2 authentication layer caught it — no valid tokens, no access. Defense in depth saved us.
