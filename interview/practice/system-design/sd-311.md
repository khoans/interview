**311.** REST vs gRPC: When should you use each? What are the trade-offs in performance, tooling, and developer experience?

========== THIS SECTION IS THE ANSWER ==========

Use REST for external-facing APIs where broad client compatibility matters. Use gRPC for internal service-to-service calls where performance and strong typing are priorities.

On our marketplace, all APIs were REST with JSON — our Next.js frontend, mobile clients, and third-party dealer integrations all consumed them easily. REST was the right choice for external traffic because every language and tool supports it, documentation is straightforward with OpenAPI/Swagger, and debugging with curl is trivial. For high-throughput internal calls (e.g., search service calling the ranking service thousands of times per second), gRPC would have given us 2-5x better performance via binary Protocol Buffers and HTTP/2 multiplexing, but we chose to keep REST internally too because our team was smaller and the operational simplicity outweighed the performance gain.

The trade-off: gRPC is faster (binary serialization, HTTP/2, streaming) and enforces strict contracts via proto files, but the tooling ecosystem is smaller — harder to debug, browser support requires gRPC-Web proxy, and the learning curve is steeper. REST is universal but verbose and slower. Choose gRPC only when the performance difference actually matters at your scale.
