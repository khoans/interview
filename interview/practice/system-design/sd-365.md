**365.** How do you configure Nginx for load balancing? Explain the algorithms: round-robin, least_conn, ip_hash, and weighted.

========== THIS SECTION IS THE ANSWER ==========

You define an `upstream` block with multiple servers and Nginx distributes requests across them. **Round-robin** (default) — rotates evenly through servers. **least_conn** — sends to the server with fewest active connections, good for variable request duration. **ip_hash** — same client IP always hits the same server, useful for sticky sessions. **Weighted** — assign weights to servers so more powerful ones get more traffic.

In our project, we used Nginx for blue-green deployment on a single EC2: `upstream app { server 127.0.0.1:8080; server 127.0.0.1:8081 backup; }`. The `backup` directive meant 8081 only received traffic when 8080 was down. During deployment, our script started the new version on 8081, health-checked it, then swapped the config to make 8081 primary and 8080 backup. For a multi-instance setup, we'd use least_conn because our search API had variable response times — simple round-robin would overload instances stuck on slow queries.

The trade-off: ip_hash enables sticky sessions without external session storage, but it causes uneven distribution if traffic comes from a few large NAT IPs (like corporate networks). least_conn is generally the safest default for APIs. Weighted is useful during rolling deployments — give new instances lower weight initially and ramp up.
