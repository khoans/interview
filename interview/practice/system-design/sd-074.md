**74.** What is a data pipeline? How do you ensure reliability and exactly-once processing?

========== THIS SECTION IS THE ANSWER ==========

A data pipeline is a series of steps that move and transform data from sources to destinations. Reliability means no data loss and correct processing despite failures. Exactly-once processing means each record is processed exactly once — not skipped (at-least-once) and not duplicated (at-most-once). True exactly-once is extremely hard in distributed systems.

In our project, our clickstream pipeline (user events to Redshift) needed reliability. We achieved effective exactly-once through at-least-once delivery + idempotent processing. SQS guaranteed at-least-once delivery — if a Lambda consumer crashed mid-processing, the message became visible again after the visibility timeout and was reprocessed. To prevent duplicates, each event had a unique eventId, and our Redshift staging table had a UNIQUE constraint on eventId — duplicate INSERTs were rejected. For the nightly aggregation job, we used Step Functions with checkpointing — if step 3 of 5 failed, the pipeline resumed from step 3 instead of restarting from scratch. Dead letter queues captured messages that failed after 3 retries for manual investigation.

The trade-off: true exactly-once (like Kafka's transactional API) requires tight coupling between the message broker and the processing system. Our approach (at-least-once + idempotent consumers) was simpler and worked with any queue (SQS, SNS), but required every consumer to handle duplicates. The lesson: design every consumer to be idempotent. Use unique IDs on every event. At-least-once delivery + idempotent consumers is the pragmatic way to achieve effectively exactly-once processing without the complexity of distributed transactions.
