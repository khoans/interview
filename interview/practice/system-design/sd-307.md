**307.** When should you use a BFF (Backend-for-Frontend) pattern? When does it add unnecessary complexity?

========== THIS SECTION IS THE ANSWER ==========

Use BFF when you have multiple frontends (web, mobile, third-party) with significantly different data needs from the same backend services. Skip it when you have a single frontend or when the data transformation is trivial.

On our marketplace, we effectively had a BFF layer — our Next.js server-side rendering layer aggregated data from multiple backend services (search, listings, dealer info) and shaped it specifically for the web frontend. Mobile clients hit the same APIs but with different query parameters for pagination and field selection. If we had a native mobile app with drastically different data needs, a dedicated mobile BFF would have made sense to avoid bloating the web BFF with mobile-specific logic.

The trade-off: BFF simplifies each frontend's integration but duplicates some backend aggregation logic across BFFs. It also adds another service to deploy and maintain. For a single frontend consuming a few microservices, a BFF is over-engineering — just let the frontend call services directly through the API gateway.
