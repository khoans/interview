**195.** What is Kubernetes audit logging? What events should you monitor?

========== THIS SECTION IS THE ANSWER ==========

Kubernetes audit logging records all requests to the API server -- who did what, when, and on which resource. Audit policies define what to log at four levels: None, Metadata (request metadata only), Request (metadata + request body), RequestResponse (full request and response). Logs are sent to a backend (file, webhook to SIEM).

In our marketplace, we used CloudTrail for the equivalent AWS audit logging. Mapped to Kubernetes audit events to monitor: (1) Secret access -- any get/list on Secrets (detect credential theft), (2) RBAC changes -- create/update/delete on Roles, ClusterRoles, and bindings (detect privilege escalation), (3) pod exec commands -- kubectl exec into production pods (detect unauthorized access), (4) failed authentication attempts (detect brute force), (5) namespace creation/deletion (detect unauthorized environments), (6) admission webhook rejections (track policy violations). We would ship audit logs to our Kibana/ELK stack for dashboarding and alerting, similar to how CloudTrail events went to CloudWatch.

The trade-off: logging everything at RequestResponse level generates enormous volume and impacts API server performance. Use a tiered approach: Metadata for most resources, Request for secrets and RBAC, RequestResponse only for specific high-value resources. Also set up automated alerts -- audit logs are useless if nobody reviews them.
