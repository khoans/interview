**94.** Design a chat application. How do you handle real-time messaging and presence?

========== THIS SECTION IS THE ANSWER ==========
Key components: WebSocket connections for real-time bidirectional messaging, a message broker (like Redis Pub/Sub or Kafka) for cross-server message routing, a message store for persistence and history, and a presence service tracking online/offline status via heartbeats.

**Real example:** On our automotive marketplace, we had a dealer-to-buyer messaging feature. We used WebSocket connections from the Next.js frontend to Spring Boot backend via a /ws endpoint secured with JWT tokens. Messages were persisted in PostgreSQL and published to Redis Pub/Sub so that if the sender and receiver were connected to different ECS instances, the message still reached the recipient in real-time. Presence was tracked with Redis keys (user:{id}:online) with a 30-second TTL, refreshed by client heartbeats every 15 seconds.

**Trade-off:** WebSocket connections are stateful, which complicates load balancing — we used ALB with sticky sessions to keep connections stable. Redis Pub/Sub has no message persistence, so if a server was temporarily disconnected, messages were lost. For mission-critical chat, Kafka would be better (durable, replayable), but for our marketplace messaging volume (~10K concurrent connections), Redis Pub/Sub was simpler and sufficient.
