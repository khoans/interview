**144.** What is CSRF (Cross-Site Request Forgery)? How do CSRF tokens work?

========== THIS SECTION IS THE ANSWER ==========
CSRF tricks a user's browser into making an unwanted request to a site where the user is authenticated (e.g., a hidden form on a malicious site POSTs to your-bank.com/transfer). The browser automatically includes cookies, so the request looks legitimate. CSRF tokens prevent this by including a server-generated random token in each form/request that the attacker's site cannot know or predict.

**Real example:** On our automotive marketplace, Spring Security's CSRF protection was enabled by default for all state-changing requests (POST, PUT, DELETE). The server generated a unique CSRF token per session, stored it in the HttpSession, and expected it in every form submission as a hidden field or X-CSRF-TOKEN header. For our REST APIs consumed by the Next.js frontend, we used the SameSite=Strict cookie attribute on session cookies, which prevented browsers from sending cookies on cross-origin requests entirely — a simpler and equally effective CSRF defense for modern browsers. We disabled CSRF for stateless JWT-based API endpoints since they do not use cookies.

**Trade-off:** CSRF tokens add complexity (token generation, storage, validation) and break if the token gets out of sync (e.g., browser back button, multiple tabs). SameSite cookies are simpler but require modern browser support and do not work for older browsers. For APIs using JWT in Authorization headers (not cookies), CSRF is not a concern because the browser does not automatically attach Authorization headers — only cookies are auto-sent.
