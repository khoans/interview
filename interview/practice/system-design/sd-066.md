**66.** What is CORS? Why does it exist and how do you configure it?

========== THIS SECTION IS THE ANSWER ==========

CORS (Cross-Origin Resource Sharing) is a browser security mechanism that restricts web pages from making requests to a different domain than the one that served the page. It exists to prevent malicious websites from making API calls on behalf of authenticated users (CSRF-like attacks). The server must explicitly allow cross-origin requests via headers: Access-Control-Allow-Origin, Access-Control-Allow-Methods, Access-Control-Allow-Headers.

In our project, we had a classic CORS setup. Our Next.js frontend was served from www.autotrader.com, and our APIs lived at api.autotrader.com. Without CORS configuration, the browser would block all API calls. We configured CORS in our Spring Boot API gateway using @CrossOrigin and a global CorsConfiguration: allowed origins were www.autotrader.com and dealer.autotrader.com (not "*" — that would allow any site to call our API). We allowed GET, POST, PUT, DELETE methods and specific headers (Authorization, Content-Type). For preflight requests (OPTIONS), we cached the CORS response for 3600 seconds to avoid the extra round trip on every API call.

The trade-off: permissive CORS (Allow-Origin: *) is easy but removes cross-origin protection — any website can call your API. Restrictive CORS requires maintaining an explicit allowlist, which can be annoying during development (localhost:3000 needs to be whitelisted). The lesson: never use wildcard origins in production. Explicitly list your frontend domains. Use a CORS filter early in your request chain so preflight requests don't hit your application logic. And remember — CORS is browser-enforced only; it doesn't protect your API from curl or server-to-server calls.
