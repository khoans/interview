**125.** What is mTLS (mutual TLS)? When would you use it?

========== THIS SECTION IS THE ANSWER ==========
Mutual TLS extends standard TLS by requiring BOTH sides to present certificates. In standard TLS, only the server proves its identity. In mTLS, the client also presents a certificate that the server verifies. This provides strong two-way authentication, commonly used for service-to-service communication in microservices.

**Real example:** On our automotive marketplace, we considered mTLS for internal microservice communication on ECS. The use case: our payment service needed to guarantee that only authorized internal services (not a compromised container) could call it. With mTLS, each ECS task would have its own certificate issued by our internal CA (AWS Private CA), and the payment service would verify the calling service's identity before accepting the request. In practice, we used a lighter approach — VPC-level network isolation plus JWT service tokens — because mTLS certificate management across 15+ services with auto-scaling was operationally complex.

**Trade-off:** mTLS provides the strongest service identity guarantee but requires certificate lifecycle management for every service instance — issuance, rotation, and revocation at scale. Service meshes like Istio or AWS App Mesh automate this with sidecar proxies, but add infrastructure complexity. For teams without a service mesh, API keys or JWT service tokens are simpler alternatives, accepting weaker identity guarantees.
