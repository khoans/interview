**342.** Synchronous writes vs async writes (eventual consistency): When can you afford to lose a few writes vs when must every write be confirmed?

========== THIS SECTION IS THE ANSWER ==========

Every write must be confirmed for financial transactions, user account changes, and any data where loss means legal or business liability. Async writes are acceptable for analytics events, activity logs, and non-critical updates where occasional loss is tolerable.

On our marketplace, listing updates from dealers were synchronous — the dealer clicked "save," we wrote to PostgreSQL, confirmed the transaction, then returned a 200 response. The dealer needed to know their changes were saved. But user analytics events (page views, search clicks, impression tracking) were fired asynchronously to SQS — if a few events were lost during a spike, our analytics dashboards would be 99.9% accurate instead of 100%, which was perfectly acceptable. This kept the user-facing response time fast by not blocking on analytics writes.

The trade-off: synchronous writes give you guaranteed durability and immediate consistency but add latency to every request and create back-pressure when the database is slow. Async writes are faster for the caller and decouple the write path, but you risk message loss (SQS is durable but not if the producer crashes before sending). The question to ask: "If this write silently disappeared, would anyone notice or care?"
