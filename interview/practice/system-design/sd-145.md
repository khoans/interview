**145.** What is SSRF (Server-Side Request Forgery)? How did the Capital One breach exploit it?

========== THIS SECTION IS THE ANSWER ==========
SSRF tricks a server into making HTTP requests to internal resources that should be inaccessible from the outside. The attacker provides a URL to an internal endpoint (like http://169.254.169.254 — the AWS metadata service), and the server fetches it, potentially leaking IAM credentials, internal data, or enabling lateral movement.

**Real example:** The Capital One breach (2019) exploited SSRF through a misconfigured WAF: the attacker sent a request that made the WAF proxy fetch the EC2 metadata service URL (169.254.169.254/latest/meta-data/iam/security-credentials/), which returned temporary IAM credentials with S3 access, exposing 100M+ customer records. On our automotive marketplace, we mitigated SSRF by: (1) using IMDSv2 (requires a session token, immune to simple SSRF), (2) validating and whitelisting URLs when our listing service fetched external vehicle data feeds, (3) blocking private IP ranges (10.x, 172.16.x, 169.254.x) in outbound URL validation, and (4) running services with least-privilege IAM roles.

**Trade-off:** URL validation is tricky — attackers use DNS rebinding (resolve to a public IP first, then switch to internal IP), IP address encoding tricks (decimal, hex), and redirect chains to bypass blacklists. A whitelist approach (only allow specific known domains) is more secure than a blacklist but limits functionality. IMDSv2 is the single most impactful mitigation for AWS environments and should be mandatory on all instances.
