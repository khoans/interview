**341.** Vertical scaling vs horizontal scaling: When should you scale up before scaling out? What are the limits of each?

========== THIS SECTION IS THE ANSWER ==========

Scale up (vertical) first when your application is stateful, single-threaded, or when the bottleneck is a single resource (CPU, memory). Scale out (horizontal) when you have hit the vertical ceiling or need redundancy.

On our marketplace, we scaled PostgreSQL vertically — upgrading the RDS instance from db.r5.large to db.r5.2xlarge when query load increased. Vertical scaling required zero code changes and zero architecture rethinking, just a few minutes of downtime during the resize. Meanwhile, our stateless Spring Boot search service scaled horizontally on ECS from 4 to 12 instances during peak hours using auto-scaling rules based on CPU. The search service was designed stateless from the start — sessions in Redis, no local state.

The trade-off: vertical scaling is simple (bigger machine, no code changes) but has a hard ceiling (the largest available instance type) and creates a single point of failure. Horizontal scaling is theoretically unlimited but requires stateless design, load balancing, distributed data management, and more complex debugging. The practical approach: scale vertically until it becomes expensive or creates availability risk, then design for horizontal scaling.
