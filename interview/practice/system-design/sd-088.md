**88.** What is geographic routing (geo-routing)? How do you serve users from the nearest region?

========== THIS SECTION IS THE ANSWER ==========
Geographic routing directs user requests to the closest data center or edge location based on their geographic location, reducing latency and improving user experience. This is typically implemented at the DNS level or through a global load balancer.

**Real example:** On our automotive marketplace, we deployed across us-east-1 and us-west-2 on AWS. Route 53 with latency-based routing directed East Coast users to us-east-1 and West Coast users to us-west-2. CloudFront served static assets (vehicle images, JS bundles) from 200+ edge locations globally. For API requests, Route 53 health checks ensured that if us-east-1 went down, all traffic automatically failed over to us-west-2 within 60 seconds. This gave us sub-100ms latency for most US users.

**Trade-off:** Multi-region adds data replication complexity — we had to ensure PostgreSQL cross-region replication for listing data, and DynamoDB global tables for session data. The operational cost roughly doubled (two regions = two sets of infrastructure). For a US-only marketplace, two regions was the sweet spot; more regions would have added diminishing returns given our user distribution.
