**60.** What is graceful degradation? How does it differ from fail-fast?

========== THIS SECTION IS THE ANSWER ==========

Graceful degradation: when a component fails, the system continues operating with reduced functionality rather than crashing entirely. Fail-fast: when a problem is detected, the system fails immediately and reports the error rather than proceeding with potentially corrupt data or cascading the failure.

In our project, we used both strategies for different scenarios. Graceful degradation: when the pricing service was unavailable, the listing page still loaded — it showed "Price Unavailable, Contact Dealer" instead of the dynamic price. When Solr search was down, we fell back to a simpler PostgreSQL query that handled basic filters (make, model, year) without advanced features (faceted search, fuzzy matching). The user experience was diminished but functional. Fail-fast: when the database connection was lost, our payment processing service immediately returned a 503 instead of queuing up requests. If it kept accepting requests with a dead database, the thread pool would exhaust, memory would spike, and the entire service would crash — taking other endpoints down too.

The trade-off: graceful degradation requires designing fallback paths for every dependency (more code to write and test), and stale/partial data can confuse users. Fail-fast is simpler but means unavailability for the user. The lesson: use graceful degradation for read paths (showing cached or partial data is better than nothing) and fail-fast for write paths (accepting a payment you can't process is worse than refusing it). The combination gives you the best user experience during partial outages.
