**186.** What are Kubernetes Secrets? Why are they not truly secret by default and how do you fix that?

========== THIS SECTION IS THE ANSWER ==========

Kubernetes Secrets store sensitive data (passwords, tokens, keys) but are only base64-encoded by default -- not encrypted. Anyone with RBAC access to read Secrets in a namespace can decode them instantly. They are stored in etcd, which by default is also unencrypted at rest. This means Secrets are not truly secret without additional measures.

In our marketplace, we stored all secrets in AWS Secrets Manager with KMS encryption and strict IAM policies -- far more secure than default K8s Secrets. To fix K8s Secrets: (1) enable etcd encryption at rest using an EncryptionConfiguration with AES-CBC or KMS provider, (2) restrict RBAC so only specific service accounts can read secrets in their namespace, (3) use External Secrets Operator to sync secrets from AWS Secrets Manager or HashiCorp Vault into K8s Secrets at runtime rather than storing them in YAML files, (4) never put Secret manifests in git -- they contain base64-encoded values that are trivially decoded.

The lesson: treat default K8s Secrets as slightly better than ConfigMaps, not as a real secret management solution. In our AWS setup, secrets were encrypted with KMS, access was audited via CloudTrail, and rotation was automated. Kubernetes should integrate with an external secret store for the same level of protection.
