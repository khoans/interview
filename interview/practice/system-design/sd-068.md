**68.** What is encryption at rest vs encryption in transit?

========== THIS SECTION IS THE ANSWER ==========

Encryption in transit protects data as it moves between systems (TLS/HTTPS — prevents eavesdropping on the network). Encryption at rest protects stored data on disk (AES-256 encryption — prevents data exposure if physical storage is stolen or improperly decommissioned).

In our project, we encrypted everything both ways. In transit: TLS termination at ALB for all API traffic, TLS on CloudFront for image delivery, TLS for RDS connections (require_ssl=true in PostgreSQL config so unencrypted connections were rejected). At rest: RDS encryption enabled with AWS KMS keys (all database files, backups, and read replicas encrypted automatically), S3 bucket encryption with SSE-S3 for vehicle images, EBS volume encryption for ECS host instances, and DynamoDB encryption enabled by default. We also encrypted ElastiCache Redis at rest — important because Redis cached sensitive data like dealer session tokens and API keys.

The trade-off: encryption at rest adds minimal performance overhead on modern hardware (~1-2% CPU for AES-NI accelerated encryption) but adds operational complexity around key management. If you lose your KMS key, you lose access to all encrypted data. Encryption in transit (TLS) adds latency for the handshake (~1-2ms for initial connection) but is negligible for persistent connections. The lesson: enable both by default on every service from day one. On AWS, it's often just a checkbox. The cost is negligible, and retroactively encrypting an existing database requires downtime and migration.
