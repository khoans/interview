**323.** DynamoDB vs MongoDB vs Cassandra: When should you use each NoSQL option?

========== THIS SECTION IS THE ANSWER ==========

Use DynamoDB for serverless, fully managed key-value/document access with predictable performance at any scale on AWS. Use MongoDB for flexible document queries with secondary indexes and aggregation pipelines. Use Cassandra for massive write-heavy workloads with multi-datacenter replication.

On our marketplace, we chose DynamoDB for session storage and user activity tracking because it was fully managed (zero ops), integrated natively with AWS (Lambda, ECS), offered single-digit millisecond latency at 30M-user scale, and had auto-scaling built in. We paid per request and never thought about capacity planning. MongoDB would have required managing a cluster (or paying for Atlas), and Cassandra would have been extreme overkill for our access patterns. If we had needed complex document queries with aggregation (like a content management system), MongoDB's query language would have been superior to DynamoDB's limited query model.

The trade-off: DynamoDB is cheap and fast for known access patterns but punishes you harshly for ad-hoc queries — you must design your partition keys and GSIs upfront. MongoDB offers flexible querying but requires cluster management. Cassandra handles massive writes across regions but has the steepest learning curve and operational burden.
