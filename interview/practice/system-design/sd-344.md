**344.** Precomputation vs on-the-fly computation: When should you precompute and store results (materialized views) vs compute at request time?

========== THIS SECTION IS THE ANSWER ==========

Precompute when the computation is expensive, the inputs change infrequently, and many users request the same result. Compute on-the-fly when inputs are unique per request, data changes rapidly, or storage cost exceeds compute cost.

On our marketplace, we precomputed several things: vehicle pricing aggregates (average price by make/model/year/region) ran as nightly batch jobs and were stored as materialized views in PostgreSQL. Computing this across millions of listings on every request would have taken seconds. The precomputed result served in milliseconds. But personalized search results were computed on-the-fly — each user's search query was unique (different filters, location, sort), making precomputation impossible for the entire query space. We cached popular search combinations in Redis but computed unique queries fresh via Solr.

The trade-off: precomputation trades storage and freshness for speed — results are fast but potentially stale until the next refresh. On-the-fly computation is always fresh but slow for complex aggregations. The hybrid approach works best: precompute common, expensive aggregations; compute unique, lightweight queries on demand; cache results for the middle ground.
