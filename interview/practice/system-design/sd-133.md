**133.** What is port forwarding and SSH tunneling? When would you use them?

========== THIS SECTION IS THE ANSWER ==========
Port forwarding maps a port on one machine to a port on another. SSH tunneling (local or remote port forwarding) creates an encrypted tunnel through SSH, letting you securely access services that are not directly reachable. Local forwarding: access a remote service through a local port. Remote forwarding: expose a local service to a remote network.

**Real example:** On our automotive marketplace, we used SSH local port forwarding regularly for database access. To connect to production PostgreSQL (in a private subnet) from a developer's laptop, we ran: `ssh -L 5433:prod-rds.internal:5432 bastion-host`, then connected our SQL client to localhost:5433. The traffic was encrypted through the SSH tunnel to the bastion, which forwarded it to RDS. This let us use GUI tools (DBeaver, pgAdmin) against production without exposing RDS to the internet. We also used this for ElastiCache Redis debugging: `ssh -L 6380:redis.internal:6379 bastion-host`.

**Trade-off:** SSH tunneling is powerful but creates shadow access paths that bypass normal access controls if not properly managed. Someone could tunnel to any internal service the bastion can reach. We mitigated this by restricting bastion SSH access to specific users, enabling jump host logging, and eventually migrating to AWS Session Manager port forwarding which provides IAM-based access control and CloudTrail audit trails without maintaining a bastion.
