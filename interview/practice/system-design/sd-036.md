**36.** What is an event-driven architecture? How does pub/sub differ from point-to-point messaging?

========== THIS SECTION IS THE ANSWER ==========

Event-driven architecture is where services communicate by producing and consuming events rather than direct API calls. Pub/sub: a publisher broadcasts events to a topic, and all subscribers receive a copy (fan-out). Point-to-point: a message goes to a queue, and only one consumer processes it (competing consumers).

In our project, we used both patterns. SNS (pub/sub) for event fan-out: when a listing was sold, an "ListingSold" event was published to an SNS topic. The search service (remove from index), analytics service (record sale metric), and notification service (email the buyer) all subscribed — each got a copy of the event independently. SQS (point-to-point) for work distribution: image processing tasks went into an SQS queue with 4 consumer instances competing for messages. Each image was processed by exactly one instance, and SQS ensured no duplicates via visibility timeout.

The trade-off: pub/sub is great for broadcasting but every subscriber gets every message — you can't selectively route without filtering logic. Point-to-point ensures exactly-one processing but only one consumer gets each message — you can't fan out. We combined them: SNS topic fanned out to multiple SQS queues (SNS-to-SQS pattern), giving us both fan-out and reliable per-consumer processing. The lesson: most real systems need both patterns — use SNS for fan-out, SQS for reliable per-consumer processing, and chain them together.
