**14.** What is database indexing? How does a B-tree index work?

========== THIS SECTION IS THE ANSWER ==========

An index is a data structure that speeds up data retrieval at the cost of extra storage and slower writes. A B-tree index organizes data in a balanced tree where each node has multiple keys — the database traverses from root to leaf in O(log n) to find a row, instead of scanning the entire table.

In our automotive marketplace, indexing was critical. Our vehicle listings table had ~15M rows. Without an index, searching for "Toyota Camry 2020 under $30K in Texas" would be a full table scan. We had composite B-tree indexes on (make, model, year) and (state, price) in PostgreSQL. A query that took 3 seconds without indexes dropped to 15ms. We also used Apache Solr with inverted indexes for full-text search and faceted filtering — Solr handled the complex multi-filter queries while PostgreSQL handled transactional writes.

The trade-off: every index speeds up reads but slows down writes because the index must be updated on every INSERT/UPDATE. We had a batch job that bulk-updated dealer inventory nightly — on tables with 8+ indexes, the bulk update took 3x longer. The lesson: index for your read patterns, but audit index usage regularly. We dropped 4 unused indexes that were slowing writes for zero read benefit.
