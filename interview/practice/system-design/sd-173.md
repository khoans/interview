**173.** What is Horizontal Pod Autoscaler (HPA)? What metrics does it use to scale?

========== THIS SECTION IS THE ANSWER ==========

HPA automatically adjusts the number of pod replicas based on observed metrics. By default, it uses CPU and memory utilization, but with the custom metrics API, it can scale on application-specific metrics like requests per second, queue depth, or response latency. It evaluates metrics every 15 seconds and calculates the desired replica count using the formula: desired = ceil(current * (currentMetric / targetMetric)).

In our marketplace, we used ECS auto-scaling with similar logic. Our listing search service scaled on CPU utilization (target 60%) and a custom CloudWatch metric for Solr query latency. When evening traffic spiked (car shoppers browsing after work), CPU exceeded the threshold, and auto-scaling added instances from 4 to 12 within 3 minutes. In Kubernetes, this would be an HPA targeting 60% CPU on the search-service Deployment, with a custom metric from Prometheus for p95 latency exceeding 200ms.

The trade-off: HPA reacts to metrics after they change, so there is always a lag during traffic spikes. For predictable patterns (our evening spike), combine HPA with scheduled scaling -- pre-scale before the rush. Also, set appropriate min/max bounds to prevent runaway scaling that could exhaust cluster resources or inflate cloud costs.
