**250.** What is a CORS preflight request? What triggers it and how does the browser handle it?

A CORS preflight is an HTTP OPTIONS request the browser automatically sends before certain cross-origin requests to check if the server allows them. It is triggered by: non-simple methods (PUT, DELETE, PATCH), custom headers (Authorization, Content-Type: application/json), or credentials.

On our marketplace, our Next.js frontend (hosted on a different domain) called our Spring Boot API backend. Every API call with an Authorization header triggered a preflight. We configured CORS in Spring Security with allowed origins (our frontend domain), allowed methods (GET, POST, PUT, DELETE), allowed headers (Authorization, Content-Type), and max-age (3600 seconds to cache preflight results). Without proper CORS config, the browser blocked all API calls -- a common issue during local development.

The trade-off: preflight caching (Access-Control-Max-Age) reduces extra OPTIONS requests but means CORS policy changes take time to propagate to users. We set it to 1 hour in production. Also, wildcard origins (*) do not work with credentials -- you must explicitly list allowed origins, which creates maintenance overhead when adding new frontend domains.
