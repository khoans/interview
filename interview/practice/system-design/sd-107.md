**107.** What is a rainbow table attack? How do salted hashes prevent it?

========== THIS SECTION IS THE ANSWER ==========
A rainbow table is a precomputed lookup table mapping common passwords to their hash values. An attacker with a stolen database can look up each hash in the table to find the original password in seconds instead of brute-forcing. Salting prevents this because each password has a unique random salt, meaning the attacker would need a separate rainbow table for every possible salt value — making precomputation infeasible.

**Real example:** On our automotive marketplace, before I joined, there was a legacy system that stored MD5 hashes without salting. During a security audit, the penetration tester cracked 40% of user passwords in under an hour using publicly available rainbow tables for MD5. We migrated to bcrypt with per-user salts: on next login, each user's password was transparently re-hashed with bcrypt. The migration took 3 months to reach 95% coverage (remaining 5% were inactive accounts that were forced to reset).

**Trade-off:** Salting eliminates rainbow table attacks but does not protect against targeted brute-force on a single hash — that is where bcrypt's slowness matters. Salts add minimal storage overhead (16 bytes per user) and no meaningful performance cost. There is really no reason not to use salts — any modern password hashing library (bcrypt, scrypt, Argon2) includes them by default.
