**267.** What is token-based authentication? Explain access token vs refresh token lifecycle.

Token-based authentication issues tokens after login instead of maintaining server-side sessions. The **access token** (short-lived, 15-30 minutes) is sent with every API request in the Authorization header. The **refresh token** (long-lived, days/weeks) is used to obtain new access tokens without re-entering credentials. When the access token expires, the client silently uses the refresh token to get a new pair.

On our marketplace, we used OAuth2 with JWT access tokens (15-minute expiry) and opaque refresh tokens (7-day expiry stored in HttpOnly cookies). The flow: user logs in, receives both tokens; access token goes in memory (React state) in our Next.js frontend; every API call includes it; when it expires, the frontend calls /oauth/token with the refresh token to get a new access token transparently. If the refresh token is expired or revoked, the user must log in again.

The trade-off: short-lived access tokens limit the damage window if stolen, but force more refresh calls. We chose 15 minutes as a balance -- short enough to limit exposure, long enough that active users rarely notice the refresh. Storing refresh tokens in HttpOnly cookies prevents XSS theft but requires CSRF protection.
