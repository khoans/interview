**75.** What is CQRS (Command Query Responsibility Segregation)? When is it useful?

========== THIS SECTION IS THE ANSWER ==========

CQRS separates the write model (commands: create, update, delete) from the read model (queries: fetch, search, aggregate). Each side can have its own data store, schema, and scaling strategy. It's useful when read and write patterns differ significantly — different data shapes, different scaling needs, different performance requirements.

In our project, we naturally practiced CQRS without calling it that. The write side: dealers created and updated listings through the listing service, which wrote to PostgreSQL (normalized schema — listings, images, features in separate tables with foreign keys, optimized for transactional integrity). The read side: buyers searched and viewed listings through the search service, which read from Apache Solr (denormalized documents — all listing data, images, dealer info flattened into a single Solr document, optimized for fast queries). When a dealer updated a listing, an SQS event triggered the search indexer to update the Solr document. This let us scale reads and writes independently — Solr handled 30M queries/month without touching PostgreSQL.

The trade-off: CQRS adds complexity — you maintain two data stores, a synchronization mechanism, and deal with eventual consistency (a listing update takes seconds to appear in search). For simple CRUD applications, it's overkill. The lesson: CQRS is worth it when your read and write models are fundamentally different (like our normalized relational write model vs denormalized search-optimized read model). If reads and writes use the same schema and scale similarly, a single model is simpler.
