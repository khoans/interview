**182.** What is a ServiceAccount in Kubernetes? Why should pods not use the default service account?

========== THIS SECTION IS THE ANSWER ==========

A ServiceAccount provides an identity for pods to authenticate with the Kubernetes API and external services. Every namespace has a default ServiceAccount that is automatically mounted into pods. Pods should not use it because the default SA often has more permissions than needed, and if one pod is compromised, the attacker inherits those permissions to interact with the cluster API.

In our marketplace, we applied the same principle with ECS task roles -- each service had its own IAM role, never a shared default role. Mapped to Kubernetes: our listing-service pod would use a dedicated ServiceAccount (sa-listing-service) with a RoleBinding granting only the permissions it needs (e.g., read ConfigMaps in its namespace). The search-service would have sa-search-service with different permissions. We would set automountServiceAccountToken: false on pods that do not need API access at all, reducing the attack surface.

The lesson: the default SA is like running everything as root -- convenient but dangerous. When we once had an ECS task with an overly broad IAM role, a vulnerability in a dependency allowed an attacker to list S3 buckets they should not have seen. Dedicated, minimal ServiceAccounts per workload contain the blast radius.
