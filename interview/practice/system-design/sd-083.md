**83.** What is a read replica? How do you handle replication lag?

========== THIS SECTION IS THE ANSWER ==========
A read replica is a copy of the primary database that serves read queries, offloading the primary and improving read throughput. Replication lag is the delay between a write on the primary and its appearance on the replica.

**Real example:** On our automotive marketplace, we had PostgreSQL with read replicas for vehicle search and listing queries (30M users/month, reads outnumbered writes 20:1). After a dealer updated a listing price, users hitting a replica might see the stale price for a few hundred milliseconds. We handled this with "read-your-own-writes" consistency: after a write, that user's subsequent reads were routed to the primary for a short window (5 seconds), while all other users continued reading from replicas.

**Trade-off:** Routing recent writers to the primary increases primary load and partially defeats the purpose of replicas. We monitored replication lag via New Relic — if lag exceeded 1 second, alerts fired and we investigated. For truly critical reads (payment, order confirmation), always read from primary regardless.
