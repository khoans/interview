**87.** What is auto-scaling? What metrics should trigger scale-out vs scale-up?

========== THIS SECTION IS THE ANSWER ==========
Auto-scaling automatically adjusts compute capacity based on demand. Scale-out adds more instances (horizontal), while scale-up increases resources on existing instances (vertical). Scale-out is preferred for stateless services; scale-up for databases or stateful workloads where adding nodes is complex.

**Real example:** On our automotive marketplace (30M users/month), we configured ECS auto-scaling for our Spring Boot microservices. Scale-out triggered on CPU > 70% sustained for 3 minutes or request count per target > 1000/min on the ALB. Scale-in triggered on CPU < 30% for 10 minutes with a cooldown of 5 minutes to prevent flapping. For our PostgreSQL RDS, we used scale-up (larger instance class) rather than scale-out because adding read replicas required application-level routing changes.

**Trade-off:** Scale-out requires stateless design and load balancer configuration, but gives near-linear capacity growth. Scale-up is simpler but has hard limits (largest instance type) and requires downtime for RDS single-AZ. We learned to set conservative scale-in policies — once we scaled in too aggressively during what looked like a traffic lull but was actually a brief dip between peak waves, causing latency spikes when traffic returned.
