**24.** What is Redis? How is it different from Memcached?

========== THIS SECTION IS THE ANSWER ==========

Redis is an in-memory data structure store supporting strings, hashes, lists, sets, sorted sets, and more. Memcached is a simpler in-memory key-value cache. Key differences: Redis supports persistence (RDB snapshots, AOF), replication, pub/sub, Lua scripting, and rich data types. Memcached is multi-threaded (better for pure caching at high concurrency) but only supports simple key-value strings.

In our project, we used AWS ElastiCache Redis (cluster mode) for multiple purposes. For search result caching, we used simple key-value (serialized JSON). For session management, we used Redis hashes to store user session attributes. For real-time leaderboards (most-viewed vehicles this week), we used sorted sets — ZADD to increment, ZREVRANGE to fetch top 10 — operations that would require complex SQL queries with PostgreSQL. We also used Redis pub/sub to broadcast cache invalidation events across service instances. Memcached couldn't have done the sorted sets or pub/sub — we would have needed separate systems.

The trade-off: Redis is single-threaded per core (uses I/O multiplexing), so extremely hot keys can bottleneck a single shard. Memcached's multi-threaded model handles raw throughput better for simple caching. But Redis's versatility meant we used one system instead of three. The lesson: Redis is almost always the right choice unless you have a pure simple-caching use case at extreme scale where Memcached's multi-threading matters.
