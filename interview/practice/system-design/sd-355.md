**355.** How did you secure your database? (network access, encryption at rest, encryption in transit, credential rotation, audit logging)

========== THIS SECTION IS THE ANSWER ==========

Five layers of database security on our PostgreSQL RDS instance serving 30M users.

**Network access**: RDS instance in a private subnet with no public accessibility. Security group allowed inbound on port 5432 only from application ECS service security groups — not even developer laptops could connect directly. We used AWS Session Manager for emergency access through a bastion pattern. **Encryption at rest**: Enabled by default on RDS using AWS KMS with a customer-managed key. All data, snapshots, and automated backups encrypted transparently. **Encryption in transit**: SSL/TLS enforced on all connections — `rds.force_ssl=1` parameter group setting. Spring Boot datasource configured with `sslmode=require`. **Credential rotation**: Database credentials stored in AWS Secrets Manager with automatic 30-day rotation. Application fetched credentials at startup and refreshed on rotation events via a Secrets Manager SDK integration. No hardcoded passwords anywhere. **Audit logging**: RDS PostgreSQL audit logging enabled via pgAudit extension, streaming to CloudWatch Logs. We tracked all DDL changes, privilege grants, and access to sensitive tables.

The trade-off: SSL on every database connection adds about 1ms latency and credential rotation requires application awareness (connection pool refresh). But for a platform handling dealer PII and financial data, these were non-negotiable baseline controls.
