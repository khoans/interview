**343.** Push model vs pull model: When should the server push data to clients vs clients pull on demand?

========== THIS SECTION IS THE ANSWER ==========

Push when updates are time-sensitive, frequent, and the server knows when new data is available. Pull when updates are infrequent, the client needs data on-demand, or you want simpler infrastructure.

On our marketplace, we used pull for most interactions — the Next.js frontend pulled listing data from the API on page load, dealers pulled their dashboard stats on refresh. This was simple, stateless, and worked well because car shopping is not a real-time activity. However, we used push (via SNS) for dealer notifications — when a lead came in on their listing, we pushed the notification immediately rather than waiting for the dealer to poll. Time-to-respond on leads directly impacted dealer sales conversion.

The trade-off: push gives low-latency delivery but requires maintaining connections or notification channels (WebSockets, push notifications, SNS), which adds infrastructure complexity and scaling challenges. Pull is simpler and more scalable (stateless HTTP) but wastes bandwidth when polling returns no changes, and creates latency between when data is available and when the client sees it. For most CRUD applications, pull with reasonable polling intervals is sufficient.
