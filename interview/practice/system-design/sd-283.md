**283.** A developer accidentally pushed an AWS access key to a public GitHub repo. What do you do?

This is a critical incident requiring immediate action measured in minutes, not hours. The key must be assumed compromised the instant it was pushed.

On our marketplace, the response plan: (1) **Immediately** -- within minutes -- disable the AWS access key in IAM console (do not delete yet, just deactivate -- you need it for forensics), (2) Generate a new key pair and update it in AWS Secrets Manager, (3) Review CloudTrail logs for any unauthorized API calls made with the compromised key -- check for EC2 instances launched (crypto mining), S3 data exfiltration, IAM changes, (4) Use git filter-branch or BFG Repo-Cleaner to remove the secret from git history (just deleting the file leaves it in history), (5) Rotate any resources the key had access to, (6) Post-mortem: add pre-commit hooks (git-secrets, GitLeaks) to prevent this, enable GitHub secret scanning, and review IAM policies to ensure keys follow least privilege.

The lesson: AWS bots scan public GitHub repos and exploit leaked keys within minutes -- sometimes faster than your incident response. We moved all secrets to AWS Secrets Manager and KMS, and our CI pulled secrets at runtime rather than storing them in code.
