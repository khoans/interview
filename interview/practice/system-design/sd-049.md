**49.** What is the two generals problem? How does it relate to distributed consensus?

========== THIS SECTION IS THE ANSWER ==========

The two generals problem: two armies must agree on an attack time by sending messengers through enemy territory. No matter how many acknowledgments they exchange, neither general can be certain the last message was received. It proves that perfect agreement over an unreliable network is impossible — there's always uncertainty about the final message.

This directly relates to our work because every distributed system faces this. In our project, when the listing service wrote to PostgreSQL and then sent an SQS message to update the search index, there was a window where the database write succeeded but the SQS send could fail (or vice versa). We couldn't achieve perfect atomicity across these two systems. Our practical solution was the transactional outbox pattern: instead of sending to SQS directly, we wrote the event to an "outbox" table in the same PostgreSQL transaction as the listing update. A separate poller read the outbox table and published to SQS. This guaranteed at-least-once delivery — if the poller failed, it would retry on the next poll.

The trade-off: the two generals problem tells us perfect consensus is impossible, but practical protocols (Raft, Paxos, 2PC) achieve "good enough" consensus with trade-offs. 2PC blocks if the coordinator fails. Raft requires a majority quorum (can't operate with fewer than half the nodes). The lesson: don't try to achieve perfect distributed consistency. Design for at-least-once delivery with idempotent consumers — it's the pragmatic sweet spot for real systems.
