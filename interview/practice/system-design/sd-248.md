**248.** What is a cookie vs session vs local storage? Security implications of each?

**Cookies** are sent with every HTTP request, can be secured with HttpOnly/Secure/SameSite flags, and have a 4KB limit. **Sessions** are server-side state referenced by a session ID (stored in a cookie). **Local storage** is client-side only (5-10MB), accessible via JavaScript, and never sent automatically with requests.

On our marketplace, we used HttpOnly Secure cookies for session management with Spring Security -- the session ID cookie could not be read by JavaScript (preventing XSS theft). Our OAuth2 access tokens were stored in memory (React state) in the Next.js frontend, and refresh tokens were in HttpOnly cookies. We never stored tokens in local storage because any XSS vulnerability would expose them. Session data was stored in ElastiCache Redis for horizontal scaling across multiple ECS instances.

The trade-off: cookies with HttpOnly protect against XSS but are vulnerable to CSRF -- which is why we used SameSite=Strict plus CSRF tokens in Spring Security. Local storage is immune to CSRF but fully exposed to XSS. There is no perfect solution; you layer defenses based on the threat model.
