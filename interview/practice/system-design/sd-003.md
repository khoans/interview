**3.** What is caching? When should you use it and what are the common pitfalls?

========== THIS SECTION IS THE ANSWER ==========

Caching stores frequently accessed data in a faster storage layer (memory) to avoid expensive recomputation or database queries. Use it when: data is read-heavy, data doesn't change every second, and the same data is requested repeatedly.

In our project, we had three caching layers: CloudFront (CDN) cached static assets and some API responses at the edge. ElastiCache (Redis) cached search results — popular queries like "Honda Civic near Houston" were served from cache in 5ms instead of hitting Solr at 200ms. Caffeine (in-process cache) cached small reference data like vehicle makes/models that rarely changed.

Common pitfalls: stale data (cache shows sold cars), cache stampede (cache expires, 1000 requests hit the database simultaneously), and over-caching (caching everything wastes memory and makes debugging harder). We got burned once when a cache invalidation bug showed sold vehicles for 60 seconds after sale — users complained. The fix was event-driven invalidation via SQS: when inventory updated, it published an event that cleared the relevant cache keys immediately.
