**33.** What is GraphQL? What problems does it solve compared to REST?

========== THIS SECTION IS THE ANSWER ==========

GraphQL is a query language for APIs where the client specifies exactly what data it needs in a single request. It solves two REST problems: over-fetching (getting fields you don't need) and under-fetching (needing multiple REST calls to assemble one view).

In our project, we considered GraphQL for the vehicle detail page. With REST, the Next.js frontend made 4 separate API calls: GET /listings/{id} (core data), GET /listings/{id}/images (photos), GET /listings/{id}/pricing (price history), and GET /dealers/{dealerId} (dealer info). That was 4 round trips. With GraphQL, a single query could fetch exactly the fields needed for the page. We ultimately stayed with REST and used a BFF (Backend for Frontend) pattern instead — a thin aggregation layer that made those 4 internal calls and returned a single combined response to the frontend. This gave us the same "one request" benefit without introducing GraphQL's complexity.

The trade-off: GraphQL eliminates over/under-fetching but introduces complexity — you need a schema, resolvers, N+1 query problems at the resolver level (solved with DataLoader), caching becomes harder (no HTTP caching since everything is POST), and it's a new paradigm for the team to learn. The lesson: GraphQL is powerful for complex, client-driven data requirements (like a mobile app needing different data than web). For a team already productive with REST, a BFF layer often delivers 80% of the benefit with 20% of the complexity.
