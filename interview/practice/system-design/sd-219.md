**219.** What is a VPC peering vs Transit Gateway? When would you use each?

========== THIS SECTION IS THE ANSWER ==========

VPC peering is a direct one-to-one connection between two VPCs -- non-transitive, meaning if VPC A peers with B and B peers with C, A cannot reach C through B. Transit Gateway is a hub-and-spoke model where multiple VPCs and on-premises networks connect to a central hub -- transitive routing, centralized management. Peering is free (data transfer charges only); Transit Gateway costs $0.05/GB plus $0.05/hour per attachment.

In our marketplace, we used VPC peering between our dev and staging accounts (simple, two VPCs, direct connection). If we had more accounts or needed on-premises connectivity, Transit Gateway would be better. Our setup: the dev VPC (10.0.0.0/16) peered with the staging VPC (10.1.0.0/16) so integration tests in dev could reach staging databases. The production VPC (10.2.0.0/16) had no peering -- it was isolated by design, accessible only through the CI/CD pipeline via IAM roles. Route tables in each VPC pointed the peer CIDR to the peering connection.

The trade-off: VPC peering is simpler and cheaper for a small number of connections but becomes unmanageable at scale -- connecting N VPCs requires N*(N-1)/2 peering connections (10 VPCs = 45 connections). Transit Gateway scales linearly and supports transitive routing, but costs more and adds a potential bottleneck. For our 3-account setup, peering was sufficient; organizations with 10+ accounts should use Transit Gateway.
