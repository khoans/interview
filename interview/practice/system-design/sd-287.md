**287.** You need to rotate database credentials without downtime. How do you do it?

The key is a dual-credential strategy: the application supports two sets of credentials simultaneously during the rotation window.

On our marketplace with PostgreSQL on RDS and multiple ECS instances, our rotation process: (1) Create a new database password in AWS Secrets Manager (Secrets Manager supports automatic rotation with Lambda functions), (2) Update the PostgreSQL user to accept the new password (ALTER USER ... SET PASSWORD), (3) Update the secret in Secrets Manager to the new value, (4) Our Spring Boot services used Secrets Manager SDK to fetch credentials at startup -- trigger a rolling restart of ECS tasks so new instances pick up the new secret, (5) Verify all instances are using the new credentials via health checks, (6) Remove the old password. During steps 3-5, both old and new passwords work, so there is zero downtime.

The trade-off: automatic rotation with short intervals (30 days) is more secure but requires robust automation -- if the rotation Lambda fails, you can lock yourself out. We used Secrets Manager built-in RDS rotation Lambda which handles the dual-password strategy automatically. The lesson: never store database credentials in application config files or environment variables directly -- always use a secrets manager with rotation support.
