**142.** What is SQL injection? How do prepared statements / parameterized queries prevent it?

========== THIS SECTION IS THE ANSWER ==========
SQL injection occurs when user input is concatenated directly into SQL queries, allowing attackers to inject malicious SQL (e.g., `' OR 1=1 --`). Prepared statements prevent this by separating the SQL structure from the data — the database engine compiles the query template first, then binds parameters as literal values, never as executable SQL.

**Real example:** On our automotive marketplace, Spring Data JPA repositories generated parameterized queries by default. For custom queries, we used @Query with named parameters: `@Query("SELECT l FROM Listing l WHERE l.dealerId = :dealerId AND l.status = :status")`. Even in our Solr search service where we built dynamic queries, we used Solr's parameterized query API rather than string concatenation. SonarCloud flagged any string concatenation in SQL contexts as a critical vulnerability in our CI/CD pipeline, blocking the PR from merging.

**Trade-off:** Prepared statements handle value injection but do not protect against injection in table names, column names, or ORDER BY clauses (these cannot be parameterized). For dynamic column sorting, we used a whitelist approach — mapping user input to predefined safe column names rather than accepting arbitrary input. Also, ORM-generated queries are generally safe, but native queries and JPQL with string concatenation remain vulnerable, so code review discipline is essential.
