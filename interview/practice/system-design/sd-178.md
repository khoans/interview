**178.** What is a Kubernetes rollback? How do you roll back a failed deployment?

========== THIS SECTION IS THE ANSWER ==========

Kubernetes rollback reverts a Deployment to a previous revision. Kubernetes keeps a history of ReplicaSets (controlled by revisionHistoryLimit). You execute kubectl rollout undo deployment/my-service to revert to the previous version, or kubectl rollout undo deployment/my-service --to-revision=3 for a specific version. The rollback is itself a rolling update, so it follows the same maxSurge/maxUnavailable rules.

In our marketplace, we had a critical incident where a listing service deployment introduced a bug that returned wrong pricing data. Our GitHub Actions pipeline deployed the broken version to ECS. We detected it within 5 minutes through New Relic alerts (error rate spike from 0.1% to 15%). The rollback in ECS involved redeploying the previous task definition revision, which took about 3 minutes for full rollout. In Kubernetes, kubectl rollout undo would achieve the same in seconds by pointing back to the previous ReplicaSet.

The lesson: always verify rollbacks work before you need them. We ran rollback drills quarterly. Also, keep revisionHistoryLimit reasonable (we kept 5) -- too many old ReplicaSets clutter the cluster. Pair rollbacks with feature flags so you can disable broken features without a full redeployment.
