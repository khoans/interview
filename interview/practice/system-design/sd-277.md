**277.** What is scope-based authorization in OAuth2? How do you design scopes for an API?

Scopes define what permissions an OAuth2 token grants -- they limit what an application can do on behalf of a user. Scopes should be designed around resources and actions, following the pattern resource:action (e.g., vehicles:read, listings:write, dealers:admin).

On our marketplace, we designed scopes hierarchically: read:vehicles (browse listings -- granted to all consumers), write:vehicles (create/update listings -- granted to dealer apps), read:dealers (view dealer profiles), admin:dealers (manage dealer accounts -- platform admin only), read:analytics (view marketplace reports). Third-party partner apps that integrated with our API were granted limited scopes -- a price comparison site got only read:vehicles, never write:vehicles. Spring Security checked scopes using @PreAuthorize annotations on controller methods.

The trade-off: too few scopes means over-privileged tokens (a token that can read can also write). Too many scopes creates complexity for API consumers who must request the right combination. We aimed for 10-15 well-defined scopes that mapped to real business operations, not one scope per endpoint.
