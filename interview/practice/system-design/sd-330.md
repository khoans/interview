**330.** ACID vs BASE: When is eventual consistency acceptable? When must you have strong consistency?

========== THIS SECTION IS THE ANSWER ==========

Strong consistency (ACID) is required for financial transactions, inventory counts where overselling is unacceptable, and any operation where reading stale data causes real business harm. Eventual consistency (BASE) is acceptable when temporary staleness is tolerable — search results, analytics, recommendations, social feeds.

On our marketplace, we used ACID (PostgreSQL transactions) for dealer account updates, listing ownership changes, and any operation involving money. But search results were eventually consistent — when a dealer updated a listing price, the PostgreSQL write was immediate (ACID), but the Solr search index update happened asynchronously via SQS, meaning users might see the old price for 2-5 seconds. This was perfectly acceptable because a slightly stale search result does not cause business harm — the listing detail page always hit PostgreSQL for the current price.

The trade-off: strong consistency limits your scalability options (single-master writes, distributed locks) and increases latency. Eventual consistency enables horizontal scaling and better availability (AP in CAP theorem) but requires your application to handle stale reads gracefully. The key question is: "What is the business cost of reading stale data for N seconds?" If the answer is "negligible," go eventual.
