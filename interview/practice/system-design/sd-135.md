**135.** What is a subnet? How do you design subnets for public-facing vs private services?

========== THIS SECTION IS THE ANSWER ==========
A subnet is a logical partition of a VPC's IP address range. Public subnets have a route to an Internet Gateway (resources can have public IPs). Private subnets route through NAT Gateways for outbound-only internet access. The design principle: put only load balancers and bastions in public subnets; everything else (application servers, databases, caches) goes in private subnets.

**Real example:** On our automotive marketplace, our VPC (10.0.0.0/16) was structured across two AZs: Public subnets (10.0.1.0/24, 10.0.2.0/24) held ALBs and NAT Gateways. Private app subnets (10.0.10.0/24, 10.0.20.0/24) held ECS tasks running Spring Boot services. Private data subnets (10.0.100.0/24, 10.0.200.0/24) held RDS PostgreSQL, ElastiCache Redis, and DynamoDB VPC endpoints. Each tier had its own Security Group rules: ALB accepted internet traffic, app tier accepted only ALB traffic, data tier accepted only app tier traffic.

**Trade-off:** More subnets provide finer isolation but consume IP address space and add routing complexity. We used /24 subnets (254 usable IPs each), which was sufficient for our ECS tasks. However, when auto-scaling spiked during peak traffic, we once ran low on IPs in the app subnet. Lesson learned: plan subnet sizes for peak capacity, not average. Use /20 or /22 for subnets that will host auto-scaling workloads.
