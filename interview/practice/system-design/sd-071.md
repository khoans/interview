**71.** What is batch processing vs stream processing? When would you use each?

========== THIS SECTION IS THE ANSWER ==========

Batch processing handles data in large chunks at scheduled intervals (hourly, daily) — high throughput, high latency. Stream processing handles data record-by-record in real-time as it arrives — low latency, continuous. Batch is for "what happened yesterday," stream is for "what's happening now."

In our project, we used both. Batch processing: a nightly job aggregated the day's search impressions, clicks, and leads per listing into a dealer performance report. It processed ~50M events, joined with listing metadata, and wrote summary tables to Redshift. This ran as an AWS Lambda + Step Functions workflow triggered by a CloudWatch cron rule at 2 AM EST. Stream processing: real-time fraud detection on lead submissions. When a buyer submitted a lead (contact request to a dealer), a Lambda consumer on the SQS stream checked for suspicious patterns in real-time — same phone number submitting 50+ leads in an hour, or leads from known spam IP ranges. Fraudulent leads were flagged immediately before reaching the dealer.

The trade-off: batch processing is simpler (no state management, easier error recovery — just rerun the batch) but results are stale. Stream processing gives real-time results but is more complex (handling late arrivals, maintaining processing state, exactly-once semantics). The lesson: use batch for analytics and reporting where freshness of hours/days is acceptable. Use stream for operations that require immediate action — fraud detection, real-time notifications, live dashboards. Many systems need both (Lambda architecture).
