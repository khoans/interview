**280.** What is Cross-Origin authentication? How do you handle auth in a microservices architecture?

Cross-origin authentication deals with authenticating users when the frontend and backend are on different domains, and multiple backend services need to verify the same user identity. In microservices, the challenge is validating auth consistently across all services without each one managing its own auth logic.

On our marketplace, we used an API Gateway pattern: CloudFront/ALB handled TLS termination, then our routing rules validated the JWT token once at the edge before forwarding requests to downstream services. Each microservice (vehicle, dealer, search) also validated the JWT independently using the shared JWKS public key -- defense in depth. The token contained the user roles and dealer_id, so each service could make authorization decisions without calling back to the auth server. CORS was configured at the gateway level to allow our Next.js frontend domain.

The trade-off: validating tokens at both the gateway and each service adds redundant work but prevents unauthorized access if the gateway is bypassed (internal network compromise). The alternative -- trusting the gateway completely -- is simpler but creates a single point of failure for security. In our architecture, the 1-2ms per-service JWT validation was negligible compared to the security benefit.
