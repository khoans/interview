**282.** You discover an API endpoint exposing user emails without authentication. How do you fix it and prevent recurrence?

Immediate action: add authentication to the endpoint within minutes -- in Spring Security, this means ensuring the endpoint is not in the permitAll() list and requires a valid JWT. Then assess the damage: check access logs in Kibana to see if anyone exploited it, how long it was exposed, and how many records were accessed.

On our marketplace, this happened when a developer added a new endpoint for user profile lookup but forgot to include it in the security configuration. Fix steps: (1) Add @PreAuthorize annotation to the controller method, (2) Deploy hotfix immediately via GitHub Actions fast-track pipeline, (3) Review Kibana logs to identify any unauthorized access, (4) Report to security team for potential data breach notification assessment. To prevent recurrence: we changed our Spring Security config to deny-by-default (all endpoints require authentication unless explicitly marked public), added integration tests that verify every endpoint requires auth, and made security review mandatory for any PR adding new endpoints.

The lesson: default-deny is critical. Our initial security config was allow-by-default with specific endpoints protected -- this meant any new endpoint was accidentally public until someone remembered to add security. Flipping to deny-by-default eliminated this entire class of vulnerability.
