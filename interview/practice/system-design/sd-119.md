**119.** What is perfect forward secrecy? Why is it important for TLS?

========== THIS SECTION IS THE ANSWER ==========
Perfect forward secrecy (PFS) ensures that even if the server's long-term private key is compromised in the future, past encrypted sessions cannot be decrypted. This is achieved by using ephemeral (temporary) key pairs for each session — the session keys are generated, used, and discarded, never stored.

**Real example:** On our automotive marketplace, our ALB and CloudFront distributions used TLS 1.2+ with ECDHE (Elliptic Curve Diffie-Hellman Ephemeral) cipher suites, which provide PFS. Each TLS handshake generated a new ephemeral key pair for session key exchange. If an attacker recorded all our encrypted traffic and later stole our server's RSA private key (from ACM, theoretically), they still could not decrypt the recorded sessions because the ephemeral keys were never stored. We enforced PFS-capable cipher suites only by configuring the ALB security policy to TLS-1-2-2017-01 or newer.

**Trade-off:** PFS cipher suites (ECDHE, DHE) are slightly more computationally expensive than non-PFS suites (RSA key exchange) due to the Diffie-Hellman computation per session. On modern hardware, this overhead is negligible (~1ms). TLS 1.3 made PFS mandatory — all cipher suites use ephemeral key exchange — eliminating the need to choose. There is essentially no reason to avoid PFS today.
