**348.** Saga pattern vs two-phase commit: When should you use each for distributed transactions? What are the failure modes?

========== THIS SECTION IS THE ANSWER ==========

Use sagas for microservices where services own separate databases and long-running transactions are the norm. Use two-phase commit (2PC) only when all participants share the same transaction manager (e.g., within a single database or XA-compatible resources) and the transaction is short-lived.

On our marketplace, when a dealer published a listing, it involved multiple services: validate listing data, update PostgreSQL, index in Solr, and notify subscribed users. We used the saga pattern with compensating transactions — if Solr indexing failed, we marked the listing as "pending" in PostgreSQL and retried via SQS dead-letter queue processing. Two-phase commit across these independent services would have been impractical — it would require all services to hold locks simultaneously, creating coupling, latency, and a coordinator single point of failure.

The trade-off: sagas handle partial failures gracefully with compensating actions but are complex to design (every step needs a rollback action) and provide only eventual consistency — there is a window where data is inconsistent. 2PC guarantees atomicity but blocks all participants until the coordinator decides, does not scale across network boundaries, and if the coordinator crashes mid-commit, all participants are stuck holding locks. In microservices, sagas are almost always the right choice.
