**313.** Synchronous (REST/gRPC) vs Asynchronous (message queue): How do you decide which communication pattern to use between services?

========== THIS SECTION IS THE ANSWER ==========

Use synchronous when the caller needs an immediate response to proceed (user-facing requests). Use asynchronous when the caller does not need to wait and the work can be processed later (background jobs, notifications, data pipelines).

On our marketplace, the search API was synchronous — a user searches for cars and needs results in under 200ms. But when a dealer updated a listing, we published an event to SQS asynchronously. The listing service did not need to wait for the search index, analytics, and notification systems to all finish processing. Each consumer processed the SQS message at its own pace. If Solr indexing was slow, it did not block the dealer's update response.

The trade-off: async decouples services and improves resilience (if a downstream service is down, messages queue up and get processed when it recovers), but you lose immediate feedback — the dealer did not see their listing update in search results instantly, there was a 2-5 second delay. For user-facing flows where immediacy matters, synchronous is necessary. For everything else, async is almost always better.
