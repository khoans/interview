**190.** What is a service mesh (Istio, Linkerd)? How does it handle mTLS between pods?

========== THIS SECTION IS THE ANSWER ==========

A service mesh is an infrastructure layer that handles service-to-service communication transparently -- mTLS encryption, traffic routing, observability, and retry logic. It works by injecting a sidecar proxy (Envoy in Istio, linkerd-proxy in Linkerd) into every pod. All traffic flows through this proxy, which handles mTLS automatically -- the application code is completely unaware of the encryption.

In our marketplace, we handled inter-service security at the network level (VPC, security groups) and application level (OAuth2 tokens between services). A service mesh would have simplified this significantly. With Istio, every Spring Boot service pod gets an Envoy sidecar that automatically encrypts traffic using certificates issued by the Istio CA. When listing-service calls search-service, Envoy on the source pod encrypts the request, and Envoy on the destination pod decrypts and verifies the certificate -- zero code changes. Istio also provides request-level metrics, distributed tracing, and traffic splitting for canary deployments.

The trade-off: service meshes add significant operational complexity and resource overhead -- each sidecar consumes CPU and memory, and the control plane (istiod) is another component to manage. Linkerd is lighter than Istio but has fewer features. For our 8-service architecture, the overhead was not justified. Consider a service mesh when you have 20+ services and need uniform mTLS, observability, and traffic management without modifying application code.
