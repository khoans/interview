**32.** What is gRPC? When would you choose it over REST?

========== THIS SECTION IS THE ANSWER ==========

gRPC is a high-performance RPC framework using Protocol Buffers (binary serialization) over HTTP/2. It supports bidirectional streaming, multiplexing, and generates strongly-typed client/server code from .proto files. Choose gRPC over REST when you need low latency between internal services, strict API contracts, or streaming.

In our project, we used REST for all external-facing APIs (frontend, dealer partners) because REST is universally understood and easy to debug with curl. For internal service-to-service communication, we evaluated gRPC for the pricing-to-inventory service calls that happened on every listing page load. The pricing service made ~50K calls/sec to inventory during peak hours. With REST/JSON, each call had ~2KB of JSON overhead (verbose field names, text encoding). Switching to gRPC with protobuf would have reduced payload size by ~60% and removed the JSON serialization overhead. Ultimately we chose to keep REST internally too, because the team was more familiar with it and the latency was acceptable at ~15ms per call with connection reuse.

The trade-off: gRPC is faster and more efficient but harder to debug (binary payloads — you can't just curl it), requires protobuf compilation in your build pipeline, and has poor browser support (needs gRPC-web proxy). REST is slower but universally toolable and debuggable. The lesson: gRPC shines for high-throughput internal microservice communication. For external APIs or small teams, REST's simplicity wins.
