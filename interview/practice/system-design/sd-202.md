**202.** What is the difference between IAM role and IAM user? When do you use each?

========== THIS SECTION IS THE ANSWER ==========

An IAM user has permanent long-lived credentials (access key + secret key or password) tied to a person. An IAM role has no permanent credentials -- it provides temporary security tokens (via STS) that expire. Use users for human identities that need console/CLI access. Use roles for everything else -- services, cross-account access, federated users, and any automated process.

In our marketplace: IAM users for our developers (with MFA and password policies). IAM roles for: (1) ECS task roles -- each microservice assumed its role at startup, getting 6-hour temporary credentials, (2) Lambda execution roles -- our image resizing Lambda assumed a role with S3 read/write, (3) cross-account roles -- our CI/CD in the dev account assumed a role in the production account for deployments, (4) GitHub Actions used OIDC federation to assume a deployment role without storing any AWS credentials in GitHub secrets. We had zero service accounts using IAM users -- everything automated used roles.

The lesson: IAM users with access keys are the number one source of credential leaks. We enforced a policy: no access keys older than 90 days, no access keys for service accounts (use roles), and mandatory MFA for all users. When AWS introduced IAM Identity Center (SSO), we migrated to federated access -- developers authenticate through corporate SSO, assume a role, and get temporary credentials. Zero long-lived keys.
