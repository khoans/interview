**25.** What is a distributed cache? How does consistent hashing help?

========== THIS SECTION IS THE ANSWER ==========

A distributed cache spreads cached data across multiple nodes to handle more data and traffic than a single machine can. Consistent hashing maps keys to nodes on a virtual ring — when a node is added or removed, only ~1/N of keys need to remap (instead of rehashing everything), minimizing cache misses during scaling events.

In our project, we used ElastiCache Redis in cluster mode with 6 shards. Redis Cluster uses hash slots (16,384 slots distributed across shards) — which is conceptually similar to consistent hashing. When we scaled from 4 to 6 shards during a traffic spike (Black Friday car deals), Redis automatically migrated only the affected hash slots to new shards. Without consistent hashing, all cached data would have been remapped, causing a massive cache miss storm hitting our PostgreSQL database simultaneously. With it, about 30% of keys remapped, and the database handled the temporary increased load without issues.

The trade-off: consistent hashing adds complexity compared to simple modulo hashing (key % N nodes), and you can still get hot spots if key distribution is skewed. We mitigated hot spots by using virtual nodes — each physical shard owned multiple points on the hash ring, evening out the distribution. The lesson: consistent hashing is essential for any distributed cache that needs to scale dynamically. Don't use modulo hashing in production.
