**192.** What is container escape? How do you prevent a container from accessing the host?

========== THIS SECTION IS THE ANSWER ==========

Container escape is when a process inside a container breaks out of its isolation and gains access to the host OS -- reading host files, accessing other containers, or executing commands on the host. This typically happens through running as root, mounting the host filesystem, using privileged mode, or exploiting kernel vulnerabilities.

Prevention measures we would apply: (1) never run containers in privileged mode (privileged: false), (2) run as non-root user (runAsNonRoot: true) -- all our Spring Boot images used UID 1000, (3) use read-only root filesystem (readOnlyRootFilesystem: true), (4) drop all Linux capabilities (capabilities.drop: [ALL]) and add back only what is needed, (5) prevent host path mounts via PSA or Gatekeeper policies, (6) disable privilege escalation (allowPrivilegeEscalation: false), (7) use seccomp and AppArmor profiles to restrict syscalls.

The trade-off: these restrictions can break applications that rely on root access or specific Linux capabilities. Our Java services had no issues because Spring Boot does not require elevated privileges. But monitoring agents (like node-exporter) and log collectors may need hostPath mounts or specific capabilities -- isolate them in a dedicated namespace with relaxed security policies while keeping application namespaces locked down.
