**21.** Explain cache eviction policies: LRU, LFU, FIFO. When would you use each?

========== THIS SECTION IS THE ANSWER ==========

Cache eviction policies decide which entries to remove when the cache is full. LRU (Least Recently Used) evicts the entry not accessed for the longest time. LFU (Least Frequently Used) evicts the entry with the fewest accesses. FIFO (First In, First Out) evicts the oldest entry regardless of access pattern.

In our project, we used ElastiCache Redis with an LRU-based eviction policy (allkeys-lru) for our search results cache. LRU was the right fit because car shoppers have temporal locality — a user searching for "Toyota Camry in Dallas" will likely refine that search multiple times in one session, so recently accessed results should stay cached. We configured Redis with a 4GB maxmemory limit. During peak evening hours, the cache was full and LRU evicted stale search results from morning sessions that nobody was looking at anymore. Cache hit rate stayed at ~85%.

The trade-off: LRU is simple and works well for most workloads but can be fooled by a one-time scan (a batch job reading all listings would evict popular items). LFU handles this better by keeping frequently accessed items, but it's slower to adapt to changing access patterns — a listing that was popular last week stays cached even if nobody cares about it now. The lesson: LRU is the safe default for web applications. Only switch to LFU if you have proven scan-pollution problems.
