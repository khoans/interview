**20.** What is the N+1 query problem? How do you solve it?

========== THIS SECTION IS THE ANSWER ==========

The N+1 problem occurs when you query a list of N entities, then execute one additional query per entity to fetch related data — resulting in N+1 total queries instead of 1 or 2. It's the most common JPA/Hibernate performance killer.

In our project, we hit this with dealer listings. Loading a dealer page fetched the dealer (1 query), then for each of their 200 listings, it lazily loaded the listing images (200 queries) — 201 queries total, taking 3 seconds. We fixed it three ways depending on the case: (1) JOIN FETCH in JPQL — `SELECT d FROM Dealer d JOIN FETCH d.listings WHERE d.id = :id` collapsed it to 1 query. (2) @EntityGraph annotation for more complex cases where we needed to specify exactly which associations to eagerly load. (3) For the search results page, we bypassed JPA entirely and wrote a native SQL query with a single JOIN, mapped to a DTO — this was the fastest approach for read-heavy endpoints.

The trade-off: JOIN FETCH can cause cartesian product explosion if you fetch multiple collections simultaneously (e.g., listings + reviews). In that case, we used @BatchSize(size=50) on the collection, which changed 200 individual queries into 4 batched queries — not perfect but much better. The lesson: always enable Hibernate SQL logging in dev and watch for N+1 patterns before they reach production.
