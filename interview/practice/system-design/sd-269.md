**269.** What is OAuth 2.0 client credentials flow? When do you use it (service-to-service auth)?

The client credentials flow is used for machine-to-machine (M2M) authentication where no user is involved. The client (service) sends its client_id and client_secret directly to the authorization server and receives an access token. There is no user consent or redirect -- the service authenticates as itself.

On our marketplace, we used client credentials flow for internal service-to-service communication. For example, our vehicle pricing Lambda function needed to call the dealer service API to fetch dealer discount tiers. The Lambda had its own client_id/secret (stored in AWS Secrets Manager), exchanged them for a JWT access token with specific scopes (read:dealer-pricing), and called the API. No user was involved -- the Lambda acted on behalf of the system, not a specific user.

The trade-off: client credentials tokens represent the service, not a user, so you lose user-level audit trails. If Service A calls Service B with a client credentials token, you know which service made the call but not which user action triggered it. We solved this by passing a correlation ID header that traced back to the original user request.
