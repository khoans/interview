**67.** What is API rate limiting and throttling? How do you implement it at scale?

========== THIS SECTION IS THE ANSWER ==========

Rate limiting caps the number of requests a client can make (e.g., 1000/minute). Throttling slows down or degrades service when limits are approached (e.g., return lower-quality results or delay responses). At scale, rate limiting must be distributed — counters shared across all API instances, not per-instance.

In our project, we implemented distributed rate limiting using ElastiCache Redis. Each API request incremented a Redis counter keyed by API-key + minute-window (e.g., rate:dealer123:202401151430). Redis INCR with EXPIRE ensured atomic increment and automatic cleanup. All 8 API gateway instances shared the same Redis counter, so the limit was global. We had tiered limits: free dealer accounts got 100 requests/minute, premium got 1000, enterprise partners got 5000. When a client exceeded the limit, we returned 429 Too Many Requests with headers: X-RateLimit-Limit (max), X-RateLimit-Remaining (left), Retry-After (seconds to wait).

The trade-off: Redis-based distributed rate limiting adds ~1ms latency per request (Redis round trip) and Redis becomes a dependency for every API call. If Redis goes down, you either block all requests (fail-closed) or allow all requests (fail-open). We chose fail-open — a brief period without rate limiting was less damaging than blocking all API traffic. The lesson: implement rate limiting at the edge (API Gateway or ALB) when possible — AWS API Gateway has built-in throttling that doesn't require managing Redis. Only use custom Redis-based limiting when you need complex per-client or per-endpoint rules.
