**318.** Choreography vs orchestration in microservices: When should each service react to events independently vs having a central coordinator?

========== THIS SECTION IS THE ANSWER ==========

Use choreography when the workflow is simple (3-4 steps) and services can react independently. Use orchestration when the workflow is complex, has conditional logic, requires compensating transactions, or needs clear visibility into the overall process state.

On our marketplace, listing publication used choreography — dealer submits listing, validation service reacts, then search indexing reacts, then notification reacts. Each service listened to events independently via SQS. Simple and decoupled. But if we had a complex workflow like a vehicle purchase transaction (payment, title transfer, shipping, insurance verification, escrow), orchestration with a central coordinator would be better because you need to track state, handle partial failures, and execute compensating actions (refund if shipping fails).

The trade-off: choreography is loosely coupled but becomes impossible to understand when you have 10+ steps — the flow is scattered across services with no single view. Orchestration gives you a clear workflow definition and easier debugging but creates a central point of failure and tighter coupling to the orchestrator. For simple event chains, choreography wins. For business-critical multi-step workflows, orchestrate.
