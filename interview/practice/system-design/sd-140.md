**140.** What is certificate pinning? When should you use it and what are the risks?

========== THIS SECTION IS THE ANSWER ==========
Certificate pinning hardcodes the expected server certificate (or its public key hash) in the client application, so the client only trusts that specific certificate rather than any certificate signed by a trusted CA. This prevents MITM attacks even if a CA is compromised or a rogue CA issues a fraudulent certificate for your domain.

**Real example:** On our automotive marketplace, we considered certificate pinning for the mobile app's communication with our API. The idea was to pin our ACM certificate's public key hash in the Android/iOS app, rejecting any other certificate even if signed by a trusted CA. We decided against it because ACM auto-rotates certificates, and if the new certificate had a different key, the pinned mobile app would break until users updated. With 30M monthly users, forcing an app update for a certificate change was too risky.

**Trade-off:** Certificate pinning provides strong MITM protection but creates a brittle deployment model. If the pinned certificate expires or rotates, the app breaks with no server-side fix — you must ship an app update. Pin the public key (SPKI hash) rather than the full certificate to survive certificate renewals that reuse the same key pair. For web applications, pinning is deprecated in browsers (HPKP removed from Chrome). For mobile apps, use it only if you have reliable OTA update mechanisms and pin backup keys.
