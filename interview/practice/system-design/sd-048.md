**48.** What is a distributed lock? How do you implement it (Redis, Zookeeper)?

========== THIS SECTION IS THE ANSWER ==========

A distributed lock ensures only one process across multiple servers can execute a critical section at a time. With Redis, you use SET key value NX EX ttl (set if not exists with expiration). The process that successfully sets the key holds the lock. Zookeeper uses ephemeral sequential znodes — the process with the lowest sequence number holds the lock, and the lock auto-releases if the process dies.

In our project, we used Redis distributed locks (Redisson library) for our nightly dealer inventory sync job. The job ran on multiple ECS instances, but we only wanted one instance processing a dealer's inventory at a time to avoid conflicting updates. Each instance tried to acquire a lock keyed by dealerId (e.g., lock:dealer:12345) with a 5-minute TTL. Only one succeeded; others skipped that dealer. The TTL was a safety net — if the instance crashed mid-processing, the lock auto-expired instead of being held forever (deadlock prevention).

The trade-off: Redis locks (even with Redlock algorithm) are not perfectly safe — clock skew, network partitions, or Redis failover can cause two processes to hold the "same" lock simultaneously. Zookeeper provides stronger guarantees (consensus-based) but is operationally heavier. For our use case (preventing duplicate processing, not financial transactions), Redis locks with TTL were good enough. The lesson: distributed locks should be a last resort — redesign to avoid them if possible (idempotent operations, partitioned work assignment). When you must lock, keep the locked section short and always set a TTL.
