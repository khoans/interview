**367.** What is Nginx caching? How do you configure proxy_cache to cache upstream responses and what are the cache invalidation strategies?

========== THIS SECTION IS THE ANSWER ==========

Nginx can cache responses from upstream servers so repeated requests don't hit your backend. You define a cache zone with `proxy_cache_path`, enable it with `proxy_cache`, and control what gets cached with `proxy_cache_valid` and `proxy_cache_key`.

Config example: `proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=api_cache:10m max_size=1g inactive=60m;` then in the location block: `proxy_cache api_cache; proxy_cache_valid 200 5m; proxy_cache_valid 404 1m; proxy_cache_key "$scheme$request_method$host$request_uri";`. This caches 200 responses for 5 minutes and 404s for 1 minute. For invalidation: **TTL-based** — cache expires after the configured time. **Purge** — use the `proxy_cache_purge` module to explicitly delete specific cache entries. **Cache-Control headers** — respect backend's Cache-Control/Expires headers with `proxy_cache_use_stale` for serving stale content during backend failures.

The trade-off: Nginx caching is fast (serves from disk/memory, no backend hit) but adds staleness risk. In our project, we cached static API responses (product categories, site config) at the Nginx level with 10-minute TTL, but never cached user-specific data (cart, saved items). At ATC scale, we used CloudFront + Redis instead — Nginx-level caching is great for single-server setups but doesn't share cache across instances.
