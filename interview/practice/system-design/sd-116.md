**116.** What is envelope encryption? How does AWS KMS use it?

========== THIS SECTION IS THE ANSWER ==========
Envelope encryption uses two layers of keys: a data encryption key (DEK) encrypts the actual data locally, then a master key (KEK) encrypts the DEK. Only the encrypted DEK is stored alongside the data. To decrypt, you first decrypt the DEK using the master key (via KMS API call), then use the DEK to decrypt the data locally.

**Real example:** On our automotive marketplace, when S3 encrypted vehicle documents with SSE-KMS, it worked exactly this way. S3 generated a unique DEK per object, encrypted the document with AES-256 using that DEK, then called KMS to encrypt the DEK with our CMK (Customer Master Key). The encrypted DEK was stored as object metadata. On read, S3 sent the encrypted DEK to KMS, got back the plaintext DEK, decrypted the document, and returned it. Our Spring Boot services never saw the DEK — it was all transparent.

**Trade-off:** Envelope encryption avoids sending large data to KMS (KMS has a 4KB plaintext limit and API call latency of ~10ms). Instead, only the small DEK travels to KMS. The downside is managing two layers of keys and understanding that compromising the CMK compromises all DEKs. KMS mitigates this with strict IAM policies, CloudTrail audit logging for every key usage, and the CMK never leaves the HSM hardware.
