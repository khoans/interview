**326.** When should you shard your database vs using read replicas? What scale triggers each?

========== THIS SECTION IS THE ANSWER ==========

Use read replicas when your bottleneck is read throughput — many queries but writes are manageable. Shard when your bottleneck is write throughput or your dataset exceeds what a single instance can store/index efficiently.

On our marketplace, we used RDS read replicas for PostgreSQL. Our workload was heavily read-biased — 30M users browsing listings (reads) vs thousands of dealers updating listings (writes). Read replicas handled the read load by distributing queries across multiple instances. We never needed sharding because our write volume was moderate and our dataset (millions of listings) fit comfortably on a single RDS instance with proper indexing. Sharding would have been necessary if we had billions of rows or write throughput exceeding what a single primary could handle.

The trade-off: read replicas are simple to set up (one RDS config change), require no application code changes (just point reads to the replica endpoint), but only solve read scaling. Sharding solves both read and write scaling but requires application-level routing logic, makes cross-shard queries expensive or impossible, and complicates Flyway migrations. Always exhaust read replicas, caching, and query optimization before considering sharding.
