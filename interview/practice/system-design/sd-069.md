**69.** What is a VPC (Virtual Private Cloud)? How do you design network isolation?

========== THIS SECTION IS THE ANSWER ==========

A VPC is a logically isolated virtual network within a cloud provider where you control IP ranges, subnets, route tables, and security rules. Network isolation is achieved through subnets (public vs private), security groups (instance-level firewall), NACLs (subnet-level firewall), and routing rules.

In our project, we designed a three-tier VPC architecture. Public subnets (3 AZs) held only the ALB and NAT Gateway — internet-facing components. Private application subnets (3 AZs) held ECS services — no direct internet access, outbound traffic went through NAT Gateway. Private data subnets (3 AZs) held RDS PostgreSQL and ElastiCache Redis — no internet access at all, only reachable from the application subnet. Security groups enforced strict rules: the database security group only accepted connections on port 5432 from the application security group. The application security group only accepted traffic from the ALB security group on ports 8080-8089. This meant even if an attacker compromised the ALB, they couldn't directly reach the database.

The trade-off: strict network isolation adds operational complexity — developers can't directly connect to the database for debugging (they need a bastion host or SSM Session Manager). More subnets mean more NAT Gateway costs ($0.045/GB). VPC peering across regions for disaster recovery adds routing complexity. The lesson: always put databases in private subnets with no internet access. Use AWS SSM Session Manager instead of bastion hosts — it's more secure (no SSH keys to manage) and auditable. Never put RDS in a public subnet, even with security groups — defense in depth matters.
