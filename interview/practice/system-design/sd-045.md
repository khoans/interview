**45.** What is the strangler fig pattern? How do you migrate from monolith to microservices?

========== THIS SECTION IS THE ANSWER ==========

The strangler fig pattern incrementally replaces a monolith by routing specific functionality to new microservices while the old system continues running. Over time, more features are strangled out until the monolith is empty and can be decommissioned. Named after the strangler fig tree that grows around a host tree and eventually replaces it.

In our project, this was exactly how we migrated. The original system was a monolithic Spring Boot application. We started by extracting the search service — the highest-traffic, most performance-critical module. We put an API gateway (ALB with path-based routing) in front: /api/search/* routed to the new search microservice, everything else still went to the monolith. The search service got its own database (Solr) and its own deployment pipeline. Over 6 months, we extracted inventory, pricing, and dealer management the same way. The monolith shrank with each extraction. We ran both old and new in parallel with feature flags to compare results before fully cutting over.

The trade-off: strangler fig is low-risk (you can roll back by routing traffic back to the monolith) but slow — each extraction requires understanding the monolith's tangled dependencies. We had cases where "simple" extractions uncovered hidden database-level coupling (shared tables, triggers). The lesson: start with the most independent, highest-value module. Don't try to extract everything at once. And keep the monolith running until you're 100% confident in the new service — we ran parallel for weeks before decommissioning each module.
