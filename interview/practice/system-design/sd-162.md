**162.** What is a Pod in Kubernetes? Why is it the smallest deployable unit (not a container)?

========== THIS SECTION IS THE ANSWER ==========

A Pod is the smallest deployable unit in Kubernetes -- it wraps one or more containers that share the same network namespace (same IP, localhost communication), storage volumes, and lifecycle. It is not a single container because some workloads need tightly coupled helper containers (sidecars) that must share resources -- like a logging agent alongside your app, or an Envoy proxy handling mTLS.

When evaluating EKS for our marketplace, we designed pods with a primary Spring Boot container plus an AWS CloudWatch logging sidecar sharing a volume mount for log files. The sidecar read logs from the shared volume and shipped them to CloudWatch, while the main container focused purely on business logic. In ECS, we achieved similar behavior with task definitions containing multiple containers sharing the same network and volumes.

The trade-off: multi-container pods simplify communication (localhost instead of service discovery) but increase pod resource requirements and make scaling less granular -- if only the main container needs more replicas, the sidecar scales too. Keep sidecar containers lightweight to minimize waste.
