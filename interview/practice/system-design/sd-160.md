**160.** What is secret scanning? How do you prevent secrets from being committed to git?

========== THIS SECTION IS THE ANSWER ==========

Secret scanning detects credentials, API keys, tokens, and other secrets accidentally committed to version control. Prevention is layered: pre-commit hooks block secrets locally, CI scanning catches anything that slips through, and GitHub built-in secret scanning alerts on known secret patterns.

In our marketplace: (1) Pre-commit -- we used git-secrets with a pre-commit hook that scanned for AWS key patterns (AKIA prefix), database connection strings, and custom regex for our internal API keys. (2) CI -- GitHub Actions ran TruffleHog on every PR, scanning the full diff for high-entropy strings and known patterns. (3) GitHub native secret scanning was enabled on the repo, alerting when it detected partner integration tokens. When a developer accidentally staged a Secrets Manager ARN in a config file, the pre-commit hook caught it before push. All secrets lived in AWS Secrets Manager, injected via environment variables in ECS task definitions.

The lesson: prevention (pre-commit hooks) is better than detection (scanning after commit). But when detection catches something, rotate the secret immediately -- assume it is compromised. We had a runbook that could rotate any credential within 30 minutes.
