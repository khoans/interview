**333.** Cache-aside vs read-through vs write-through: How do your read/write patterns determine the right strategy?

========== THIS SECTION IS THE ANSWER ==========

Cache-aside (lazy loading) is best for read-heavy workloads where cache misses are acceptable. Read-through automates cache population on miss. Write-through ensures cache is always updated on writes, best for write-then-read patterns. Choose based on whether you tolerate stale data and who controls the cache logic.

On our marketplace, we used cache-aside with ElastiCache Redis. The application checked Redis first; on a cache miss, it queried PostgreSQL, then wrote the result to Redis with a TTL. This was ideal because our reads were 100x more frequent than writes, and a cache miss (cold start or TTL expiry) just added one extra database query — acceptable for our latency requirements. We did not use write-through because dealers updated listings infrequently, and eagerly populating the cache on every write would have wasted Redis memory on listings that might never be read again.

The trade-off: cache-aside gives you full control but the application owns all caching logic (check, miss, populate, invalidate). Write-through guarantees cache freshness but adds write latency (write to DB and cache on every mutation). Read-through simplifies application code but requires a cache library that supports it. For most Spring Boot applications, cache-aside with `@Cacheable` annotations is the simplest and most flexible.
