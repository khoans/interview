**191.** What is the principle of least privilege for Kubernetes RBAC? Give a real example.

========== THIS SECTION IS THE ANSWER ==========

Least privilege in Kubernetes RBAC means granting each user, service account, and CI/CD pipeline only the minimum API permissions they need. Every Role should specify exact resources and verbs -- never use wildcards (*) in production. This limits the blast radius if credentials are compromised.

In our marketplace, we applied this rigorously on AWS IAM and would do the same in Kubernetes. Example: our GitHub Actions deployment service account would get a Role allowing only create, update, and patch on Deployments, Services, and ConfigMaps in the production namespace -- it cannot read Secrets, delete pods, or access other namespaces. A developer service account gets get, list, and logs on pods in the dev namespace -- they can debug but not modify anything. The monitoring system (Prometheus) gets a ClusterRole with only get and list on pods, endpoints, and nodes -- read-only metrics collection.

The lesson: we once had an overly broad IAM role that could describe all resources across all services. When that role's temporary credentials leaked through a debug log, the attacker could enumerate our entire infrastructure. After that incident, we scoped every role to exactly the resources it operated on. The extra effort in RBAC configuration pays off when a breach happens.
