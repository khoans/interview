**306.** When should you use a monorepo vs polyrepo for microservices? What are the CI/CD implications of each?

========== THIS SECTION IS THE ANSWER ==========

Use monorepo when teams share libraries, need atomic cross-service changes, or are under 50 engineers. Use polyrepo when services are truly independent, owned by separate teams, and have different tech stacks.

On our marketplace, we used a polyrepo approach — each microservice (search, listings, user service) had its own GitHub repository with its own GitHub Actions pipeline. This made sense because teams operated independently, deployed on different schedules, and had clear ownership boundaries. The downside was shared libraries (common DTOs, auth utilities) — we published them to a private Maven repository, and version bumps required coordinated PRs across repos.

The trade-off: monorepo gives you atomic commits across services and easier code sharing, but CI gets complex — you need smart build tools to only build/test what changed, or every commit triggers all pipelines. Polyrepo gives clean isolation but shared code becomes a versioning nightmare. We mitigated polyrepo pain by keeping shared libraries minimal and using contract testing between services.
