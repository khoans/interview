**279.** What is a session store? Compare in-memory, Redis, and database-backed sessions.

A session store persists user session data between requests. **In-memory** stores sessions in the application process (fast, but lost on restart and not shareable across instances). **Redis** stores sessions in a distributed cache (fast, shared, supports TTL). **Database-backed** stores sessions in PostgreSQL/MySQL (persistent, queryable, but slowest).

On our marketplace, we used Spring Session with ElastiCache Redis as our session store. This was essential because we had multiple ECS instances behind an ALB -- without a shared session store, a user request hitting a different instance would lose their session. Redis gave us sub-millisecond read latency, automatic TTL-based expiration (sessions expired after 30 minutes of inactivity), and the ability to scale horizontally. We configured Redis in cluster mode across two AZs for high availability.

The trade-off: Redis adds infrastructure complexity and cost (our ElastiCache cluster was about $200/month). In-memory sessions are free and fastest but only work with sticky sessions (which cause uneven load). Database sessions are the most durable but add 5-10ms latency per request. For our traffic volume (30M monthly users), Redis was the clear winner.
