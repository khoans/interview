**369.** How do you configure Nginx for WebSocket proxying? What headers (Upgrade, Connection) need to be forwarded and why?

========== THIS SECTION IS THE ANSWER ==========

WebSocket starts as an HTTP request with an `Upgrade: websocket` header, then the connection is upgraded from HTTP to a persistent bidirectional TCP connection. Nginx must forward these upgrade headers to the backend, otherwise it treats the request as a normal HTTP request and the WebSocket handshake fails.

Config: `location /ws/ { proxy_pass http://backend; proxy_http_version 1.1; proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection "upgrade"; proxy_read_timeout 3600s; }`. `proxy_http_version 1.1` is required because WebSocket needs HTTP/1.1 (not 1.0). `Upgrade` and `Connection` headers tell the backend to switch protocols. `proxy_read_timeout` must be increased from the default 60 seconds — WebSocket connections are long-lived, and Nginx will close idle connections after the timeout.

In our ecommerce project, we considered WebSockets for real-time order status updates but opted for polling instead due to simplicity. On the ATC project, analytics events used WebSockets on some microsites. The trade-off: WebSocket proxying through Nginx works well but each open connection consumes a file descriptor. For thousands of persistent connections, you need to tune `worker_connections` and OS-level `ulimit` — otherwise Nginx runs out of file descriptors.
