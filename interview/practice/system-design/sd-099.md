**99.** Design a task scheduler (like cron at scale). How do you ensure exactly-once execution?

========== THIS SECTION IS THE ANSWER ==========
Key components: a task registry (database storing task definitions, schedules, ownership), a scheduler service that polls for due tasks, a distributed lock to ensure only one instance picks each task, and an execution tracker that records outcomes. Exactly-once is achieved through idempotent execution plus distributed locking.

**Real example:** On our automotive marketplace, we had scheduled jobs: nightly Solr re-indexing, daily dealer report generation, hourly stale-listing cleanup. We used Spring's @Scheduled with ShedLock (backed by PostgreSQL) to ensure only one ECS instance executed each job. ShedLock acquires a row-level lock in a shedlock table with lock_until, locked_at, and locked_by columns. If a node crashed mid-execution, the lock expired after the configured lockAtMostFor duration and another node could pick it up. Each job was designed to be idempotent — re-running the stale-listing cleanup twice had no adverse effect.

**Trade-off:** Database-based locking (ShedLock) adds DB load per scheduled check and has lock granularity limited to the configured interval. For high-frequency scheduling (every second), Redis-based locking is faster. Also, exactly-once is technically impossible in distributed systems — we achieved effectively-once through idempotent operations plus at-least-once delivery. The key lesson: always design jobs to be safe to re-run.
