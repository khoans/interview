**274.** What is JWT structure? Explain header, payload, and signature. What should never be in a JWT?

A JWT has three Base64URL-encoded parts separated by dots: **Header** (algorithm and token type, e.g., {"alg":"RS256","typ":"JWT"}), **Payload** (claims -- iss, sub, exp, iat, plus custom claims like roles and scopes), and **Signature** (cryptographic signature of header+payload using the specified algorithm and secret/key).

On our marketplace, our JWTs contained: iss (our auth server URL), sub (user ID), exp (15 min from issuance), roles (CONSUMER, DEALER_ADMIN), dealer_id (for dealer-scoped operations), and scopes (read:vehicles, write:listings). Spring Security validated the signature using the public key from our JWKS endpoint, checked expiration, and extracted claims for authorization decisions.

What should NEVER be in a JWT: passwords, credit card numbers, PII like SSN or full addresses, internal system secrets, or sensitive business data. JWTs are Base64-encoded (not encrypted) -- anyone can decode the payload. We made this mistake early by including user emails in tokens; after a security review, we moved to opaque user IDs only, with email lookups done server-side when needed.
