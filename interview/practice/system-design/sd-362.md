**362.** How does Nginx handle concurrent connections? Explain its event-driven, non-blocking architecture vs Apache's process/thread-per-connection model.

========== THIS SECTION IS THE ANSWER ==========

Nginx uses an event-driven, non-blocking architecture. A single worker process handles thousands of connections using an event loop (epoll on Linux) — it doesn't block waiting for I/O. When a request needs to wait for a backend response or disk read, Nginx moves on to handle other connections and comes back when the I/O is ready. Apache's traditional model (prefork/worker MPM) spawns a process or thread per connection — each connection consumes memory and CPU context, so 10,000 concurrent connections means 10,000 threads.

In our project, we ran Nginx on a single EC2 instance handling the frontend static files and proxying to Spring Boot. Even with modest traffic, Nginx used ~20MB of memory while serving hundreds of concurrent connections. Apache in the same role would have consumed significantly more memory and required tuning MaxClients/ThreadsPerChild to avoid exhaustion.

The trade-off: Nginx's event-driven model is vastly more efficient for I/O-bound workloads (proxying, serving files) — it can handle 10,000+ concurrent connections with minimal resources. But for CPU-heavy processing (like running PHP via mod_php), Apache's per-process model can be simpler since the processing happens inline. For a reverse proxy / static file server role, Nginx wins hands down.
