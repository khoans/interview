**358.** How did you handle security logging and alerting? What security events did you monitor and what triggered alerts?

========== THIS SECTION IS THE ANSWER ==========

We monitored security events at three layers — application, infrastructure, and network — with escalating alert severity.

**Application layer** (New Relic + Kibana): Failed authentication attempts (>10 per minute from same IP triggered alert), authorization failures (user attempting to access another dealer's data), JWT token anomalies (expired/invalid tokens spike), and API rate limit breaches. **Infrastructure layer** (CloudTrail + CloudWatch): IAM credential usage from unusual locations, root account login attempts (immediate PagerDuty alert), security group changes, KMS key usage anomalies, and S3 bucket policy modifications. **Network layer** (VPC Flow Logs): Outbound traffic to unknown IPs, port scanning patterns, and rejected connection spikes indicating reconnaissance. We used CloudWatch Alarms with SNS topics — low-severity alerts went to a Slack channel, high-severity (root login, security group changes, data exfiltration patterns) triggered PagerDuty pages to the on-call engineer.

Lesson learned: the hardest part was tuning thresholds to avoid alert fatigue. Initially we alerted on every failed login, which generated hundreds of false positives daily from users mistyping passwords. We tuned to velocity-based alerts (rate of failures) rather than individual events, which reduced noise by 90% while still catching real brute-force attempts.
