**215.** What is RDS encryption? How do you encrypt data at rest and in transit for databases?

========== THIS SECTION IS THE ANSWER ==========

RDS encryption at rest uses KMS to encrypt the underlying storage, automated backups, snapshots, and read replicas. You enable it at instance creation time (cannot be added later without migration). Encryption in transit uses SSL/TLS connections between the application and database -- configured by requiring SSL in the RDS parameter group and using SSL certificates in the JDBC connection.

In our marketplace: (1) At rest -- our PostgreSQL RDS instance was encrypted with a customer-managed KMS key from day one. All automated backups and cross-region snapshot copies were encrypted. When we created read replicas, they inherited the encryption. (2) In transit -- we set rds.force_ssl=1 in the parameter group, rejecting any non-SSL connection. Our Spring Boot JDBC URL included sslmode=verify-full with the RDS CA certificate bundled in the application, ensuring both encryption and server identity verification. (3) We also encrypted specific columns containing PII (SSN, email) at the application level using KMS envelope encryption -- even a DBA with full database access could not read raw PII without the KMS key.

The trade-off: encryption at rest has negligible performance impact (AES-256 is hardware-accelerated). SSL in transit adds ~5-10% overhead due to the TLS handshake and encryption, but connection pooling (HikariCP in our Spring Boot apps) minimizes this by reusing connections. Application-level column encryption adds complexity but provides true defense-in-depth when combined with storage encryption.
