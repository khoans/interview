**95.** Design a file storage system (like Google Drive). How do you handle uploads and sharing?

========== THIS SECTION IS THE ANSWER ==========
Key components: object storage (S3) for actual files, a metadata service (PostgreSQL) tracking file ownership/permissions/versions, a chunked upload service for large files with resume capability, and a sharing/permission model with ACLs or role-based access.

**Real example:** On our automotive marketplace, dealers uploaded vehicle photos and documents (inspection reports, Carfax). We used S3 presigned URLs for direct browser-to-S3 uploads, bypassing our servers entirely — this was critical for handling hundreds of concurrent photo uploads without overwhelming our ECS instances. Metadata (file owner, listing association, upload timestamp) was stored in PostgreSQL. Sharing was controlled through IAM-style permissions: dealer admins could share document access with sub-users via a permission table (user_id, file_id, permission_level).

**Trade-off:** Presigned URLs expose S3 paths and have time-limited validity, so we set a 15-minute expiration. Chunked upload (multipart in S3) supports resume for large files but requires cleanup of abandoned uploads — we ran a lifecycle policy deleting incomplete multipart uploads after 7 days. Versioning in S3 provides file history but doubles storage costs if not managed with lifecycle rules to expire old versions.
