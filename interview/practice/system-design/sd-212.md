**212.** What is VPC security? Explain security groups, NACLs, VPC Flow Logs, and VPC endpoints.

========== THIS SECTION IS THE ANSWER ==========

VPC security layers: Security Groups are stateful firewalls at the instance/ENI level (allow rules only, return traffic automatically allowed). NACLs (Network ACLs) are stateless firewalls at the subnet level (allow and deny rules, must explicitly allow return traffic). VPC Flow Logs capture IP traffic metadata for auditing. VPC Endpoints allow private connectivity to AWS services without traversing the internet.

In our marketplace VPC design: (1) Security Groups -- the ALB SG allowed inbound 443 from the internet; the ECS SG allowed inbound 8080 only from the ALB SG; the RDS SG allowed inbound 5432 only from the ECS SG. This created a chain: internet -> ALB -> ECS -> RDS, no skipping layers. (2) NACLs on the database subnet denied all traffic except from the application subnet on port 5432. (3) VPC Flow Logs sent to CloudWatch detected unusual traffic patterns -- we once found an ECS task making unexpected outbound connections to an unknown IP (turned out to be a misconfigured health check). (4) VPC Endpoints for S3, DynamoDB, Secrets Manager, and KMS kept all AWS API traffic private, never hitting the public internet.

The trade-off: layering SGs and NACLs provides defense in depth but increases complexity. NACLs are hard to manage because they are stateless (you must define both inbound and outbound rules with ephemeral port ranges). We relied primarily on Security Groups and used NACLs only as a coarse secondary layer on sensitive subnets.
