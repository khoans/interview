**30.** What is data lake vs data warehouse? When would you use each?

========== THIS SECTION IS THE ANSWER ==========

A data warehouse stores structured, pre-processed data optimized for analytical queries (star schema, columnar storage — Redshift, BigQuery). A data lake stores raw data in any format (structured, semi-structured, unstructured) at low cost, processing it on read (S3 + Athena, Databricks).

In our project, we used S3 as a data lake for raw event data — every user click, search query, page view, and dealer action was streamed as JSON events into S3 partitioned by date. That was terabytes of raw data monthly. For business analytics, our data engineering team ran ETL jobs that transformed this raw data into structured tables in Redshift (our data warehouse) — aggregated metrics like "average days to sell by make/model," "search-to-lead conversion rates," and "dealer performance rankings." Business analysts queried Redshift directly with SQL. Data scientists queried the raw S3 data lake with Athena for ad-hoc exploration that didn't fit the warehouse schema.

The trade-off: a data warehouse gives you fast, predictable query performance but requires upfront schema design — if you didn't model a dimension, you can't query it. A data lake keeps everything raw (schema-on-read) but queries are slower and require more technical skill. The lesson: use both — lake for raw storage and exploration, warehouse for repeatable business reporting. Don't try to make one serve both purposes.
