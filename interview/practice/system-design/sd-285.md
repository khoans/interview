**285.** A penetration test reveals your application is vulnerable to SQL injection. Walk through the remediation.

Immediate action: assess the severity -- can the injection read/write/delete data? Which endpoint is affected? On our marketplace, the remediation steps:

(1) **Hotfix** -- replace any raw SQL concatenation with parameterized queries. In Spring Data JPA, this means using @Query with named parameters (:param) or JpaRepository methods, never String concatenation. For our Solr search queries, we used SolrJ parameterized query builder instead of raw query strings. (2) **Deploy immediately** via GitHub Actions. (3) **Assess damage** -- check Kibana logs for suspicious query patterns on the affected endpoint, look for UNION SELECT, OR 1=1, or time-based blind injection patterns. (4) **Comprehensive scan** -- run SonarCloud across all repositories to find similar patterns. Search codebase for any raw SQL string concatenation. (5) **Prevention** -- add SQL injection detection rules to SonarCloud quality gate (block PRs with injection vulnerabilities), add integration tests with SQL injection payloads, use an ORM (JPA/Hibernate) as the default data access layer.

The lesson: SQL injection is a solved problem with parameterized queries -- it should never appear in modern code. When it does, it usually indicates a shortcut someone took for a "quick query" that bypassed the ORM layer.
