**81.** What is consistent hashing? How does it minimize data redistribution?

========== THIS SECTION IS THE ANSWER ==========
Consistent hashing maps both data keys and nodes onto a virtual ring, so adding or removing a node only redistributes keys adjacent to that node — on average K/N keys move instead of rehashing everything.

**Real example:** On our automotive marketplace, we used ElastiCache Redis cluster for caching vehicle search results and sessions across 30M monthly users. Redis Cluster distributes 16,384 hash slots across nodes. When we scaled from 3 to 6 Redis nodes during peak traffic (new model release events), only about half the slots migrated, keeping search latency under 200ms during the scaling window.

**Trade-off:** Without virtual nodes, you get uneven key distribution and hotspots. We configured multiple hash slots per physical node to achieve roughly even distribution. The cost is added complexity in slot migration management — during one scaling event, a stuck slot migration caused a 2-minute cache miss spike before auto-recovery kicked in. Always set appropriate timeouts and monitor slot migration progress.
