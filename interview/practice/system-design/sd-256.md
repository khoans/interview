**256.** What is a socket? How does socket programming work at a high level?

A socket is an endpoint for bidirectional communication between two processes over a network, identified by an IP address and port number. Socket programming involves: creating a socket, binding it to an address/port (server), listening for connections (server), connecting to a server (client), then sending/receiving data through read/write operations.

On our marketplace, every HTTP request from users to our Spring Boot services used sockets under the hood. Tomcat (our embedded server) created a server socket on port 8080, accepted incoming connections from the ALB, and spawned threads to handle each request. For our Redis connections, Spring Data Redis maintained a pool of persistent socket connections to ElastiCache (avoiding the overhead of creating new TCP connections per request). Our SQS long-polling also used sockets with extended timeout to efficiently wait for new messages.

The trade-off: persistent socket connections (connection pooling) reduce latency but consume memory and file descriptors. We tuned our HikariCP connection pool for PostgreSQL to max 20 connections per ECS instance -- too many and we would exhaust database connections, too few and requests would queue waiting for a connection.
