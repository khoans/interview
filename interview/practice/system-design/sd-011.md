**11.** What is database sharding? When and why would you shard a database?

========== THIS SECTION IS THE ANSWER ==========

Sharding is splitting a single database into multiple smaller databases (shards), each holding a subset of the data. You shard when a single database can no longer handle the write volume or data size — vertical scaling has hit its limit.

In our automotive marketplace, we didn't shard PostgreSQL because our write volume (~5K writes/sec for listing updates) was handled by a single RDS instance. However, we used DynamoDB for user activity tracking (search history, saved vehicles), which is inherently sharded by partition key. We chose userId as the partition key so all activity for one user lived on the same shard, making reads fast. At 30M monthly users generating clickstream data, this distributed the load across hundreds of partitions automatically.

The trade-off: sharding eliminates single-database bottlenecks but makes cross-shard queries painful — joins across shards are essentially application-level joins. It also complicates rebalancing when shards grow unevenly (hot shards). The lesson: don't shard prematurely. We kept PostgreSQL unsharded for transactional data and only used a natively-sharded store (DynamoDB) for high-volume, key-based access patterns.
