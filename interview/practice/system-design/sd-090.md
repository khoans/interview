**90.** What is the thundering herd problem? How do you prevent it?

========== THIS SECTION IS THE ANSWER ==========
The thundering herd problem occurs when a cache entry expires and hundreds or thousands of concurrent requests simultaneously hit the backend to regenerate it, potentially overwhelming the database or downstream service.

**Real example:** On our automotive marketplace, our homepage featured "most popular vehicles" cached in ElastiCache Redis with a 5-minute TTL. When the cache expired during peak hours, 200+ concurrent requests all tried to query Solr and rebuild the cache simultaneously. Solr CPU spiked to 100% and response times degraded for all users. We fixed this with a cache stampede lock: the first request to find an expired key acquired a distributed lock (Redis SETNX with TTL), rebuilt the cache, and released it. All other requests either waited briefly or received a slightly stale value using a "stale-while-revalidate" pattern.

**Trade-off:** The lock approach can add latency for the waiting requests and introduces a single point of failure (what if the lock holder crashes?). We set a short lock TTL (2 seconds) as a safety net. An alternative approach — randomizing TTLs — is simpler but less effective because it only spreads expirations, it does not eliminate the problem entirely.
