**79.** What is a materialized view? How does it improve read performance?

========== THIS SECTION IS THE ANSWER ==========

A materialized view is a precomputed query result stored as a physical table. Unlike a regular view (which re-executes the query every time), a materialized view serves the cached result directly — turning an expensive multi-table join or aggregation into a simple table scan. It must be refreshed periodically to stay current.

In our project, we used materialized views in PostgreSQL for the dealer dashboard analytics. The "listing performance" page showed each dealer their listings' total views, clicks, and lead conversion rates — computed by joining listings, page_views, clicks, and leads tables and aggregating. This 4-table join with GROUP BY across millions of rows took ~8 seconds live. We created a materialized view that precomputed this aggregation and refreshed it every 15 minutes via a pg_cron job (REFRESH MATERIALIZED VIEW CONCURRENTLY). Dashboard queries dropped from 8 seconds to 50ms. The CONCURRENTLY option allowed reads during refresh — dealers didn't see downtime.

The trade-off: materialized views consume storage (it's a full copy of the query result), and data is stale between refreshes. REFRESH MATERIALIZED VIEW is expensive — it re-executes the entire query, which can spike database load. We scheduled refreshes during low-traffic periods. Also, you can't parameterize a materialized view (it's a fixed query), so we created separate views per common access pattern. The lesson: materialized views are a simple, powerful optimization for read-heavy dashboards with complex aggregations. If 15-minute staleness is acceptable, they're often the simplest solution before reaching for Redis caching or CQRS.
