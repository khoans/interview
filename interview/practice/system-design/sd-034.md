**34.** What are WebSockets? When would you use them instead of HTTP polling?

========== THIS SECTION IS THE ANSWER ==========

WebSockets provide full-duplex, persistent connections between client and server over a single TCP connection. Unlike HTTP (request-response), either side can send messages at any time. Use WebSockets when you need real-time, bidirectional communication — chat, live notifications, collaborative editing, live dashboards.

In our project, we used WebSockets for the dealer dashboard's live notification feed. When a buyer submitted a lead (inquiry about a vehicle), the dealer needed to see it within seconds — not minutes. With HTTP polling every 5 seconds from 50K active dealer sessions, that would be 10K requests/second to the server — most returning "no new leads." Instead, we used WebSocket connections via Spring WebSocket with STOMP protocol. When a lead was created, the backend pushed a notification to the dealer's WebSocket session instantly. This reduced server load by ~90% compared to polling and gave dealers sub-second notification latency.

The trade-off: WebSockets require maintaining persistent connections (memory per connection on the server), handling reconnections (mobile networks drop connections frequently), and load balancer configuration (sticky sessions or Redis-backed session sharing). HTTP polling is simpler to implement, stateless, and works through any proxy. The lesson: use WebSockets only when you truly need real-time push. For "near real-time" (seconds of delay is fine), long polling or SSE is simpler.
