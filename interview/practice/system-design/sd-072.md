**72.** What is MapReduce? How does it process large datasets in parallel?

========== THIS SECTION IS THE ANSWER ==========

MapReduce is a programming model for processing large datasets in parallel across a cluster. Two phases: Map — each node processes a chunk of data and emits key-value pairs. Reduce — values with the same key are grouped and aggregated. The framework handles distribution, parallelism, and fault tolerance automatically. Example: counting word frequency — map emits (word, 1) for each word, reduce sums the counts per word.

In our project, we used the MapReduce concept (though not Hadoop directly) for our monthly dealer analytics report. The "map" phase: Lambda functions processed S3 event files in parallel — each Lambda handled one day's worth of clickstream data (~2M events), extracting (listingId, {impressions, clicks, leads}) tuples. The "reduce" phase: results were aggregated by listingId across all days, computing totals and conversion rates, then written to Redshift. Processing 30 days x 2M events each completed in ~15 minutes with 30 parallel Lambda invocations, versus the 8+ hours it would take sequentially on a single machine.

The trade-off: MapReduce is powerful for embarrassingly parallel tasks but inefficient for iterative algorithms (machine learning, graph processing) because it writes intermediate results to disk between iterations. It's also batch-only — no real-time processing. Modern alternatives (Spark, Flink) keep data in memory between stages, making them 10-100x faster for iterative workloads. The lesson: understand MapReduce as a foundational concept, but in practice, use Spark/Flink for complex data processing. We used Lambda + S3 as a serverless "MapReduce" because our data volume didn't justify a Spark cluster.
