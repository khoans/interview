**98.** Design a logging and monitoring system. How do you handle millions of log entries per second?

========== THIS SECTION IS THE ANSWER ==========
Key components: log agents on each service shipping logs to a message buffer (Kafka), a log processing pipeline that parses/filters/enriches, a time-series store for metrics, a search index (Elasticsearch) for log queries, and dashboards/alerting on top. The message buffer decouples producers from consumers, absorbing spikes.

**Real example:** On our automotive marketplace, all 15+ Spring Boot microservices on ECS shipped structured JSON logs via Fluentd sidecar containers to Kibana (ELK stack). Application metrics went to New Relic via the Java agent. We ingested roughly 2-3 GB of logs per hour across services. Key design choices: log levels were configurable per service via environment variables (no redeployment needed), correlation IDs propagated via Spring Sleuth across service calls for distributed tracing, and we set a 30-day retention policy in Elasticsearch with index lifecycle management to control storage costs.

**Trade-off:** Centralized logging adds infrastructure cost and can become a bottleneck. When our Elasticsearch cluster was undersized, log ingestion backed up and we lost visibility during an incident — exactly when we needed logs most. We learned to size the logging infrastructure for 3x normal throughput and to always have a fallback (CloudWatch Logs as secondary). Sampling high-volume debug logs (1 in 100) also helped reduce costs without losing signal.
