**334.** Short TTL vs long TTL: How do you balance freshness vs performance? What's the cost of stale data?

========== THIS SECTION IS THE ANSWER ==========

Set TTL based on the business cost of stale data. If stale data causes user confusion or financial errors, use short TTLs (seconds to minutes). If stale data is invisible to users, use long TTLs (hours to days).

On our marketplace, we used different TTLs for different data types. Search results in Redis had a 5-minute TTL — a user seeing a listing that was sold 5 minutes ago was mildly annoying but not harmful (the detail page showed the real-time status). Vehicle images on CloudFront CDN had a 24-hour TTL because images rarely changed and serving them from cache saved massive bandwidth. But dealer inventory counts had a 30-second TTL because showing "50 cars available" when it is actually 45 would mislead shoppers actively comparing dealers.

The trade-off: short TTLs keep data fresh but increase backend load (more cache misses, more database queries) and reduce the performance benefit of caching. Long TTLs maximize cache hit rate but risk showing stale data. There is no universal answer — map each data type to its staleness tolerance and set TTLs accordingly. Also consider cache invalidation on write as a complement to TTL.
