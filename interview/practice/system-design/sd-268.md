**268.** What is token revocation? How do you invalidate JWTs (blacklist, short expiry, reference tokens)?

Token revocation is the ability to invalidate a previously issued token before it expires. JWTs are stateless by design (no server-side lookup), making revocation challenging. Three approaches: **Blacklist** (store revoked token IDs in Redis, check on every request -- adds latency), **Short expiry** (15-minute tokens limit the damage window without revocation), **Reference tokens** (token is just an opaque ID, server looks up the actual claims -- fully revocable but stateful).

On our marketplace, we combined short expiry with a Redis blacklist for critical situations. Access tokens expired in 15 minutes (acceptable damage window). For immediate revocation (compromised account, user logout), we added the token JTI (JWT ID) to an ElastiCache Redis blacklist checked by Spring Security on each request. The blacklist entry TTL matched the token remaining lifetime, so entries auto-cleaned. For refresh tokens, we used opaque tokens stored in PostgreSQL -- revocation was simply deleting the database row.

The trade-off: a blacklist check on every request adds 1-2ms latency (Redis round trip). Pure short-expiry approach has zero overhead but means a stolen token works for up to 15 minutes. For a marketplace handling financial data, the Redis check was worth the latency cost.
