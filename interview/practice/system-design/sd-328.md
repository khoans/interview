**328.** When should you use a search engine (Elasticsearch/Solr) vs database full-text search?

========== THIS SECTION IS THE ANSWER ==========

Use a dedicated search engine when you need faceted search, fuzzy matching, relevance scoring, synonyms, autocomplete, or when searching across millions of documents with complex queries. Use database full-text search (PostgreSQL tsvector) for simple keyword search on moderate datasets.

On our marketplace, we chose Apache Solr for vehicle search — 30M users searching across millions of listings with filters on make, model, year, price range, mileage, location radius, dealer rating, and sorting by relevance or price. Solr gave us faceted navigation (show count per make/model), fuzzy matching (typo tolerance), boosting (promoted listings), and sub-50ms query times. PostgreSQL full-text search could handle keyword matching but would have choked on our faceted filtering across 15+ dimensions with relevance scoring at that scale.

The trade-off: a dedicated search engine adds operational complexity — you maintain a separate data store, build an indexing pipeline (we synced from PostgreSQL to Solr via SQS events), and handle consistency lag. PostgreSQL full-text search is zero operational overhead since it is built in, but it lacks faceting, sophisticated relevance tuning, and degrades at scale. If your search is a simple LIKE query on a small table, do not bring in Elasticsearch.
