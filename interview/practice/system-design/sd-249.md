**249.** What is the difference between stateful and stateless protocols?

Stateful protocols maintain connection context between requests (e.g., TCP, FTP) -- the server remembers previous interactions. Stateless protocols treat each request independently (e.g., HTTP) -- no server-side memory of previous requests.

On our marketplace, our Spring Boot REST APIs were designed to be stateless -- every request carried its own authentication (OAuth2 Bearer token in the Authorization header) and context. This was critical for horizontal scaling: with multiple ECS instances behind an ALB, any instance could handle any request without session affinity. When we needed state (like user sessions), we externalized it to ElastiCache Redis rather than keeping it in-memory on the application server. Our SQS message processing was also stateless -- each message contained all the information needed to process it.

The trade-off: stateless design requires passing more data per request (larger tokens, repeated context), which increases bandwidth. But the scalability benefit far outweighs this cost. We learned this the hard way when an early design used sticky sessions on the ALB, causing uneven load distribution when one instance had all the heavy dealer sessions.
