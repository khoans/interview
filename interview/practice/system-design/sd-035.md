**35.** What is a message queue (e.g., SQS, RabbitMQ)? How does it decouple services?

========== THIS SECTION IS THE ANSWER ==========

A message queue is a buffer between producers and consumers — the producer sends a message and moves on without waiting for the consumer to process it. This decouples services in time (producer and consumer don't need to be available simultaneously), speed (they can process at different rates), and knowledge (producer doesn't need to know who consumes the message).

In our project, we used AWS SQS extensively. When a dealer uploaded a new listing, the listing service published a message to an SQS queue. Three independent consumers processed it: (1) the image processing service resized and optimized photos, (2) the search indexing service updated Apache Solr, and (3) the analytics service recorded the event. Without SQS, the listing service would have needed to call all three synchronously — if the image service was slow, the dealer's upload would hang. With SQS, the dealer got an instant response, and the downstream processing happened asynchronously. If a consumer was temporarily down, messages waited in the queue (SQS retains messages for up to 14 days).

The trade-off: message queues add eventual consistency (the search index lags behind the database), operational complexity (dead letter queues, retry logic, monitoring queue depth), and debugging difficulty (tracing a request across async boundaries). The lesson: use queues for anything that doesn't need an immediate response. If the user is waiting for the result, keep it synchronous.
