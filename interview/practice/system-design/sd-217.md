**217.** What is the shared responsibility model in AWS? What is your responsibility vs AWS's?

========== THIS SECTION IS THE ANSWER ==========

The shared responsibility model divides security between AWS (security OF the cloud) and the customer (security IN the cloud). AWS is responsible for physical infrastructure, hypervisor, managed service internals, network infrastructure, and global infrastructure availability. The customer is responsible for data encryption, IAM configuration, network controls (security groups, NACLs), OS patching (for EC2), application code security, and data classification.

In our marketplace, the boundary was clear: AWS managed the physical data centers, the ECS control plane, RDS engine patching, and S3 durability. We were responsible for: (1) configuring IAM roles and policies correctly (least privilege), (2) encrypting data at rest (KMS) and in transit (TLS), (3) securing our application code (input validation, SQL injection prevention), (4) network configuration (VPC, security groups, private subnets for databases), (5) patching our Docker base images and application dependencies, (6) managing secrets (Secrets Manager rotation), (7) monitoring and incident response (GuardDuty, CloudTrail alerts).

The lesson: the model shifts based on the service. With EC2, you patch the OS. With RDS, AWS patches the database engine but you manage access controls. With Lambda, AWS manages everything except your code and IAM. We chose managed services (ECS Fargate, RDS, ElastiCache) specifically to shift more responsibility to AWS, reducing our operational security burden.
