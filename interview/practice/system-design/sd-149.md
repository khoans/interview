**149.** What is sensitive data exposure? How do you prevent it in API responses and logs?

========== THIS SECTION IS THE ANSWER ==========
Sensitive data exposure occurs when applications inadvertently reveal PII, credentials, tokens, or internal system details through API responses, logs, error messages, or insufficient encryption. Common examples: returning full credit card numbers in API responses, logging passwords in plaintext, or including stack traces in production error responses.

**Real example:** On our automotive marketplace, we implemented multiple layers to prevent data exposure. API responses: DTOs explicitly controlled which fields were serialized — we used @JsonIgnore on sensitive fields (password hashes, SSNs) and response projection classes that only included fields the client needed. Logs: we built a custom Logback pattern layout that redacted patterns matching SSN (XXX-XX-XXXX), credit card numbers, and email addresses before writing to Kibana. Error responses: Spring Boot's server.error.include-stacktrace=never in production ensured internal exceptions never reached clients — they got a generic error ID that mapped to the full trace in our internal logs. We also enforced KMS encryption for all PII at rest in DynamoDB.

**Trade-off:** Aggressive redaction in logs can make debugging harder — a redacted email address in logs means you cannot trace a specific user's issue without additional steps. We balanced this by logging user IDs (non-sensitive) instead of email/names and providing an internal tool to look up user details by ID when needed. The key principle: never log anything you would not want to see in a breach disclosure.
