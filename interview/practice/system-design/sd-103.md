**103.** What is a hash collision? Why does it matter for security?

========== THIS SECTION IS THE ANSWER ==========
A hash collision occurs when two different inputs produce the same hash output. For security, this is dangerous because an attacker could substitute a malicious file/message that has the same hash as a legitimate one, bypassing integrity checks.

**Real example:** On our automotive marketplace, we used SHA-256 checksums to verify that deployment artifacts from GitHub Actions were not tampered with before deploying to ECS. If we had used MD5 instead (where practical collision attacks exist since 2004), an attacker could theoretically craft a malicious JAR file with the same MD5 hash as our legitimate artifact, bypassing the integrity check. The 2017 SHAttered attack demonstrated a practical SHA-1 collision, which is why our SonarCloud rules flagged SHA-1 usage as a vulnerability.

**Trade-off:** Collision-resistant algorithms (SHA-256, SHA-3) have longer digests and are computationally heavier. For security-sensitive operations like code signing and certificate validation, this overhead is non-negotiable. For non-security scenarios like cache keys or data partitioning, collisions are handled programmatically (chaining, open addressing), and faster hash functions are acceptable.
