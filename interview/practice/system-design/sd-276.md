**276.** What is the "none" algorithm attack in JWT? How do you prevent it?

The "none" algorithm attack exploits JWT libraries that accept alg:"none" in the header, which means no signature verification. An attacker modifies the JWT payload (changing roles to admin, for example), sets the algorithm to "none," removes the signature, and the server accepts it as valid because "none" means no verification needed.

On our marketplace, we prevented this in Spring Security OAuth2 by explicitly configuring accepted algorithms -- only RS256 was allowed, and any token with alg:"none" or HS256 was rejected immediately. The Spring Security JwtDecoder was configured with a specific JWK Set URI and algorithm whitelist. We also validated the issuer claim to ensure tokens came from our auth server, not a forged source.

The lesson: this is a well-known vulnerability (CVE-2015-9235) that should not exist in production, yet it still appears in custom JWT implementations. Never trust the algorithm specified in the token header -- always enforce the expected algorithm server-side. Use well-maintained libraries (Spring Security, nimbus-jose-jwt) rather than hand-rolling JWT validation.
