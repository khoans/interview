**335.** CDN caching vs application caching vs database caching: At which layer should you cache for your use case?

========== THIS SECTION IS THE ANSWER ==========

Cache at the layer closest to the consumer for maximum impact. CDN for static assets and public content, application cache (Redis) for user-specific or dynamic data, database cache (query cache, materialized views) for expensive aggregations.

On our marketplace, we cached at all three layers. CloudFront CDN cached vehicle images, JavaScript bundles, and public listing pages — this served 70%+ of traffic without hitting our servers at all. ElastiCache Redis cached API responses like listing details and search results at the application layer — reducing PostgreSQL load by 80%. At the database layer, we used PostgreSQL materialized views for nightly analytics aggregations (average prices by make/model/year) that would take minutes to compute on the fly.

The trade-off: CDN caching is the most impactful (eliminates server load entirely) but only works for public, non-personalized content and is harder to invalidate. Application caching (Redis) handles personalized data but adds infrastructure. Database caching (query cache) is the simplest but least effective — it only helps repeated identical queries. Layer your caching: CDN first, Redis second, database optimization last.
