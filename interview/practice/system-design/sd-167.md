**167.** What is a ConfigMap vs Secret in Kubernetes? How do you inject configuration into pods?

========== THIS SECTION IS THE ANSWER ==========

ConfigMap stores non-sensitive configuration data as key-value pairs (application properties, feature flags, URLs). Secrets store sensitive data (passwords, API keys, TLS certs) with base64 encoding (not encryption by default). Both can be injected into pods as environment variables or mounted as files in the container filesystem.

In our marketplace on ECS, we used the equivalent pattern: SSM Parameter Store for configuration and Secrets Manager for secrets, both injected as environment variables in task definitions. Mapped to Kubernetes: our Spring Boot application.yml overrides (server.port, solr.url, feature flags) would be a ConfigMap mounted as a file at /config/application.yml. Database passwords and API keys would be Secrets injected as environment variables. We preferred env vars for simple values and file mounts for complex configs. Spring Boot externalized configuration picks up both automatically.

The trade-off: Secrets are only base64-encoded, not encrypted at rest by default -- anyone with kubectl access can decode them. You must enable etcd encryption at rest and use tools like Sealed Secrets or External Secrets Operator to manage secrets securely. We avoided this concern on AWS by using Secrets Manager with KMS encryption and IAM-based access control.
