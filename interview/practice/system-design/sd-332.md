**332.** Local cache (in-memory) vs distributed cache (Redis): When do you use each? What about consistency between instances?

========== THIS SECTION IS THE ANSWER ==========

Use local cache (Caffeine, Guava) for small, frequently accessed, rarely changing data like configuration or reference data. Use distributed cache (Redis) for large datasets, shared state, or when consistency across multiple application instances matters.

On our marketplace running on ECS with 8-12 instances, we used both. Local Caffeine cache held static reference data — car makes/models list, state abbreviations — that changed maybe once a month and was tiny (a few MB). This avoided a network round-trip to Redis for every request. For listing data and search results, we used ElastiCache Redis because all instances needed the same cached view — if instance A cached a listing update, instance B should see it too.

The trade-off: local cache is the fastest (no network hop, nanosecond access) but each instance has its own copy, leading to inconsistency — instance A might have fresh data while instance B has stale data until its TTL expires. For frequently changing data across multiple instances, this inconsistency is unacceptable. Redis adds 1-2ms network latency but gives you a single consistent view and saves memory by not duplicating across all instances.
