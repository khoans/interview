**12.** What is database replication? Explain master-slave vs master-master replication.

========== THIS SECTION IS THE ANSWER ==========

Database replication copies data from one database server to others for redundancy and read scaling. Master-slave (primary-replica): one master handles all writes, replicas handle reads. Master-master: multiple nodes accept writes, syncing changes bidirectionally.

In our project, we used AWS RDS PostgreSQL with master-slave replication — one primary in us-east-1, a read replica in the same AZ for read scaling, and a cross-region read replica in us-west-2 for disaster recovery. Our read-to-write ratio was about 100:1 (users browsing listings vs dealers updating inventory), so offloading reads to replicas reduced primary load by ~80%. The Spring Boot services used a routing DataSource — writes went to primary, reads went to replicas.

The trade-off: master-slave is simpler but the master is a write bottleneck and single point of failure (failover takes 1-2 minutes with RDS). Master-master eliminates the write bottleneck but introduces conflict resolution headaches — two nodes updating the same row simultaneously. We stuck with master-slave because our write volume was manageable, and the simplicity was worth the brief failover window.
