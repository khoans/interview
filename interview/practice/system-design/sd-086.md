**86.** What is the fan-out pattern? Explain fan-out-on-write vs fan-out-on-read (e.g., Twitter timeline).

========== THIS SECTION IS THE ANSWER ==========
Fan-out is distributing a single event to multiple recipients. Fan-out-on-write (push model) pre-computes and writes to each recipient's feed at publish time. Fan-out-on-read (pull model) assembles the feed at read time by querying each followed source.

**Real example:** On our automotive marketplace, when a dealer published a new listing, we used fan-out-on-write via SNS/SQS: the listing event fanned out to the search indexing queue (Solr update), the notification queue (email alerts to saved-search users), the analytics queue, and the CDN invalidation queue. Each consumer processed independently. This was a natural fit because a listing publish was infrequent relative to reads, and we needed guaranteed delivery to each downstream system.

**Trade-off:** Fan-out-on-write is expensive for users with millions of followers (the "celebrity problem" in Twitter's case) — one post triggers millions of writes. Fan-out-on-read avoids that write amplification but makes reads slower and more complex. Twitter uses a hybrid: fan-out-on-write for normal users, fan-out-on-read for celebrities. For our marketplace, fan-out-on-write via SQS was fine since dealers had at most thousands of subscribers, not millions.
