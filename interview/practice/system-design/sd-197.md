**197.** What is DNS in Kubernetes? How does CoreDNS resolve service names?

========== THIS SECTION IS THE ANSWER ==========

CoreDNS is the default DNS server in Kubernetes, running as a Deployment in kube-system namespace. It watches the Kubernetes API for Services and creates DNS records automatically. When a pod calls search-service, CoreDNS resolves it to the ClusterIP. The full DNS format is <service>.<namespace>.svc.cluster.local -- pods in the same namespace can use the short name.

In our marketplace, we used AWS Cloud Map for service discovery in ECS, which served the same purpose. Mapped to Kubernetes: when our listing-service Spring Boot app called http://search-service:8080/api/search, CoreDNS resolved search-service to its ClusterIP (e.g., 10.96.0.15). kube-proxy then load-balanced the request across healthy pod IPs behind that service. For external DNS, CoreDNS forwards to upstream resolvers (VPC DNS in AWS). We configured our Spring Boot applications with service URLs as environment variables, making them easy to override between environments.

The trade-off: DNS-based discovery has TTL caching issues -- if a service IP changes, clients may cache the old IP. Kubernetes mitigates this with ClusterIP stability, but for headless services (StatefulSets), individual pod IPs change on restart. Set appropriate DNS TTLs and use connection pooling with health checks. Also, CoreDNS is a single point of failure -- run at least 2 replicas with anti-affinity across nodes.
