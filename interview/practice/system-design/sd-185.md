**185.** What is a Security Context in Kubernetes? How do you run containers as non-root?

========== THIS SECTION IS THE ANSWER ==========

A Security Context defines privilege and access control settings for a pod or container -- user/group IDs, filesystem permissions, Linux capabilities, and privilege escalation rules. To run as non-root: set runAsNonRoot: true, runAsUser: 1000, and allowPrivilegeEscalation: false in the pod spec. The container image must also support running as a non-root user.

In our marketplace, all our Spring Boot Docker images ran as a non-root user (appuser, UID 1000) created in the Dockerfile. In Kubernetes terms, the Security Context would enforce this: securityContext.runAsNonRoot=true rejects any container trying to run as root, readOnlyRootFilesystem=true prevents writing to the container filesystem (forces use of volumes for writable data), and dropping all Linux capabilities (capabilities.drop=[ALL]) removes unnecessary kernel privileges. Our application wrote logs and temp files to mounted volumes, not the root filesystem.

The trade-off: running as non-root breaks some legacy applications that assume root access (writing to /etc, binding to port 80). Our Spring Boot services ran on port 8080 (non-privileged port) and had no issues. For images that require root during build but not runtime, use multi-stage Dockerfiles -- build as root, run as non-root. This is a non-negotiable security practice for production.
