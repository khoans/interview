**165.** What is an Ingress in Kubernetes? How does it route external traffic to services?

========== THIS SECTION IS THE ANSWER ==========

An Ingress is a Kubernetes resource that defines HTTP/HTTPS routing rules to map external URLs to internal services. It requires an Ingress Controller (like NGINX, AWS ALB Ingress Controller, or Traefik) that implements these rules. You define host-based routing (api.example.com vs www.example.com) and path-based routing (/api/listings to listing-service, /api/search to search-service) in a single resource.

In our marketplace, we used an AWS ALB as the equivalent of an Ingress Controller. It routed /api/listings/* to the listing service, /api/search/* to the Solr-backed search service, /api/users/* to the user service, and /* to the Next.js frontend. TLS termination happened at the ALB with ACM certificates. In an EKS setup, the AWS Load Balancer Controller would create ALB rules directly from Ingress annotations, giving the same path-based routing we had.

The trade-off: Ingress consolidates routing into one external endpoint (cost-effective, one LB instead of many) but becomes a single point of failure. We mitigated this by using a multi-AZ ALB with health checks. Also, Ingress is limited to Layer 7 HTTP -- for TCP/UDP workloads, you need a LoadBalancer Service directly.
