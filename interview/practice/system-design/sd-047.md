**47.** What is idempotency? Why is it critical in distributed systems?

========== THIS SECTION IS THE ANSWER ==========

An operation is idempotent if performing it multiple times produces the same result as performing it once. In distributed systems, network failures, retries, and duplicate messages are inevitable — idempotency ensures these don't cause duplicate side effects (double charges, duplicate records).

In our project, idempotency was critical for the dealer payment flow. When a dealer purchased a featured listing, the payment API could timeout without the client knowing if the payment succeeded. The client would retry. Without idempotency, the dealer could be charged twice. We solved this with an idempotency key: the client generated a unique UUID per request and sent it in the X-Idempotency-Key header. The server stored this key in Redis with the response. On retry, the server checked Redis — if the key existed, it returned the stored response without processing again. For SQS message consumers, we used a DynamoDB table to track processed message IDs — before processing, we checked if the messageId was already in the table.

The trade-off: implementing idempotency requires extra storage (for tracking processed request IDs) and adds latency (the Redis/DynamoDB lookup on every request). You also need to decide the TTL for idempotency keys — too short and late retries cause duplicates, too long and you waste storage. We used a 24-hour TTL. The lesson: every POST/PUT endpoint that mutates data should be idempotent. Design it from the start — retrofitting idempotency onto existing APIs is painful.
