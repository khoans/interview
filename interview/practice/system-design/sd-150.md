**150.** What is broken access control? Explain IDOR (Insecure Direct Object Reference).

========== THIS SECTION IS THE ANSWER ==========
Broken access control means users can act outside their intended permissions — accessing other users' data, modifying resources they do not own, or escalating privileges. IDOR is a specific type where the application exposes internal object references (IDs) in URLs or parameters and does not verify the requesting user owns that object. For example, changing /api/orders/123 to /api/orders/124 to view another user's order.

**Real example:** On our automotive marketplace, IDOR was a real threat — dealers managed listings via /api/listings/{listingId}. A malicious dealer could try to modify another dealer's listing by guessing the ID. We prevented this with ownership validation in Spring Security: a custom @PreAuthorize expression checked that the authenticated dealer's ID matched the listing's dealer_id before any mutation. For read operations, we added a global JPA filter: `@Filter(name = "dealerFilter", condition = "dealer_id = :currentDealerId")` so queries automatically scoped to the authenticated dealer. We also used UUIDs instead of sequential integers for public-facing IDs, making ID guessing impractical.

**Trade-off:** Authorization checks on every request add latency (database lookup to verify ownership). We cached authorization contexts in Redis (user -> owned resource IDs) with short TTL. UUIDs prevent enumeration but are less human-readable for debugging and support. The most important lesson: never rely solely on the frontend to hide resources — always enforce authorization server-side, because API endpoints can be called directly.
