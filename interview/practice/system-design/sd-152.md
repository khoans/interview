**152.** What is input validation vs input sanitization? Where should each be applied?

========== THIS SECTION IS THE ANSWER ==========

Input validation checks whether data conforms to expected rules (type, length, format, range) and rejects bad input. Input sanitization transforms data to remove or escape dangerous characters. Validation should be applied at the API entry point (controller layer), and sanitization should happen right before output or storage -- especially before rendering HTML or constructing queries.

In our marketplace, we validated all listing submissions at the Spring Boot controller using Bean Validation annotations -- @Size(max=500) for descriptions, @Pattern for VIN numbers (17 alphanumeric characters), @Min(0) for price. For search queries going into Apache Solr, we sanitized by escaping Solr special characters to prevent query injection. User-submitted dealer reviews were sanitized with an HTML whitelist (OWASP Java HTML Sanitizer) before storing to PostgreSQL, stripping any script tags.

The lesson: never rely on frontend validation alone -- we caught requests bypassing our Next.js forms hitting the API directly. Server-side validation is non-negotiable; sanitization is defense-in-depth for output contexts.
