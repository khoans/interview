**10.** What is the difference between monolithic and microservices architecture?

========== THIS SECTION IS THE ANSWER ==========

A monolith is a single deployable unit — all features share the same codebase, same database, same deployment. Microservices split the system into independent services, each owning its own data, deployed separately, communicating over the network (REST, gRPC, messages).

In our project, the system started as a Spring Boot monolith. As the team grew to 30+ developers, deployments became painful — one team's change could break another team's feature, and we couldn't deploy the search module without also deploying inventory. We extracted the search service first (Strangler Fig pattern), then inventory, then pricing — each became an independent microservice with its own database and deployment pipeline.

The trade-off: monolith is simpler — one codebase, one deployment, no network calls between modules, easy debugging. Microservices give you independent deployment, team autonomy, and technology flexibility — but add complexity (distributed tracing, eventual consistency, network failures, operational overhead). The rule: start with a monolith, split when team size or deployment frequency makes it painful. Don't start with microservices for a 3-person team — that's over-engineering.
