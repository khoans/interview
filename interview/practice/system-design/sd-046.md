**46.** What is the bulkhead pattern? How does it isolate failures?

========== THIS SECTION IS THE ANSWER ==========

The bulkhead pattern isolates components so a failure in one doesn't bring down others — like watertight compartments in a ship. In practice, this means using separate thread pools, connection pools, or even separate service instances for different workloads so that one overloaded path doesn't exhaust shared resources.

In our project, we applied bulkheads at the thread pool level using Resilience4j. The listing service made calls to both the pricing service (critical — user sees the price) and the analytics service (non-critical — tracking impressions). Initially, both used the same HTTP connection pool (50 connections). When the analytics service had a slowdown, its calls consumed all 50 connections waiting for timeouts, starving the pricing calls — users couldn't see prices even though the pricing service was healthy. We fixed this by creating separate bulkheads: 40 connections for pricing (critical path), 10 for analytics (best effort). When analytics slowed down again, only its 10 connections were consumed. Pricing continued unaffected.

The trade-off: bulkheads reduce resource utilization efficiency — you're pre-allocating resources per path that might not always be fully used. Over-isolating can also waste capacity (20 idle threads reserved for a rarely-used service). The lesson: isolate critical paths from non-critical ones. Don't bulkhead everything — identify which downstream calls, if they fail, would cause user-visible impact, and protect those with dedicated resource pools.
