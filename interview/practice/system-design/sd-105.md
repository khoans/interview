**105.** What is a salt in password hashing? Why is it necessary?

========== THIS SECTION IS THE ANSWER ==========
A salt is a random value generated per password and prepended/appended before hashing. It ensures that even if two users have the same password, their hashes are different. Without salting, identical passwords produce identical hashes, enabling pre-computed dictionary attacks.

**Real example:** On our automotive marketplace, Spring Security's BCryptPasswordEncoder automatically generates a unique 16-byte salt per password and embeds it in the hash output string (format: $2a$12$<22-char-salt><31-char-hash>). When verifying a login, bcrypt extracts the salt from the stored hash and re-hashes the input with the same salt for comparison. This meant even our most common password ("Password123" — used by multiple dealer accounts, unfortunately) produced completely different hashes in the database, preventing attackers from identifying accounts with the same password.

**Trade-off:** Salts must be stored alongside the hash (bcrypt does this automatically in the hash string). They do not need to be secret — their purpose is uniqueness, not secrecy. The main risk is using a static/shared salt across all users, which defeats the purpose entirely. Some frameworks require manual salt generation, increasing the chance of developer error — bcrypt's built-in salt management eliminates this.
