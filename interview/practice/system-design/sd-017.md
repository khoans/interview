**17.** What is database partitioning? Explain horizontal vs vertical partitioning.

========== THIS SECTION IS THE ANSWER ==========

Partitioning splits a table into smaller pieces for performance. Horizontal partitioning splits rows — e.g., data from January in partition 1, February in partition 2. Vertical partitioning splits columns — frequently accessed columns in one table, rarely accessed in another.

In our project, we used both approaches in PostgreSQL. We horizontally partitioned the vehicle_analytics table by month using PostgreSQL's native range partitioning — each month's clickstream data (page views, search impressions) lived in its own partition. This made queries like "show me last month's performance" fast because PostgreSQL only scanned one partition instead of the entire 2-year history. We also used vertical partitioning on the listings table — the core listing data (make, model, year, price) was in the main table, while the large description text and dealer notes were in a separate table joined by listing_id. This kept the main table's row size small, fitting more rows per page in memory.

The trade-off: horizontal partitioning speeds up range queries and makes data archival easy (just drop old partitions), but cross-partition queries are slower. Vertical partitioning reduces I/O for common queries but requires joins for full records. The lesson: partition based on actual query patterns, not theoretical optimization.
