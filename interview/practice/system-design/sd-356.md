**356.** How did you secure your CI/CD pipeline? What prevents a malicious actor from injecting code through your build process?

========== THIS SECTION IS THE ANSWER ==========

We secured the pipeline at four points: code review, build integrity, secret management, and deployment authorization.

On our marketplace using GitHub Actions: **Code review**: Branch protection required at least 2 PR approvals before merge to main. No direct pushes to main allowed. SonarCloud quality gate had to pass (no critical vulnerabilities, no security hotspots) or the PR was blocked. **Build integrity**: GitHub Actions workflows were version-pinned (no `@latest` tags on actions) to prevent supply chain attacks. We used only verified marketplace actions or our own custom actions. Dependency scanning with Dependabot flagged vulnerable transitive dependencies. **Secret management**: CI secrets (AWS credentials, Docker registry tokens) stored in GitHub encrypted secrets and injected as environment variables at runtime — never in code or logs. We used OIDC federation with AWS to avoid long-lived access keys entirely. **Deployment authorization**: Production deployments required manual approval in the GitHub Actions workflow — a senior engineer had to click "approve" before the deploy step executed.

Lesson learned: the weakest link was often developer laptops — if a developer's GitHub account was compromised, they could approve their own malicious PR. Enforcing MFA on all GitHub accounts and requiring 2 reviewers (not the author) closed this gap.
