**210.** What is AWS KMS (Key Management Service)? Explain CMKs, data keys, and key policies.

========== THIS SECTION IS THE ANSWER ==========

KMS manages encryption keys for your AWS resources. Customer Master Keys (CMKs, now called KMS keys) are the top-level keys that never leave KMS hardware. Data keys are generated by KMS for envelope encryption -- KMS generates a plaintext data key and an encrypted copy; you encrypt your data with the plaintext key, then discard it, storing only the encrypted data key alongside the ciphertext. Key policies control who can use and manage each key.

In our marketplace, KMS was used everywhere: (1) S3 server-side encryption -- vehicle images encrypted with a KMS key (SSE-KMS), each object encrypted with a unique data key. (2) RDS encryption -- PostgreSQL encrypted at rest with a KMS key. (3) Secrets Manager -- all secrets (DB passwords, API keys) encrypted with a dedicated KMS key. (4) EBS volumes encrypted with KMS. Key policies granted: the listing-service role could only Encrypt/Decrypt with the listings-key, the backup role could only GenerateDataKey for backup encryption. No role had kms:* wildcard.

The trade-off: KMS adds ~5-10ms latency per API call (Encrypt/Decrypt) and has a request rate limit (5,500 requests/second per region, soft limit). For high-throughput encryption, use envelope encryption with data key caching (AWS Encryption SDK) to minimize KMS calls. We cached data keys for 5 minutes, reducing KMS calls by 99% for our high-volume listing image encryption.
