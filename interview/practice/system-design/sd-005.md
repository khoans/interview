**5.** What is the CAP theorem? Explain with real-world examples.

========== THIS SECTION IS THE ANSWER ==========

CAP theorem states that a distributed system can only guarantee two of three properties simultaneously: Consistency (every read returns the most recent write), Availability (every request gets a response), and Partition tolerance (the system works even when network splits occur). Since network partitions are inevitable in distributed systems, you're really choosing between CP (consistent but may reject requests) and AP (available but may serve stale data).

Real-world examples: a banking system is CP — when you transfer money, you'd rather get an error than have two accounts show incorrect balances. A social media feed is AP — it's fine if you see a post 5 seconds late, but the feed should always load.

In our project, our search system was AP — if one region had stale inventory data for a few seconds, that was acceptable. But our order/payment processing was CP — we couldn't allow two users to purchase the same vehicle. We used PostgreSQL with synchronous replication for payment data (CP) and DynamoDB with eventual consistency for search metadata (AP). The lesson: don't apply one consistency model to your entire system — choose per use case.
