**41.** What is service discovery? How do services find each other in a microservices architecture?

========== THIS SECTION IS THE ANSWER ==========

Service discovery is how microservices locate each other's network addresses dynamically, since instances scale up/down and IPs change. Client-side discovery: the client queries a service registry (Eureka, Consul) and picks an instance. Server-side discovery: a load balancer (ALB, Kubernetes Service) handles routing — clients just call a fixed DNS name.

In our project, we used server-side discovery via AWS ECS with ALB. Each microservice registered its ECS tasks with a target group on the ALB. When the listing service needed to call the pricing service, it called pricing-service.internal.autotrader.com — Route 53 private hosted zone resolved this to the internal ALB, which routed to healthy pricing service instances. When ECS scaled the pricing service from 4 to 8 instances during peak traffic, the new instances automatically registered with the ALB target group. No code changes, no registry to manage.

The trade-off: server-side discovery (ALB) adds a network hop but is simple — no service registry to operate, no client library needed. Client-side discovery (Eureka) eliminates the extra hop but requires a registry cluster (another thing to operate and keep highly available) and a client library in every service. The lesson: on AWS with ECS/EKS, lean on ALB + Route 53 for service discovery. It's one less component to manage. We only considered Eureka when we had complex routing needs that ALB couldn't handle.
