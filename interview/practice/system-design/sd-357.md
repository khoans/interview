**357.** How did you secure your Docker containers in production? (non-root, image scanning, minimal base images, read-only filesystem)

========== THIS SECTION IS THE ANSWER ==========

We applied four hardening practices for our ECS containers on the automotive marketplace.

**Non-root user**: Every Dockerfile created a dedicated application user and switched to it with `USER appuser`. Our Spring Boot JARs ran as this non-root user — even if an attacker gained code execution, they could not install packages or modify system files. **Image scanning**: ECR scanned every image on push. GitHub Actions failed the build if HIGH or CRITICAL CVEs were found. We also ran weekly scheduled scans to catch newly discovered vulnerabilities in existing images. **Minimal base images**: We used `eclipse-temurin:17-jre-alpine` for production — no compiler, no shell tools, no package manager. This reduced the attack surface from hundreds of installed packages to a handful. Debug images with full JDK were only used in development. **Read-only filesystem**: ECS task definitions set `readonlyRootFilesystem: true` with specific tmpfs mounts for `/tmp` where Spring Boot needed to write temporary files. This prevented any file modification on the container filesystem.

The trade-off: these hardening measures make debugging production issues harder — no shell, no tools, read-only filesystem means you cannot exec into a container and poke around. We accepted this trade-off by investing in good logging (structured JSON to Kibana) and monitoring (New Relic) instead.
