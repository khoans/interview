**368.** How do you implement rate limiting in Nginx? Explain limit_req_zone, limit_req, burst, and nodelay directives.

========== THIS SECTION IS THE ANSWER ==========

Nginx rate limiting uses the leaky bucket algorithm. `limit_req_zone` defines the shared memory zone tracking request rates per key (usually client IP). `limit_req` applies the limit to a location. `burst` allows a temporary spike above the rate. `nodelay` processes burst requests immediately instead of queuing them.

Config: `limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;` then `location /api/ { limit_req zone=api_limit burst=20 nodelay; }`. This allows 10 requests/second per IP, with a burst buffer of 20. Without `nodelay`, excess requests within the burst are queued and processed at the base rate. With `nodelay`, they're processed immediately but still count against the burst budget. Once burst is exhausted, requests get 503'd.

In our Korean ecommerce project, we set rate limiting on the AI rendering endpoint — 5 requests/minute per IP with burst of 2, because each AI rendering call was expensive (5-10 seconds of GPU compute). Without rate limiting, a single user could overload the AI engine by spamming the "visualize" button.

The trade-off: rate limiting by IP is simple but affects users behind shared NATs (corporate offices, mobile carriers). For authenticated APIs, rate limiting by user ID or API key is more accurate. Nginx rate limiting is per-instance — for distributed rate limiting, you'd need Redis.
