**216.** What is AWS Config? How does it detect non-compliant resources?

========== THIS SECTION IS THE ANSWER ==========

AWS Config continuously monitors and records your AWS resource configurations and evaluates them against desired rules. Config Rules define compliance standards (e.g., all S3 buckets must be encrypted, all security groups must not allow 0.0.0.0/0 on port 22). When a resource drifts from the rule, Config marks it non-compliant and can trigger automatic remediation via SSM Automation or Lambda.

In our marketplace, we had ~30 Config Rules: (1) s3-bucket-server-side-encryption-enabled -- flagged any unencrypted bucket. (2) restricted-ssh -- detected security groups allowing SSH from anywhere. (3) rds-storage-encrypted -- verified all database instances were encrypted. (4) iam-user-mfa-enabled -- flagged users without MFA. (5) ec2-instances-in-vpc -- ensured no EC2 instances existed outside our VPC. When a developer created a test security group with 0.0.0.0/0 inbound, Config detected it within 15 minutes, triggered an SNS notification to Slack, and an auto-remediation Lambda removed the offending rule.

The trade-off: Config evaluations are not real-time (periodic or change-triggered with slight delay), so there is a window where non-compliant resources exist. For critical rules, combine with SCPs (preventive) and Config (detective). Also, Config has a cost per rule evaluation and per configuration item recorded -- at scale, this adds up. Prioritize rules that address your highest-risk compliance requirements.
