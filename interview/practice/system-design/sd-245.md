**245.** What is HTTP/2 vs HTTP/3? What are the key improvements in each?

HTTP/2 introduced multiplexing (multiple requests over one TCP connection), header compression (HPACK), and server push -- eliminating HTTP/1.1 head-of-line blocking at the application layer. HTTP/3 replaces TCP with QUIC (UDP-based), solving TCP-level head-of-line blocking, enabling 0-RTT connection establishment, and handling network switching (Wi-Fi to cellular) gracefully.

On our marketplace, CloudFront supported HTTP/2 by default for our Next.js frontend, which significantly improved page load times since vehicle listing pages loaded many assets (images, JS bundles, API calls) simultaneously over a single connection. HTTP/3 support was enabled on CloudFront as well, benefiting mobile users who frequently switched between networks while browsing vehicle listings. Our backend ALB-to-ECS communication still used HTTP/1.1 since internal latency was already sub-millisecond.

The trade-off: HTTP/3 over QUIC can be blocked by corporate firewalls that drop UDP traffic. Browsers gracefully fall back to HTTP/2, but this means some users never benefit from HTTP/3. You must support both protocols.
