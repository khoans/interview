**350.** Feature flags vs branching: When should you use feature flags to control releases? When do they become tech debt?

========== THIS SECTION IS THE ANSWER ==========

Use feature flags when you need to decouple deployment from release — ship code to production but control who sees it. Use branching for short-lived development isolation. Feature flags become tech debt when they are never cleaned up after full rollout.

On our marketplace, we used feature flags for major UI changes and search algorithm updates. When we redesigned the vehicle detail page, we deployed the new version behind a flag — 10% of users saw the new design while we monitored engagement metrics in New Relic. If metrics dropped, we flipped the flag off instantly without a rollback deployment. This was critical for a 30M-user site where a bad deployment could impact millions. We managed flags in our Spring Boot configuration, toggled via environment variables in ECS task definitions.

The trade-off: feature flags enable safe, gradual rollouts and instant kill switches, but they litter your codebase with if/else branches. If you do not clean up flags after 100% rollout, you end up with dead code paths that confuse developers and make testing harder. We had a rule: every feature flag had an owner and an expiration date. If not removed within 2 sprints of full rollout, it became a tracked tech debt ticket.
