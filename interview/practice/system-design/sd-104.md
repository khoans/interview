**104.** What is bcrypt? Why is it preferred over SHA-256 for password hashing?

========== THIS SECTION IS THE ANSWER ==========
Bcrypt is an adaptive password hashing function that is intentionally slow and includes a configurable work factor (cost). Unlike SHA-256 which is designed to be fast (billions of hashes per second on GPUs), bcrypt's slowness makes brute-force attacks impractical — each guess takes ~100ms instead of nanoseconds.

**Real example:** On our automotive marketplace, we used Spring Security's BCryptPasswordEncoder with a work factor of 12 for all user passwords (dealers and buyers). At cost 12, each hash took ~250ms — imperceptible during login but meaning an attacker with a stolen database would need ~8 years to brute-force a single 8-character password. SHA-256 at GPU speeds could try 10 billion hashes per second, cracking the same password in minutes. We also built in work factor migration: when users logged in, if their stored hash used an older cost factor, we re-hashed with the current factor transparently.

**Trade-off:** Bcrypt's deliberate slowness means it consumes more CPU per login. During peak hours with thousands of concurrent logins, this added measurable CPU load. The mitigation is to avoid re-hashing on every request (only at login) and to size your auth service appropriately. Also, bcrypt has a 72-byte input limit — very long passwords get truncated silently, which is a known gotcha.
