**270.** What is OAuth 2.0 PKCE? Why is it required for SPAs and mobile apps?

PKCE (Proof Key for Code Exchange) protects the authorization code flow for public clients (SPAs, mobile apps) that cannot securely store a client secret. The client generates a random code_verifier, hashes it into a code_challenge sent with the auth request, then proves possession by sending the original code_verifier when exchanging the authorization code for tokens.

On our marketplace, our Next.js frontend (a single-page application) used PKCE for the OAuth2 authorization code flow. Without PKCE, an attacker who intercepted the authorization code (via a malicious browser extension or compromised redirect URI) could exchange it for tokens. With PKCE, the code is useless without the code_verifier that only the legitimate client holds in memory. Spring Security OAuth2 supported PKCE out of the box on the authorization server side.

The trade-off: PKCE adds complexity to the client-side auth implementation (generating verifier, computing SHA-256 challenge, managing state). But it is now mandatory per OAuth 2.1 for all public clients. The old implicit flow (which returned tokens directly in the URL fragment) is deprecated because tokens in URLs get logged in browser history and server logs.
