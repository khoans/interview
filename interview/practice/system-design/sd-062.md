**62.** What is JWT (JSON Web Token)? What are its pros and cons vs session-based auth?

========== THIS SECTION IS THE ANSWER ==========

JWT is a self-contained token with three parts: header (algorithm), payload (claims like userId, roles, expiry), and signature (verifying integrity). Session-based auth stores session state on the server (in memory or Redis) and sends a session ID cookie to the client.

In our project, we used JWTs for API authentication because our microservices architecture made server-side sessions impractical. With 15 different services behind an ALB, session-based auth would require sticky sessions (limiting load balancing) or a shared session store (extra infrastructure). JWTs solved this — each service independently validated the token by checking the signature with the public key. No shared state needed. We used short-lived access tokens (15 minutes) paired with refresh tokens (7 days) stored in HttpOnly, Secure, SameSite cookies. The JWT payload included userId, dealerId, and roles (DEALER_ADMIN, DEALER_VIEWER) so services could authorize without a database lookup.

The trade-off: JWTs can't be revoked before expiry (unlike sessions where you delete the server-side record). If a dealer's account is compromised, we can't instantly invalidate their token — we have to wait up to 15 minutes. We mitigated this with a short TTL and a Redis-based token blocklist for critical cases (account compromise). The lesson: JWTs are ideal for stateless microservices, but keep the TTL short and have a revocation strategy for security-critical scenarios. Don't put sensitive data in the JWT payload — it's base64-encoded, not encrypted.
