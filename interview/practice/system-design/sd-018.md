**18.** What is a write-ahead log (WAL)? How does it ensure data durability?

========== THIS SECTION IS THE ANSWER ==========

A write-ahead log records changes to a sequential log file before applying them to the actual data pages. If the database crashes mid-write, it replays the WAL on startup to recover committed transactions and roll back incomplete ones. This guarantees durability without flushing every data page to disk immediately.

In our project, PostgreSQL's WAL was critical for our RDS setup. We configured WAL-level replication (wal_level=replica) to stream changes to our read replicas in real-time. The WAL served double duty — crash recovery and replication. When we had an incident where an RDS instance restarted unexpectedly during a dealer bulk-upload of 10K listings, zero data was lost because all committed transactions were in the WAL. The recovery took about 45 seconds — PostgreSQL replayed the WAL from the last checkpoint.

The trade-off: WAL adds write amplification — every change is written twice (once to WAL, once to data files). This increases I/O, which matters on I/O-constrained instances. We sized our RDS instance with provisioned IOPS (io1 storage) specifically because the WAL write overhead was significant during peak dealer upload hours. The lesson: WAL is non-negotiable for durability, but plan your storage IOPS budget around it.
