**214.** What is S3 bucket security? Explain bucket policies, ACLs, encryption, and Block Public Access.

========== THIS SECTION IS THE ANSWER ==========

S3 security layers: Block Public Access is the first defense -- an account-level and bucket-level setting that overrides any policy or ACL that would make the bucket public. Bucket policies are resource-based JSON policies controlling access (who, what actions, conditions). ACLs are legacy access controls (prefer policies instead). Encryption options include SSE-S3 (S3-managed keys), SSE-KMS (KMS-managed keys with audit trail), and SSE-C (customer-provided keys).

In our marketplace: (1) Block Public Access was enabled at the account level -- no bucket could ever be made public accidentally. (2) Our vehicle-images bucket policy allowed only the CloudFront OAI to GetObject (users accessed images through CloudFront, never directly from S3), and only the listing-service role to PutObject. (3) All buckets used SSE-KMS encryption with dedicated KMS keys per data classification -- customer PII had a stricter key policy than public listing images. (4) S3 access logging was enabled on sensitive buckets, forwarding to a central audit bucket. (5) Object versioning was enabled to protect against accidental deletion. We disabled ACLs entirely (BucketOwnerEnforced) to simplify the permission model.

The trade-off: SSE-KMS adds cost ($0.03/10K requests) and has throughput limits (5,500 Decrypt calls/sec). For high-volume image serving, we used CloudFront caching to minimize S3 requests. SSE-S3 is free and sufficient when you do not need per-key access control or audit trails.
