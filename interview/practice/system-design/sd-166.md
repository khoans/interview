**166.** What is a Namespace in Kubernetes? How do you use it for multi-tenancy?

========== THIS SECTION IS THE ANSWER ==========

A Namespace is a virtual cluster within Kubernetes that provides logical isolation for resources. It enables you to partition a single physical cluster into separate environments with independent resource quotas, RBAC policies, and network policies. Default namespaces include default, kube-system, and kube-public.

For our marketplace, if we ran on EKS, we would use namespaces to separate environments: dev, staging, production each in their own namespace. Each namespace would have ResourceQuotas (dev: 4 CPU, 8GB RAM; production: 32 CPU, 64GB RAM) to prevent a dev workload from starving production. RBAC RoleBindings would restrict developers to the dev namespace while only the CI/CD pipeline (GitHub Actions service account) could deploy to production. NetworkPolicies would prevent pods in dev from accidentally calling production databases.

The trade-off: namespaces provide soft multi-tenancy, not hard isolation -- they share the same control plane, nodes, and kernel. For true tenant isolation (different customers on the same cluster), you need additional tools like network policies, pod security standards, and resource quotas. For hard multi-tenancy, separate clusters per tenant may be necessary despite the higher cost.
