**53.** What is a single point of failure (SPOF)? How do you eliminate them?

========== THIS SECTION IS THE ANSWER ==========

A SPOF is any component whose failure brings down the entire system. You eliminate SPOFs through redundancy — multiple instances, failover mechanisms, and eliminating dependencies on a single resource.

In our project, we systematically identified and eliminated SPOFs. The database was a SPOF — solved with RDS Multi-AZ (automatic failover to a standby in another AZ). The load balancer was a potential SPOF — solved by using AWS ALB which is inherently redundant across AZs. DNS was a SPOF — solved by using Route 53 which is distributed globally. The one SPOF we struggled with was our Apache Solr search cluster — we ran a single Solr master with replicas. If the master died, indexing stopped (reads still worked from replicas). We eventually migrated the Solr write path to use SolrCloud with distributed indexing, eliminating the single master dependency. The trickiest SPOF was "human knowledge" — one engineer who understood the legacy pricing algorithm. We documented it and paired two other engineers on the service.

The trade-off: eliminating every SPOF costs money and adds complexity. Multi-AZ doubles database cost. Multi-region triples it. At some point, the cost of redundancy exceeds the cost of downtime. The lesson: do a SPOF analysis regularly — list every component and ask "what happens if this dies?" Eliminate SPOFs on the critical path first, accept risk on non-critical components.
