**320.** When should you use a shared event bus (Kafka) vs direct service-to-service events? What are the coupling trade-offs?

========== THIS SECTION IS THE ANSWER ==========

Use a shared event bus when multiple services need to consume the same events and you want a single source of truth for event history. Use direct events (point-to-point SQS, HTTP callbacks) when there is a single known consumer and the interaction is simple.

On our marketplace, we used SNS as a shared event bus for listing lifecycle events — multiple services (search, analytics, notifications) subscribed to the same topic. The producer (listings service) published once and did not care who consumed. If we had used direct calls, the listings service would need to know every consumer and maintain connections to each. For simpler interactions — like the search service requesting a specific dealer's details — a direct REST call was cleaner than publishing an event and waiting for a response.

The trade-off: a shared event bus (Kafka or SNS) decouples producers from consumers and enables event replay (Kafka) or easy fan-out (SNS), but adds infrastructure to manage and makes the data flow harder to trace. Direct communication is simpler to debug and lower latency, but creates point-to-point coupling that does not scale well as the number of consumers grows.
