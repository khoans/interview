**84.** What is a connection multiplexer? How does it reduce database connections?

========== THIS SECTION IS THE ANSWER ==========
A connection multiplexer (like PgBouncer) sits between your application and database, pooling a small number of actual database connections and multiplexing many application requests through them. Instead of each microservice instance holding its own pool, the multiplexer shares connections across all clients.

**Real example:** On our automotive marketplace, we had 15+ Spring Boot microservices on ECS, each with HikariCP pools of 10-20 connections. With auto-scaling, we sometimes had 50+ ECS tasks, meaning 500-1000 connections to PostgreSQL — dangerously close to the max_connections limit. We introduced PgBouncer in transaction pooling mode, which let each ECS task think it had 20 connections, but PgBouncer mapped them to just 100 actual PostgreSQL connections total. CPU on the database dropped 15% from reduced connection overhead.

**Trade-off:** Transaction pooling mode breaks prepared statements and session-level features (SET statements, temp tables). We had to refactor some services that relied on server-side prepared statements in JPA. Session pooling mode preserves these features but is less efficient.
