**163.** What is a Deployment vs StatefulSet vs DaemonSet? When do you use each?

========== THIS SECTION IS THE ANSWER ==========

Deployment manages stateless replicas with rolling updates -- pods are interchangeable. StatefulSet manages stateful workloads where each pod has a stable identity, persistent storage, and ordered startup/shutdown (e.g., databases, Kafka brokers). DaemonSet ensures exactly one pod runs on every node -- used for cluster-wide infrastructure like log collectors, monitoring agents, or network plugins.

In our marketplace architecture mapped to Kubernetes: our Spring Boot microservices (listing, search, user services) would be Deployments -- any pod can handle any request, they scale horizontally, and rolling updates replace pods without downtime. PostgreSQL and Redis would be StatefulSets -- each replica needs stable DNS names (postgres-0, postgres-1) and persistent volumes that survive pod restarts. Our New Relic infrastructure agent and Fluentd log forwarder would be DaemonSets, running on every node to collect metrics and logs.

The trade-off: StatefulSets are harder to manage (complex scaling, backup strategies, operator patterns) compared to Deployments. We kept our databases on managed AWS services (RDS, ElastiCache) rather than running them in containers precisely to avoid StatefulSet operational complexity.
