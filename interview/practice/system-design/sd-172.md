**172.** What is a ReplicaSet? How does Kubernetes maintain the desired number of pods?

========== THIS SECTION IS THE ANSWER ==========

A ReplicaSet ensures a specified number of identical pod replicas are running at all times. It uses a control loop: continuously compare actual pod count vs desired count, then create or delete pods to match. You rarely create ReplicaSets directly -- Deployments manage them, creating a new ReplicaSet for each rollout.

In our marketplace on ECS, we used the equivalent concept: ECS service desired count. When we set desired count to 4 for our listing service, the ECS scheduler ensured 4 tasks were always running. If a container crashed or an EC2 instance died, ECS detected it within seconds and launched a replacement. The same principle applies to ReplicaSets -- if a node fails, the ReplicaSet controller notices fewer pods than desired and schedules new ones on healthy nodes. During our peak traffic (evenings/weekends), we scaled from 4 to 12 replicas.

The trade-off: ReplicaSets ensure availability but all pods are identical -- they cannot handle stateful workloads where each instance is unique. For that, you need StatefulSets. Also, pod disruption budgets (PDBs) should be configured to prevent all replicas from being evicted simultaneously during node maintenance.
