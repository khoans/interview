**73.** What is an ETL pipeline? How does it differ from ELT?

========== THIS SECTION IS THE ANSWER ==========

ETL (Extract, Transform, Load): extract data from sources, transform it (clean, enrich, aggregate) in a processing layer, then load the result into the destination. ELT (Extract, Load, Transform): extract and load raw data into the destination first, then transform it using the destination's compute power. ELT is common with modern cloud warehouses (Redshift, BigQuery) that have powerful built-in compute.

In our project, we used ELT for our analytics pipeline. Extract: Lambda functions pulled raw clickstream events from SQS and raw listing data from PostgreSQL. Load: raw data was loaded into S3 (data lake) and then COPY'd into Redshift staging tables — no transformation yet. Transform: dbt (data build tool) ran SQL transformations inside Redshift — joining clickstream events with listing metadata, calculating conversion rates, building dealer performance dimensions. We chose ELT over ETL because Redshift's columnar engine was faster at transforming data than doing it in Lambda. Transformations were SQL (analysts could write and debug them) rather than Python scripts in Lambda (requiring developer involvement).

The trade-off: ETL reduces the data before loading (less storage, cleaner destination) but requires a separate transformation engine and can't reprocess if transformation logic changes. ELT loads everything raw (more storage cost) but lets you re-transform without re-extracting — when we changed how conversion rates were calculated, we just re-ran the dbt models on existing raw data. The lesson: ELT is the modern default with cloud warehouses. Storage is cheap; re-extracting from production databases is expensive and risky.
