**289.** Your Kubernetes cluster has pods running as root. How do you fix this at scale?

Running containers as root means a container escape vulnerability gives the attacker root on the host. This must be fixed systematically, not pod by pod.

On our ECS-based platform, we faced the equivalent issue with Docker containers. The approach for Kubernetes at scale: (1) **Audit** -- use kubectl to find all pods running as root: check securityContext.runAsUser across all namespaces. Tools like Kubescape or kube-bench automate this CIS benchmark check. (2) **Enforce policy** -- deploy OPA Gatekeeper or Kyverno with a policy that rejects any pod spec with runAsUser:0 or missing runAsNonRoot:true. Start in audit mode (log violations) before switching to enforce mode (block deployments). (3) **Fix images** -- update Dockerfiles to use USER directive with a non-root user (e.g., USER 1001). Ensure the application can run as non-root (file permissions, port binding above 1024). (4) **Set Pod Security Standards** -- apply the "restricted" profile at the namespace level.

The trade-off: some applications genuinely need root (binding to port 80, accessing host devices). For those, use specific exceptions in the policy engine and document why. We found that 95% of our containers could run as non-root once we fixed file permission issues. The remaining 5% needed NET_BIND_SERVICE capability instead of full root.
