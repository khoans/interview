**300.** Design a security architecture for a new microservices application: authentication, authorization, encryption, secrets management, network security, and monitoring.

========== THIS SECTION IS THE ANSWER ==========

I would design six layers based on what we built for our 30M-user automotive marketplace.

**Authentication**: OAuth2 with JWT via Spring Security. Frontend obtains tokens from an identity provider, passes Bearer tokens to the API. Service-to-service uses client credentials flow with unique client ID/secret per service. **Authorization**: RBAC enforced at the API gateway and within each service using Spring Security method-level annotations — dealer, consumer, and admin roles with distinct permissions. **Encryption**: TLS 1.2+ everywhere in transit; ALB terminates external TLS, internal traffic encrypted between services. At rest, KMS-managed keys for RDS, DynamoDB, S3, and SQS. **Secrets management**: AWS Secrets Manager with automatic rotation for database credentials and API keys — injected at runtime, never hardcoded or in environment variables. **Network security**: all services in private subnets with no public IPs. Only the ALB sits in the public subnet. Security groups restrict service-to-service traffic to specific ports. VPC endpoints for AWS services to avoid internet traversal. **Monitoring**: CloudTrail for API audit, VPC Flow Logs for network anomalies, New Relic for application-level metrics, Kibana dashboards for failed auth attempts with SNS alerting on thresholds.

The trade-off: full-stack security adds latency (token validation, encryption overhead) and operational complexity (secret rotation, certificate management). But handling 30M users' personal and financial data, every layer prevented a real category of attack we saw attempted in production.
