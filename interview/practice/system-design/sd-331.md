**331.** When should you add caching vs optimizing the underlying query? How do you decide?

========== THIS SECTION IS THE ANSWER ==========

Optimize the query first. Caching should be a last resort after you have exhausted indexing, query rewriting, and schema improvements. Add caching when the optimized query is still too slow for your latency requirements, or when the data is read far more often than written.

On our marketplace, we first optimized PostgreSQL queries — added composite indexes, rewrote N+1 queries with JOINs, used EXPLAIN ANALYZE to eliminate sequential scans. That brought our listing detail page from 300ms to 50ms. But for the homepage showing "trending listings" (an expensive aggregation across millions of rows), even the optimized query took 800ms. We added ElastiCache Redis with a 5-minute TTL, dropping it to under 5ms. The query only ran once every 5 minutes instead of on every page load.

The trade-off: caching hides slow queries and creates stale data problems, cache invalidation complexity, and thundering herd risks on expiry. If you cache a bad query, you are just deferring the problem and adding a layer of complexity. But for genuinely expensive computations that do not change frequently, caching is the right tool. Always check: "Can I make this query fast enough without caching?" first.
