**58.** What is a health check? How do liveness and readiness probes differ?

========== THIS SECTION IS THE ANSWER ==========

A health check is an endpoint that reports whether a service is operational. Liveness probe: "Is the process alive?" — if it fails, the container is restarted (detects deadlocks, infinite loops). Readiness probe: "Can this instance handle traffic?" — if it fails, the load balancer stops sending traffic to it (detects startup in progress, dependency unavailable).

In our project, we implemented Spring Boot Actuator health checks for both. Our liveness endpoint (/actuator/health/liveness) checked that the JVM was responsive — if the service was stuck in a GC death spiral or a thread deadlock, this would fail, and ECS would restart the container. Our readiness endpoint (/actuator/health/readiness) checked that the database connection pool was active, the Redis connection was alive, and the Solr cluster was reachable. During deployments, a new instance took ~15 seconds to initialize (Spring context startup, connection pool warming). The readiness probe ensured the ALB didn't send traffic until the instance was fully ready. Without it, users would get 503 errors during rolling deployments.

The trade-off: overly aggressive health checks (checking every dependency in liveness) can cause unnecessary restarts — if Redis is briefly unreachable, you don't want to restart your service (it won't help). Put dependency checks in readiness, not liveness. Keep liveness simple (is the process responsive?). The lesson: liveness = restart me if I'm broken. Readiness = don't send me traffic if I'm not ready. Mixing them up causes cascading restarts during dependency outages.
