**106.** What is a pepper? How does it differ from a salt?

========== THIS SECTION IS THE ANSWER ==========
A pepper is a secret value added to the password before hashing, but unlike a salt, it is NOT stored in the database — it is stored separately (e.g., in application config, environment variable, or a secrets manager). If the database is compromised, the attacker has the salts but not the pepper, making cracking much harder.

**Real example:** On our automotive marketplace, we stored the pepper in AWS Secrets Manager, injected as an environment variable into our auth service on ECS. The hashing process was: hash = bcrypt(pepper + password, salt). Even if an attacker dumped the entire PostgreSQL database including all bcrypt hashes and salts, they could not crack passwords without the pepper value from Secrets Manager. This added a second layer of defense beyond bcrypt's built-in salt.

**Trade-off:** Pepper management adds operational complexity — if you lose the pepper, ALL existing passwords become unverifiable and every user must reset their password. We mitigated this by keeping the pepper in Secrets Manager with versioning and backup. Pepper rotation is also harder than salt rotation: you must either re-hash all passwords (requires users to log in) or try multiple pepper versions during verification, which multiplies hash computation time.
