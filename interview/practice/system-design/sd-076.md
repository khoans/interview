**76.** What is event sourcing? How does it differ from traditional CRUD?

========== THIS SECTION IS THE ANSWER ==========

In traditional CRUD, you store the current state — an UPDATE overwrites the previous value. In event sourcing, you store every state change as an immutable event in sequence. The current state is derived by replaying all events. Instead of "listing price = $25,000" you store "PriceSet($30K)", "PriceReduced($27K)", "PriceReduced($25K)."

In our project, we used event sourcing for the dealer audit trail. Compliance required tracking every change to a listing — who changed what, when, and the previous value. With traditional CRUD, we would have needed separate audit tables with triggers, and we'd lose the full history if the trigger logic changed. With event sourcing on the listing modification flow, every change was stored as an immutable event in DynamoDB (ListingCreated, PriceChanged, PhotoAdded, StatusChanged). The current listing state was materialized in PostgreSQL for queries. When a dealer disputed a billing charge, we could replay events to prove exactly when their listing was active and at what price — invaluable for resolving disputes.

The trade-off: event sourcing makes writes simple (append-only, no conflicts) and gives you a complete audit history, but reads are expensive if you replay events on every query. You need projections (materialized views of current state) which add eventual consistency. The event store also grows forever — we used DynamoDB TTL to archive events older than 2 years to S3. The lesson: event sourcing is overkill for most CRUD applications. Use it when audit history is a business requirement or when you need to rebuild state from different perspectives (e.g., showing a listing's price history graph).
