**272.** What is RBAC vs ABAC vs ReBAC? When would you use each access control model?

**RBAC** (Role-Based Access Control) assigns permissions to roles, users get roles -- simple and auditable. **ABAC** (Attribute-Based Access Control) evaluates policies based on attributes of user, resource, and environment -- flexible but complex. **ReBAC** (Relationship-Based Access Control) grants access based on relationships between entities -- natural for social/collaborative apps (e.g., Google Docs sharing).

On our marketplace, we used RBAC through Spring Security: roles included CONSUMER (browse/save vehicles), DEALER_ADMIN (manage listings, pricing), DEALER_STAFF (view-only dealer dashboard), and PLATFORM_ADMIN (full system access). RBAC was sufficient because our permission model was hierarchical and well-defined. If we needed rules like "dealers can only edit listings in their own region" or "discount pricing only available during business hours," we would need ABAC. ReBAC would be useful if we added features like shared watchlists between users.

The trade-off: RBAC is easy to understand and audit but suffers from role explosion when permissions get granular (needing roles like DEALER_ADMIN_EAST_COAST_PREMIUM). ABAC handles this elegantly but policies are harder to debug and audit. Start with RBAC and evolve to ABAC only when role explosion becomes a real problem.
