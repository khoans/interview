**174.** What is a Kubernetes Job vs CronJob? When would you use each?

========== THIS SECTION IS THE ANSWER ==========

A Job runs a pod to completion (one-time or parallel batch processing) -- it ensures the task succeeds by retrying on failure. A CronJob creates Jobs on a schedule (cron syntax) -- like a cluster-native cron. Jobs are for one-off tasks; CronJobs are for recurring tasks.

In our marketplace, we used AWS Lambda and CloudWatch Events for equivalent patterns. Our nightly price analytics aggregation (compute average prices by make/model/year across millions of listings) would be a CronJob running at 2 AM daily. Our one-time data migration when we restructured the DynamoDB table schema would be a Job -- run once, retry on failure, report completion. We also had a CronJob-equivalent for generating daily dealer performance reports, and a Lambda-based Job for processing S3-uploaded bulk listing CSVs from large dealers.

The trade-off: Jobs are not designed for long-running processes -- if your batch takes hours, you need proper checkpointing and restart logic. CronJobs can create overlapping runs if the previous run has not finished; use concurrencyPolicy: Forbid to prevent this. For complex workflows with dependencies between steps, consider Argo Workflows or AWS Step Functions instead.
