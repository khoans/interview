**16.** What is eventual consistency? How does it differ from strong consistency?

========== THIS SECTION IS THE ANSWER ==========

Strong consistency means a read always returns the most recent write — every node sees the same data at the same time. Eventual consistency means after a write, replicas will converge to the same value eventually, but reads might temporarily return stale data.

In our project, we lived with both models. PostgreSQL primary gave us strong consistency for critical data — when a dealer changed a vehicle price, the next buyer query to the primary saw the updated price immediately. But our read replicas had ~100-200ms replication lag, so a buyer hitting a replica might see the old price briefly — that was acceptable. Our ElastiCache Redis layer was eventually consistent by design — we cached search results for 60 seconds. If a dealer added a new listing, it wouldn't appear in search for up to a minute. We accepted this because search freshness within a minute was fine for car shopping.

The trade-off: strong consistency requires coordination (slower writes, less availability during network partitions — CAP theorem). Eventual consistency is faster and more available but requires your business logic to tolerate stale reads. The lesson: classify your data by consistency needs. Price display on a detail page? Strong. Search results listing? Eventual is fine. Don't pay the strong-consistency tax everywhere.
