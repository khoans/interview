**288.** A third-party library you depend on has a critical CVE. What is your process?

Immediate assessment: what is the CVSS score, is there a known exploit in the wild, and does our usage pattern expose us to the vulnerability?

On our marketplace, our process for a critical CVE (like Log4Shell): (1) **Assess exposure** -- does our code trigger the vulnerable code path? For Log4Shell, any service using Log4j for logging was affected. Check mvn dependency:tree to find which services include it (direct or transitive). (2) **Patch** -- if a fixed version exists, update the dependency in pom.xml immediately. If no patch exists, apply the vendor workaround (for Log4Shell, setting the JVM flag -Dlog4j2.formatMsgNoLookups=true). (3) **Deploy** -- fast-track deployment via GitHub Actions, skipping non-critical checks but keeping SonarCloud and tests. (4) **Verify** -- confirm the vulnerable version is no longer present in deployed containers. (5) **Communicate** -- notify stakeholders of the exposure window and remediation. (6) **Post-mortem** -- add Dependabot alerting for this library, review if SCA would have caught it earlier.

The lesson: having a fast deployment pipeline is a security feature. Our ability to go from CVE disclosure to production patch in under 2 hours was only possible because GitHub Actions CI/CD was fully automated. Teams with manual deployment gates take days.
