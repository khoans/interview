**22.** What is a write-through cache vs write-back cache vs write-around cache?

========== THIS SECTION IS THE ANSWER ==========

Write-through: write to cache and database simultaneously — consistent but slower writes. Write-back (write-behind): write to cache first, asynchronously flush to database — fast writes but risk data loss if cache crashes. Write-around: write directly to database, skip cache — avoids cache pollution but first reads are always cache misses.

In our project, we used different strategies for different data. For dealer profile data (name, contact info, logo), we used write-through — when a dealer updated their profile via the admin API, we wrote to PostgreSQL and updated Redis in the same request. Profile updates were rare (maybe once a month per dealer), so the extra write latency was negligible. For search result caching, we used write-around — search results were cached on first read with a 60-second TTL, and listing writes went straight to the database and Solr index. We didn't pre-populate the search cache because the query combinations were too vast to predict.

The trade-off: write-through gives you consistency but doubles your write latency and is wasteful if the data is rarely read after being written. Write-back gives you the fastest writes but you can lose data on cache failure — we never used it for anything requiring durability. The lesson: match the strategy to your read/write ratio. High-read, low-write data (dealer profiles) benefits from write-through. High-write, unpredictable-read data (analytics events) benefits from write-around.
