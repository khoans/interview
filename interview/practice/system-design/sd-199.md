**199.** What is etcd encryption? How do you encrypt secrets at rest in the Kubernetes data store?

========== THIS SECTION IS THE ANSWER ==========

etcd encryption encrypts resources (primarily Secrets) stored in etcd so that even if someone gains access to the etcd data files or backups, they cannot read sensitive data. You configure this via an EncryptionConfiguration file on the API server, specifying which resources to encrypt and which encryption provider to use.

Providers from weakest to strongest: identity (no encryption, default), secretbox (AES-CBC with local key), kms (envelope encryption using an external KMS like AWS KMS). In our marketplace on EKS, AWS encrypted etcd by default using their managed keys. For self-managed Kubernetes: create an EncryptionConfiguration with kms provider pointing to AWS KMS, pass it to the API server via --encryption-provider-config flag, then run kubectl get secrets --all-namespaces -o json | kubectl replace -f - to re-encrypt existing secrets. The KMS approach is best because keys are managed externally and can be rotated without re-encrypting etcd.

The trade-off: encryption adds latency to every Secret read/write (KMS call overhead). Local key encryption (secretbox) is faster but the key is stored on the API server disk -- if the server is compromised, so is the key. KMS envelope encryption keeps the master key external but adds a dependency on the KMS service. For production, KMS is the right choice despite the slight latency overhead.
