**284.** Your Redis instance is accessible from the public internet. How do you secure it?

This is a critical security issue -- Redis has no authentication by default and supports dangerous commands like FLUSHALL and CONFIG SET. An exposed Redis instance can lead to data theft, data destruction, or even remote code execution.

On our marketplace, our ElastiCache Redis was secured by design: (1) **Network isolation** -- Redis ran in a private subnet within our VPC, with no public IP and no internet gateway route. Security groups only allowed inbound on port 6379 from our ECS application security group. (2) **Authentication** -- we enabled Redis AUTH with a strong password stored in AWS Secrets Manager. (3) **Encryption** -- in-transit encryption (TLS) and at-rest encryption via AWS KMS were enabled. (4) **Dangerous commands** -- FLUSHALL and CONFIG were renamed/disabled via ElastiCache parameter groups.

If I discovered an exposed Redis: immediate steps are (1) restrict security group to block public access, (2) assume data is compromised -- rotate all cached session tokens, (3) check Redis MONITOR logs for unauthorized commands, (4) enable AUTH immediately. The trade-off: enabling TLS on Redis adds 10-20% latency overhead, but for a publicly accessible instance, security trumps performance every time.
