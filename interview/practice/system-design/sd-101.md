**101.** What is hashing? How does it differ from encryption? Give examples of each use case.

========== THIS SECTION IS THE ANSWER ==========
Hashing is a one-way function that produces a fixed-size digest from input — you cannot reverse it back to the original. Encryption is a two-way function using a key — you can decrypt back to plaintext. Hashing is for verification (passwords, data integrity), encryption is for confidentiality (protecting data in transit/at rest).

**Real example:** On our automotive marketplace, we used bcrypt hashing for user passwords in PostgreSQL — even if the database was compromised, passwords could not be reversed. For encryption, we used AWS KMS with AES-256 to encrypt PII (Social Security Numbers for dealer identity verification) stored in DynamoDB. The critical difference: we never needed to "un-hash" a password (just compare hashes), but we absolutely needed to decrypt SSNs for the verification workflow.

**Trade-off:** Hashing is faster and simpler since there is no key management, but it is irreversible by design. Encryption requires secure key management (rotation, access control, audit logging via KMS) which adds operational complexity. A common mistake is using encryption where hashing suffices (e.g., encrypting passwords instead of hashing them), which means a key compromise reveals all passwords.
