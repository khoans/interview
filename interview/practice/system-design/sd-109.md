**109.** What is asymmetric encryption (RSA, ECDSA)? Explain public/private key pairs.

========== THIS SECTION IS THE ANSWER ==========
Asymmetric encryption uses a mathematically linked key pair: a public key (shared openly) for encryption and a private key (kept secret) for decryption. RSA is based on factoring large primes; ECDSA uses elliptic curve math for shorter keys with equivalent security. You encrypt with the recipient's public key; only their private key can decrypt.

**Real example:** On our automotive marketplace, asymmetric encryption was used in several places. TLS certificates on our ALB used RSA-2048 key pairs — browsers encrypted the session key with our server's public key during the TLS handshake. For JWT tokens, we used RS256 (RSA + SHA-256): the auth service signed tokens with its private key, and all microservices verified tokens using the public key (distributed via a JWKS endpoint). This meant only the auth service could create tokens, but any service could verify them without sharing the private key.

**Trade-off:** Asymmetric encryption is ~1000x slower than AES, which is why it is only used for key exchange and signatures, not bulk data. RSA-2048 keys are 256 bytes, while ECDSA P-256 provides equivalent security with 32-byte keys — smaller keys mean faster operations and smaller certificates. We considered switching to ECDSA for JWT signing but RSA had broader library support at the time.
