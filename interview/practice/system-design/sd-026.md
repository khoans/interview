**26.** What is object storage (e.g., S3)? How does it differ from block storage and file storage?

========== THIS SECTION IS THE ANSWER ==========

Object storage stores data as objects (blob + metadata + unique ID) in a flat namespace — no directory hierarchy. Block storage provides raw storage volumes (like a hard drive) attached to a server. File storage provides a hierarchical filesystem (NFS, EFS). Object storage is best for unstructured data at scale; block for databases; file for shared filesystems.

In our project, we used S3 extensively. Vehicle images (20+ photos per listing, 15M listings) lived in S3 — that's hundreds of millions of objects. S3 gave us 11 nines of durability, unlimited scale, and we served images through CloudFront CDN for low-latency delivery to buyers across the US. Our PostgreSQL RDS used EBS block storage (gp3 volumes) because databases need low-latency random I/O that only block storage provides. We never used file storage (EFS) — our microservices were stateless and didn't need shared filesystem access.

The trade-off: object storage has high latency for small random reads (HTTP overhead per request) — you'd never run a database on S3. Block storage has low latency but is tied to a single server and costs more per GB. The lesson: S3 for images, documents, backups, logs — anything you write once and read many times. EBS for databases. Don't store images in your database or run databases on S3.
