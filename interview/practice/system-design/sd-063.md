**63.** What is HTTPS/TLS? How does the TLS handshake work?

========== THIS SECTION IS THE ANSWER ==========

HTTPS is HTTP over TLS (Transport Layer Security), encrypting data in transit. The TLS handshake: (1) Client sends "ClientHello" with supported cipher suites and TLS version. (2) Server responds with its certificate and chosen cipher suite. (3) Client verifies the certificate against trusted CAs. (4) They exchange keys (via Diffie-Hellman or RSA) to establish a shared session key. (5) All subsequent communication is encrypted with symmetric encryption using that session key.

In our project, TLS termination happened at the ALB — the ALB handled the TLS handshake with clients, decrypted the traffic, and forwarded unencrypted HTTP to our ECS services internally. This offloaded the computationally expensive TLS operations from our Java services. We used AWS Certificate Manager (ACM) for free, auto-renewing TLS certificates — no manual certificate rotation. For CloudFront (serving vehicle images), we configured TLS 1.2 minimum (dropping TLS 1.0/1.1 for security) and HSTS headers to force browsers to always use HTTPS.

The trade-off: TLS termination at the ALB means internal traffic between ALB and ECS is unencrypted. For most applications this is fine (internal VPC traffic), but for compliance-heavy environments (PCI-DSS), you'd need end-to-end encryption (TLS between ALB and backend too), adding latency and operational overhead. The lesson: TLS at the edge (ALB/CloudFront) is the right default. Only add internal TLS if compliance requires it. And always enforce TLS 1.2+ — older versions have known vulnerabilities.
