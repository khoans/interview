**169.** What are liveness, readiness, and startup probes? What happens when each fails?

========== THIS SECTION IS THE ANSWER ==========

Liveness probe checks if the container is alive -- if it fails, Kubernetes kills and restarts the container (useful for deadlock recovery). Readiness probe checks if the container can serve traffic -- if it fails, Kubernetes removes the pod from Service endpoints but does not restart it (useful during temporary overload). Startup probe checks if the application has started -- it disables liveness/readiness until it succeeds, preventing slow-starting apps from being killed prematurely.

In our marketplace on ECS, we configured health checks similarly. Our Spring Boot services used Spring Actuator endpoints: /actuator/health/liveness returned 200 when the JVM was running, /actuator/health/readiness returned 200 only when database connections and Solr connections were established. During deployments, a new instance needed 30-45 seconds to warm up (JVM, Spring context, connection pools). Without a startup probe equivalent, ECS would mark the instance unhealthy and kill it before it finished starting. We set the health check grace period to 60 seconds to handle this.

The trade-off: aggressive probe settings (short intervals, low thresholds) catch failures fast but cause unnecessary restarts during GC pauses or temporary network blips. We used 10-second intervals with 3 failure thresholds, giving services 30 seconds of tolerance before action.
