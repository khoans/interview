**176.** What is a Kubernetes node drain? When and how do you drain a node safely?

========== THIS SECTION IS THE ANSWER ==========

Node drain gracefully evicts all pods from a node, respecting PodDisruptionBudgets (PDBs) and graceful termination periods. You drain nodes before maintenance (OS patches, instance type changes, decommissioning). The command kubectl drain <node> --ignore-daemonsets --delete-emptydir-data cordons the node first (prevents new pods from scheduling), then evicts existing pods to other nodes.

In our marketplace on ECS, we handled this during EC2 instance rotation. When patching the underlying AMI, we would set the instance to draining state in ECS, which stopped new task placement and waited for existing tasks to complete or be moved. We configured a 120-second deregistration delay on the ALB target group so in-flight requests completed before the instance was terminated. In Kubernetes, the equivalent is setting terminationGracePeriodSeconds on pods and PDBs to ensure at least N replicas stay available during drain.

The trade-off: draining takes time -- if you have many pods with long grace periods or PDBs requiring high availability, a drain can take minutes. For urgent security patches, you might need to force-drain (--force --grace-period=0), which kills pods immediately but risks dropped requests. Always have enough spare capacity across other nodes to absorb drained workloads.
