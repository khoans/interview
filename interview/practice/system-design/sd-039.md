**39.** What is an API gateway? What responsibilities does it handle?

========== THIS SECTION IS THE ANSWER ==========

An API gateway is a single entry point for all client requests, sitting between clients and backend microservices. It handles cross-cutting concerns: request routing, authentication/authorization, rate limiting, request/response transformation, SSL termination, logging, and sometimes response caching and circuit breaking.

In our project, we used AWS API Gateway in front of our microservices. It handled: (1) Authentication — validated JWT tokens before requests reached our services, rejecting unauthenticated calls at the edge. (2) Rate limiting — dealer API partners were limited to 1000 requests/minute per API key. (3) Request routing — /api/v2/listings/* routed to the listing service on ECS, /api/v2/search/* routed to the search service. (4) API key management — each dealer partner got a unique API key tracked for usage billing. Without the gateway, every microservice would have needed to implement auth, rate limiting, and logging independently — duplicated effort and inconsistent behavior.

The trade-off: an API gateway adds latency (an extra network hop, ~5-10ms) and is a potential single point of failure. It can also become a bottleneck if all traffic funnels through one instance. We mitigated this by using AWS-managed API Gateway (auto-scales) rather than self-hosted (Kong, Zuul). The lesson: an API gateway is essential for microservices architectures — the alternative (every service handling cross-cutting concerns) is worse. But keep it thin — business logic belongs in services, not the gateway.
