**28.** What is cache-aside (lazy loading) pattern? When would you use read-through instead?

========== THIS SECTION IS THE ANSWER ==========

Cache-aside: the application checks the cache first; on a miss, it fetches from the database, writes the result to cache, and returns it. The application manages both cache and database. Read-through: the cache itself fetches from the database on a miss — the application only talks to the cache, and the cache handles the data loading logic.

In our project, we used cache-aside for almost everything because Spring Boot made it simple with @Cacheable annotation. For the vehicle detail page, the service checked Redis first; on a miss, queried PostgreSQL, stored the result in Redis with a 5-minute TTL, and returned it. The developer had full control over what got cached, the TTL, and the serialization format. We chose cache-aside over read-through because we needed flexibility — some queries required joining data from PostgreSQL and Solr before caching the combined result. A read-through cache couldn't handle that multi-source assembly.

The trade-off: cache-aside puts caching logic in your application code (potential for inconsistency if you forget to invalidate), while read-through centralizes it in the cache layer (cleaner code but less flexibility). Read-through also guarantees the cache is always populated — with cache-aside, the first request after eviction always hits the database. The lesson: cache-aside is the pragmatic default for most Spring Boot services. Consider read-through only if you have a simple single-source data model and want to keep caching logic out of application code.
