**38.** What is long polling vs short polling vs Server-Sent Events (SSE)?

========== THIS SECTION IS THE ANSWER ==========

Short polling: client sends requests at fixed intervals (e.g., every 5 seconds) — simple but wasteful when there's no new data. Long polling: client sends a request, server holds it open until new data is available or timeout — reduces empty responses but still creates a new connection each time. SSE (Server-Sent Events): server pushes events over a single persistent HTTP connection — efficient, one-directional (server to client only).

In our project, we used different approaches for different features. For the dealer lead notifications, we initially used short polling (every 10 seconds) — it worked but generated massive unnecessary traffic. We upgraded to SSE for the dealer dashboard because notifications were server-to-client only (dealer doesn't send data back). SSE was simpler than WebSockets — it worked over standard HTTP, auto-reconnected on connection drops, and our ALB handled it without sticky session configuration. For the "is this listing still available?" real-time indicator on the buyer's detail page, we used long polling with a 30-second timeout — simple, worked through all proxies, and the update frequency was low.

The trade-off: short polling is simplest but wasteful. Long polling is better but each response requires a new connection. SSE is efficient for server-push but only one-directional. WebSockets handle bidirectional but are most complex. The lesson: SSE is the underrated sweet spot — if you only need server-to-client push (which covers most notification use cases), SSE is simpler than WebSockets and more efficient than polling.
