**198.** What is a Kubernetes Ingress Controller? How do you secure it with TLS and rate limiting?

========== THIS SECTION IS THE ANSWER ==========

An Ingress Controller is the actual component that implements Ingress routing rules -- it is a reverse proxy (NGINX, Traefik, AWS ALB) running inside the cluster that reads Ingress resources and configures routing accordingly. Securing it involves TLS termination (encrypting external traffic) and rate limiting (preventing abuse).

In our marketplace, we used AWS ALB with ACM certificates for TLS and WAF for rate limiting. Mapped to Kubernetes NGINX Ingress Controller: (1) TLS -- store the certificate in a K8s Secret (type: kubernetes.io/tls), reference it in the Ingress spec (tls.secretName), or use cert-manager to auto-provision and renew Let's Encrypt certificates. (2) Rate limiting -- use NGINX annotations: nginx.ingress.kubernetes.io/limit-rps=50 (50 requests per second per IP), nginx.ingress.kubernetes.io/limit-connections=10 (max 10 concurrent connections). We also configured the Ingress to redirect HTTP to HTTPS, set HSTS headers, and use OWASP ModSecurity WAF rules.

The trade-off: NGINX Ingress Controller is a single point of ingress, so it must be highly available (multiple replicas, pod anti-affinity, HPA). Rate limiting at the Ingress level is per-IP, which can be bypassed with distributed attacks or may block legitimate users behind a shared NAT. Combine with AWS WAF or Cloudflare in front for more sophisticated rate limiting and DDoS protection.
