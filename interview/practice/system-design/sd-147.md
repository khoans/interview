**147.** What is insecure deserialization? How can it lead to remote code execution?

========== THIS SECTION IS THE ANSWER ==========
Insecure deserialization occurs when an application deserializes untrusted data without validation. In Java, when ObjectInputStream.readObject() processes a malicious byte stream, the deserialization process can trigger gadget chains — sequences of existing class methods that chain together to execute arbitrary code. This led to critical vulnerabilities like Apache Commons Collections RCE.

**Real example:** On our automotive marketplace, we took this seriously because Java serialization vulnerabilities were a known risk for Spring Boot applications. Our mitigations: (1) we never used Java native serialization for external-facing APIs — all APIs used JSON via Jackson, which does not execute arbitrary code during deserialization; (2) for Jackson, we disabled default typing (ObjectMapper.enableDefaultTyping() was banned in our code standards) because it allows polymorphic deserialization attacks; (3) SonarCloud flagged any usage of ObjectInputStream in code reviews; (4) we kept all dependencies updated via Dependabot to patch known gadget chain vulnerabilities (e.g., Commons Collections, Spring framework CVEs).

**Trade-off:** Avoiding Java native serialization means you lose some framework features that rely on it (HttpSession serialization for session replication, some JMS messaging). We used Redis-based sessions with JSON serialization instead of Java serialization. For inter-service communication, JSON over HTTP was slightly less efficient than binary serialization (protobuf, Kryo), but the security simplicity was worth the ~10% performance overhead.
