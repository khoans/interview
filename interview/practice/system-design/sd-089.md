**89.** What is multi-tenancy? Explain shared database vs database-per-tenant.

========== THIS SECTION IS THE ANSWER ==========
Multi-tenancy is an architecture where a single application instance serves multiple customers (tenants) with data isolation. Shared database stores all tenants' data in the same database with a tenant_id column for isolation. Database-per-tenant gives each tenant a dedicated database for stronger isolation.

**Real example:** On our automotive marketplace, we used a shared database approach with PostgreSQL. Dealer groups (tenants) shared the same database, with every table having a dealer_group_id column. All queries included a WHERE clause filtering on this column, enforced through Spring Data JPA's @Filter annotation and a tenant context set via a request interceptor. This let us serve thousands of dealer groups efficiently without managing thousands of database instances.

**Trade-off:** Shared database is cost-effective and operationally simpler, but a noisy neighbor (one large dealer running heavy reports) can impact others. We mitigated this with query timeouts and resource governor settings. Database-per-tenant provides perfect isolation and easier compliance (GDPR deletion = drop database), but operational overhead scales linearly with tenants. For SaaS products with strict compliance requirements (healthcare, finance), database-per-tenant is often worth the cost.
