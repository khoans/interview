**359.** How did you secure sensitive data in your cloud environment? (S3 buckets, environment variables, database backups, log files)

========== THIS SECTION IS THE ANSWER ==========

Every data store was secured with encryption, access control, and auditing.

**S3 buckets**: Default encryption enabled with KMS, bucket policies denied public access (S3 Block Public Access enabled at the account level), and ACLs disabled. Vehicle images served through CloudFront with Origin Access Identity — no direct S3 URL access. Versioning enabled for accidental deletion recovery. **Environment variables**: Never stored secrets in environment variables or code. All sensitive config (database credentials, API keys, OAuth secrets) lived in AWS Secrets Manager and were fetched at application startup. ECS task roles had IAM permissions scoped to only the specific secrets each service needed. **Database backups**: RDS automated backups encrypted with KMS (same key as the database). Backup retention set to 30 days. Cross-region backup replication to us-west-2 for disaster recovery, also encrypted. Snapshot sharing disabled — no cross-account access. **Log files**: CloudWatch log groups encrypted with KMS. Log retention policies set (90 days for application logs, 1 year for security audit logs). PII never logged — filtered at the application layer before shipping to CloudWatch.

Lesson learned: the most common mistake was developers using environment variables for secrets in ECS task definitions — these are visible in the AWS console and API. We enforced a policy: all secrets through Secrets Manager references in task definitions, validated by a pre-deployment check in GitHub Actions.
