**97.** Design a search autocomplete system. How do you make suggestions fast?

========== THIS SECTION IS THE ANSWER ==========
Key components: a trie or prefix-based index for fast prefix matching, precomputed top suggestions per prefix cached in Redis or at edge, and an async pipeline updating suggestion rankings based on search frequency. Target latency: under 50ms for each keystroke.

**Real example:** On our automotive marketplace, search autocomplete was critical — users typed "Toyota Cam" and expected "Toyota Camry 2024" instantly. We used Apache Solr's suggester component backed by an AnalyzingInfixLookupFactory that matched within words, not just prefixes. The top 10 suggestions per popular prefix were cached in ElastiCache Redis with a 1-hour TTL. CloudFront also cached GET /api/suggest?q=toyota responses at the edge. Search frequency data from Kibana logs was fed back nightly to re-rank suggestions. Average autocomplete latency was 30ms for cached prefixes.

**Trade-off:** Precomputing all prefixes is memory-intensive — for a vocabulary of 100K terms, the prefix space explodes. We only cached the top 5000 most-searched prefixes and fell back to Solr for rare queries. Also, autocomplete can inadvertently suggest sensitive or inappropriate terms, so we maintained a blocklist that filtered suggestions before caching.
