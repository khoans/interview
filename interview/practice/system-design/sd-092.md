**92.** Design a rate limiter. How would you implement it for a distributed system?

========== THIS SECTION IS THE ANSWER ==========
A distributed rate limiter uses a centralized counter store (typically Redis) to track request counts per client across all application instances. Common algorithms: token bucket (allows bursts), sliding window log (precise but memory-heavy), or sliding window counter (good balance).

**Real example:** On our automotive marketplace, we implemented API rate limiting using Redis with a sliding window counter. Each dealer API key was limited to 100 requests/minute. We used a Redis key like `ratelimit:{apiKey}:{windowMinute}` with INCR and EXPIRE. The Spring Boot gateway checked Redis before routing — if the count exceeded the limit, it returned HTTP 429 with a Retry-After header. Since all ECS instances shared the same ElastiCache Redis, the limit was enforced globally regardless of which instance handled the request.

**Trade-off:** Centralized Redis adds a network hop to every request (~1ms). If Redis goes down, you must decide: fail open (allow all requests, risk abuse) or fail closed (block all, risk legitimate traffic). We chose fail open with local in-memory fallback rate limiting — less precise but keeps the service available. Another option is a token bucket algorithm which handles burst traffic more gracefully than fixed windows.
