**309.** When should you use sidecar pattern vs library-based approach for cross-cutting concerns (logging, auth, tracing)?

========== THIS SECTION IS THE ANSWER ==========

Use a sidecar when you have polyglot services (different languages) that all need the same cross-cutting concern. Use a library when all services share the same language and framework, which makes a shared library simpler and faster.

On our marketplace, all services were Java/Spring Boot, so we used the library approach — a shared Spring Boot starter that included structured JSON logging, Spring Security OAuth2 configuration, and distributed tracing with correlation IDs. Every service added one Maven dependency and got consistent logging, auth, and tracing. A sidecar (like Envoy or Istio) would have added container overhead and operational complexity for no polyglot benefit since we were 100% Java.

The trade-off: sidecars provide language-agnostic consistency and can be updated independently of the application (no redeployment needed), but add memory/CPU overhead per container and introduce network complexity. Libraries are faster (in-process, no network hop) but require every service to update the dependency version — which can drift if teams are slow to upgrade.
