**175.** What is container resource management? Explain CPU/memory requests vs limits.

========== THIS SECTION IS THE ANSWER ==========

Requests are the guaranteed minimum resources a container gets -- the scheduler uses them to decide which node to place the pod on. Limits are the maximum resources a container can use -- exceeding memory limit causes an OOMKill, exceeding CPU limit causes throttling. Setting requests without limits allows bursting; setting both creates a bounded resource envelope.

In our marketplace on ECS, we configured equivalent CPU/memory allocations per task. Our Spring Boot listing service was set to request 512MB RAM and 0.5 vCPU (guaranteed minimum), with limits of 1GB RAM and 1 vCPU. The search service needed more due to Solr query caching -- 1GB request, 2GB limit. We learned the hard way when a memory leak in the listing service caused OOMKills every few hours; New Relic showed steadily growing heap usage. Setting the memory limit correctly exposed the leak early rather than letting it consume the entire node.

The trade-off: setting requests too high wastes cluster resources (you are reserving capacity you do not use). Setting them too low causes pod eviction under pressure (Kubernetes evicts pods whose usage exceeds requests when the node is under memory pressure). Profile your applications under realistic load before setting these values.
