**102.** Explain MD5, SHA-1, SHA-256, and SHA-3. Which are still considered secure and why?

========== THIS SECTION IS THE ANSWER ==========
MD5 (128-bit) and SHA-1 (160-bit) are broken — practical collision attacks exist. SHA-256 (256-bit, SHA-2 family) and SHA-3 (256-bit, Keccak-based) are currently secure. SHA-256 is the industry standard; SHA-3 is an alternative with a fundamentally different internal structure, providing algorithm diversity.

**Real example:** On our automotive marketplace, SonarCloud flagged any usage of MD5 or SHA-1 in our codebase as a security vulnerability. We used SHA-256 for content integrity checks (verifying file uploads matched expected hashes) and bcrypt (which internally uses Blowfish, not SHA) for passwords. In our CI/CD pipeline with GitHub Actions, artifact checksums used SHA-256. We also saw SHA-256 in our TLS certificates — our AWS Certificate Manager certs used SHA-256 with RSA for the signature algorithm.

**Trade-off:** SHA-256 is computationally heavier than MD5 (~3-4x slower), which matters at high throughput. For non-security use cases like hash table distribution or checksumming where collision resistance is not critical, faster algorithms like xxHash or MurmurHash are better choices. For password hashing, even SHA-256 is too fast — you want deliberately slow algorithms like bcrypt to resist brute-force attacks.
