**70.** What is zero-trust architecture? How does it differ from perimeter-based security?

========== THIS SECTION IS THE ANSWER ==========

Perimeter-based security: trust everything inside the network, protect the boundary (castle-and-moat). Zero-trust: trust nothing, verify everything — every request must be authenticated and authorized regardless of network location. "Never trust, always verify."

In our project, we evolved toward zero-trust incrementally. Initially, internal service-to-service calls were unauthenticated — if you were inside the VPC, you were trusted. This was perimeter-based thinking. The risk: if one service was compromised (e.g., through a dependency vulnerability), the attacker could call any internal API without authentication. We moved toward zero-trust by: (1) Adding JWT-based service-to-service authentication — each service had its own client credentials and included a JWT on every internal call. (2) Implementing least-privilege IAM roles — each ECS task had its own IAM role with only the permissions it needed (the listing service could read S3 images but not access DynamoDB user data). (3) Encrypting internal traffic between services. (4) Using VPC endpoints for AWS service access instead of routing through the public internet.

The trade-off: zero-trust adds latency (token validation on every internal call), operational complexity (managing service credentials, rotating secrets), and development friction (local development becomes harder when everything requires authentication). Perimeter-based is simpler but one breach compromises everything. The lesson: implement zero-trust gradually. Start with IAM least-privilege (easy win), then add service authentication, then internal encryption. Don't try to go from castle-and-moat to full zero-trust overnight.
