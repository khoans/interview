**353.** How did you handle API authentication between your frontend and backend? What about service-to-service auth?

========== THIS SECTION IS THE ANSWER ==========

Frontend-to-backend used OAuth2 with JWT Bearer tokens. Service-to-service used OAuth2 client credentials flow with unique credentials per service.

On our marketplace, the Next.js frontend obtained a JWT from our identity provider after user login. Every API request included the token as a Bearer header. Our Spring Security configuration validated the token signature (RS256), checked expiration, verified the audience claim matched our API, and extracted user roles for authorization. Tokens had a 30-minute expiration with refresh token rotation. For service-to-service calls (e.g., listings service calling dealer service), each service had its own client ID and secret stored in Secrets Manager. It obtained a machine-to-machine token via the client credentials grant and included it in internal API calls. This way, every internal call was authenticated and we could audit which service called which endpoint.

The trade-off: JWT validation on every request adds a few milliseconds, and managing per-service credentials increases operational overhead (rotation, provisioning). But the alternative — no internal auth and trusting the network — is a single misconfigured security group away from a breach. The few milliseconds of token validation cost is negligible compared to the security it provides.
