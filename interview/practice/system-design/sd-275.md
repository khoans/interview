**275.** What is a JWT algorithm (HS256 vs RS256 vs ES256)? Which is recommended and why?

**HS256** (HMAC-SHA256) is symmetric -- the same secret signs and verifies. **RS256** (RSA-SHA256) is asymmetric -- private key signs, public key verifies. **ES256** (ECDSA-SHA256) is also asymmetric but uses elliptic curves -- smaller keys and faster than RSA for equivalent security.

On our marketplace, we used RS256 because in a microservices architecture, multiple services need to verify tokens but only the auth server should sign them. With HS256, every service would need the shared secret (increasing the attack surface -- any compromised service could forge tokens). With RS256, only the auth server holds the private key; all other services verify using the public key from our JWKS endpoint. Spring Security OAuth2 Resource Server fetched the public key automatically.

The trade-off: RS256 signatures are larger (256 bytes vs 32 bytes for HS256) and slower to compute, which adds a few bytes to every API request and slight CPU overhead on the auth server. ES256 offers the best balance (smaller signatures than RS256, same security guarantees) and is recommended for new systems. We stayed with RS256 because it was already established and widely supported.
