**308.** When should you use an API gateway vs direct service-to-service calls? What's the performance cost?

========== THIS SECTION IS THE ANSWER ==========

Use an API gateway for external-facing traffic where you need centralized auth, rate limiting, and routing. Use direct service-to-service calls for internal communication where latency matters and services already trust each other.

On our marketplace, external traffic from 30M users hit CloudFront then our ALB acting as an API gateway — handling SSL termination, routing to the correct ECS service, and WAF rate limiting. But internal service-to-service calls (e.g., listings service calling the dealer service) went direct via internal ALB or service discovery, bypassing the external gateway. Adding the gateway hop to internal calls would have added 5-10ms per call for no security benefit since internal services already authenticated via client credentials.

The trade-off: an API gateway centralizes cross-cutting concerns (auth, logging, rate limiting) in one place, but it becomes a single point of failure and adds latency. For internal calls, the gateway overhead is pure waste. Keep external traffic through the gateway and internal traffic direct.
