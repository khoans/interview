**181.** What is RBAC (Role-Based Access Control) in Kubernetes? Explain Role, ClusterRole, RoleBinding.

========== THIS SECTION IS THE ANSWER ==========

RBAC controls who can do what in a Kubernetes cluster. A Role defines permissions (verbs like get, list, create, delete on resources like pods, services) within a single namespace. A ClusterRole defines permissions cluster-wide or across all namespaces. RoleBinding binds a Role to a user/group/service account within a namespace; ClusterRoleBinding binds a ClusterRole globally.

In our marketplace, we applied the same RBAC concepts to AWS IAM. Mapped to Kubernetes: developers would get a Role in the dev namespace allowing get, list, logs on pods (read-only debugging) -- bound via RoleBinding. The CI/CD service account (GitHub Actions) would get a Role in the production namespace with create, update, delete on deployments -- only it can deploy. Cluster admins would get a ClusterRole with full access, bound via ClusterRoleBinding. The principle: developers debug but do not deploy; the pipeline deploys but only what it is allowed to.

The trade-off: RBAC is powerful but verbose -- each team, environment, and service account needs specific bindings. Without tooling, managing dozens of Role/RoleBinding YAML files becomes error-prone. Use Helm or Kustomize to template RBAC manifests, and audit bindings regularly with kubectl auth can-i --list to verify actual permissions match intent.
