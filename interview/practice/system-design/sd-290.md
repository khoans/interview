**290.** You need to implement end-to-end encryption for a chat feature. What is your approach?

End-to-end encryption (E2EE) means only the sender and recipient can read messages -- the server only sees ciphertext and cannot decrypt. This requires client-side encryption using asymmetric key pairs.

My approach for adding E2EE to a chat feature: (1) **Key generation** -- each user generates an RSA or X25519 key pair on their device. The public key is uploaded to the server, the private key stays on the device (stored in secure storage). (2) **Message encryption** -- sender fetches recipient public key, generates a symmetric AES-256 session key, encrypts the message with AES, encrypts the AES key with the recipient public key (hybrid encryption). (3) **Storage** -- server stores only encrypted messages. (4) **Key exchange** -- for group chats, use the Signal Protocol (Double Ratchet algorithm) for forward secrecy. (5) **Key backup** -- encrypted key backup to allow device recovery.

The trade-off is significant: E2EE means the server cannot search messages, moderate content, or recover messages if the user loses their device key. On our marketplace, if we added dealer-consumer chat, E2EE would prevent us from moderating conversations for fraud detection (fake listings, phishing). Most business chat features use transport encryption (TLS) rather than E2EE because the business needs searchability and compliance archiving.
