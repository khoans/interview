**91.** Design a URL shortener (like bit.ly). What are the key components?

========== THIS SECTION IS THE ANSWER ==========
Key components: an API gateway, a short-code generation service, a key-value store for URL mappings, a redirect service, and an analytics pipeline. The short code is generated via base62 encoding of an auto-increment ID or a hash of the long URL, stored in a fast lookup store, and redirects use HTTP 301/302.

**Real example:** If I were designing this with our marketplace stack, I would use DynamoDB for the URL mapping (single-digit ms reads, scales infinitely), a Spring Boot service generating 7-character base62 codes from a Snowflake-style ID generator (avoiding collisions without coordination), CloudFront for caching popular redirects at the edge, and SQS feeding click analytics into a separate pipeline. For 30M monthly users worth of shortened URLs, DynamoDB on-demand would cost under $50/month.

**Trade-off:** Base62 encoding of sequential IDs is simple but predictable — users can guess other URLs by incrementing. Hashing the URL avoids this but requires collision handling. 301 (permanent redirect) is faster due to browser caching but prevents accurate click tracking. 302 (temporary redirect) ensures every click hits your server for analytics but adds latency. For analytics-focused use cases like marketing campaigns, 302 is the right choice despite the extra hop.
