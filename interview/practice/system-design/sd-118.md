**118.** What is a nonce? How does it prevent replay attacks?

========== THIS SECTION IS THE ANSWER ==========
A nonce (number used once) is a unique, single-use value included in a cryptographic protocol or API request. It prevents replay attacks because even if an attacker captures a valid request, replaying it with the same nonce will be rejected since the server tracks used nonces.

**Real example:** On our automotive marketplace, our OAuth2 implementation with Spring Security included a nonce in the authorization request. The auth server generated a random nonce, included it in the ID token, and our client verified the nonce matched what it originally sent — preventing an attacker from replaying a captured authorization response. For API idempotency, we used a similar concept: each payment-related API call included an X-Idempotency-Key header (effectively a nonce). The server stored used keys in Redis with a 24-hour TTL, rejecting duplicates. This prevented double-charging when network retries occurred.

**Trade-off:** Nonce tracking requires server-side storage (which nonces have been used), adding state management complexity. For high-throughput APIs, storing every nonce can consume significant memory. We used Redis with TTL-based expiration — nonces older than 24 hours were evicted automatically. The time-window approach is a pragmatic compromise: if a replay happens within 24 hours, it is blocked; beyond that, the original transaction context is likely invalid anyway.
