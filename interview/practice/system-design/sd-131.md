**131.** What are IP tables / security groups / NACLs? How do they filter traffic at different levels?

========== THIS SECTION IS THE ANSWER ==========
These are firewall mechanisms at different layers. IPtables operates at the OS level on individual hosts (Linux kernel packet filtering). Security Groups are stateful firewalls at the instance/ENI level in AWS — if inbound is allowed, outbound response is automatic. NACLs (Network ACLs) are stateless firewalls at the subnet level — you must explicitly allow both inbound and outbound traffic.

**Real example:** On our automotive marketplace, we used all three in a layered defense. NACLs at the subnet level: public subnets allowed inbound 443/80, private subnets blocked all direct internet access. Security Groups per service: the ALB SG allowed inbound 443 from 0.0.0.0/0, the ECS task SG allowed inbound 8080 only from the ALB SG, the RDS SG allowed inbound 5432 only from ECS task SGs. This meant even if an attacker compromised an ECS task, they could not reach the database on a different port or from an unauthorized source. IPtables were used on bastion hosts for additional host-level restrictions.

**Trade-off:** NACLs are stateless, meaning you must manage both inbound and outbound rules explicitly, including ephemeral port ranges for return traffic — this is error-prone. Security Groups are stateful and easier to manage, so we used NACLs only as a coarse outer layer and Security Groups for fine-grained control. The lesson: Security Groups referencing other Security Groups (SG-to-SG rules) is the cleanest approach in AWS and automatically handles IP changes during auto-scaling.
