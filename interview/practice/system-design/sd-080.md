**80.** What is change data capture (CDC)? How does it sync data between systems?

========== THIS SECTION IS THE ANSWER ==========

Change data capture (CDC) tracks row-level changes (inserts, updates, deletes) in a database and streams them as events to other systems in real-time. Instead of periodic batch ETL (which misses changes between runs), CDC captures every change as it happens by reading the database's transaction log (WAL in PostgreSQL, binlog in MySQL).

In our project, we used a CDC-like approach to keep our PostgreSQL database and Apache Solr search index in sync. When a dealer updated a listing in PostgreSQL, we needed Solr to reflect the change within seconds. Our approach: the listing service published an event to SQS after every write (application-level CDC). A better approach we evaluated was Debezium — an open-source CDC tool that reads PostgreSQL's WAL directly and publishes every row change to Kafka topics. Debezium would have caught changes we missed (direct database migrations, bulk updates via SQL scripts that bypassed the application layer). We didn't adopt it because the operational overhead of running Kafka + Debezium outweighed the benefit at our scale.

The trade-off: application-level CDC (publishing events in code) is simpler but misses changes that bypass the application (direct SQL, database triggers, migrations). Log-based CDC (Debezium/WAL) captures everything but requires infrastructure (Kafka, connectors) and adds complexity. There's also latency — CDC adds seconds of delay, so downstream systems are eventually consistent. The lesson: for most microservices, application-level events (publish after write) are sufficient. Move to log-based CDC when you have multiple consumers needing the same data changes or when changes can come from outside the application layer.
