**77.** What is a time-series database? When would you use it over a relational database?

========== THIS SECTION IS THE ANSWER ==========

A time-series database (TSDB) is optimized for data indexed by time — metrics, sensor readings, stock prices. It excels at: high write throughput (append-only), time-range queries, automatic downsampling (aggregate old data), and compression of sequential time data. Examples: InfluxDB, TimescaleDB, Amazon Timestream.

In our project, we stored application metrics and monitoring data in a time-series fashion. New Relic (our APM tool) used a time-series backend to store request latency, error rates, and throughput per service — millions of data points per minute across 15 services. For our custom business metrics (search volume trends, peak traffic hours, listing view counts over time), we used CloudWatch custom metrics, which is essentially a managed TSDB. We considered TimescaleDB (PostgreSQL extension) for storing vehicle price history (tracking how listing prices changed over weeks/months for market analysis) but opted to store price change events in DynamoDB and aggregate in Redshift instead.

The trade-off: a TSDB handles time-indexed writes and range queries 10-100x faster than a relational database, and compresses time-series data significantly. But it's terrible for non-time queries (joins, ad-hoc filtering by non-time dimensions). You'd never run your application's transactional data on a TSDB. The lesson: use a TSDB for monitoring, metrics, IoT sensor data, and any workload where every record has a timestamp and you primarily query by time ranges. For everything else, stick with relational or document databases.
