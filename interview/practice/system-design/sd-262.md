**262.** What is SSO (Single Sign-On)? How does it work with SAML and OIDC?

SSO allows users to authenticate once and access multiple applications without re-entering credentials. **SAML** (Security Assertion Markup Language) uses XML-based assertions -- the IdP sends a signed XML token to the SP via browser redirect (common in enterprises). **OIDC** (OpenID Connect) is built on OAuth2, uses JSON/JWT tokens, and is more modern and developer-friendly (common in web/mobile apps).

On our marketplace, we used OAuth2/OIDC for SSO across our microservices. A user logged in once through our identity provider, received a JWT access token, and that token was accepted by all backend services (vehicle service, dealer service, search service) without re-authentication. For internal tools, enterprise customers used SAML-based SSO through their corporate IdP (Okta, Azure AD) to access our dealer portal. Spring Security supported both protocols.

The trade-off: SAML is verbose and complex but deeply entrenched in enterprise environments -- you cannot avoid supporting it for B2B integrations. OIDC is simpler and better for modern apps but some large enterprises still only support SAML. We ended up supporting both, which doubled our auth configuration and testing effort.
