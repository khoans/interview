**205.** What is AWS Organizations and SCPs (Service Control Policies)? How do they enforce guardrails?

========== THIS SECTION IS THE ANSWER ==========

AWS Organizations manages multiple AWS accounts centrally. Service Control Policies (SCPs) are permission guardrails applied to accounts or OUs (Organizational Units) -- they restrict the maximum permissions available, even for the root user. SCPs do not grant permissions; they set boundaries. IAM policies within an account cannot exceed SCP limits.

In our marketplace, we had separate AWS accounts for dev, staging, and production under an Organization. SCPs enforced: (1) production account could only use us-east-1 and us-west-2 regions (data residency compliance), (2) no account could disable CloudTrail or GuardDuty (security baseline), (3) dev account could not create expensive instance types (cost control -- no p4d.24xlarge GPU instances), (4) no account could make S3 buckets public (prevent data leaks). Even if a developer had AdministratorAccess in the dev account, the SCP prevented them from creating resources in unauthorized regions.

The trade-off: SCPs are blunt instruments -- they apply to the entire account, including automation and service-linked roles. We once had an SCP blocking a specific API call that AWS Config needed internally, which broke compliance monitoring. Test SCPs thoroughly in a sandbox OU before applying to production. Use deny-list approach (allow everything, explicitly deny dangerous actions) rather than allow-list for flexibility.
