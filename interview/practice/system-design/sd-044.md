**44.** What is the sidecar pattern? How is it used in service meshes?

========== THIS SECTION IS THE ANSWER ==========

The sidecar pattern deploys a helper process alongside your main application container — same pod/host, separate process. The sidecar handles cross-cutting concerns (logging, monitoring, networking, security) so the application code doesn't have to. In service meshes (Istio, Linkerd), a sidecar proxy (Envoy) intercepts all network traffic to/from the service, handling mTLS, retries, circuit breaking, and observability transparently.

In our project, we didn't use a full service mesh, but we used the sidecar pattern for log aggregation. Each ECS task ran a Fluentd sidecar container alongside the application container. The application wrote logs to stdout, and the Fluentd sidecar shipped them to our ELK stack (Elasticsearch + Kibana). This meant our Java services didn't need a logging library configured for remote shipping — they just logged to stdout, and the sidecar handled the rest. When we needed to change the log destination (from ELK to CloudWatch), we updated the sidecar config without touching application code.

The trade-off: sidecars add resource overhead (each one consumes CPU/memory) and operational complexity (now you're managing two containers per service instead of one). In a service mesh, the sidecar proxy adds ~1-2ms latency per hop. At our scale (~15 microservices), we didn't need a full service mesh — the overhead wasn't justified. The lesson: sidecars are great for isolating infrastructure concerns from application code. But a full service mesh is overkill until you have 50+ services where manual configuration of mTLS, retries, and circuit breaking becomes unmanageable.
