**130.** What is a SYN flood attack? How do SYN cookies help?

========== THIS SECTION IS THE ANSWER ==========
A SYN flood sends massive numbers of TCP SYN packets without completing the 3-way handshake, filling the server's half-open connection table until legitimate connections are refused. SYN cookies eliminate the need to store state for half-open connections: the server encodes the connection parameters (MSS, timestamp, source) into the SYN-ACK sequence number, and only allocates resources when a valid ACK comes back containing the correctly incremented sequence number.

**Real example:** On our automotive marketplace, our ECS instances running Spring Boot had Linux kernel SYN cookies enabled by default (net.ipv4.tcp_syncookies=1). During normal operation, the kernel used the standard connection table. Only when the SYN backlog was nearly full did SYN cookies activate automatically. At the AWS level, Shield Standard absorbed volumetric SYN floods at the network edge before they reached our instances, and our ALB handled TCP connection management for web traffic, providing an additional buffer.

**Trade-off:** SYN cookies sacrifice some TCP features during an attack: TCP options like window scaling and selective ACK are lost because there is no state to store them in. This reduces throughput for connections established during the attack. Once the attack subsides, normal connection handling resumes automatically. The alternative — increasing the backlog queue (tcp_max_syn_backlog) — is a stop-gap that just delays exhaustion rather than solving the problem.
