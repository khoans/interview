**183.** What is a NetworkPolicy in Kubernetes? How do you restrict pod-to-pod communication?

========== THIS SECTION IS THE ANSWER ==========

A NetworkPolicy is a Kubernetes resource that controls traffic flow between pods using label selectors. By default, all pods can communicate with all other pods (flat network). NetworkPolicies act as a firewall -- you define ingress (incoming) and egress (outgoing) rules specifying which pods, namespaces, or IP ranges are allowed. You need a CNI plugin that supports NetworkPolicies (Calico, Cilium, Weave).

In our marketplace, we applied equivalent network isolation using AWS security groups and VPC design. Mapped to Kubernetes: the listing-service pod would have a NetworkPolicy allowing ingress only from the API gateway pod (label: app=api-gateway) on port 8080, and egress only to PostgreSQL (port 5432) and Redis (port 6379). The database pods would allow ingress only from application pods, never from the internet. Default-deny policies on the namespace ensure no unexpected communication paths exist.

The trade-off: NetworkPolicies are namespace-scoped and can become complex with many services. A default-deny policy (deny all, then whitelist) is the safest approach but requires you to explicitly define every allowed communication path. Start by mapping your service communication graph, then codify it. Missing a rule means broken functionality -- test NetworkPolicies in staging before production.
