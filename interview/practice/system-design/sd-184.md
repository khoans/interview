**184.** What is Pod Security Admission (PSA)? What replaced PodSecurityPolicy?

========== THIS SECTION IS THE ANSWER ==========

Pod Security Admission (PSA) replaced the deprecated PodSecurityPolicy (removed in K8s 1.25) as the built-in mechanism to enforce security standards on pods. PSA defines three security levels: Privileged (unrestricted), Baseline (prevents known privilege escalations), and Restricted (hardened best practices). You apply these levels per namespace using labels with three modes: enforce (reject violations), audit (log violations), and warn (show warnings).

In our marketplace context, we would label our production namespace with pod-security.kubernetes.io/enforce=restricted, which prevents pods from running as root, using host networking, or mounting host paths. Our monitoring namespace (running Prometheus, node-exporter) might need pod-security.kubernetes.io/enforce=baseline since some monitoring agents need slightly elevated privileges. The kube-system namespace stays privileged for system components.

The trade-off: PSA is simpler than PodSecurityPolicy but less flexible -- it is namespace-wide, not per-pod. If you need granular per-workload policies (e.g., one pod in a namespace needs host networking), you either create a separate namespace or use a third-party solution like OPA/Gatekeeper or Kyverno for fine-grained control. PSA covers 80% of use cases; use policy engines for the remaining 20%.
