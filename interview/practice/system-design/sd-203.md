**203.** What is STS (Security Token Service)? How does AssumeRole work?

========== THIS SECTION IS THE ANSWER ==========

AWS STS issues temporary security credentials (access key, secret key, session token) with a configurable expiration (15 min to 12 hours). AssumeRole is the most common STS operation -- it lets an identity (user, service, or another role) assume a different role and receive temporary credentials with that role's permissions. The role must have a trust policy allowing the caller to assume it.

In our marketplace, AssumeRole was central to our security model: (1) Cross-account deployment -- GitHub Actions in our CI account called sts:AssumeRole on a deployment role in the production account, getting 1-hour temp credentials to deploy ECS tasks. (2) Service-to-service -- our listing service assumed a role to access a partner's S3 bucket in their AWS account. (3) Developer access -- devs assumed a read-only production role from their dev account for debugging, requiring MFA (aws:MultiFactorAuthPresent condition). (4) ECS task roles -- behind the scenes, ECS uses STS to provide temp credentials to each container via the container credential provider.

The trade-off: temporary credentials are inherently more secure (they expire) but add complexity -- you must handle credential refresh in long-running processes, and STS calls add latency on startup. The AWS SDK handles refresh automatically for most cases. Also, the role's trust policy is critical -- a misconfigured trust policy can allow unintended principals to assume the role.
