**123.** What is a TCP 3-way handshake? How can it be exploited (SYN flood)?

========== THIS SECTION IS THE ANSWER ==========
The TCP 3-way handshake establishes a connection: (1) Client sends SYN, (2) Server responds SYN-ACK and allocates resources for the half-open connection, (3) Client sends ACK to complete. A SYN flood attack exploits step 2 — the attacker sends millions of SYN packets with spoofed source IPs, the server allocates resources for each half-open connection but never receives the final ACK, exhausting its connection table.

**Real example:** On our automotive marketplace, we were protected against SYN floods at multiple layers. AWS Shield Standard (free, automatic) mitigated volumetric SYN floods at the AWS network edge before traffic even reached our VPC. Our ALB handled TCP connections and had built-in SYN flood protection. For our ECS instances that were not behind ALB (internal services), we relied on Security Groups and the Linux kernel's SYN cookies (net.ipv4.tcp_syncookies=1), which avoid allocating resources for half-open connections by encoding connection state in the SYN-ACK sequence number.

**Trade-off:** SYN cookies eliminate the half-open connection table exhaustion but sacrifice some TCP options (window scaling, selective acknowledgment) because that state is not stored. For most applications, this is an acceptable trade-off during an attack. AWS Shield Advanced ($3K/month) provides additional DDoS protection with dedicated response team, but Shield Standard was sufficient for our threat profile.
