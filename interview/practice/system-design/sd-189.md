**189.** What is OPA/Gatekeeper? How does it enforce policies in a Kubernetes cluster?

========== THIS SECTION IS THE ANSWER ==========

OPA (Open Policy Agent) is a general-purpose policy engine, and Gatekeeper is its Kubernetes-native integration. Gatekeeper runs as an admission controller that evaluates policies (written in Rego language) against every resource creation/update request. If the resource violates a policy, the request is rejected before it reaches etcd. You define ConstraintTemplates (reusable policy logic) and Constraints (where to apply them).

In our marketplace, we enforced similar policies through CI/CD checks and AWS Config rules. Mapped to Gatekeeper: (1) all pods must have resource requests and limits set -- prevents noisy neighbors, (2) no containers running as root, (3) images must come from our private ECR registry, (4) all deployments must have at least 2 replicas in production namespace, (5) no LoadBalancer services in non-production namespaces (cost control). Gatekeeper evaluates these on every kubectl apply and rejects violations instantly with a clear error message.

The trade-off: Rego language has a steep learning curve, and overly strict policies slow down development. Start with audit mode (report violations without blocking) to understand the impact, then switch to enforcement. Also, Gatekeeper adds latency to every API request (typically 10-50ms) since it is in the admission webhook path. Keep policies focused and well-tested.
