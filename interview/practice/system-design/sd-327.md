**327.** When should you use Redis as a primary data store vs just a cache?

========== THIS SECTION IS THE ANSWER ==========

Use Redis as a primary store for ephemeral data where loss is acceptable (sessions, rate limiters, leaderboards, real-time counters). Use it only as a cache when the data must be durable and Redis is accelerating access to a persistent database.

On our marketplace, Redis (ElastiCache) served dual roles. As a cache, it stored frequently accessed listing details and search results with a 5-minute TTL — miss the cache, hit PostgreSQL. As a primary store, it held user session data and rate-limiting counters. Session loss meant a user had to log in again — inconvenient but not catastrophic. We would never store order or payment data in Redis because even with AOF persistence, Redis is not designed for the durability guarantees that financial data demands.

The trade-off: Redis as a primary store gives you sub-millisecond reads and rich data structures (sorted sets, HyperLogLog, streams), but durability depends on persistence configuration — RDB snapshots can lose the last few minutes of data, and AOF with fsync-every-write kills performance. If losing data would be a business problem, Redis is your cache layer, not your database.
