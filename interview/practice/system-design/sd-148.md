**148.** What is security misconfiguration? Give examples in web servers, databases, and cloud.

========== THIS SECTION IS THE ANSWER ==========
Security misconfiguration is any improperly configured security setting that leaves a system vulnerable. It is the most common vulnerability because there are so many configuration points across the stack. Examples: web servers with directory listing enabled or default pages exposed, databases with default credentials or public access, cloud resources with overly permissive IAM policies or public S3 buckets.

**Real example:** On our automotive marketplace, we caught several misconfigurations through security audits. Web server: Spring Boot Actuator endpoints (/actuator/health, /actuator/env) were initially exposed without authentication — /actuator/env could leak environment variables including database credentials. We locked it down with Spring Security to allow only /actuator/health publicly. Database: a staging RDS instance was accidentally created with public accessibility enabled — caught by AWS Config rule within minutes. Cloud: an IAM role for a Lambda function had s3:* permissions when it only needed s3:GetObject on one bucket — we enforced least-privilege with SCPs and IAM Access Analyzer.

**Trade-off:** Security hardening takes ongoing effort — every new service, database, or cloud resource is a potential misconfiguration. Automation is key: we used AWS Config rules for continuous compliance checking, SonarCloud for code-level security scanning, and infrastructure-as-code (CloudFormation) with security review for all resource provisioning. The challenge is balancing speed of development with security review — too many gates slow delivery, too few gates miss vulnerabilities.
