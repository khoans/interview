**296.** You need to implement logging that captures security events but not PII. How do you design it?

========== THIS SECTION IS THE ANSWER ==========

Design structured logging with explicit field classification. Security events get full context; PII fields are masked or excluded at the logging layer before reaching the aggregator.

On our marketplace, we used SLF4J with Logback and a custom JSON layout. Security events — failed logins, permission denials, token expirations, unusual access patterns — were logged with full context: timestamp, user ID (not email), IP, action, resource, and result. PII fields like email, phone, and payment info were excluded using a custom annotation-based filter. Logs shipped to Kibana via CloudWatch Logs, and the log group had restricted IAM policies so only the security team could query raw logs.

Lesson learned: it is far easier to never log PII than to scrub it after the fact. Once PII enters your log pipeline, it spreads to backups, search indices, and dashboards. Use a whitelist approach — only log explicitly approved fields — rather than trying to blacklist sensitive ones.
