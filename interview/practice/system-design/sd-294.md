**294.** A new microservice needs access to another team's database. What is the secure approach?

========== THIS SECTION IS THE ANSWER ==========

Never grant direct database access. The owning team exposes data through a well-defined API with proper authentication, authorization, rate limiting, and a versioned contract.

On our marketplace, the inventory team owned the listings database in PostgreSQL. When the analytics service needed listing data, we did not give them a read replica connection. Instead, we exposed a paginated REST API with OAuth2 service-to-service authentication (client credentials flow). The analytics team consumed the API, and we controlled exactly what data was exposed, enforced rate limits, and could evolve our schema independently using Flyway migrations without breaking their service.

The trade-off: an API adds latency compared to direct DB queries and you maintain an API contract. But direct DB access creates tight coupling — schema changes break consumers silently, you lose row-level security control, and you have zero observability of who reads what. The small latency cost is always worth the decoupling.
