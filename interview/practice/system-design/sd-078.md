**78.** What is a search engine (e.g., Elasticsearch, Solr)? How does inverted indexing work?

========== THIS SECTION IS THE ANSWER ==========

A search engine indexes documents for fast full-text and structured search. An inverted index maps each unique term to the list of documents containing it — like a book's index. Instead of scanning every document for "Toyota," the engine looks up "Toyota" in the inverted index and instantly gets document IDs [123, 456, 789]. This makes search O(1) for term lookup instead of O(n) for full scan.

In our project, Apache Solr was the backbone of our vehicle search — the most critical feature for 30M monthly users. Each listing was indexed as a Solr document with fields: make, model, year, price, mileage, features, dealer location, full-text description. The inverted index for "Toyota" contained ~2M document IDs. When a user searched "Toyota Camry under $30K in Dallas," Solr intersected the inverted indexes for each term, applied range filters (price < 30K), geo-filtered by location, and returned ranked results in ~50ms across 15M documents. We also used Solr's faceted search — showing "Toyota (150K results), Honda (120K results)" — which was computed from the inverted index without scanning all documents.

The trade-off: search engines are phenomenal for read performance but not designed for transactions or frequent updates. Updating a Solr document required re-indexing the entire document (no partial field update in older Solr versions). Our indexing pipeline had ~5-second latency between a PostgreSQL write and the Solr update. The lesson: use a search engine alongside your database, not instead of it. Database is the source of truth for writes; search engine is the optimized read layer. Keep them in sync via events.
