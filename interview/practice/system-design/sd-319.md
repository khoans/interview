**319.** When should you use batch processing vs stream processing? What if you need both (Lambda architecture)?

========== THIS SECTION IS THE ANSWER ==========

Use batch when latency tolerance is hours (daily reports, ETL jobs, ML model training). Use stream when you need near-real-time reactions (fraud detection, live dashboards, event-driven updates).

On our marketplace, we used both. Listing updates were stream-processed — when a dealer changed a price, an SQS event triggered near-real-time Solr reindexing so search results updated within seconds. But analytics reports (daily active listings, price trends, dealer performance) ran as nightly batch jobs that aggregated the full day's data from PostgreSQL. If we needed both real-time dashboards AND accurate historical reports, that is the Lambda architecture — stream for speed, batch for accuracy, and a serving layer that merges both views.

The trade-off: stream processing gives low latency but is harder to debug, replay, and guarantee exactly-once semantics. Batch is simpler, easier to rerun if something goes wrong, and produces accurate aggregates. Lambda architecture gives you both but doubles your processing pipeline — maintaining two code paths for the same data is expensive. Consider the Kappa architecture (stream-only with replayable logs like Kafka) as a simpler alternative.
