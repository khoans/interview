**164.** What is a Kubernetes Service? Explain ClusterIP, NodePort, and LoadBalancer types.

========== THIS SECTION IS THE ANSWER ==========

A Kubernetes Service provides a stable network endpoint to access a group of pods, since pods are ephemeral and their IPs change. ClusterIP (default) exposes the service only inside the cluster on an internal IP -- used for service-to-service communication. NodePort exposes the service on a static port on every node -- accessible from outside but not production-ready. LoadBalancer provisions an external cloud load balancer (AWS ALB/NLB) -- the production way to expose services externally.

Mapped to our marketplace: our internal microservices (listing-service calling search-service) would use ClusterIP -- no external access needed, just internal DNS resolution like search-service.default.svc.cluster.local. Our API gateway would use LoadBalancer type, which in EKS provisions an AWS ALB to route internet traffic. In our actual ECS setup, we achieved the same pattern: internal services used service discovery (AWS Cloud Map) and the public-facing API used an ALB with target groups.

The trade-off: LoadBalancer creates a cloud LB per service, which gets expensive with many services. The common pattern is one LoadBalancer for an Ingress controller, which then routes to multiple ClusterIP services internally -- this is exactly what we did with our single ALB routing to multiple ECS services by path.
