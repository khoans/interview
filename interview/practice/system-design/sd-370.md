**370.** How do you tune Nginx for high performance? Explain worker_processes, worker_connections, keepalive, gzip compression, and buffer settings.

========== THIS SECTION IS THE ANSWER ==========

Key tuning parameters: **worker_processes** — set to the number of CPU cores (`auto` detects this). **worker_connections** — max connections per worker (default 512, set to 1024-4096 for high traffic). Total capacity = workers x connections. **keepalive** — reuse TCP connections to backends instead of opening new ones per request, critical for reducing latency. **gzip** — compress responses to reduce bandwidth. **Buffers** — control how Nginx buffers upstream responses.

Our config: `worker_processes auto; events { worker_connections 2048; } http { keepalive_timeout 65; keepalive_requests 1000; gzip on; gzip_types text/plain application/json application/javascript text/css; gzip_min_length 256; proxy_buffer_size 16k; proxy_buffers 4 32k; }`. For the upstream keepalive: `upstream backend { server 127.0.0.1:8080; keepalive 32; }` — maintains 32 idle connections to the backend, avoiding TCP handshake overhead per request. We also set `proxy_buffering on` so Nginx buffers the full response from the backend before sending it to the client — this frees up the backend thread faster.

The trade-off: more worker_connections means more memory per worker. Gzip saves bandwidth but costs CPU — don't gzip already-compressed content (images, videos). Proxy buffering uses memory but improves backend throughput. For our single-EC2 setup, these defaults handled our traffic comfortably. At ATC scale, the ALB handled these concerns instead.
