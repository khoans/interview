**196.** What is a sidecar proxy for security? How does Envoy handle mTLS in Istio?

========== THIS SECTION IS THE ANSWER ==========

A sidecar proxy is a container injected alongside your application container in the same pod. For security, it transparently intercepts all inbound and outbound traffic to enforce mTLS, authorization policies, and observability without modifying application code. In Istio, the Envoy sidecar handles the entire TLS handshake -- certificate management, rotation, and verification.

The flow in Istio: (1) istiod (control plane) issues short-lived X.509 certificates to each Envoy sidecar via SDS (Secret Discovery Service), (2) when listing-service calls search-service, the source Envoy initiates a TLS handshake using its certificate, (3) the destination Envoy verifies the source identity against the Istio CA, (4) if authorized, encrypted communication proceeds. Certificates auto-rotate every 24 hours. In our marketplace, we would configure Istio PeerAuthentication to require STRICT mTLS across the production namespace, ensuring no plaintext service-to-service traffic.

The trade-off: the sidecar adds ~50MB memory and ~10ms latency per hop. For our latency-sensitive search service (p95 target of 200ms), adding 10ms per downstream call across 3 hops adds 30ms -- significant. Linkerd sidecar is lighter (~10MB, ~1ms latency). Evaluate whether the security benefit justifies the performance overhead for each service, or use mTLS only for sensitive communication paths.
