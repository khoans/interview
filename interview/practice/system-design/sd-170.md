**170.** What is Helm? How does it simplify Kubernetes deployments?

========== THIS SECTION IS THE ANSWER ==========

Helm is the package manager for Kubernetes -- it bundles related K8s manifests (Deployments, Services, ConfigMaps, Ingress) into reusable charts with templated values. Instead of managing 10+ YAML files per service, you install a chart with custom values.yaml. Helm also provides versioning, rollback (helm rollback), and a chart repository for sharing across teams.

For our marketplace, we had around 8 microservices that would each need Deployment, Service, ConfigMap, HPA, and ServiceAccount manifests. Without Helm, that is 40+ YAML files with duplicated boilerplate. With Helm, we would create one base chart with templates parameterized by service name, replicas, image, resource limits, and environment variables. Deploying a new service becomes: helm install search-service ./base-chart -f search-values.yaml. In our actual setup, we achieved similar templating with ECS task definition templates in our GitHub Actions CI/CD pipeline, parameterized per service.

The trade-off: Helm adds a templating layer that can become complex (Go templates with conditionals, loops, subcharts). Debugging a rendered template is harder than reading plain YAML. Some teams prefer Kustomize for simpler overlay-based customization without a templating engine. Choose Helm when you need chart reusability across many services or teams.
