**40.** What is the difference between synchronous and asynchronous communication in microservices?

========== THIS SECTION IS THE ANSWER ==========

Synchronous: the caller sends a request and blocks until the response arrives (REST call, gRPC). The caller and callee must both be available. Asynchronous: the caller sends a message and continues without waiting (message queue, event bus). The callee processes it later independently.

In our project, we used synchronous communication for operations where the user needed an immediate response. When a buyer loaded a listing page, the listing service synchronously called the pricing service to get the current price — the user was waiting, so we needed the response in real-time (~50ms). We used asynchronous communication (SQS) for operations where the user didn't need to wait. When a dealer updated a listing, we responded "accepted" immediately and asynchronously triggered search reindexing, image reprocessing, and analytics recording via SQS messages. This kept the dealer's API response time under 200ms instead of the 2+ seconds it would take to do everything synchronously.

The trade-off: synchronous is simpler to reason about (request-response, easy error handling) but creates tight coupling — if the pricing service is down, the listing page fails. Asynchronous decouples services and improves resilience but adds complexity (eventual consistency, dead letter queues, message ordering). The lesson: use sync for user-facing operations where the response is needed immediately, async for everything else. Our rule of thumb: if the user is waiting, synchronous. If not, asynchronous.
