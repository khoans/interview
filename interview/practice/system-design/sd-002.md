**2.** What is a load balancer? How does it distribute traffic across servers?

========== THIS SECTION IS THE ANSWER ==========

A load balancer sits between clients and servers, distributing incoming requests across multiple backend instances to prevent any single server from being overwhelmed. Common algorithms: round-robin (rotate evenly), least connections (send to the server with fewest active connections), weighted (send more to stronger machines), IP hash (same client always hits the same server).

In our project, we used AWS ALB (Application Load Balancer) at Layer 7. It routed requests based on URL path — `/api/search/*` went to the search service fleet, `/api/inventory/*` went to inventory. We used least-connections algorithm because search queries had variable processing time (simple queries: 50ms, complex multi-filter queries: 2 seconds). Round-robin would have overloaded instances stuck processing slow queries.

The trade-off: L7 load balancers (ALB) can inspect HTTP content and do smart routing, but add ~1-2ms latency. L4 load balancers (NLB) are faster but can only route based on IP/port, no path-based routing. For most web APIs, L7 is the right choice. We used L4 (NLB) only for internal gRPC traffic where we needed raw TCP performance.
