**55.** What is rate limiting? What algorithms exist (token bucket, leaky bucket, sliding window)?

========== THIS SECTION IS THE ANSWER ==========

Rate limiting controls how many requests a client can make in a given time window. Token bucket: tokens are added at a fixed rate, each request consumes a token — allows bursts up to bucket capacity. Leaky bucket: requests enter a queue processed at a fixed rate — smooths traffic but doesn't allow bursts. Sliding window: counts requests in a rolling time window — more accurate than fixed windows, no boundary burst issues.

In our project, we implemented rate limiting at the API Gateway level for our dealer partner APIs. Each partner got 1000 requests/minute using a token bucket algorithm — this allowed short bursts (a dealer syncing inventory might send 200 requests in 10 seconds) while enforcing the overall limit. For our public search API, we used sliding window rate limiting at 100 requests/minute per IP to prevent scraping. We stored counters in ElastiCache Redis using INCR with TTL — simple and fast. When a partner exceeded their limit, we returned 429 Too Many Requests with a Retry-After header.

The trade-off: token bucket is forgiving (allows bursts, good UX) but can let through sudden spikes. Leaky bucket is strict (constant rate, protects the backend) but punishes legitimate bursts. Sliding window is most accurate but requires more memory (tracking per-request timestamps). The lesson: token bucket is the best default for APIs — it accommodates real-world usage patterns while still enforcing limits. Use Redis for distributed rate limiting so the limit applies across all service instances, not per instance.
