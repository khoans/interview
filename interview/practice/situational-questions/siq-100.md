**100.** How did you handle production incidents? Walk me through your incident response process from alert to resolution.

========== THIS SECTION IS THE ANSWER ==========

We had a structured incident response process: Detect, Triage, Mitigate, Resolve, Post-mortem.

A real example: New Relic alert fired at 2 PM — search API error rate jumped from 0.1% to 15% in us-east-1. The on-call engineer (rotating weekly, PagerDuty) acknowledged within 5 minutes. Triage: checked New Relic dashboard, saw that errors were all coming from the Solr cluster — connection refused. Checked AWS Console — one of three Solr EC2 instances had an EBS volume failure. Mitigate (within 15 minutes): updated Solr load balancer to remove the unhealthy node, restarted the application instances to clear stale connection pool entries. Error rate dropped to 0.5%. Resolve: launched a replacement Solr instance from AMI, replicated data from a healthy node, re-added to the cluster. Full resolution within 90 minutes. Post-mortem (next day): documented timeline, root cause (single EBS volume failure), impact (15% error rate for 15 minutes), and action items — enable Solr replication factor 2 for all collections, add Solr node health check to monitoring, create runbook for Solr node replacement.

Lesson: the biggest time-saver in incidents is good observability. Because we had New Relic transaction traces, Kibana logs with correlation IDs, and CloudWatch metrics, we identified the root cause in minutes instead of hours. The post-mortem action items are where real improvement happens — without them, you'll have the same incident again.
