## 282. What is the Iterator pattern?

**Iterator pattern provides way to access elements of collection sequentially without exposing underlying representation.**

It's about uniform traversal — same interface for lists, trees, graphs, streams.

**When to use:**
- Different collections — client doesn't care if it's list, tree, hash map
- Custom traversal — filtered iteration, sorted iteration, reverse iteration
- Lazy evaluation — iterate over stream, not all elements in memory

**Trade-off:**
- Uniform interface — same iteration code for any collection
- Encapsulation — collection internals hidden
- But: extra object — iterator allocation per iteration
- Limited to sequential access — can't jump to position easily

**Real-world example:**
Our analytics platform had 5 data sources — SQL database, NoSQL, CSV files, API streams, in-memory cache. Each had different iteration API. Query code: 5 different iteration patterns, 100 lines per source. Implemented Iterator: each source provides Iterator interface. Query code: same iteration pattern, 20 lines. Adding new source: implement Iterator, no query changes. But iterator allocation added overhead — 1M iterations × 100ns = 100 seconds. Added object pool for iterators — overhead: 100s → 1s.

**Rule of thumb:** Iterator for uniform collection traversal. Not for simple arrays/lists — use built-in iteration. Consider iterator pooling for high-frequency iteration.
