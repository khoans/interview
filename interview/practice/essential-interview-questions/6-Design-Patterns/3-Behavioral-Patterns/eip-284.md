## 284. What is the Mediator pattern?

**Mediator pattern encapsulates how objects interact — objects communicate through mediator, not directly.**

It's about reducing coupling — many-to-many communication becomes many-to-one (mediator).

**When to use:**
- Many-to-many communication — 10 objects interacting, 100 interaction paths
- Complex interactions — object A affects B, which affects C, which affects A
- Reuse objects independently — objects don't know about each other

**Trade-off:**
- Reduced coupling — objects don't reference each other
- Centralized control — mediator knows all interactions
- But: mediator becomes god class — knows everything, does everything
- Single point of failure — mediator bug affects all interactions

**Real-world example:**
Our chat application had direct object references — User → ChatRoom → User → Message → Notification. 20 classes, 50+ references between them. Change one class, 5 others broke. Implemented Mediator: ChatMediator handles all communication. User → Mediator → User. Classes: 20, references: 20 (all to mediator). But mediator grew to 500 lines — handled chat, notifications, user status, typing indicators. Split into 4 mediators (ChatMediator, NotificationMediator, StatusMediator, TypingMediator). Each mediator: 125 lines, testable independently.

**Rule of thumb:** Mediator for complex many-to-many interactions. Not for simple one-to-one — over-engineering. Split mediator when it exceeds 200 lines — multiple focused mediators.
