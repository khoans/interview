## 288. What is the Null Object pattern?

**Null Object provides do-nothing implementation — eliminates null checks by using neutral object instead of null.**

It's about avoiding NullPointerException — return null object instead of null, callers don't need null checks.

**When to use:**
- Optional behavior — feature may not be available, default is do-nothing
- Eliminate null checks — 50+ if (x != null) checks
- Default behavior — null means "use default", not "no value"

**Trade-off:**
- No null checks — cleaner code, no NPE
- Explicit behavior — null object does something (nothing)
- But: hides absence — caller doesn't know if real object or null object
- Debugging — null object silently does nothing, bugs harder to find

**Real-world example:**
Our logging system had if-else everywhere — if logger != null, log; else skip. 100+ null checks. Implemented NullLogger (implements Logger, all methods empty). Default logger: NullLogger. Callers: logger.log() — no null check. Null checks: 100+ → 0. But bugs were hidden — logger not configured, NullLogger silently did nothing. Added debug mode: NullLogger logs warning "NullLogger used — logger not configured". Caught 5 misconfigurations in first week.

**Rule of thumb:** Null Object for optional behavior with sensible default (do nothing). Not for required dependencies — null should fail fast. Add logging to null object for debugging misconfigurations.
