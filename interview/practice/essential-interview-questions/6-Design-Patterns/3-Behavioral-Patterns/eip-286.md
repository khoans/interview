## 286. What is the Visitor pattern?

**Visitor pattern separates algorithm from object structure — add operations without modifying classes.**

It's about adding behavior to existing class hierarchy — new operations without touching existing code.

**When to use:**
- Stable hierarchy, new operations — class structure rarely changes, frequently add operations
- Cross-cutting operations — same operation across many classes (export, validate, serialize)
- Avoid polluting classes — operation logic doesn't belong in domain classes

**Trade-off:**
- Add operations without modifying classes — open/closed principle
- Related operations grouped — all export logic in ExportVisitor
- But: fragile — adding new class requires updating all visitors
- Violates encapsulation — visitor accesses internal state

**Real-world example:**
Our AST (Abstract Syntax Tree) had 20 node types — Expression, Statement, Variable, Function... Needed 5 operations: evaluate, serialize, validate, optimize, pretty-print. Without visitor: 20 classes × 5 operations = 100 methods scattered. Implemented Visitor: 20 node types accept(visitor), 5 visitor classes (EvaluateVisitor, SerializeVisitor...). Adding operation: 1 new visitor class (20 methods). But adding node type (IfStatement): update all 5 visitors — 100 methods to add. Chose visitor because operations added weekly, node types added yearly.

**Rule of thumb:** Visitor for stable hierarchies with frequent new operations. Not for evolving hierarchies — every new class breaks all visitors. Document which is stable — operations or class structure.
