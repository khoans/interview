## 278. What is the Command pattern?

**Command pattern encapsulates request as object — parameterize clients with different requests, queue/log requests, support undo.**

It's about turning operations into first-class objects — commands can be stored, queued, executed later.

**When to use:**
- Undo/redo — command stores state, can reverse operation
- Command queues — background processing, job scheduling
- Macro operations — group multiple commands as one

**Trade-off:**
- Undo capability — commands store state for reversal
- Queuing — commands executed later, asynchronously
- But: more classes — one class per command type
- State management — commands must store enough state for undo

**Real-world example:**
Our text editor needed undo/redo — 50+ operations (insert, delete, format, cut, paste). Each operation stored before/after state. Command interface: execute(), undo(). 50 command classes. Undo stack: last 100 commands. Redo stack: undone commands. But memory grew — 100 commands × 1KB = 100KB per document. Added command compression — merge consecutive insert commands. Memory: 100KB → 20KB average.

**Rule of thumb:** Command for undo/redo, job queues, macro operations. Not for simple operations — over-engineering. Store minimal state for undo — enough to reverse, not full object copies.
