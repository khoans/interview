## 290. What is the Double Dispatch pattern?

**Double Dispatch pattern allows method selection based on runtime type of two objects — not just receiver, but also argument.**

It's about runtime polymorphism on two dimensions — visitor pattern is the most common implementation.

**When to use:**
- Two-dimensional polymorphism — behavior depends on types of both objects
- Collision detection — shape A collides with shape B (circle-circle, circle-rectangle, rectangle-rectangle)
- Operation on heterogeneous collections — different operation based on both operand types

**Trade-off:**
- Runtime type selection — behavior based on both object types
- Extensible — add new types without changing existing code
- But: complex — requires two method calls, visitor infrastructure
- Type safety — compiler can't verify all combinations

**Real-world example:**
Our game had collision detection — 10 shape types (Circle, Rectangle, Triangle...). 10 × 10 = 100 collision combinations. If-else: if circle && circle, use circle-circle collision; if circle && rectangle, use circle-rectangle collision. 500 lines of if-else. Implemented Double Dispatch: Shape.collidesWith(Shape) calls visitor.visit(this). Each shape has visit methods for all other shapes. 100 collision methods, but organized (Circle.collidesWithCircle, Circle.collidesWithRectangle...). Adding new shape: 10 new methods vs. 50 if-else branches.

**Rule of thumb:** Double Dispatch for two-dimensional type-based behavior. Not for single-dimension — use regular polymorphism. Consider Visitor pattern — it implements double dispatch cleanly.
