## 279. What is the State pattern?

**State pattern allows object to change behavior when internal state changes — appears to change class.**

It's about state-driven behavior — different state, different method implementations.

**When to use:**
- State machines — document lifecycle (draft → review → published → archived)
- Conditional behavior — 10+ if-else based on state field
- State-specific logic — each state has unique behavior

**Trade-off:**
- Clean state logic — each state is separate class
- But: more classes — one per state, state management overhead
- State transitions — must manage state changes explicitly

**Real-world example:**
Our order system had Order class with state field (NEW, PAID, SHIPPED, DELIVERED, CANCELLED). 200 lines of if-else — if state == PAID, allow ship; if state == CANCELLED, reject all operations. Bugs: invalid state transitions (NEW → DELIVERED). Implemented State pattern: 5 state classes, Order delegates to current state. Invalid transitions throw exception. Bugs: 10/month → 0. But added 5 state classes + context. State transition logic distributed across 5 classes — harder to see all transitions at once. Added state transition diagram documentation.

**Rule of thumb:** State for complex state machines with 4+ states. Not for simple flags (active/inactive) — over-engineering. Document state transitions — which states can transition to which.
