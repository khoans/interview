## 275. What is the Strategy pattern? Give an example.

**Strategy pattern defines a family of algorithms, encapsulates each one, and makes them interchangeable — client chooses which strategy to use.**

It's about runtime algorithm selection — same interface, different implementations for different contexts.

**When to use:**
- Multiple algorithms for same task — sorting (quick, merge, bubble), compression (zip, gzip, lz4)
- Algorithm varies by context — different pricing strategies for different customer tiers
- Avoid if-else hell — 10+ conditions choosing algorithm

**Trade-off:**
- Flexible algorithm selection — change at runtime, test independently
- But: more classes — one per strategy, plus context class
- Indirection — one extra method call through strategy interface

**Real-world example:**
Our shipping calculator had if-else for 8 shipping methods — FedEx, UPS, USPS, DHL, each with different calculation logic. 200 lines of if-else, bugs everywhere. Implemented Strategy: ShippingStrategy interface, 8 implementations. Context class holds strategy. Adding new carrier: 1 new class, no changes to existing code. Bugs: 5/month → 0. But added 8 strategy classes + 1 context — 9 classes vs. 1 monolith. Worth it for maintainability.

**Rule of thumb:** Strategy for interchangeable algorithms with same interface. Not for one-of-a-kind logic — over-engineering. Use when you have 3+ variations of same operation.
