## 276. What is the Observer pattern? Give an example.

**Observer pattern defines one-to-many dependency — when one object changes state, all dependents are notified automatically.**

It's about event-driven architecture — subjects publish events, observers subscribe and react.

**When to use:**
- Event propagation — UI events (button click → multiple handlers)
- Decoupled notification — sender doesn't know receivers
- Automatic updates — data changes, multiple views update

**Trade-off:**
- Loose coupling — subject doesn't know observers
- But: unexpected updates — observer chain hard to trace
- Memory leaks — observers not unsubscribed, prevent garbage collection

**Real-world example:**
Our dashboard had 5 widgets showing same data — stock price, order count, user activity. Data fetch polled 5 times, 5x database load. Implemented Observer: data model is subject, 5 widgets are observers. Single data fetch, notify all observers. Database load: 5 queries → 1 query. But widget A had memory leak — didn't unsubscribe on destroy. After 1 hour, 1000 dead observers, dashboard crashed. Added automatic unsubscribe in onDestroy lifecycle. Zero leaks since.

**Rule of thumb:** Observer for one-to-many notifications. Always implement unsubscribe — prevent memory leaks. Document observer execution order — don't assume synchronous or ordered.
