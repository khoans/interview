## 283. What is the Chain of Responsibility pattern?

**Chain of Responsibility pattern passes request along chain of handlers — each handler decides to process or pass to next.**

It's about decoupled request handling — sender doesn't know which handler will process.

**When to use:**
- Multiple handlers — request can be handled by different objects
- Handler order matters — try handler A, then B, then C
- Dynamic chain — add/remove handlers at runtime

**Trade-off:**
- Decoupled sender/receiver — sender doesn't know which handler processes
- Flexible chain — add/remove handlers without changing sender
- But: no guarantee handled — request may reach end of chain unhandled
- Debugging complexity — trace through chain to find which handler processed

**Real-world example:**
Our authentication system had 5 auth methods — API key, JWT, session, OAuth, SAML. If-else chain: 100 lines, order hardcoded. Implemented Chain of Responsibility: AuthHandler chain (APIKeyHandler → JWTHandler → SessionHandler → OAuthHandler → SAMLHandler). Request passes through chain, first valid handler processes. Adding new auth: insert handler in chain. But debugging auth failures required tracing all 5 handlers — added logging at each handler ("APIKeyHandler: no key found, passing to next").

**Rule of thumb:** Chain of Responsibility for multiple potential handlers. Ensure chain has default/fallback handler — don't let requests go unhandled. Log each handler decision for debugging.
