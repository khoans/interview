## 285. What is the Memento pattern?

**Memento pattern captures and externalizes object's internal state — can restore object to this state later.**

It's about state snapshots — save state without exposing internals, restore later.

**When to use:**
- Undo/redo — save state before each operation
- Checkpoints — save state at milestones, restore on failure
- Snapshots — point-in-time state for comparison

**Trade-off:**
- Encapsulation — state saved without exposing internals
- Undo capability — restore to previous state
- But: memory cost — each memento stores full state
- Serialization overhead — deep copy of object graph

**Real-world example:**
Our game had save/load system — player state (level, inventory, position, quests, achievements). Direct serialization exposed internals, broke encapsulation. Implemented Memento: GameOriginator creates Memento with full state. Save: originator.createMemento(), store memento. Load: originator.restoreMemento(). 10 save slots × 50KB = 500KB per game. But memory grew — players saved 100+ times. Added save limit (10 slots), compression (500KB → 100KB). Zero encapsulation violations.

**Rule of thumb:** Memento for undo/checkpoints with encapsulation. Limit number of mementos — memory cost adds up. Compress mementos if large — trade CPU for memory.
