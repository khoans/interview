## 287. What is the Interpreter pattern?

**Interpreter pattern defines grammar for language and interpreter to interpret sentences — parse and execute domain-specific languages.**

It's about building DSLs (Domain-Specific Languages) — grammar rules as classes, parse trees as objects.

**When to use:**
- Simple DSLs — rule engines, query languages, formula evaluators
- Grammar-based parsing — well-defined grammar with rules
- Custom scripting — users define behavior in DSL

**Trade-off:**
- Flexible DSL — users define rules, behavior
- Maintainable grammar — each rule is separate class
- But: complex — grammar classes, parse tree, interpreter
- Performance — interpretation slower than compiled code

**Real-world example:**
Our pricing engine had business rules in code — "if customer is premium AND order > $100, apply 10% discount". Business changed rules weekly, developer deployment required. Implemented Interpreter: grammar (AndRule, OrRule, GreaterThanRule, DiscountAction), parser builds rule tree from business-readable syntax. Business changed rules via UI, no deployment. But interpreter added 200ms overhead per calculation. Cached parsed rule trees — overhead: 200ms → 5ms.

**Rule of thumb:** Interpreter for business rules, simple DSLs. Not for complex languages — use parser generators (ANTLR). Cache parsed expressions — interpretation is expensive.
