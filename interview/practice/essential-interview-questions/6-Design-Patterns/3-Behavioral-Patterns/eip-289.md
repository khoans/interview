## 289. What is the Specification pattern?

**Specification pattern encapsulates business rules as composable predicates — combine rules with AND, OR, NOT.**

It's about reusable business logic — rules are first-class objects, can be combined, reused, tested independently.

**When to use:**
- Complex business rules — customer eligibility, product filtering, validation
- Rule composition — combine rules dynamically (premium AND active AND verified)
- Reusable predicates — same rule used in multiple places

**Trade-off:**
- Composable rules — AND, OR, NOT combinations
- Testable — each rule tested independently
- But: more classes — one class per rule
- Query translation — in-memory rules may not translate to SQL

**Real-world example:**
Our user filtering had if-else everywhere — if active AND verified AND premium, show feature. 50+ filtering locations, inconsistent logic. Implemented Specification: IsActive, IsVerified, IsPremium specifications. Combine: IsActive.and(IsVerified).and(IsPremium). Reused in 50 locations, consistent logic. Adding rule: 1 new specification class. But database queries couldn't use specifications — in-memory filtering after fetch. Added specification-to-SQL translator (200 lines). Now specifications work for both in-memory and database queries.

**Rule of thumb:** Specification for complex, composable business rules. Not for simple validation — over-engineering. Consider database translation — specifications should work for both in-memory and persisted data.
