## 277. What is the difference between Observer and Pub/Sub?

**Observer is in-process, direct method calls — subject knows observers. Pub/Sub is out-of-process, message-based — publisher doesn't know subscribers.**

Same concept (one-to-many notification), different scope and coupling.

**Observer:**
- In-process — same application, direct method calls
- Tight coupling — subject holds references to observers
- Synchronous — notify all observers in same thread
- Example: Java Swing event listeners, React state updates

**Pub/Sub:**
- Out-of-process — different applications, message broker
- Loose coupling — publisher sends to topic, subscribers unknown
- Asynchronous — message queue, subscribers process independently
- Example: Kafka topics, Redis Pub/Sub, AWS SNS/SQS

**Trade-off:**
- Observer: simple, fast — but limited to single application
- Pub/Sub: scalable, decoupled — but requires message broker, eventual consistency

**Real-world example:**
Our app started with Observer — order placed, notify inventory, shipping, email observers (same JVM). Worked for monolith. Split to microservices: Observer broke (different JVMs). Implemented Pub/Sub with Kafka: order service publishes OrderCreated event, inventory/shipping/email services subscribe. Services independent, can fail independently. But introduced eventual consistency — email sometimes sent before inventory reserved. Added saga pattern for ordering.

**Rule of thumb:** Observer for in-process event handling. Pub/Sub for cross-service communication. Don't use Pub/Sub for single application — message broker overhead unnecessary.
