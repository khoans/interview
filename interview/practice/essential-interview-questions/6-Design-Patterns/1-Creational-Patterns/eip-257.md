## 257. What is the Prototype pattern?

**Prototype pattern creates new objects by copying existing instances — clone instead of new.**

It's about avoiding expensive initialization — copy pre-configured prototype instead of building from scratch.

**When to use:**
- Expensive initialization — database connections, cached data, complex calculations
- Runtime configuration — object configured at runtime, need copies with same config
- Undo/redo — save prototype state, restore by copying

**Trade-off:**
- Fast cloning — copy existing state vs. full initialization
- But: shallow copy issues — nested objects shared between original and clone
- Complexity — deep copy requires careful implementation

**Real-world example:**
Our report generator created complex report templates — loaded database schema, calculated metrics, configured styling. 2 seconds per report. Users needed 10 variations — 20 seconds total. Implemented Prototype: create one template (2 seconds), clone 9 times (50ms each). Total time: 20 seconds → 2.5 seconds. But shallow copy caused bug — modifying clone's title changed original's title. Fixed with deep copy — clone nested objects too.

**Rule of thumb:** Prototype for expensive-to-create objects needed in multiple variations. Ensure deep copy for nested mutable objects. Not needed for simple objects — new is faster than clone overhead.
