## 261. What is Lazy Initialization?

**Lazy Initialization delays object creation until first access — create on-demand, not upfront.**

It's about avoiding work you might not need — expensive objects only created if actually used.

**When to use:**
- Expensive initialization — 2 seconds to load, may never be used
- Optional features — premium features only loaded for premium users
- Circular dependencies — A needs B, B needs A — lazy break the cycle

**Trade-off:**
- Faster startup — don't initialize unused features
- Memory savings — only used objects consume memory
- But: first access is slow — initialization happens on first use
- Complexity — thread-safe lazy init requires synchronization

**Real-world example:**
Our enterprise app loaded all 50 modules at startup — 2 minutes startup time. Most modules rarely used (admin, reporting). Implemented lazy initialization: core modules at startup, admin/reporting on first access. Startup: 2 minutes → 15 seconds. Admin module (used once/day): 2-minute cost paid once, not every startup. But first admin access took 2 seconds (user noticed). Added progress indicator: "Loading admin module..." — user tolerance improved.

**Rule of thumb:** Lazy initialization for expensive, rarely-used objects. Not for always-used objects — adds complexity without benefit. Consider user experience — first-access delay should be acceptable or show progress.
