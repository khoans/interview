## 253. How do you make Singleton thread-safe?

**Thread-safe Singleton requires synchronization — double-checked locking, static initialization, or enum-based approach.**

The goal: one instance created even when multiple threads access simultaneously.

**Approaches:**
- **Eager initialization** — create instance at class load time (simple, always thread-safe)
- **Double-checked locking** — check, lock, check again (lazy, efficient)
- **Static holder** — inner static class holds instance (lazy, no synchronization)
- **Enum** — single-element enum (simplest, serialization-safe)

**Trade-off:**
- Eager: simple, no synchronization cost — but instance created even if never used
- Double-checked: lazy, efficient — but complex, error-prone (volatile required)
- Enum: simplest, safe — but can't extend, limited flexibility

**Real-world example:**
Our configuration Singleton had race condition — two threads created two instances simultaneously. Configuration got out of sync — half services used old config, half used new. Bug took 2 days to trace. Fixed with double-checked locking: check if null, synchronize, check again, create. Zero duplicate instances since. But synchronization added 5ms overhead on first access. Switched to static holder pattern — lazy initialization, no synchronization overhead.

**Rule of thumb:** Use enum Singleton for simplicity (effective Java recommendation). Use static holder for lazy initialization without synchronization. Avoid double-checked locking unless you understand volatile semantics.
