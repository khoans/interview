## 252. What are the problems with Singleton?

**Singleton causes hidden coupling, testability issues, and violates Single Responsibility Principle — it's often a workaround for poor dependency injection.**

The problems outweigh benefits in modern applications with DI containers.

**Key problems:**
- **Hidden coupling** — classes depend on global state, not explicit dependencies
- **Testability** — Singleton state persists across tests, tests interfere with each other
- **Single Responsibility violation** — class manages both business logic and its own lifecycle
- **Concurrency** — thread-safe initialization requires synchronization (performance cost)

**Trade-off:**
- Convenient global access — no DI setup needed
- But: tests require special handling — reset state, use test doubles
- Hard to refactor — changing Singleton affects entire codebase

**Real-world example:**
Our logging Singleton worked great until tests ran in parallel — test A configured logger to DEBUG, test B expected ERROR, both failed. 30% of tests were flaky due to Singleton state. Refactored to dependency injection — logger passed as constructor parameter. Tests: each test gets fresh logger instance. Flaky tests: 30% → 0%. But required changing 200 classes — constructor injection everywhere. Worth it — tests reliable, codebase maintainable.

**Rule of thumb:** Avoid Singleton in new code — use dependency injection. For existing Singletons: add reset method for tests, or refactor to DI gradually. Never use Singleton for business logic — only for truly shared resources.
