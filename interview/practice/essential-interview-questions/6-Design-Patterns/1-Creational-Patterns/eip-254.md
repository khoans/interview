## 254. What is the Factory pattern?

**Factory pattern delegates object creation to a separate class — factory decides which concrete class to instantiate based on input.**

It's about encapsulating object creation logic — callers depend on abstraction, not concrete classes.

**When to use:**
- Multiple implementations — PaymentProcessor (CreditCard, PayPal, BankTransfer)
- Creation logic complex — validation, configuration, dependency setup
- Decouple caller from concrete classes — test with mocks easily

**Trade-off:**
- Flexibility — add new implementations without changing callers
- But: more classes — factory + products, increased complexity
- Indirection — harder to trace which concrete class is created

**Real-world example:**
Our payment system had if-else for payment types — if credit card, create CreditCardProcessor; if PayPal, create PayPalProcessor. 500 lines of creation logic scattered across 20 classes. Added PaymentFactory — single responsibility: create processor based on type. Creation logic: 500 lines → 100 lines in factory. Adding new payment type: change 1 class (factory), not 20. But introduced indirection — debugging required tracing through factory.

**Rule of thumb:** Factory for multiple implementations with complex creation logic. Not needed for simple new ClassName() — over-engineering. Use when creation logic changes frequently or varies by context.
