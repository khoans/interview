## 262. What is Dependency Injection pattern?

**Dependency Injection provides dependencies from outside — objects receive dependencies via constructor/setter, don't create them internally.**

It's about inversion of control — framework creates and wires objects, not your code.

**Key benefits:**
- Testability — inject mocks instead of real dependencies
- Loose coupling — depend on interfaces, not concrete classes
- Single Responsibility — class focuses on business logic, not dependency creation

**Trade-off:**
- Testable, decoupled code — easy to swap implementations
- But: more boilerplate — constructors with 5+ parameters
- Framework dependency — Spring, Guice become critical infrastructure
- Hidden complexity — wiring happens in configuration, not visible in code

**Real-world example:**
Our service classes created dependencies internally — new UserRepository(), new EmailService(). Unit tests required real database, real email server — slow, flaky, 30% failure rate. Refactored to constructor injection — UserRepository passed in constructor. Tests: inject mock repository, no database needed. Test failure rate: 30% → 0%. Test time: 10 seconds → 100ms. But required Spring framework — 200MB overhead, 30-second startup. For simple apps, overkill. For enterprise apps, worth it.

**Rule of thumb:** Dependency Injection for any non-trivial application — testability benefits outweigh framework overhead. Constructor injection preferred (explicit, immutable). Setter injection for optional dependencies. Avoid service locator anti-pattern — hiding dependencies.
