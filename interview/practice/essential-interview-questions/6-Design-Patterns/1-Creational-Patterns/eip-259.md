## 259. When would you use Factory vs Builder?

**Factory chooses which class to create. Builder constructs a complex object step-by-step.**

Different problems: Factory is "which type?", Builder is "how to configure?"

**Factory:**
- Problem: multiple implementations, choose at runtime
- Example: PaymentProcessor — CreditCard vs. PayPal vs. BankTransfer
- Decision: which concrete class to instantiate

**Builder:**
- Problem: one class, many configuration options
- Example: Email — to, from, CC, subject, body, attachments (all optional)
- Decision: how to configure this instance

**Trade-off:**
- Factory: flexibility in implementations — add new payment types without changing callers
- Builder: readability in construction — fluent API, no telescoping constructors
- Can combine: Factory creates Builder — PaymentFactory.getBuilder(type)

**Real-world example:**
Our notification system started with Factory — EmailNotifier, SMSNotifier, PushNotifier. Worked for choosing type. But EmailNotifier had 10 optional fields (CC, BCC, attachments, priority, HTML vs. plain). Constructor explosion: 8 constructors. Refactored: Factory chooses notifier type, Builder configures it. NotificationFactory.create(type).with(...).build(). Type selection + configuration: both clean.

**Rule of thumb:** Factory for choosing implementations. Builder for configuring complex objects. Use both when you need both — Factory picks type, Builder configures it.
