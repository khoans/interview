## 256. What is the Builder pattern? When would you use it?

**Builder pattern constructs complex objects step-by-step — fluent API for objects with many optional parameters.**

It's about readable object construction — no more telescoping constructors with 10 parameters.

**When to use:**
- Many optional parameters — Email with CC, BCC, attachments, priority, headers
- Immutable objects — all fields set during build, immutable after
- Complex validation — builder validates before build, object always valid

**Trade-off:**
- Readable construction — emailBuilder.to("a@b.com").cc("c@d.com").subject("Hi").build()
- But: verbose for simple objects — 4 fields need 4 methods + builder class
- More code — builder class duplicates fields from main class

**Real-world example:**
Our Email class had 4 constructors — 2 params, 5 params, 8 params, 10 params. Developers passed null for unused params, wrong order caused bugs (subject as body). Implemented Builder: EmailBuilder.with().to().cc().subject().body().build(). Constructor bugs: 10/month → 0. Readability improved — parameters named, order doesn't matter. But added 80 lines of builder code for 10-field class. For simple 3-field DTOs, not worth it.

**Rule of thumb:** Builder for objects with 4+ parameters, especially optional ones. Not for simple DTOs with 2-3 fields. Use when object should be immutable after construction.
