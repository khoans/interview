## 251. What is the Singleton pattern? When would you use it?

**Singleton ensures a class has only one instance and provides global access to it.**

Use it for shared resources where multiple instances would cause problems — configuration, thread pools, database connections.

**When to use:**
- Shared resources — single database connection pool, cache manager
- Coordination across system — event bus, logger, configuration
- Expensive initialization — load once, share everywhere

**Trade-off:**
- Global access — convenient, no dependency injection needed
- But: hidden coupling — tests can't isolate, hard to mock
- Violates Single Responsibility — controls both business logic and lifecycle

**Real-world example:**
Our application had 50+ database connections — each service created its own pool. Database hit max connections (500), new connections rejected. Switched to Singleton connection pool — single pool shared across all services. Connections: 50 → 1 pool of 50. Database errors: 100/day → 0. But unit tests failed — Singleton persisted state between tests. Fixed: reset Singleton in test teardown, or use dependency injection for testability.

**Rule of thumb:** Singleton for truly shared resources (connection pools, caches). Avoid for business logic — use dependency injection instead. Always consider testability — can you reset state between tests?
