## 255. What is the difference between Factory and Abstract Factory?

**Factory creates one type of product. Abstract Factory creates families of related products without specifying concrete classes.**

Factory is for single product variations. Abstract Factory is for product families that must work together.

**Factory:**
- Creates one product type — PaymentProcessor (CreditCard, PayPal, BankTransfer)
- One factory method — createProcessor(type)
- Use case: choose implementation at runtime

**Abstract Factory:**
- Creates product families — UI widgets (Button, Checkbox, Menu) for Windows/Mac/Linux
- Multiple factory methods — createButton(), createCheckbox(), createMenu()
- Use case: ensure compatible products across themes/platforms

**Trade-off:**
- Factory: simple, one decision point — which processor to create
- Abstract Factory: ensures compatibility — Windows button works with Windows checkbox
- But: more complexity — abstract factory + concrete factories + product families

**Real-world example:**
Our e-commerce had Factory for payment processors — worked for single product. Expanded to checkout UI — buttons, forms, modals needed consistent styling (modern, classic, minimal). Factory wasn't enough — button style didn't match form style. Implemented Abstract Factory: ModernFactory creates ModernButton + ModernForm + ModernModal. ClassicFactory creates ClassicButton + ClassicForm + ClassicModal. UI consistency: 100%. But added 6 factory classes — complexity increased.

**Rule of thumb:** Factory for single product variations. Abstract Factory for product families that must be consistent. Don't use Abstract Factory unless you have 2+ related products needing consistent themes.
