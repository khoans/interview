## 260. What is Object Pool pattern?

**Object Pool pattern reuses expensive-to-create objects — borrow from pool, use, return to pool instead of creating new each time.**

It's about amortizing initialization cost across multiple uses — database connections, thread pools, byte buffers.

**When to use:**
- Expensive initialization — database connections (network handshake, authentication)
- Frequent create/destroy — short-lived objects in tight loops
- Resource limits — max 100 connections, pool enforces limit

**Trade-off:**
- Performance — reuse avoids initialization cost (100ms → 1ms)
- Resource control — pool size limits concurrent usage
- But: complexity — pool management, leak detection, stale object handling
- State leakage — borrowed object may have state from previous user

**Real-world example:**
Our API created new database connections per request — 50ms connection overhead. 1000 requests/second = 50 seconds just connecting. Implemented connection pool (HikariCP): 50 connections, reused. Connection time: 50ms → 1ms (borrow from pool). Throughput: 1000 → 10,000 requests/second. But connection leaks — developers forgot to return connections. Pool exhausted after 1 hour. Added try-with-resources, leak detection (log if not returned in 30 seconds). Zero leaks since.

**Rule of thumb:** Object Pool for expensive resources (database connections, threads). Not for cheap objects (String, simple DTOs) — pooling overhead exceeds creation cost. Always implement proper return mechanism — try-finally or try-with-resources.
