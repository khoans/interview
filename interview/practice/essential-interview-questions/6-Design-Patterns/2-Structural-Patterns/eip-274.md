## 274. What is the Front Controller pattern?

**Front Controller provides single entry point for all requests — centralizes request handling, routing, cross-cutting concerns.**

It's the foundation of MVC frameworks — one servlet handles all URLs, dispatches to appropriate handlers.

**When to use:**
- Centralized processing — authentication, logging, routing for all requests
- Consistent handling — same pipeline for every request
- Framework development — build request processing pipeline

**Trade-off:**
- Centralized control — one place for security, logging, routing
- But: single point of failure — Front Controller bug affects all requests
- Can become god class — handles everything, hard to maintain

**Real-world example:**
Our web app had 20 servlets — each handled authentication, logging, error handling independently. Inconsistent: 15 servlets logged requests, 5 didn't; 10 checked authentication, 10 assumed authenticated. Security breach: unauthenticated access to admin servlet (missing auth check). Implemented Front Controller (Spring DispatcherServlet): single entry point, authentication filter, logging filter, routing to handlers. Security: 100% authenticated. Logging: 100% coverage. But Front Controller became bottleneck — 50ms overhead per request. Optimized: async processing, connection pooling. Overhead: 50ms → 5ms.

**Rule of thumb:** Front Controller for web applications needing consistent request handling. Use framework implementations (Spring DispatcherServlet, Struts ActionServlet) — don't build your own unless necessary. Keep filters thin — single responsibility each.
