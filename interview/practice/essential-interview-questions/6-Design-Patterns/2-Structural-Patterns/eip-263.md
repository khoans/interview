## 263. What is the Adapter pattern? Give an example.

**Adapter pattern converts one interface to another — makes incompatible interfaces work together.**

It's a wrapper that translates calls between different APIs without changing the underlying code.

**When to use:**
- Third-party integration — their API doesn't match your interface
- Legacy system modernization — new code expects new interface, old code can't change
- Multiple implementations — unify different APIs behind common interface

**Trade-off:**
- Integration without modification — wrap, don't rewrite
- But: extra layer — slight performance overhead, debugging complexity
- Can hide problems — adapter masks fundamental API mismatches

**Real-world example:**
Our payment system used Stripe API. Business required PayPal support — PayPal API completely different (method names, authentication, response format). Created PaymentAdapter: our code calls processPayment(), adapter translates to PayPal's DoDirectPayment(). Integration time: 2 days vs. 2 weeks to refactor entire system. But adapter added 15ms latency (translation overhead). Acceptable for PayPal integration speed.

**Rule of thumb:** Adapter for integrating incompatible systems. Not for new development — design consistent interfaces from start. Keep adapters thin — translation only, no business logic.
