## 273. What is the difference between Proxy and Decorator?

**Proxy controls access to object — adds behavior before delegating. Decorator adds behavior to object — enhances functionality.**

Same structure (both wrap objects), different intent.

**Proxy:**
- Intent: control access — lazy loading, permissions, caching
- Relationship: proxy represents real object — one-to-one
- Example: ProtectionProxy checks permission before allowing operation

**Decorator:**
- Intent: add behavior — new functionality, cross-cutting concerns
- Relationship: decorator enhances object — can stack multiple
- Example: LoggingDecorator logs, CompressionDecorator compresses

**Trade-off:**
- Proxy: access control without changing real object — but can block legitimate access
- Decorator: flexible behavior composition — but stack depth makes debugging hard

**Real-world example:**
Our document service had both. Proxy: DocumentProxy checked read/write permissions before delegating to real Document. Decorator: DocumentDecorator added versioning (save old version before update), auditing (log all changes), caching (cache read results). Same structure — both wrapped Document. Different purpose: Proxy (security), Decorator (enhancement). Used together: Document → Proxy (access control) → VersioningDecorator → AuditingDecorator → CachingDecorator. Clear separation: security in proxy, features in decorators.

**Rule of thumb:** Proxy for access control (should this call proceed?). Decorator for behavior enhancement (what else should happen?). Can stack both — proxy closest to real object, decorators outside.
