## 270. What is the Bridge pattern?

**Bridge pattern separates abstraction from implementation — both can vary independently without subclass explosion.**

It's about decoupling two dimensions of variation — shape (circle, square) and color (red, blue) without 6 subclasses.

**When to use:**
- Two independent variations — UI (Windows/Mac) × Component (Button/Checkbox)
- Runtime switching — change implementation without changing abstraction
- Avoid subclass explosion — 10 shapes × 10 colors = 100 subclasses without bridge

**Trade-off:**
- Independent evolution — add shapes without touching colors, vice versa
- But: more classes — abstraction + implementation hierarchies
- Indirection — one extra method call through implementation reference

**Real-world example:**
Our charting library had Chart type (Bar, Line, Pie) and Renderer type (SVG, Canvas, PDF). Without bridge: 9 subclasses (BarChartSVG, BarChartCanvas, BarChartPDF, LineChartSVG...). Implemented Bridge: Chart holds reference to Renderer. BarChart uses any Renderer. Adding new renderer: 1 class vs. 10 chart subclasses. But debugging chart rendering required understanding two class hierarchies — added factory methods (Chart.withSVG(), Chart.withCanvas()) for clarity.

**Rule of thumb:** Bridge when two dimensions vary independently. Not for single variation — use inheritance or strategy. Document which is abstraction (what) vs. implementation (how).
