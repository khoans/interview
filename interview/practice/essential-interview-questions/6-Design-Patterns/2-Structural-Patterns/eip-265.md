## 265. What is the difference between Decorator and Inheritance?

**Inheritance adds behavior at compile-time — fixed at class definition. Decorator adds behavior at runtime — dynamic, composable.**

Inheritance is "is-a" relationship. Decorator is "has-a" relationship (wrapping).

**Inheritance:**
- Static — behavior fixed when class is written
- Deep hierarchies — Animal → Mammal → Dog → GuardDog → AttackGuardDog
- Tight coupling — change parent, all children affected

**Decorator:**
- Dynamic — behavior chosen at runtime
- Flat structure — Dog + AttackBehavior + GuardBehavior
- Loose coupling — add/remove decorators independently

**Trade-off:**
- Inheritance: simple, performant — but inflexible, fragile base class problem
- Decorator: flexible, composable — but more objects, debugging complexity

**Real-world example:**
Our notification system started with inheritance — EmailNotification, SMSNotification, EncryptedEmailNotification, EncryptedSMSNotification, PriorityEmailNotification... 15 subclasses for 3 features. Adding logging required 5 new subclasses. Refactored to decorators: Notification → encrypt() → log() → prioritize(). Any combination: 3 decorators vs. 15 subclasses. Adding logging: 1 decorator class vs. 5 subclasses. But stack traces showed 5 decorator calls — added correlation IDs for tracing.

**Rule of thumb:** Inheritance for core "is-a" relationships (Dog is Animal). Decorator for optional "can-do" behaviors (can encrypt, can log). Prefer composition over inheritance — decorators are composition.
