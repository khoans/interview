## 267. What is the Proxy pattern?

**Proxy pattern provides surrogate or placeholder for another object — control access to real object.**

It's about indirection — proxy intercepts calls, adds behavior before/after delegating to real object.

**When to use:**
- Lazy initialization — create expensive object only when needed
- Access control — check permissions before allowing operation
- Remote services — local proxy represents remote object (RPC, web services)
- Caching — proxy returns cached result if available

**Trade-off:**
- Controlled access — add behavior without changing real object
- But: extra layer — performance overhead, debugging complexity
- Can become bottleneck — all calls go through proxy

**Real-world example:**
Our image gallery loaded full-resolution images on page load — 50 images × 2MB = 100MB, page load 10 seconds. Implemented virtual proxy: load thumbnail first, full image on click. Page load: 10 seconds → 500ms (thumbnails only). Bandwidth: 100MB → 5MB (users click 2-3 images). But proxy added complexity — had to handle loading states, failed loads, retry logic. Added progress indicator for large images.

**Rule of thumb:** Proxy for access control, lazy loading, caching. Not for simple objects — overhead exceeds benefit. Document when proxy creates real object (first access vs. background).
