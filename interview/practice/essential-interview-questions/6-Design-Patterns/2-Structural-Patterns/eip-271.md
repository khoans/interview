## 271. What is the Flyweight pattern?

**Flyweight pattern shares common state between objects — reduces memory by extracting intrinsic (shared) state from extrinsic (unique) state.**

It's about memory optimization — 10,000 objects sharing 100 unique states instead of duplicating.

**When to use:**
- Many similar objects — text characters, game entities, map markers
- Memory constraints — too many objects causing OOM or GC pressure
- State can be separated — some state shared, some unique per object

**Trade-off:**
- Memory reduction — 10,000 objects → 100 shared + 10,000 extrinsic references
- But: complexity — separate intrinsic vs. extrinsic state management
- CPU overhead — look up shared state on each operation

**Real-world example:**
Our text editor created Character object per character — 100,000 characters × 100 bytes = 10MB. Most characters shared font, size, color (intrinsic), only position and char value unique (extrinsic). Implemented Flyweight: CharacterFactory returns shared Character for each (font, size, color) combination. Memory: 10MB → 500KB (100 unique styles × 100 bytes + 100,000 × 10 bytes extrinsic). But rendering slowed 20% — lookup shared state per character. Added LRU cache for flyweight lookup — back to original speed.

**Rule of thumb:** Flyweight for memory-constrained scenarios with many similar objects. Profile first — don't optimize prematurely. Ensure intrinsic state is truly immutable — shared mutable state causes bugs.
