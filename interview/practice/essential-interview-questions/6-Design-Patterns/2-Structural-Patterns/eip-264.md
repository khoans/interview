## 264. What is the Decorator pattern? Give an example.

**Decorator pattern adds behavior to objects dynamically — wrap object with additional functionality without modifying the class.**

It's about runtime composition over compile-time inheritance — stack behaviors flexibly.

**When to use:**
- Cross-cutting concerns — logging, caching, authentication around core logic
- Optional features — base object + optional decorators (compression, encryption)
- Avoid subclass explosion — 10 features × 10 options = 100 subclasses without decorators

**Trade-off:**
- Flexible composition — mix and match decorators at runtime
- But: debugging complexity — stack trace shows 5 decorator layers
- Object identity — decorated object is different instance (== comparison fails)

**Real-world example:**
Our API response needed optional compression, encryption, caching. Without decorators: 8 subclasses (Base, Compressed, Encrypted, Cached, Compressed+Encrypted, etc.). With decorators: Response → compress() → encrypt() → cache(). Any combination possible. Added compression feature: 1 new class vs. 4 new subclasses. But debugging response issues required unwrapping 3 decorator layers — added toString() to each decorator showing wrapped type.

**Rule of thumb:** Decorator for optional, stackable behaviors. Not for required features — use inheritance instead. Document decorator order matters — compress then encrypt vs. encrypt then compress.
