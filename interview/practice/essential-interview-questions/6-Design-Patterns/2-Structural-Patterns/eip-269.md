## 269. What is the Composite pattern?

**Composite pattern treats individual objects and compositions uniformly — tree structure where leaf and composite have same interface.**

It's about part-whole hierarchies — file/folder, menu/menu item, organization/department.

**When to use:**
- Tree structures — hierarchical data (org chart, file system)
- Uniform treatment — client doesn't care if it's leaf or composite
- Recursive operations — calculate size, render, validate entire tree

**Trade-off:**
- Uniform interface — same methods for leaf and composite
- But: leaf has methods that don't make sense (folder operations on file)
- Complexity — composite must manage children, iterate, aggregate results

**Real-world example:**
Our file system had File and Folder classes — different interfaces. Code had if-else everywhere: if File, get size; if Folder, iterate children, sum sizes. 50+ locations with type checks. Implemented Composite: FileSystemComponent with getSize(), both File and Folder implement. Folder.getSize() sums children. Type checks: 50 → 0. But File had to implement add/remove (folder operations) — threw UnsupportedOperationException. Trade-off accepted for uniform interface.

**Rule of thumb:** Composite for tree structures with uniform operations. Not for flat structures — over-engineering. Consider safety (throw for unsupported operations) vs. transparency (all methods on all nodes).
