## 221. What is Docker volume?

**Docker volume is persistent storage for containers — data survives container restarts, deletions, and recreations.**

Containers are ephemeral — filesystem changes are lost on restart. Volumes solve this for databases, logs, uploaded files.

**Volume types:**
- **Named volumes** — Docker-managed (docker volume create), best for databases
- **Bind mounts** — host path mapped to container — good for development (code sync)
- **Tmpfs mounts** — in-memory storage — fast, temporary data

**Trade-off:**
- Data persistence — database survives container deletion
- But: backup complexity — volumes are outside container, separate backup needed
- Portability — bind mounts tie containers to specific host paths

**Real-world example:**
Our PostgreSQL container stored data in container filesystem. Container crashed, recreated — all data lost. 2 hours to restore from database backup. Added Docker volume: postgres_data:/var/lib/postgresql/data. Container recreated — data persisted, zero data loss. But backup was manual — docker run with volume mount, pg_dump. Automated: nightly backup script, dump to S3, 7-day retention. Zero data loss since 18 months.

**Rule of thumb:** Volumes for all stateful data — databases, caches, uploaded files. Named volumes for production, bind mounts for local development (hot reload). Backup volumes separately from containers — automate dumps to external storage.
