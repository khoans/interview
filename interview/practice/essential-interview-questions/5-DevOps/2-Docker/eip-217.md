## 217. What is a Dockerfile?

**A Dockerfile is a text file with instructions to build a Docker image — step-by-step recipe for assembling layers.**

It's infrastructure as code — version controlled, reviewed, tested like application code.

**Key instructions:**
- FROM — base image (ubuntu, alpine, openjdk)
- RUN — execute commands (install packages, compile code)
- COPY/ADD — copy files into image
- EXPOSE — document which ports container listens on
- CMD/ENTRYPOINT — default command when container starts

**Trade-off:**
- Reproducible builds — same Dockerfile = same image every time
- But: instruction order matters — affects layer caching, build speed
- Security — running as root, hardcoded secrets are common mistakes

**Real-world example:**
Our Dockerfile had 20 RUN instructions — each created a layer, image had 25 layers. Build time: 10 minutes (no cache). Image size: 800MB. Optimized: combined RUN instructions with &&, used multi-stage build. Layers: 25 → 8. Build time: 10 min → 2 min (cached layers). Image size: 800MB → 150MB. But Dockerfile became harder to read — long RUN commands. Added comments, split into logical sections with comments.

**Rule of thumb:** Order instructions by change frequency — dependencies first (rarely change), app code last (changes often). Combine RUN commands with &&. Use specific versions (python:3.9-slim), not latest. Never store secrets in Dockerfile.
