## 222. What is Docker network?

**Docker network enables container-to-container communication — isolated network stacks, DNS-based service discovery.**

Containers on same network can communicate by name — no hardcoded IPs.

**Network types:**
- **Bridge** — default, containers on same host communicate via Docker bridge
- **Host** — container shares host network — no isolation, better performance
- **Overlay** — multi-host networking — containers across hosts communicate
- **None** — no network — isolated, security-sensitive workloads

**Trade-off:**
- Service discovery — containers find each other by name, not IP
- Isolation — separate networks for different environments (dev, prod)
- But: network debugging complexity — DNS, routing, firewall rules
- Performance — bridge networking adds slight latency

**Real-world example:**
Our app had API container and PostgreSQL container. Initially: API connected to host:5432 — required host PostgreSQL, not containerized. Moved PostgreSQL to container: API couldn't connect (different networks). Created docker network: app-network, attached both containers. API connected to postgres:5432 (DNS resolution). But developer added new service — forgot to attach to network, connection timeouts. Added network requirement to docker-compose.yml — all services auto-attached. Zero network issues since.

**Rule of thumb:** Create custom bridge network per application — default bridge has no DNS. Use service names, not IPs. For production: overlay networks (Swarm) or CNI plugins (Kubernetes). Isolate sensitive services (databases) on separate networks.
