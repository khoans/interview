## 216. What is the difference between image and container?

**Image is the template (read-only). Container is the running instance (read-write, ephemeral).**

Image is to container what a class is to an object — one defines, the other executes.

**Image:**
- Read-only — built once, never modified
- Stored in registry — Docker Hub, ECR, GCR
- Versioned — myapp:v1.0, myapp:v1.1
- Reusable — same image creates multiple containers

**Container:**
- Read-write layer — changes during execution (not persisted)
- Running process — has PID, network, filesystem
- Ephemeral — deleted, recreated freely
- One-to-many — one image → many containers

**Trade-off:**
- Immutable infrastructure — replace containers, don't modify them
- But: debugging requires exec into container — changes lost on restart
- Data persistence requires volumes — container filesystem is temporary

**Real-world example:**
Our team confused images and containers — SSH'd into containers, installed hotfixes manually. Container restarted — hotfixes gone, production broken. Fixed: immutable containers — no SSH, no manual changes. Bug fix: rebuild image, redeploy container. Deployment time: instant (already had CI/CD). Zero "where's my fix" incidents since. But required culture change — developers couldn't "just fix" production directly.

**Rule of thumb:** Images are immutable artifacts — version, store, audit. Containers are disposable — replace, don't repair. For debugging: exec temporarily, but never rely on container modifications.
