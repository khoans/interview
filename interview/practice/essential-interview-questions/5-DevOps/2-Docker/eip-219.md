## 219. What is multi-stage build?

**Multi-stage build uses multiple FROM statements in Dockerfile — build artifacts in early stages, copy only runtime necessities to final stage.**

It's the single best technique for reducing image size — build tools stay in build stage, not production image.

**How it works:**
- Stage 1 (build): full SDK, build tools, source code — compile, test, package
- Stage 2 (runtime): minimal JRE/runtime, copy built artifact from stage 1
- Final image: only runtime + artifact — no build tools, no source code

**Trade-off:**
- Dramatically smaller images — 1GB → 100MB typical
- Better security — fewer packages = fewer vulnerabilities
- But: build takes longer — two stages to build
- Debugging harder — no build tools in final image

**Real-world example:**
Our Java microservice: single-stage Dockerfile with OpenJDK 11. Image: 700MB (JDK + app). Deployed 20 microservices: 14GB total disk space. Multi-stage build: stage 1 (Maven build with JDK), stage 2 (JRE only, copy JAR). Image: 700MB → 180MB. Total storage: 14GB → 3.6GB. Deployment time: 5 min → 1 min (smaller images pull faster). But build time increased: 3 min → 5 min (two stages). Trade-off accepted — deployment speed matters more than build speed.

**Rule of thumb:** Multi-stage for all compiled languages (Java, Go, .NET, Node with native modules). Name stages (AS builder, AS runtime) for clarity. Copy only production artifacts — exclude tests, source maps unless needed.
