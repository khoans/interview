## 214. What is a Docker image?

**A Docker image is a read-only template containing application code, runtime, libraries, and configuration — the blueprint for creating containers.**

It's a layered filesystem — each instruction in Dockerfile creates a layer, cached and reused across images.

**Key characteristics:**
- Immutable — once built, image doesn't change (tag new version for changes)
- Layered — base layer (OS), middle layers (dependencies), top layer (app)
- Versioned — tagged with version numbers (myapp:1.0, myapp:latest)
- Registry-stored — pushed to Docker Hub, ECR, GCR for distribution

**Trade-off:**
- Layer caching — rebuilds are fast if only app code changes
- But: large images — 1GB+ images slow to pull, storage expensive
- Security — outdated base images have vulnerabilities

**Real-world example:**
Our Java app Docker image was 1.2GB — full JDK, all dependencies, build tools. Pull time: 5 minutes on slow networks. Storage: 50 versions × 1.2GB = 60GB. Optimized: multi-stage build, JRE instead of JDK, slim base image. Image size: 1.2GB → 150MB. Pull time: 5 min → 30 seconds. Storage: 60GB → 7.5GB. But optimization took 2 days — worth it for teams pulling images daily.

**Rule of thumb:** Keep images under 200MB — use slim/alpine base images, multi-stage builds. Tag with versions, not just "latest". Scan images weekly for vulnerabilities.
