## 218. How do you optimize a Dockerfile?

**Optimize Dockerfiles for smaller images, faster builds, and better security — multi-stage builds, layer caching, minimal base images.**

The goal: smallest image that runs your application correctly.

**Optimization techniques:**
- **Multi-stage builds** — build in one stage, copy artifacts to minimal runtime stage
- **Layer caching** — order instructions: OS packages → dependencies → source code
- **Minimal base images** — alpine (5MB), slim variants, distroless (no shell)
- **Combine RUN commands** — reduce layers, clean up in same layer (rm -rf /var/cache)
- **.dockerignore** — exclude node_modules, .git, build artifacts

**Trade-off:**
- Smaller images — faster pulls, less storage, fewer vulnerabilities
- But: debugging harder — alpine has no bash, distroless has no shell
- Build complexity — multi-stage builds harder to understand

**Real-world example:**
Our Node.js app Dockerfile: FROM node:14, copy everything, npm install. Image: 900MB, build: 8 minutes. Vulnerabilities: 50+ (full OS included). Optimized: multi-stage — build in node:14, runtime in node:14-alpine. Copy only node_modules (production), exclude dev dependencies. Image: 900MB → 120MB. Build: 8 min → 2 min. Vulnerabilities: 50+ → 5. But alpine uses musl libc — one native module failed. Switched to node:14-slim (Debian-based). Image: 150MB, zero compatibility issues.

**Rule of thumb:** Multi-stage for compiled languages (Java, Go, Node with native modules). Alpine for simple apps, slim for compatibility. Always use .dockerignore — exclude .git, node_modules, logs.
