## 223. What is the difference between COPY and ADD in Dockerfile?

**Both copy files into Docker images, but ADD has extra features (URL download, tar extraction) that cause unexpected behavior. COPY is simpler and preferred.**

Use COPY unless you specifically need ADD's features.

**COPY:**
- Copies files from host to image
- Simple, predictable — what you see is what you get
- Recommended for 95% of use cases

**ADD:**
- Copies files from host to image
- Can download from URLs — ADD https://example.com/file.tar.gz /app/
- Auto-extracts tarballs — ADD app.tar.gz /app/ extracts automatically
- Less predictable — behavior changes based on file type

**Trade-off:**
- ADD seems convenient — download and extract in one instruction
- But: hidden behavior — tar extraction not obvious to readers
- Security — downloading from URLs in build is non-reproducible
- Best practice — COPY for clarity, separate RUN for downloads

**Real-world example:**
Our Dockerfile used ADD for config files — ADD config.tar.gz /etc/app/. Worked until someone updated the tarball without changing filename — Docker cache didn't invalidate, old config used in production. Incident: 4 hours to debug (config looked correct in repo, but not in image). Fixed: use COPY for all files, explicit RUN curl/wget for downloads, explicit RUN tar -xzf for extraction. Build time: +30 seconds. Debugging time: 4 hours → 10 minutes.

**Rule of thumb:** Always use COPY unless you need ADD's tar extraction or URL download. For downloads: RUN curl + COPY — explicit, reproducible. For tarballs: COPY + RUN tar -xzf — clear what's happening.
