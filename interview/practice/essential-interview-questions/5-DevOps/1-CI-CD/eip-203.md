## 203. What is continuous delivery?

**Continuous Delivery ensures code is always in a deployable state — every commit that passes tests can be deployed to production with a manual approval.**

It's about making deployments boring — predictable, low-risk, one-click operations.

**Key characteristics:**
- Automated testing — unit, integration, acceptance, performance tests
- Automated deployment to staging — production-like environment
- Manual approval for production — business decides when to deploy
- Deployment in minutes — not hours of manual steps

**Trade-off:**
- Deploy on demand — no scheduling, no deployment windows
- But: requires discipline — no skipping tests, no manual hotfixes
- Infrastructure cost — staging environment must mirror production

**Real-world example:**
Our e-commerce site deployed every 2 weeks — Friday night deployments, 4-hour windows, on-call engineers. Business wanted to deploy before Black Friday — couldn't, deployment freeze. Implemented continuous delivery: automated tests, one-click deployment to production. Deployed 3 times/week, 15 minutes each. Black Friday deployment: scheduled for 10 AM, deployed in 5 minutes during lunch. Zero downtime. But required 3 months of test automation — 80% code coverage, performance tests, smoke tests.

**Rule of thumb:** Continuous delivery for business-critical apps requiring approval gates. Regulated industries (finance, healthcare) need manual approval for compliance. Automate everything except the deploy button.
