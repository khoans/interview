## 210. What is blue-green deployment?

**Blue-green deployment runs two identical production environments — one serving traffic (blue), one idle (green). Deploy to idle, test, switch traffic.**

It's zero-downtime deployments with instant rollback — switch back to blue if green fails.

**How it works:**
- Blue environment serves production traffic
- Deploy new version to green (idle) environment
- Test green environment (smoke tests, integration tests)
- Switch load balancer — all traffic goes to green
- Blue becomes idle — rollback target if needed

**Trade-off:**
- Zero downtime — users experience no interruption
- Instant rollback — switch back to blue in seconds
- But: 2x infrastructure cost — two full production environments

**Real-world example:**
Our e-commerce site had 30-minute deployment windows — maintenance mode, users saw "site down". Implemented blue-green: AWS ALB routing to blue (v1) or green (v2). Deploy v2 to green, test, switch ALB. Downtime: 30 minutes → 0 seconds. Rollback: 30 minutes (redeploy) → 10 seconds (ALB switch). But infrastructure cost doubled — 2× EC2, 2× RDS. Optimized: green environment scaled to minimum during idle (50% capacity), scale up after switch. Cost: 2x → 1.5x.

**Rule of thumb:** Blue-green for zero-downtime requirements, critical applications. Use auto-scaling to reduce idle environment cost. Database migrations must be backward-compatible — both versions run simultaneously during switch.
