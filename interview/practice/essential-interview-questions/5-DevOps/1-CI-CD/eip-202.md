## 202. What is continuous integration?

**Continuous Integration is the practice of merging code changes frequently (daily) with automated builds and tests — catching integration issues early.**

It's about finding bugs within minutes of introduction, not weeks later in production.

**Key practices:**
- Commit to main branch daily — no long-lived feature branches
- Automated build on every commit — compile, lint, package
- Automated tests on every commit — unit, integration, smoke tests
- Fast feedback — developers know within 10 minutes if they broke something

**Trade-off:**
- Early bug detection — fix bugs when context is fresh, 10x cheaper
- But: test maintenance — 500+ tests to maintain, flaky tests erode trust
- Build time — pipelines slow down as codebase grows

**Real-world example:**
Our team had 2-week sprint cycles — feature branches lived 10 days, merge day was "integration hell" — 2 days resolving conflicts, fixing broken tests. Implemented CI: daily merges, Jenkins ran 200 tests in 8 minutes. Merge conflicts: 2 days → 10 minutes. Bug detection: 10 days → 8 minutes. But test suite grew to 500 tests, build time 8 → 25 minutes. Developers waited too long. Split tests: fast tests (unit) run on every commit (5 min), slow tests (integration) run nightly.

**Rule of thumb:** CI is mandatory for team development. Keep build under 10 minutes — split fast/slow tests. Fix broken builds immediately — don't let red builds persist.
