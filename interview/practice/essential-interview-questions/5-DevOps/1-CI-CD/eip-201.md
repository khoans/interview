## 201. What is CI/CD?

**CI/CD is Continuous Integration and Continuous Delivery/Deployment — automated pipelines that build, test, and deploy code changes frequently and reliably.**

It's about reducing deployment risk through automation and small, incremental changes.

**Key components:**
- **CI** — developers merge code daily, automated builds and tests run on every commit
- **CD (Delivery)** — code is always deployable, manual approval triggers deployment
- **CD (Deployment)** — code deploys to production automatically after passing tests

=> CI is about catching problems early — every commit triggers an automated build and test run so integration issues surface within minutes, not days before release.

=>   CD is about making releases boring — the pipeline automates everything from build to deployment so that shipping to production is a routine, low-risk event rather than a stressful 8-hour window.


**Trade-off:**
- Faster, safer deployments — small changes, automated testing, quick rollback
- But: pipeline maintenance — infrastructure as code, test flakiness, environment management
- Initial setup cost — 2-4 weeks to build robust pipeline

**Real-world example:**
Our team deployed manually once per month — 8-hour deployment windows, 30% rollback rate, production outages. Implemented CI/CD with Jenkins: automated builds, 500 automated tests, one-click deployment. Deployment frequency: monthly → daily. Rollback rate: 30% → 5%. Deployment time: 8 hours → 30 minutes. But pipeline took 3 weeks to build, 2 weeks to stabilize flaky tests. Worth it — zero weekend deployments since.

**Rule of thumb:** CI is non-negotiable — run tests on every commit. CD depends on risk — delivery (manual approval) for regulated industries, deployment (automatic) for low-risk web apps.
