## 209. What are the stages in a typical CI/CD pipeline?

**A typical CI/CD pipeline has 5-7 stages: checkout, build, test, security scan, package, deploy to staging, deploy to production.**

Each stage gates the next — failure stops the pipeline, preventing bad code from progressing.

**Common stages:**
1. **Checkout** — pull code from Git, checkout commit/branch
2. **Build** — compile code, resolve dependencies, create artifacts
3. **Test** — unit tests, integration tests, code coverage
4. **Security scan** — SAST, dependency vulnerabilities, secrets detection
5. **Package** — create Docker image, version artifact, push to registry
6. **Deploy to staging** — deploy to production-like environment
7. **Deploy to production** — manual approval or automatic (continuous deployment)

**Trade-off:**
- More stages = more safety — catch bugs, security issues before production
- But: longer pipelines — 50+ minutes discourages frequent commits
- Parallelization helps — run independent stages concurrently

**Real-world example:**
Our pipeline had 3 stages: build, test, deploy. Production bugs slipped through — security vulnerabilities, performance regressions. Added stages: security scan (SonarQube), performance tests, staging deployment. Pipeline time: 15 → 45 minutes. Production bugs: 10/month → 2/month. But 45-minute feedback was too long — developers committed less frequently. Optimized: parallelized tests (unit + security + lint), cached dependencies. Pipeline time: 45 → 18 minutes. Commit frequency: 3/day → 10/day.

**Rule of thumb:** Minimum 4 stages: build, test, security, deploy. Keep total time under 20 minutes — parallelize where possible. Fail fast — run quick tests first, slow tests later.
