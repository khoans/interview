## 204. What is continuous deployment?

**Continuous Deployment takes delivery further — every commit that passes tests deploys to production automatically, no manual approval.**

It's the ultimate automation — developers push code, users get features within minutes.

**Key characteristics:**
- No manual approval — tests are the gatekeeper
- Feature flags — deploy code, enable for specific users
- Automated rollback — if metrics degrade, revert automatically
- Requires mature testing — 90%+ test coverage, comprehensive monitoring

**Trade-off:**
- Fastest time to market — features in production same day
- But: requires exceptional test coverage — bugs reach production if tests miss them
- Not for all domains — regulated industries require manual approval

**Real-world example:**
Our SaaS startup used continuous delivery — manual approval bottlenecked deployments (CEO traveled, couldn't approve). Switched to continuous deployment: 500 tests, automated canary analysis (error rates, latency). Deployment frequency: 3/week → 20/week. Feature lead time: 2 days → 2 hours. But first week: bug slipped through tests, caused 500 errors before detection (15 minutes). Added automated rollback on error spike. Zero customer-impacting incidents since 6 months.

**Rule of thumb:** Continuous deployment for web/SaaS with mature testing and monitoring. Not for embedded systems, medical devices, financial trading (regulatory requirements). Start with continuous delivery, evolve to deployment when tests are trustworthy.
