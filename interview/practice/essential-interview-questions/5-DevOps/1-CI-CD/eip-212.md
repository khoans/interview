## 212. What is rolling deployment?

**Rolling deployment replaces instances gradually — update 1 of 10 instances, wait for health check, update next, until all instances are new version.**

It's the default Kubernetes deployment strategy — no downtime, no extra infrastructure cost.

**How it works:**
- Start with 10 instances of v1
- Terminate 1 instance, start 1 instance of v2
- Wait for v2 to pass health checks
- Repeat until all 10 instances are v2
- Traffic gradually shifts from v1 to v2

**Trade-off:**
- Zero downtime — some instances always serving traffic
- No extra cost — same number of instances throughout deployment
- But: slow deployment — 10 instances × 2 minutes each = 20 minutes
- Mixed versions — during deployment, users may hit v1 or v2

**Real-world example:**
Our API had 20 instances behind load balancer. Blue-green was too expensive (2x infrastructure). Switched to rolling deployment: Kubernetes updated 1 pod at a time, 30-second health check between updates. Deployment time: 20 instances × 30 seconds = 10 minutes. Zero downtime. But during deployment, users reported inconsistent behavior — v1 and v2 had different API responses. Fixed: made API changes backward-compatible, deployed v2 with feature flag. Zero inconsistency complaints since.

**Rule of thumb:** Rolling deployment for stateless services with backward-compatible changes. Not for breaking API changes — users may hit old version during rollout. Use maxSurge (extra instances) to speed up deployment without downtime.
