## 208. What is GitLab CI?

**GitLab CI is GitLab's integrated CI/CD platform — pipelines defined in .gitlab-ci.yml, runners execute jobs, integrated with GitLab's DevOps suite.**

It's CI/CD plus everything else — issue tracking, code review, container registry, monitoring — all in one platform.

**Key characteristics:**
- YAML configuration — .gitlab-ci.yml defines stages, jobs, scripts
- GitLab Runners — self-hosted or shared runners
- Auto DevOps — pre-configured pipelines for common frameworks
- Integrated features — container registry, Kubernetes integration, monitoring

**Trade-off:**
- All-in-one platform — single tool for entire SDLC
- But: monolithic — if GitLab goes down, everything (repo + CI + registry) unavailable
- Resource intensive — self-hosted GitLab requires 8GB+ RAM

**Real-world example:**
Our startup used GitHub + Jenkins + Docker Hub — 3 tools, 3 logins, integration glue. Migrated to GitLab: repo, CI, registry, Kubernetes — single platform. Developer onboarding: 2 days → 2 hours. But GitLab server crashed during peak usage — 8GB RAM insufficient for 50 developers. Upgraded to 16GB, added Redis cache. Zero crashes since. CI/CD costs: Jenkins ($200/month EC2) → GitLab ($0, included with platform).

**Rule of thumb:** GitLab CI for teams wanting all-in-one DevOps platform. Self-hosted for control, GitLab.com for managed service. Size infrastructure properly — 16GB RAM minimum for 50+ developers.
