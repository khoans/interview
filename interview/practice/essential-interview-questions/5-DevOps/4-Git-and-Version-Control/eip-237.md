## 237. What is trunk-based development?

**Trunk-based development has developers merge small changes to main (trunk) multiple times daily — no long-lived feature branches.**

It's the workflow behind continuous deployment — features always deployable, no integration hell.

**Key practices:**
- Branch from trunk, merge back within 24 hours
- Small commits — one feature per commit, or partial feature
- Feature flags — incomplete features hidden behind flags
- No code freeze — trunk always deployable

**Trade-off:**
- Fast integration — no merge conflicts, features tested together
- Continuous deployment — trunk always ready to release
- But: discipline required — no "almost done" branches
- Feature flags complexity — manage flags, remove after launch

**Real-world example:**
Our startup used feature branches — 2-week cycles, merge conflicts, deployment anxiety. Switched to trunk-based: branch lifetime max 1 day, feature flags for incomplete work. Deployment frequency: 2/week → 20/day. Merge conflicts: weekly → never. But first week: developer merged half-built feature, flag didn't work, users saw broken UI. Fixed: mandatory code review + automated flag tests. Zero broken features since 12 months.

**Rule of thumb:** Trunk-based for web/SaaS with CI/CD — fast feedback, continuous deployment. Requires discipline — max 1-day branches, feature flags, code review. Not for regulated industries requiring approval gates — use Git flow instead.
