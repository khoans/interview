## 235. What is Git flow?

**Git flow is a branching model with feature branches, develop branch, release branches, and hotfix branches — structured workflow for versioned releases.**

It's designed for release-driven development with scheduled releases, not continuous deployment.

**Branch structure:**
- **main/master** — production-ready code, tagged releases
- **develop** — integration branch for features
- **feature branches** — branch from develop, merge back when complete
- **release branches** — stabilize before production, bug fixes only
- **hotfix branches** — branch from main, fix production, merge to main and develop

**Trade-off:**
- Clear structure — everyone knows where to branch, when to merge
- But: merge complexity — long-lived branches, integration hell
- Slow releases — features wait for release window, not deployable independently

**Real-world example:**
Our enterprise software used Git flow — 2-week release cycles, features merged to develop, release branch created, 3 days stabilization, deploy. Worked for quarterly releases. Startup adopted same model — features sat in develop 2 weeks before release. Business lost deals waiting for features. Switched to trunk-based development — features merged daily, deployable anytime. Release cycle: 2 weeks → same day. But Git flow still used for mobile apps — App Store releases benefit from release branches, stabilization.

**Rule of thumb:** Git flow for scheduled releases (enterprise software, mobile apps with review). Trunk-based for web/SaaS with continuous deployment. Don't use Git flow "because it's best practice" — match workflow to deployment model.
