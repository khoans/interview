## 236. What is feature branch workflow?

**Feature branch workflow creates a branch per feature — develop in isolation, merge via pull request when complete.**

It's the most common Git workflow — simple, intuitive, works for teams of any size.

**How it works:**
- Branch from main: `git checkout -b feature/user-auth`
- Develop feature — commit locally, push to remote
- Pull request — code review, automated tests
- Merge to main — feature complete, deployable

**Trade-off:**
- Isolation — features don't interfere with each other
- Code review — pull requests enforce peer review
- But: integration delays — long-lived branches diverge from main
- Merge conflicts — multiple features conflict on merge

**Real-world example:**
Our team used feature branches — average branch lifetime 3 days. Merge day: 5 features merged, 3 had conflicts, 2 broke tests (worked on branch, not main). Integration time: 4 hours. Switched to short-lived feature branches — max 1 day, rebase on main daily. Merge conflicts: 3 → 0. Broken builds: 2 → 0. But some features took 5 days — forced early merge of incomplete code. Added feature flags — merge incomplete features, hidden behind flag. Zero merge conflicts since.

**Rule of thumb:** Feature branches for all work — isolation, code review. Keep branches short-lived (max 1-2 days). Rebase on main daily — avoid divergence. Use feature flags for multi-day features — merge early, hide behind flag.
