## 241. What is semantic versioning?

**Semantic versioning is a versioning scheme — MAJOR.MINOR.PATCH — communicating breaking changes, new features, and bug fixes.**

It's a contract with users — version numbers signal upgrade risk.

**Version components:**
- **MAJOR** — breaking changes, API incompatible (1.x → 2.x)
- **MINOR** — new features, backward-compatible (1.2 → 1.3)
- **PATCH** — bug fixes, backward-compatible (1.2.0 → 1.2.1)

**Trade-off:**
- Clear communication — users know upgrade impact from version
- Dependency management — tools auto-update patches, not majors
- But: discipline required — accidental breaking changes violate semver
- Version inflation — major versions scare users, teams delay majors

**Real-world example:**
Our API library used arbitrary versions — v1.0, v1.1, v2.0 (no pattern). Users upgraded v1.0 → v1.1, breaking changes, production incidents. Complaints flooded in. Adopted semver: v1.0.0, breaking changes = v2.0.0, features = v1.1.0, fixes = v1.0.1. Users configured auto-update: patches only. Breaking changes: 10% of users affected → 0% (opt-in major upgrades). But we accidentally released breaking change in minor version — user reports caught it. Reverted, released as v3.0.0. Now automated API diff checks before release.

**Rule of thumb:** Semantic versioning for all public APIs, libraries, SDKs. Automate semver checks — breaking change detection in CI. For internal apps — version based on release date (2024.01.15) or build number.
