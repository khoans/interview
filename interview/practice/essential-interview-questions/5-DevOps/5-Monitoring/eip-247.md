## 247. What are metrics vs logs vs traces?

**Metrics are numeric measurements over time. Logs are timestamped event records. Traces are request flows across services.**

Together they form the three pillars of observability — each answers different questions.

**Metrics:**
- Numeric, aggregated — CPU 75%, 1000 requests/sec, 2% error rate
- Answer: "Is the system healthy?"
- Tools: Prometheus, Datadog, CloudWatch
- Low cardinality — dimensions like service, environment

**Logs:**
- Text records — "User 123 logged in", "Database connection failed"
- Answer: "What happened?"
- Tools: ELK, Loki, Splunk
- High cardinality — full context, stack traces

**Traces:**
- Request journeys — API → Service A → Service B → Database
- Answer: "Where is the bottleneck?"
- Tools: Jaeger, Zipkin, X-Ray
- Distributed — spans across services

**Trade-off:**
- Metrics: cheap storage, fast queries — but limited context
- Logs: full context — but expensive storage, slow queries
- Traces: distributed visibility — but sampling required, high overhead

**Real-world example:**
Our e-commerce site had latency spike — p99 200ms → 2 seconds. Metrics told us which service (checkout). Traces showed bottleneck (payment service → external gateway, 1.5 seconds). Logs revealed why (gateway timeout, SSL handshake failure). Fixed: increased timeout, added connection pooling. Each pillar alone insufficient — together, 10-minute diagnosis.

**Rule of thumb:** Collect all three — metrics for alerting, logs for debugging, traces for distributed systems. Sample traces (1-10%) — full traces too expensive. Correlate with trace IDs — click metric → see traces → see logs.
