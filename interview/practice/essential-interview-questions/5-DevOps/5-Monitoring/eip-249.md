## 249. What is SLA, SLO, SLI?

**SLA is the contract with customers (uptime guarantee). SLO is internal target (99.9% availability). SLI is the measurement (current uptime).**

They're a hierarchy — SLIs measure reality, SLOs set targets, SLAs commit to customers.

**Definitions:**
- **SLI (Indicator)** — what you measure (availability, latency, throughput)
- **SLO (Objective)** — target value (99.9% availability, p99 <200ms)
- **SLA (Agreement)** — contractual commitment (99.9% uptime or refund)

**Trade-off:**
- Clear expectations — business knows what "reliable" means
- Error budgets — spend downtime on features vs. reliability
- But: SLOs require discipline — don't set 99.99% if you can't deliver
- SLA penalties — missed SLA = refunds, lost contracts

**Real-world example:**
Our SaaS had no SLOs — "as reliable as possible." Business sold SLA: 99.99% uptime. Reality: 99.5% — $100K in refunds. Implemented SLIs: availability, latency, error rate. Set SLOs: 99.9% (achievable with current architecture). Error budget: 0.1% downtime = 43 minutes/month. Spend budget on risky deployments — if budget exhausted, freeze deployments, focus on reliability. Refunds: $100K → $0 (SLA matched SLO).

**Rule of thumb:** Start with SLIs — measure reality first. Set SLOs based on actual performance + 10% improvement. SLAs should match SLOs — don't overpromise. Track error budgets — guide deployment risk decisions.
