## 173. What is retry pattern?

**Retry pattern automatically re-executes failed operations — effective for transient failures (network blips, temporary overload) but dangerous for permanent failures.**

It's about distinguishing "try again, it'll work" from "this will never work."

**When to retry:**
- Network timeouts — connection reset, DNS resolution failures
- HTTP 503 (Service Unavailable) — temporary overload
- Database deadlocks — retry transaction with new lock order
- Rate limiting (HTTP 429) — wait for retry-after header

**When NOT to retry:**
- HTTP 4xx errors (bad request, unauthorized) — client error won't fix itself
- Validation failures — data is invalid, retry won't help
- Business logic failures — insufficient funds, out of stock

**Trade-off:**
- Retries fix 80% of transient failures — automatic recovery
- But: retry storms can overwhelm recovering services
- Exponential backoff essential — don't hammer the service

**Real-world example:**
Our payment gateway had 5% transient failures (network timeouts). No retry: 5% payment failures, customer support flooded. Added retry with exponential backoff: 100ms → 200ms → 400ms. Payment success: 95% → 99.5%. But we retried validation failures (invalid card) — same error 3 times, wasted API calls. Added retry-only-on-transient logic: retry timeouts, 503s. Don't retry 400s. Payment success stayed 99.5%, API costs dropped 30%.

**Rule of thumb:** Retry transient failures only (timeouts, 503s). Exponential backoff: 100ms, 200ms, 400ms, 800ms. Max 3-4 retries. Never retry 4xx errors or business logic failures.
