## 176. What is rate limiting?

**Rate limiting restricts how many requests a client can make in a time window — protecting services from overload, abuse, and accidental DDoS.**

It's about fair resource allocation and preventing one client from starving others.

**Common algorithms:**
- **Fixed window** — 1000 requests per minute (resets at minute boundary)
- **Sliding window** — 1000 requests in any 60-second period (smoother)
- **Token bucket** — tokens added at fixed rate, requests consume tokens (allows bursts)
- **Leaky bucket** — requests processed at fixed rate, excess queued or rejected

**Why it matters:**
- Prevents accidental overload — buggy client in infinite loop
- Stops abuse — API scraping, credential stuffing
- Fair usage — one client doesn't consume all resources

**Trade-off:**
- Protection vs. user frustration — legitimate users hit limits
- Complexity — distributed rate limiting requires Redis or similar
- Tuning — too strict = angry users, too lenient = no protection

**Real-world example:**
Our API had no rate limits. Customer's buggy script entered infinite loop — 10,000 requests/second. Database CPU spiked to 100%, legitimate users got 5-second responses. 30-minute outage. Added rate limiting: 1000 requests/minute per API key, 10,000/minute for enterprise. Next infinite loop: hit limit after 1000 requests, auto-blocked for 1 minute. Zero impact on other users. But we had false positives — legitimate batch jobs hit limits. Added rate limit tiers: standard (1K/min), premium (10K/min), enterprise (100K/min).

**Rule of thumb:** Rate limit all public APIs. Default: 1000 requests/minute per client. Higher limits for authenticated/enterprise users. Return 429 (Too Many Requests) with retry-after header.
