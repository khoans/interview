## 175. What is fallback mechanism?

**Fallback mechanism provides an alternative response when a service fails — cached data, default value, or graceful degradation instead of an error.**

It's about giving users something usable instead of "service unavailable."

**Fallback strategies:**
- **Cached data** — return last known good response (stale but usable)
- **Default value** — return empty list, zero, or "unavailable"
- **Alternative service** — switch to backup provider (PaymentGateway A → B)
- **Graceful message** — "Recommendations unavailable, continue shopping?"

**Trade-off:**
- Better UX — partial functionality vs. complete error
- But: stale data may be wrong (cached price vs. current price)
- Complexity — maintain fallback logic, test failure paths

**Real-world example:**
Our product page called 5 services: product details, reviews, recommendations, inventory, pricing. Recommendations service failed (10% failure rate), entire page showed error. 10% of users saw errors, conversion dropped 15%. Added fallback: recommendations fail → show "Popular products" (cached, updated hourly). Page error rate: 10% → 0.5%. Conversion recovered. But cached recommendations were sometimes out of stock — added real-time inventory check on cached items. Zero stock issues since.

**Rule of thumb:** Fallback for non-critical features (recommendations, reviews). Never fallback for critical data (pricing, inventory availability). Always indicate to users when showing fallback data ("Showing popular products instead").
