## 178. What is health check endpoint?

**Health check endpoint is a lightweight API that reports service health — used by load balancers, orchestrators, and monitoring to detect failures and route traffic.**

It's the difference between "service is running" and "service is actually working."

**Health check levels:**
- **Liveness** — is the process running? (basic ping)
- **Readiness** — is the service ready to accept traffic? (dependencies healthy)
- **Deep health** — is everything working? (database, cache, external services)

**Why it matters:**
- Load balancers remove unhealthy instances — no traffic to broken services
- Orchestrators restart failed containers — automatic recovery
- Monitoring alerts on degradation — proactive incident response

**Trade-off:**
- Early detection — catch failures before users notice
- But: health check overhead — every check consumes resources
- False positives — transient failure marks healthy service as unhealthy

**Real-world example:**
Our payment service had no health check. Database connection pool exhausted — service hung, but load balancer kept sending traffic. 50% of payment requests failed. 20-minute detection time (user reports). Added /health endpoint: checks database connectivity, cache connection, external payment gateway. Load balancer polls every 10 seconds. Next DB issue: health check failed, instance removed from rotation in 10 seconds. Zero failed requests — traffic routed to healthy instances. But health check was too aggressive — 1-second DB timeout caused false failures. Increased timeout to 3 seconds, added consecutive failure threshold (3 failures = unhealthy).

**Rule of thumb:** Health check for every service. Liveness: simple ping. Readiness: check critical dependencies. Deep health: check everything, but use for monitoring, not traffic routing. Poll interval: 10-30 seconds. Failure threshold: 3 consecutive failures.
