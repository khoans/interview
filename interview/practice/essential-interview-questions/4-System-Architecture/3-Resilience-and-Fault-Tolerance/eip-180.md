## 180. What is chaos engineering?

**Chaos engineering is the practice of intentionally breaking things in production to test system resilience — finding weaknesses before they cause unplanned outages.**

It's about building confidence through controlled failure injection.

**Common experiments:**
- **Kill instances** — terminate random pods/VMs, verify auto-recovery
- **Network latency** — add 500ms delay between services
- **Service failures** — inject 50% error rate, test circuit breakers
- **Resource exhaustion** — fill disk, exhaust memory, test monitoring

**Why it matters:**
- Find weaknesses before customers do — proactive vs. reactive
- Validate resilience patterns — circuit breakers, retries, fallbacks actually work
- Build muscle memory — teams practice incident response

**Trade-off:**
- Find bugs early — prevent real outages
- But: risk of causing actual outages — experiments can cascade
- Requires investment — tooling, monitoring, rollback plans

**Real-world example:**
We ran quarterly chaos experiments. First experiment: killed random pod in Kubernetes. Expected: auto-recovery in 30 seconds. Reality: stateful service lost in-flight orders, 50 orders corrupted. Found: no graceful shutdown hook, no order recovery mechanism. Fixed: added shutdown hook to complete in-flight requests, order reconciliation job. Next real node failure: zero order loss. Second experiment: added 1-second latency between payment service and gateway. Found: no timeout configured, requests hung indefinitely. Fixed: added 5-second timeout, circuit breaker. Chaos engineering prevented 6 potential outages in 18 months.

**Rule of thumb:** Start small — kill one non-critical service. Automate experiments — GameDays quarterly. Always have rollback — kill switch for experiments. Monitor during experiments — alert on unexpected behavior. Never experiment during peak traffic.
