## 179. What is graceful degradation?

**Graceful degradation reduces functionality under load or failure — keeping core features working while disabling non-essential features to preserve system stability.**

It's about failing well — users get limited functionality instead of complete outage.

**Strategies:**
- **Disable non-critical features** — turn off recommendations, analytics during overload
- **Reduce quality** — lower image resolution, fewer search results
- **Queue instead of process** — accept requests, process when load decreases
- **Read-only mode** — allow browsing, disable checkout during database issues

**Trade-off:**
- Partial service vs. complete outage — users can still accomplish core tasks
- But: complexity — feature flags, degradation logic, monitoring
- User communication — must inform users what's degraded

**Real-world example:**
Black Friday: 10x normal traffic. Database CPU hit 100%, checkout slowed to 30 seconds, entire site became unresponsive. Revenue loss: $200K/hour. Implemented graceful degradation: CPU > 80% → disable recommendations, reviews, analytics. CPU > 90% → read-only mode (browse only, no checkout). Next Black Friday: CPU hit 80%, features disabled automatically. Checkout stayed under 2 seconds. Revenue: zero loss. But customers complained about missing features — added banner: "High traffic — some features temporarily disabled." Complaints dropped 80%.

**Rule of thumb:** Identify core features (checkout, login) vs. nice-to-have (recommendations, reviews). Add feature flags for each. Automate degradation based on metrics (CPU, latency). Communicate degradation to users — transparency reduces frustration.
