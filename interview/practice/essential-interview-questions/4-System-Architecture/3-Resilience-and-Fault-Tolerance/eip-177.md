## 177. What is service discovery?

**Service discovery automatically tracks where services are running — mapping service names to network addresses as instances scale up/down or fail.**

It's essential for dynamic environments where IPs change constantly (Kubernetes, cloud).

**How it works:**
- Services register on startup — "I'm order-service, running at 10.0.1.5:8080"
- Services deregister on shutdown — clean removal
- Clients query discovery system — "Where is order-service?" → get list of instances
- Health checks remove unhealthy instances automatically

**Approaches:**
- **Client-side discovery** — client queries registry, picks instance (Eureka + Ribbon)
- **Server-side discovery** — load balancer queries registry (Kubernetes Service, AWS ALB)
- **Service mesh** — sidecar proxy handles discovery (Istio, Linkerd)

**Trade-off:**
- Dynamic scaling — instances come/go without manual config
- But: discovery system is critical infrastructure — if it fails, nothing works
- Complexity — eventual consistency, stale entries, health check tuning

**Real-world example:**
Our monolith had hardcoded service URLs — payment-service.internal:8080. Migrated to microservices, 5 instances of each service. Manually updated load balancer config every deployment — 20 minutes, error-prone. Implemented Consul for service discovery: services auto-register, load balancer queries Consul. Deployment time: 20 minutes → instant. But Consul went down during maintenance — services couldn't discover each other, 10-minute outage. Added Consul clustering (3 nodes), DNS fallback. Zero discovery outages since.

**Rule of thumb:** Service discovery for any dynamic environment (Kubernetes, auto-scaling). Use managed solutions (AWS Cloud Map, Kubernetes Services) before self-hosting. Always have fallback (DNS, hardcoded backup).
