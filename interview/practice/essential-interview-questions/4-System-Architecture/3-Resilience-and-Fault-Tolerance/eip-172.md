## 172. What is circuit breaker pattern?

**Circuit breaker prevents cascading failures by failing fast when a service is unhealthy — like an electrical circuit breaker that trips when current exceeds safe limits.**

It's the most important resilience pattern for synchronous service calls.

**Three states:**
- **Closed** — normal operation, requests pass through
- **Open** — failure threshold exceeded, requests fail immediately (no timeout wait)
- **Half-open** — test if service recovered, allow one request through

**Why it matters:**
- Without circuit breaker: 60-second timeout per failed request
- With circuit breaker: instant failure, user gets immediate error
- Prevents thread pool exhaustion — no waiting for dead services

**Trade-off:**
- Fast failure vs. occasional false positives (service recovering)
- Need to tune thresholds — too sensitive = unnecessary failures, too lenient = no protection
- State management — circuit state lost on restart

**Real-world example:**
Our inventory service had 2-second latency spike (GC pause). Checkout called inventory — 500 concurrent requests, all waiting 2 seconds. Thread pool exhausted (200 threads), checkout became unresponsive. 15-minute outage. Added circuit breaker: 5 failures in 10 seconds → open for 30 seconds. Next GC pause: circuit opened after 5 failures, requests failed instantly (100ms error vs. 2-second hang). Zero thread pool exhaustion since. Thresholds: 5 failures, 10-second window, 30-second open timeout.

**Rule of thumb:** Circuit breaker for all synchronous service calls. Default thresholds: 5 failures, 10-second window, 30-second timeout. Monitor circuit state — alert on open circuits.
