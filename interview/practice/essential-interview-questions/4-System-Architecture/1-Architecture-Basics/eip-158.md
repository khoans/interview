## 158. What is layered architecture?

**Layered architecture organizes code into horizontal layers — presentation, business logic, data access — where each layer depends only on the layer below.**

It's the most common architecture pattern (MVC, n-tier) — simple, familiar, but can become rigid.

**Key characteristics:**
- **Presentation layer** — UI, API endpoints, request/response handling
- **Business logic layer** — domain rules, workflows, validations
- **Data access layer** — database queries, ORM, repositories
- Strict dependency rule — layers only call the layer directly below

**Trade-off:**
- Separation of concerns — easy to understand, test, replace layers
- But: layers become anemic — business logic leaks to presentation
- Standardization — everyone knows where to find code
- But: "architecture by directory" — same pattern forced on everything

**Real-world example:**
Our enterprise app followed strict 3-tier: controllers → services → repositories. Worked well for CRUD operations. But complex workflows required 10+ service methods calling each other — business logic scattered across layers. Testing required mocking 5 layers. Refactored to hexagonal architecture — domain logic in the center, adapters on the outside. Business logic became testable without mocks. But team took 3 months to learn the pattern — layered was simpler for junior developers.

**Rule of thumb:** Layered architecture for standard CRUD applications with clear data flow. Consider hexagonal or clean architecture for complex domain logic. Don't over-architect simple applications.
