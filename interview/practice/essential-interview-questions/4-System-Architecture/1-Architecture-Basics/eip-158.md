## 158. What is layered architecture?

**Layered architecture organizes code into horizontal layers — presentation, business logic, data access — where each layer depends only on the layer below.**

It's the most common architecture pattern (MVC, n-tier) — simple, familiar, but can become rigid.

---

### Key characteristics:
- **Presentation layer** — UI, API endpoints, request/response handling
- **Business logic layer** — domain rules, workflows, validations
- **Data access layer** — database queries, ORM, repositories
- Strict dependency rule — layers only call the layer directly below

---

### Trade-off:
- Separation of concerns — easy to understand, test, replace layers
- But: layers become anemic — business logic leaks to presentation
- Standardization — everyone knows where to find code
- But: "architecture by directory" — same pattern forced on everything

---

### Real-world example: The Testing Problem

**Layered architecture (what we had):**

```
Controller → Service → Repository → JPA → Database
```

Testing a discount calculation rule required mocking all 5 layers:

```java
// Test for: "Orders over $1000 get 10% discount"

@Mock
Database database;

@Mock
EntityManager entityManager;

@Mock
OrderRepository repository;

@Mock
OrderService service;

@InjectMocks
OrderController controller;

@Test
void testDiscount() {
    Order mockOrder = new Order(1500.00);
    when(repository.findById(1L)).thenReturn(mockOrder);
    when(service.calculateDiscount(mockOrder)).thenReturn(150.00);
    
    Response response = controller.getDiscount(1L);
    
    verify(service).calculateDiscount(mockOrder); // Testing mock behavior, not logic
    assertEquals(150.00, response.discount);
}
```

**Problems:**
- Test verifies mocks were called, not that discount logic is correct
- Change repository method signature → 50 tests break
- Test takes 2 seconds (Spring context initialization)
- Can't test edge cases without setting up 5 layers of mocks

---

**Hexagonal architecture (what we refactored to):**

```
Domain Core (pure Java, no dependencies)
       ↑
    Ports (interfaces)
       ↑
  Adapters (Spring, JPA, REST)
```

Same discount test, no mocks:

```java
// Test for: "Orders over $1000 get 10% discount"

@Test
void testDiscount() {
    // Domain logic is pure Java - no dependencies
    Order order = new Order(1500.00);
    DiscountPolicy policy = new DiscountPolicy();
    
    // Call domain logic directly
    Money discount = policy.calculateDiscount(order);
    
    // Assert business rule
    assertEquals(Money.of(150.00), discount);
}

@Test
void testDiscount_belowThreshold() {
    Order order = new Order(500.00);
    DiscountPolicy policy = new DiscountPolicy();
    
    Money discount = policy.calculateDiscount(order);
    
    assertEquals(Money.ZERO, discount);
}

@Test
void testDiscount_exactlyAtThreshold() {
    Order order = new Order(1000.00);
    DiscountPolicy policy = new DiscountPolicy();
    
    Money discount = policy.calculateDiscount(order);
    
    assertEquals(Money.of(100.00), discount);
}
```

**Benefits:**
- No mocks — domain has no dependencies
- Test runs in 5ms (no Spring, no database)
- Tests actual business logic, not mock interactions
- Refactor repository → zero domain tests break
- Can test 100 edge cases in 2 seconds

---

### The Key Difference

| Aspect | Layered | Hexagonal |
|--------|---------|-----------|
| Domain dependencies | Spring, JPA, Database | None (pure Java) |
| Testing | Mock 5+ layers | No mocks needed |
| Test speed | 2 seconds per test | 5ms per test |
| Test stability | Brittle (breaks on refactor) | Stable (tests behavior) |
| What tests verify | Mock interactions | Actual business rules |

---

### Why This Matters

**Layered architecture:**
```java
@Service  // ← Spring dependency
public class OrderService {
    
    @Autowired  // ← Dependency injection
    private OrderRepository repository;
    
    @Transactional  // ← Spring annotation
    public Money calculateDiscount(Long orderId) {
        Order order = repository.findById(orderId);  // ← Database call required
        // Business logic mixed with infrastructure
    }
}
```

Can't test `calculateDiscount` without:
1. Spring context (2 seconds)
2. Mocked repository (brittle)
3. Mocked database (more brittleness)
4. Mocked transaction manager (complexity)

**Hexagonal architecture:**
```java
// Domain core - pure Java, no annotations
public class DiscountPolicy {
    
    public Money calculateDiscount(Order order) {
        if (order.getTotal().isGreaterThan(Money.of(1000))) {
            return order.getTotal().multiply(0.10);
        }
        return Money.ZERO;
    }
}

// Adapter layer - handles infrastructure
@Service
public class OrderServiceAdapter {
    
    private final OrderRepository port;  // ← Interface, not implementation
    private final DiscountPolicy policy; // ← Domain core
    
    public Money calculateDiscount(Long orderId) {
        Order order = port.findById(orderId);  // ← Database call in adapter
        return policy.calculateDiscount(order); // ← Domain logic, testable
    }
}
```

Domain logic is **isolated** — no Spring, no database, no annotations. Test it like a normal Java class.

---

### Rule of thumb

**Layered architecture** for standard CRUD applications with clear data flow.

**Hexagonal architecture** for complex domain logic (pricing, underwriting, trading) where:
- Business rules change frequently
- Testing is critical (financial, medical, regulatory)
- You want to swap frameworks without rewriting business logic

**Don't use hexagonal** just because it's trendy — match architecture to complexity. Simple CRUD doesn't need ports and adapters.
