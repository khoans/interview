## 160. What is CQRS pattern?

**CQRS (Command Query Responsibility Segregation) separates read and write operations — different models, different data stores, optimized independently.**

It's about trading consistency and complexity for read performance and scalability.

**Key characteristics:**
- **Command side** — handles writes (INSERT, UPDATE, DELETE), validates business rules
- **Query side** — handles reads (SELECT), optimized for fast retrieval
- Separate models — write model normalized, read model denormalized
- Eventual consistency — read model syncs from write model asynchronously

**Trade-off:**
- Read performance — denormalized views, no JOINs, 10-100x faster queries
- But: eventual consistency — reads may be seconds behind writes
- Independent scaling — scale reads and writes separately
- But: complexity — two models to maintain, sync mechanism, debugging harder

**Real-world example:**
Our analytics dashboard had 50 complex JOINs across 10 tables — 8 seconds per query. Implemented CQRS: write model (normalized, 3NF), read model (denormalized, one table per dashboard view). Write: insert to normalized tables + publish event. Read: event handler updates denormalized view. Query time: 8 seconds → 50ms (160x faster). But reads lagged 2-3 seconds behind writes. For analytics, acceptable. For account balance? Unacceptable. We kept CQRS for dashboards, traditional CRUD for user-facing data.

**Rule of thumb:** CQRS for read-heavy workloads (analytics, reporting, dashboards) where eventual consistency is acceptable. Traditional CRUD for transactional systems requiring strong consistency.
