## 157. What is event-driven architecture?

**Event-driven architecture decouples services through asynchronous events — producers emit events, consumers react, no direct coupling between them.**

It's about trading immediate consistency for scalability and loose coupling.

**Key characteristics:**
- Event producers emit events (OrderCreated, PaymentProcessed)
- Event consumers subscribe and react (send email, update inventory)
- Event broker in the middle (Kafka, RabbitMQ, SNS/SQS)
- Asynchronous — producer doesn't wait for consumer

**Trade-off:**
- Loose coupling — services don't know about each other
- But: eventual consistency — consumer may process events minutes later
- Scalability — handle traffic spikes by buffering events
- But: debugging complexity — trace event flow across 5+ consumers

**Real-world example:**
Our checkout flow was synchronous: place order → charge payment → reserve inventory → send email → notify warehouse. 2 seconds total, but email service slowdowns blocked checkout. Switched to event-driven: OrderCreated event → payment, inventory, email, warehouse all react asynchronously. Checkout dropped to 400ms (5x faster). But email sometimes delayed 5 minutes, inventory occasionally out of sync. Added event monitoring, alerting on lag >1 minute. Zero checkout slowdowns since.

**Rule of thumb:** Event-driven for non-critical workflows (notifications, analytics, reporting). Synchronous for critical flows (payment, inventory reservation). Hybrid approach works best — sync for core, async for peripherals.
