## 159. What is hexagonal architecture (ports and adapters)?

**Hexagonal architecture puts business logic at the center, surrounded by ports (interfaces) and adapters (implementations) — isolating core logic from external concerns.**

It's about making the domain testable without databases, APIs, or frameworks.

**Key characteristics:**
- **Domain core** — business logic, entities, value objects (no dependencies)
- **Ports** — interfaces defining how external world interacts with core
- **Adapters** — implementations (REST controller, database repository, message listener)
- Dependency inversion — adapters depend on ports, core depends on nothing

**Trade-off:**
- Testable domain — unit tests without mocks, 100% coverage achievable
- But: more boilerplate — ports, adapters, mapping between layers
- Framework agnostic — swap Spring for Quarkus, no domain changes
- But: learning curve — teams used to layered architecture struggle

**Real-world example:**
Our pricing engine had complex rules (discounts, promotions, volume pricing) buried in Spring controllers and JPA entities. Testing required full Spring context, 30 seconds per test. Refactored to hexagonal: domain core (pure Java), ports (PricingRepository, DiscountCalculator), adapters (JPA, REST). Domain tests: 2 seconds, no Spring, no database. But we added 40% more code — ports, adapters, mappers. For simple CRUD, overkill. For complex pricing logic, worth it.

**Rule of thumb:** Hexagonal for complex domain logic (pricing, underwriting, trading). Layered architecture for simple CRUD. Don't use hexagonal just because it's trendy — match architecture to complexity.
