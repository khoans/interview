## 183. What is database replication?

**Database replication copies data from primary database to replicas — enabling read scaling, failover, and geographic distribution.**

It's about trading consistency for availability and performance.

**Replication types:**
- **Synchronous** — primary waits for replica acknowledgment (strong consistency, slower writes)
- **Asynchronous** — primary doesn't wait (eventual consistency, faster writes, replication lag)
- **Semi-synchronous** — wait for at least one replica (balance between consistency and performance)

**Use cases:**
- Read scaling — 90% reads go to replicas, 10% writes to primary
- Failover — primary dies, replica promoted (seconds of downtime)
- Geographic — replicas in different regions, lower latency for global users

**Trade-off:**
- Read performance — 10x more read capacity with 10 replicas
- But: replication lag — replicas 1-5 seconds behind primary
- Users may see stale data — just-updated record shows old value on replica

**Real-world example:**
Our social media feed: 100K reads/second, 5K writes/second. Single database: CPU at 100%, 5-second query times. Added 5 read replicas, async replication. Reads distributed across replicas, primary handles writes only. Query time: 5s → 200ms. But users complained: posted update, didn't see it in feed for 3 seconds (replication lag). Fixed: user's own posts read from primary (strong consistency), others from replicas (eventual consistency). Zero complaints since.

**Rule of thumb:** Async replication for read-heavy workloads. Route user's own data to primary (strong consistency), others to replicas. Monitor replication lag — alert if >10 seconds.
