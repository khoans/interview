## 181. What is horizontal vs. vertical scaling?

**Vertical scaling adds more power to existing machines (CPU, RAM). Horizontal scaling adds more machines to distribute load.**

Vertical is simpler but has hard limits. Horizontal is complex but virtually unlimited.

**Vertical scaling:**
- Upgrade server: 8GB → 64GB RAM, 4 cores → 32 cores
- Simple — no code changes, just bigger hardware
- Limited — eventually hit ceiling (largest instance available)
- Single point of failure — one big server goes down, everything down

**Horizontal scaling:**
- Add more servers: 2 → 10 → 100 instances
- Complex — need load balancing, stateless design, data partitioning
- Virtually unlimited — add servers as needed
- Better resilience — one server down, others handle traffic

**Trade-off:**
- Vertical: 10x faster in 10 minutes, but costs 10x more, hits ceiling at 1-2TB RAM
- Horizontal: weeks to implement properly, but scale to millions of users

**Real-world example:**
Our monolith on 16GB server handled 1K concurrent users. Traffic grew to 5K — server maxed out. Vertical: upgraded to 128GB, 32 cores — handled 10K users, cost $2K/month. Traffic hit 50K — no bigger instances available. Horizontal: split into 10 × 16GB instances behind load balancer — handled 100K users, cost $1K/month. Zero downtime during traffic spikes.

**Rule of thumb:** Vertical scale first (quick win). Horizontal scale when you hit limits or need high availability. Design for horizontal from day one — stateless services, shared nothing architecture.
