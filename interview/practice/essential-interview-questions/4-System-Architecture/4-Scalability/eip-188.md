## 188. What is connection pooling?

**Connection pooling reuses database connections instead of creating new ones per request — reducing connection overhead from 50ms to <1ms.**

It's essential for high-throughput applications with frequent database access.

**How it works:**
- Pool initialized with N connections (e.g., 20)
- Request borrows connection from pool, uses it, returns it
- Connection stays open, reused by next request
- Pool manages max size, idle timeout, health checks

**Trade-off:**
- Performance — connection reuse: <1ms vs. 50ms for new connection
- But: pool exhaustion — all connections in use, requests queue up
- Tuning required — too small = queueing, too large = database overload

**Real-world example:**
Our API: no connection pooling, new connection per request (50ms overhead). Peak traffic: 1000 requests/second, database maxed at 500 connections. Requests queued, latency spiked to 5 seconds. Added HikariCP pool: 50 connections, reused. Latency: 5s → 100ms. But pool exhausted during traffic spike — 50 connections, 200 concurrent requests. 150 requests queued. Increased pool to 100, added connection timeout (5 seconds). Zero queueing since.

**Rule of thumb:** Always use connection pooling. Pool size: (CPU cores × 2) + effective_spindle_count for DB. Monitor pool utilization — alert at 80%. Set connection timeout — fail fast instead of indefinite queue.
