## 184. What is database sharding?

**Database sharding splits data across multiple databases based on a shard key — each shard holds a subset of data, reducing load per database.**

It's horizontal scaling for databases when replication isn't enough.

**Sharding strategies:**
- **Range-based** — users 1-1M on shard 1, 1M-2M on shard 2 (simple, but uneven distribution)
- **Hash-based** — hash(user_id) % num_shards (even distribution, but hard to rebalance)
- **Directory-based** — lookup table maps user to shard (flexible, but single point of failure)
- **Geographic** — US users on US shard, EU users on EU shard (data residency, low latency)

**Trade-off:**
- Unlimited scale — add shards as data grows
- But: cross-shard queries are expensive — JOINs across shards require application logic
- Operational complexity — backup, restore, rebalance across multiple databases

**Real-world example:**
Our messaging platform: 100 million users, 1 billion messages/day. Single database: 2TB, queries slowing (10+ seconds). Sharded by user_id (hash-based) across 20 shards. Each shard: 5 million users, 100GB. Query time: 10s → 100ms. But analytics queries (all users) required querying all 20 shards — 20x slower. Added separate analytics database (replicated from all shards). Operational overhead: 20 databases to manage, backup rotation, monitoring.

**Rule of thumb:** Shard when single database hits limits (1TB+, slow queries). Choose shard key carefully — queries should access single shard. Avoid cross-shard queries — denormalize or use separate analytics store.
