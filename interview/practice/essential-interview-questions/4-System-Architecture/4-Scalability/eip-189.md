## 189. What is database connection pool tuning?

**Connection pool tuning balances between too few connections (queueing, timeouts) and too many (database overload, context switching).**

The right size depends on workload type, database capacity, and concurrency patterns.

**Key parameters:**
- **Maximum pool size** — max connections (too small = queueing, too large = DB overload)
- **Minimum idle** — connections kept warm (faster response, but resources reserved)
- **Connection timeout** — how long to wait for connection (fail fast vs. wait)
- **Idle timeout** — when to close unused connections (resource cleanup)
- **Max lifetime** — force connection refresh (prevents stale connections)

**Trade-off:**
- Larger pool: more concurrent requests, but database context switching increases
- Smaller pool: efficient DB usage, but requests queue during spikes

**Real-world example:**
Our API had default pool size (10 connections). Traffic spike: 100 concurrent requests, 90 queued. Latency: 100ms → 10 seconds. Increased pool to 200 — database CPU spiked to 100%, all queries slowed. Tuned based on workload: our app is I/O-bound (mostly waiting for external APIs), not CPU-bound. Set pool to 50 (appropriate for I/O), added queue monitoring. Latency: 10s → 200ms, database CPU at 60%. Sweet spot: pool size matches concurrent database work, not total concurrent requests.

**Rule of thumb:** Start with 10-20 connections per app instance. Monitor: active connections, queue depth, wait time. Increase if queueing, decrease if DB CPU >80%. I/O-bound apps need larger pools than CPU-bound.
