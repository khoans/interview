## 182. What is load balancing?

**Load balancing distributes incoming requests across multiple servers — preventing any single server from becoming a bottleneck.**

It's essential for horizontal scaling and high availability.

**Common algorithms:**
- **Round-robin** — cycle through servers evenly (simple, ignores load)
- **Least connections** — send to server with fewest active connections (better for varying request times)
- **IP hash** — same client always hits same server (session affinity, but uneven distribution)
- **Weighted** — more powerful servers get more traffic (heterogeneous hardware)

**Trade-off:**
- Better resource utilization — no hot spots, all servers used evenly
- But: load balancer becomes single point of failure
- Session management — sticky sessions simplify state, but reduce flexibility

**Real-world example:**
Our e-commerce site had 5 app servers. No load balancer — DNS round-robin, uneven distribution. Server 1: 80% traffic, 2-second latency. Server 5: 5% traffic, idle. Added NGINX load balancer with least-connections algorithm. All servers at 20% traffic, p95 latency: 2s → 400ms. But NGINX became single point of failure — added backup (keepalived), automatic failover in 3 seconds. Zero load balancer outages since.

**Rule of thumb:** Load balancer for 2+ servers. Least-connections for varying request times. Sticky sessions only when required (legacy apps). Always have backup load balancer — it's critical infrastructure.
