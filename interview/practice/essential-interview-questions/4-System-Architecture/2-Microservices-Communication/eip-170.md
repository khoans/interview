## 170. What is API Gateway pattern?

**API Gateway is a single entry point for all client requests — routing, authentication, rate limiting, aggregation handled centrally, not in each service.**

It's about trading a potential bottleneck for simplified client integration and centralized cross-cutting concerns.

**Key characteristics:**
- **Request routing** — /orders → order-service, /products → product-service
- **Authentication/authorization** — JWT validation, API key checks at gateway
- **Rate limiting** — 1000 requests/minute per API key
- **Response aggregation** — combine multiple service responses into one
- **Protocol translation** — external REST, internal gRPC

**Trade-off:**
- Simplified clients — one endpoint, not 10 service URLs
- But: gateway becomes bottleneck — single point of failure
- Centralized concerns — auth, logging, rate limiting in one place
- But: gateway can become a monolith — logic creep over time

**Real-world example:**
Our mobile app called 8 microservices directly — 8 URLs to manage, auth logic duplicated, rate limiting inconsistent. Implemented Kong API Gateway: single endpoint, JWT validation at gateway, rate limiting (1000 req/min), request aggregation for home screen (4 services → 1 call). Mobile latency: 800ms → 200ms. But gateway became bottleneck — 10K requests/second, CPU at 90%. Added gateway clustering (3 instances), load balancer. Zero downtime since. Now we monitor gateway latency — alert if p95 > 100ms.

**Rule of thumb:** API Gateway for microservices with multiple clients (mobile, web, partners). Use managed solutions (Kong, AWS API Gateway) before building your own. Monitor gateway performance — it's critical infrastructure.
