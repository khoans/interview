## 162. What is synchronous vs asynchronous communication?

**Synchronous communication waits for a response before continuing. Asynchronous sends a message and continues without waiting — response handled later via callback or polling.**

Choose based on whether the caller needs an immediate answer.

**Synchronous:**
- Caller blocks until response arrives
- Simple mental model — request, response, done
- Latency adds up — 5 services × 200ms = 1 second total
- Cascading failures — one slow service blocks entire chain

**Asynchronous:**
- Caller sends message, continues immediately
- Response handled via callback, event, or polling
- Latency hidden — work happens in background
- Better resilience — sender doesn't block if receiver is down

**Trade-off:**
- Sync: simpler code, immediate consistency — but poor resilience, latency compounds
- Async: better resilience, decoupling — but eventual consistency, harder debugging

**Real-world example:**
Our notification system was synchronous: order placed → send email → send SMS → push notification. 2 seconds total, but email service timeouts caused order failures. Switched to async: order placed → publish OrderCreated event → email/SMS/push consume independently. Order confirmation: 200ms → 50ms. But SMS sometimes delayed 5 minutes. Added priority queue for SMS (high priority), email (low priority). SMS now <30 seconds, email <5 minutes. Order failures dropped to zero.

**Rule of thumb:** Sync for user-facing operations needing immediate feedback (payment confirmation). Async for background tasks (notifications, analytics, reporting). Hybrid approach works best.
