## 165. What is GraphQL?

**GraphQL is a query language for APIs where clients request exactly what they need — no over-fetching, no under-fetching, single endpoint.**

It solves REST's chatty API problem but introduces complexity in caching and query optimization.

**Key characteristics:**
- **Client-specified queries** — request only needed fields
- **Single endpoint** — POST /graphql, no /users/:id vs. /users/:id/orders
- **Strong typing** — schema defines available types, queries, mutations
- **Introspection** — clients can discover schema dynamically

**Trade-off:**
- No over-fetching — mobile apps get minimal payloads
- But: complex queries can cause N+1 database problems
- Single round-trip — fetch related data in one request
- But: caching harder — every query is unique, CDN caching ineffective
- Query complexity — malicious queries can overload backend

**Real-world example:**
Our e-commerce mobile app had 15 REST endpoints for different screens. Product screen: 3 calls (product, reviews, related items). Checkout: 5 calls (cart, user, addresses, payment methods, promotions). GraphQL: single query per screen, 60% less data transferred. But we had a DoS incident — nested query { products { reviews { user { orders { products { ... } } } } } } caused 10,000 database queries. Added query depth limit (max 5), complexity analysis, persisted queries. Zero incidents since.

**Rule of thumb:** GraphQL for complex UIs with varying data needs (mobile, dashboards). REST for simple CRUD, public APIs, caching-heavy workloads. Always add query depth limits and complexity analysis.
