## 161. How do microservices communicate with each other?

**Microservices communicate via HTTP/REST, gRPC, message brokers (async), or service mesh — choice depends on coupling, latency, and reliability requirements.**

The communication pattern determines your system's resilience and complexity.

**Common patterns:**
- **HTTP/REST** — simple, universal, but chatty and latency-prone
- **gRPC** — fast, typed contracts, but requires protobuf and code generation
- **Message brokers** — async, decoupled, but eventual consistency
- **Service mesh** — handles retries, circuit breaking, observability — but adds infrastructure complexity

**Trade-off:**
- Synchronous (REST, gRPC): simple, immediate response — but cascading failures if services go down
- Asynchronous (messages): resilient, decoupled — but harder to debug, eventual consistency

**Real-world example:**
Our order flow used synchronous REST: cart → checkout → payment → inventory → shipping. One service slow (inventory, 2 seconds), entire chain blocked. Average latency: 3 seconds. Implemented async for non-critical steps: payment (sync) → inventory (async event) → shipping (async event). Checkout dropped to 400ms. But inventory occasionally out of sync — added reconciliation job running every 5 minutes. Zero checkout timeouts since.

**Rule of thumb:** Sync for critical flows (payment, authentication). Async for peripherals (notifications, analytics, inventory updates). Use gRPC for internal service-to-service, REST for external APIs.
