## 163. What is REST? What are its constraints?

**REST is an architectural style for HTTP APIs based on resources, stateless communication, and standard HTTP methods — simple, universal, but chatty.**

It's the default for public APIs and most internal services.

**Key constraints:**
- **Client-server** — separation of concerns, independent evolution
- **Stateless** — each request contains all context, no server-side session
- **Cacheable** — responses explicitly marked cacheable or not
- **Uniform interface** — resources identified by URIs, manipulated via HTTP verbs
- **Layered system** — client doesn't know if connected directly or through proxy

**Trade-off:**
- Simple, universal — any language can call REST APIs
- But: chatty — fetching related data requires multiple round-trips
- Over-fetching/under-fetching — get entire resource or make more calls
- Versioning complexity — /v1/users vs. /v2/users

**Real-world example:**
Our mobile app's home screen needed: user profile, recent orders, recommendations, notifications. REST required 4 separate API calls — 800ms total on 3G. Users complained about slow loading. Implemented API composition endpoint (BFF pattern) — single call returned all data, 200ms. But endpoint became monolithic — any change required redeploying entire composition layer. Split into GraphQL for mobile, kept REST for internal services. Mobile latency: 800ms → 150ms.

**Rule of thumb:** REST for simple CRUD, public APIs, internal service-to-service. GraphQL for complex UIs with varying data needs. gRPC for high-performance internal communication.
