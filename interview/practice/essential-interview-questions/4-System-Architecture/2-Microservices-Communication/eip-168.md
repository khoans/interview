## 168. What is event sourcing?

**Event sourcing stores state changes as a sequence of events — not current state, but the history of how we got there. Current state is derived by replaying events.**

It's about trading query simplicity for complete audit trail and temporal queries.

**Key characteristics:**
- **Events as source of truth** — OrderCreated, OrderUpdated, OrderShipped
- **Append-only** — events never modified, only added
- **State reconstruction** — current state = sum of all events
- **Time travel** — query state as of any point in time

**Trade-off:**
- Complete audit trail — every change tracked, regulatory compliance built-in
- But: complex queries — need CQRS, read models for efficient queries
- Temporal queries — "what was the order status last Tuesday?"
- But: event schema evolution — changing event structure breaks replay

**Real-world example:**
Our banking system stored account balances in a table. Bug: balance update failed mid-transaction, balance corrupted. 3 days to reconcile. Switched to event sourcing: balance = sum of all transactions (deposits, withdrawals, transfers). Corruption impossible — events immutable, balance always derivable. Audit queries instant: "show all transactions between dates." But queries like "find accounts with balance > $10K" required full event replay — slow. Added CQRS: write events, update read model (account_balances table). Query: 30 seconds → 50ms.

**Rule of thumb:** Event sourcing for financial systems, audit-heavy domains, regulatory compliance. Traditional CRUD for simple applications. Always pair with CQRS for query performance.
