## 169. What is saga pattern?

**Saga pattern manages distributed transactions across microservices — each step has a compensating action to undo changes if a later step fails.**

It's the alternative to two-phase commit (2PC) in microservices — eventual consistency instead of atomic commits.

**Key characteristics:**
- **Choreography** — services emit events, next service reacts (decentralized)
- **Orchestration** — central coordinator tells services what to do (centralized)
- **Compensating transactions** — explicit rollback logic (CancelOrder, RefundPayment)
- **Eventual consistency** — system converges to consistent state over time

**Trade-off:**
- No distributed locks — services don't block waiting for each other
- But: complex rollback logic — every action needs a compensating action
- Better resilience — partial failures don't cascade
- But: debugging complexity — trace saga across 5+ services

**Real-world example:**
Our order flow: cart → payment → inventory → shipping. Payment succeeded, inventory failed — order in limbo, payment charged, no inventory reserved. Implemented saga with orchestration: Payment → if success, Inventory → if fail, CancelPayment. Compensation: CancelPayment, ReleaseInventory. Order failures dropped 90%. But saga took 3 weeks to implement — every service needed compensating logic. For simple flows, worth it. For complex flows with 10+ steps, consider redesigning boundaries.

**Rule of thumb:** Saga for multi-service transactions requiring rollback (orders, bookings). Two-phase commit only for financial systems needing strong consistency. Keep sagas short — 3-5 steps max.
