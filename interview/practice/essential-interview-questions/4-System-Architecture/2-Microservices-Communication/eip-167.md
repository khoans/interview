## 167. What is the difference between RabbitMQ and Kafka?

**RabbitMQ is a traditional message broker with queues and exchanges. Kafka is a distributed event streaming platform with ordered logs and replayability.**

Different tools for different use cases — messaging vs. event streaming.

**RabbitMQ:**
- Message queues with routing (direct, fanout, topic)
- Messages deleted after consumption
- Low latency (sub-millisecond)
- Best for: task queues, RPC, request/response patterns

**Kafka:**
- Append-only event logs, messages retained (days to forever)
- Consumers track offset — can replay historical events
- High throughput (millions of messages/second)
- Best for: event sourcing, CQRS, stream processing, audit logs

**Trade-off:**
- RabbitMQ: simpler, lower latency — but no replay, limited retention
- Kafka: replayable, scalable — but higher latency, operational complexity

**Real-world example:**
We used RabbitMQ for notifications — email, SMS, push. Worked perfectly: low latency, reliable delivery. Then tried RabbitMQ for event sourcing (order history). Problem: messages deleted after consumption, couldn't rebuild state. Switched to Kafka for events: order events retained 90 days, could replay for debugging, rebuild read models. Kept RabbitMQ for notifications. Right tool for each job: RabbitMQ for tasks, Kafka for events.

**Rule of thumb:** RabbitMQ for task queues, job distribution, RPC. Kafka for event streaming, audit logs, CQRS, event sourcing. Don't force one tool to do both jobs.
