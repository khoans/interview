## 164. What is gRPC? When would you use it?

**gRPC is a high-performance RPC framework using HTTP/2 and Protocol Buffers — faster than REST, strongly typed contracts, but requires code generation.**

It's ideal for internal service-to-service communication where performance matters.

**Key characteristics:**
- **Protocol Buffers** — binary serialization, smaller payloads, faster parsing
- **HTTP/2** — multiplexing, bidirectional streaming, header compression
- **Strong typing** — .proto files generate client/server code in 10+ languages
- **Built-in features** — authentication, load balancing, retries, deadlines

**Trade-off:**
- 5-10x faster than REST — smaller payloads, binary protocol
- But: requires protobuf compilation, less browser-friendly than JSON
- Better contracts — explicit schemas, versioning built-in
- But: learning curve, tooling less mature than REST

**Real-world example:**
Our payment processing had 50 internal service calls per transaction. REST: 200ms per call, 10 seconds total. Switched to gRPC: 20ms per call, 1 second total (10x faster). Payload size: 2KB JSON → 500 bytes protobuf. But mobile SDK integration was painful — protobuf added 5MB to app size, code generation broke on every proto change. Kept gRPC for internal services, REST for public/mobile APIs.

**Rule of thumb:** gRPC for internal microservices (performance, strong contracts). REST for public APIs, browser clients, simple integrations. Don't use gRPC for external APIs — clients don't want protobuf complexity.
