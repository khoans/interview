## 166. What is a message broker?

**A message broker is middleware that enables asynchronous communication between services — producers send messages, consumers receive them, decoupled in time and space.**

It's the backbone of event-driven architectures.

**Key characteristics:**
- **Decoupling** — producer doesn't know about consumers
- **Buffering** — handles traffic spikes, smooths load
- **Reliability** — messages persist, survive consumer failures
- **Routing** — publish/subscribe, point-to-point, topic-based

**Trade-off:**
- Resilience — consumer down, messages queue up, process when back
- But: eventual consistency — consumer may process minutes later
- Scalability — multiple consumers process in parallel
- But: operational complexity — broker becomes critical infrastructure

**Real-world example:**
Our order system had synchronous calls to inventory, shipping, notifications. Shipping service slow (500ms), blocked order confirmation. Added RabbitMQ: order → publish OrderCreated → inventory/shipping/notifications consume async. Order confirmation: 2 seconds → 200ms. But during Black Friday, queue grew to 50K messages, processing lagged 10 minutes. Added auto-scaling consumers (1 → 20 instances), cleared backlog in 15 minutes. Now we monitor queue depth, scale at 1K messages.

**Rule of thumb:** Message brokers for async workflows (notifications, analytics, batch processing). Not for real-time user-facing operations. Monitor queue depth and consumer lag — set alerts before backlog becomes critical.
