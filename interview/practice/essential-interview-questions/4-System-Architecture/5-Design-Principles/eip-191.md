## 191. What is SOLID principles?

**SOLID is five object-oriented design principles that make code more maintainable, extensible, and testable — Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion.**

They're guidelines, not rules — apply judiciously, not dogmatically.

**The five principles:**
- **S** — Single Responsibility: one class, one reason to change
- **O** — Open/Closed: open for extension, closed for modification
- **L** — Liskov Substitution: subclasses must be substitutable for base class
- **I** — Interface Segregation: many small interfaces > one large interface
- **D** — Dependency Inversion: depend on abstractions, not concretions

**Trade-off:**
- Maintainable code — easier to change, test, understand
- But: over-engineering risk — 10 interfaces for simple logic
- Apply incrementally — refactor when pain appears, not preemptively

**Real-world example:**
Our payment processor: 2000-line PaymentService class — handled credit cards, PayPal, bank transfers, refunds, validations, notifications. Every change broke something. Refactored with SOLID: split into PaymentProcessor (S), added new payment methods via extension not modification (O), unified PaymentMethod interface (L, I), injected dependencies (D). Code: 2000 lines → 200 lines per class. Adding new payment method: 2 days → 2 hours. But took 3 weeks to refactor — worth it for long-term maintainability.

**Rule of thumb:** SOLID for complex, long-lived code. Don't over-engineer simple scripts. Refactor when you feel pain (hard to change, frequent bugs), not as premature optimization.
