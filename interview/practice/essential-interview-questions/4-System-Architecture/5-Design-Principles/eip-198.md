## 198. What is strong consistency?

**Strong consistency means that after a write completes, all subsequent reads return the updated value — the system appears as a single, atomic data store.**

It's what developers expect from a single database — but hard to achieve in distributed systems.

**How it works:**
- Write locks all replicas (or uses consensus like Raft/Paxos)
- Read waits for latest write to be visible
- Linearizability — operations appear to happen in a single global order

**Trade-off:**
- Simple programming model — reads always return latest write
- But: higher latency — must wait for consensus across replicas
- Lower availability — if any replica is unreachable, writes may block

**Real-world example:**
Our banking system: account balances replicated across 3 data centers. Initially used eventual consistency — deposit in DC1, withdraw in DC2 before replication completed. Customer withdrew same money twice — $10K fraud. Switched to strong consistency (distributed transactions with 2PC). Deposit: waits for all 3 DCs to acknowledge. Withdraw: reads latest balance from all DCs. Zero double-spend incidents since. But latency increased: 50ms → 200ms (cross-DC consensus). Acceptable trade-off for financial integrity.

**Rule of thumb:** Strong consistency for financial transactions, inventory, access control. Eventual consistency for social, content, analytics. Don't use eventual consistency for money — the complexity of reconciliation isn't worth it.
