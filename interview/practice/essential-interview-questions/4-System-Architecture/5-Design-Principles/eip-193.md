## 193. What is Open/Closed Principle?

**Open/Closed Principle: software entities should be open for extension but closed for modification — add new behavior by adding code, not changing existing code.**

It's about protecting tested code from breaking changes.

**How to achieve:**
- **Abstraction** — depend on interfaces, not concrete classes
- **Polymorphism** — new implementations of existing interfaces
- **Strategy pattern** — swap algorithms without changing client code
- **Template method** — override specific steps, not entire algorithm

**Trade-off:**
- Safe extension — new features don't break existing functionality
- But: more abstraction — interfaces, base classes, factories
- Over-engineering risk — abstracting before understanding requirements

**Real-world example:**
Our report generator: monolithic class with if-else for each report type (PDF, Excel, CSV). New report type required modifying the class — broke existing reports 3 times. Refactored: ReportGenerator interface with generate() method. PdfReportGenerator, ExcelReportGenerator, CsvReportGenerator implementations. Adding new report type: create new class, register in factory. Zero changes to existing code. Added Word report: 1 day, zero bugs in PDF/Excel/CSV. But added complexity — 4 classes + factory vs. 1 monolith. Worth it for frequently-changing code.

**Rule of thumb:** Open/Closed for stable core logic with frequent extensions (plugins, report types, payment methods). Don't abstract prematurely — wait until you have 2-3 variations before extracting interface.
