## 195. What is domain-driven design (DDD)?

**DDD is an approach to software design that centers the domain model — aligning code structure with business concepts, using ubiquitous language shared by developers and domain experts.**

It's about making code reflect business reality, not technical abstractions.

**Key concepts:**
- **Ubiquitous language** — developers and business use same terms (no translation)
- **Bounded context** — explicit boundaries where terms have specific meaning
- **Entities** — objects with identity (Customer, Order)
- **Value objects** — objects defined by attributes (Address, Money)
- **Aggregates** — consistency boundaries (Order + OrderItems)
- **Repositories** — abstraction for data access
- **Domain services** — business logic that doesn't fit entities

**Trade-off:**
- Business-aligned code — easier to understand, change with business
- But: learning curve — developers must learn domain deeply
- Overhead — not worth it for simple CRUD applications

**Real-world example:**
Our insurance system: anemic domain model — data classes, business logic in services. Business changes required finding logic scattered across 50 service classes. Refactored with DDD: rich domain model (Policy, Claim, Premium entities), business logic in entities, ubiquitous language (underwriter, policyholder, endorsement). Business changes: find relevant entity, change logic in one place. Onboarding new developers: 3 months → 6 weeks (code matches business terminology). But took 6 months to refactor — worth it for complex, long-lived domain.

**Rule of thumb:** DDD for complex domains (insurance, finance, healthcare). Skip for simple CRUD (blogs, basic e-commerce). Invest in learning the domain — code will follow.
