## 196. What is bounded context?

**Bounded context is a boundary within which a domain model has a specific, consistent meaning — the same term can mean different things in different contexts.**

It's about managing complexity by splitting large domains into manageable pieces.

**Why it matters:**
- Same term, different meaning — "Customer" in Sales vs. "Customer" in Support
- Explicit boundaries — no ambiguity about which model applies
- Independent evolution — change one context without affecting others

**Integration patterns:**
- **Shared kernel** — two contexts share some model (tight coupling, use sparingly)
- **Customer-supplier** — one context provides data to another (upstream/downstream)
- **Anti-corruption layer** — translate between contexts, protect your model
- **Open host service** — public API for other contexts to integrate

**Trade-off:**
- Clear boundaries — teams work independently, models stay clean
- But: integration complexity — need translation between contexts
- Duplication — Customer entity in multiple contexts, slightly different

**Real-world example:**
Our e-commerce platform: one monolithic "Product" model — 200 fields, used by catalog, inventory, pricing, shipping. Every change affected all teams. Split into bounded contexts: Catalog (name, description, images), Inventory (quantity, warehouse location), Pricing (price, discounts), Shipping (dimensions, weight). "Product" in each context has only relevant fields. Catalog team adds SEO fields — zero impact on Inventory. But needed integration: product page aggregates data from all 4 contexts. Added anti-corruption layer — translates between contexts, each team owns their model.

**Rule of thumb:** Bounded context for large domains with multiple teams. Define context boundaries early — align with team structure (Conway's Law). Use anti-corruption layers for integration — don't let contexts leak into each other.
