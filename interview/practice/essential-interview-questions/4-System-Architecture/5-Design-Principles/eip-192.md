## 192. What is Single Responsibility Principle?

**Single Responsibility Principle: a class should have one reason to change — one job, one responsibility, one area of expertise.**

It's the most important SOLID principle — violations cause ripple effects across the codebase.

**Signs of violation:**
- Class names with "And" — UserManagerAndValidator, OrderProcessorAndNotifier
- God classes — 1000+ lines, knows about everything
- Change amplification — one feature change requires modifying 10 classes

**Trade-off:**
- Focused classes — easier to test, understand, maintain
- But: more classes — 20 small classes vs. 1 large class
- Navigation complexity — need to understand class relationships

**Real-world example:**
Our OrderService: 3000 lines — validated orders, calculated totals, applied discounts, persisted to DB, sent emails, updated inventory, generated invoices. Every feature request touched OrderService. Merge conflicts daily. Bug in discount logic broke email sending (unrelated code). Split into: OrderValidator, PricingCalculator, DiscountEngine, OrderRepository, OrderNotificationService, InventoryUpdater, InvoiceGenerator. Each class: 100-200 lines. Changes isolated — discount changes only touch DiscountEngine. Zero merge conflicts since.

**Rule of thumb:** One responsibility per class. If you can't describe the class in one sentence without "and", split it. Refactor when class exceeds 300-500 lines or has 3+ reasons to change.
