## 402. What is Kafka used for?

**Kafka is used for event streaming — real-time data pipelines, event sourcing, stream processing, and microservices communication.**

Common patterns: event sourcing, CQRS, real-time analytics, log aggregation.

**Use cases:**
- **Event sourcing** — store state changes as immutable events
- **Real-time analytics** — process events as they happen
- **Log aggregation** — centralize logs from multiple services
- **Microservices communication** — decouple services via events
- **CDC (Change Data Capture)** — replicate database changes to other systems

**Trade-off:**
- Decoupling — producers don't know consumers
- But: eventual consistency — consumers process events with delay
- Schema evolution — changing event format requires careful migration

**Real-world example:**
Our order system was monolithic — order placement, inventory, shipping, notifications all in one transaction. 5-second response time, database locked during peak. Split to microservices with Kafka: OrderService publishes OrderCreated → InventoryService reserves stock, ShippingService creates shipment, NotificationService sends email. Response time: 5s → 200ms. But eventual consistency — inventory showed available for 500ms after order. Added "pending inventory" status — users see accurate availability.

**Rule of thumb:** Kafka for decoupling microservices, event sourcing, real-time processing. Not for synchronous request-response — use REST/gRPC. Plan for eventual consistency — design UI/business logic accordingly.
