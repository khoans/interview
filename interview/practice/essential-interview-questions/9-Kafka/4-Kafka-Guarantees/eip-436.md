## 436. What is exactly-once delivery?

**Exactly-once delivery means messages processed exactly once — no loss, no duplicates.**

It's the strongest guarantee — requires coordination between producer and consumer.

**How to achieve:**
- Producer: transactional producer, isolation.level=read_committed
- Consumer: read_committed — only read committed transactions
- End-to-end: idempotent processing + transactional writes

**Trade-off:**
- No loss, no duplicates — strongest guarantee
- But: performance overhead — transaction coordination adds latency
- Complexity — transaction management in application code

**Real-world example:**
Our payment system needed exactly-once — duplicate charges unacceptable, lost payments unacceptable. Implemented transactional producer + transactional consumer. Duplicate charges: eliminated. Lost payments: eliminated. But transaction overhead added 30ms latency — acceptable for payment integrity. Also, transaction coordinator became bottleneck — single coordinator for all payments. Scaled to multiple coordinators — partitioned by payment_id range. Throughput: maintained. Exactly-once: preserved.

**Rule of thumb:** Exactly-once for financial transactions — payments, debits, credits. Not needed for most use cases — at-least-once + idempotent consumer suffices. Monitor transaction coordinator — potential bottleneck.
