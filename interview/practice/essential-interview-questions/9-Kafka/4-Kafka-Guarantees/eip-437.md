## 437. How do you achieve exactly-once semantics in Kafka?

**Exactly-once semantics requires: idempotent producer, transactional writes, read_committed consumer, idempotent processing.**

It's end-to-end exactly-once — from producer to consumer processing.

**Requirements:**
- **Idempotent producer** — no duplicates from producer retries
- **Transactional producer** — atomic multi-partition writes
- **read_committed consumer** — only read committed transactions
- **Idempotent consumer** — handle duplicates from rebalancing

**Trade-off:**
- End-to-end exactly-once — strongest guarantee
- But: complexity — transaction management, idempotent processing
- Performance — 20-30ms overhead for transactions

**Real-world example:**
Our banking system implemented exactly-once — debit from account A, credit to account B. Non-transactional: debit succeeded, credit failed — money vanished. Transactional producer: both operations in single transaction. Money lost: 0. But consumer rebalance caused duplicate processing — transaction committed, offset not committed, reprocessed. Added idempotent consumer — track processed transaction IDs. End-to-end exactly-once: achieved. Overhead: 25ms per transaction — acceptable for banking.

**Rule of thumb:** Exactly-once for financial — banking, payments, trading. Combine idempotent producer + transactions + read_committed + idempotent consumer. Test failure scenarios — rebalancing, crashes, retries.
