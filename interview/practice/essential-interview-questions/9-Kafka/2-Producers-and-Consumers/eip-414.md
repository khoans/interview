## 414. How does Kafka ensure message ordering?

**Kafka ensures ordering within a partition — messages are appended sequentially, consumers read in order.**

Ordering across partitions is not guaranteed — only within single partition.

**How it works:**
- Producer sends to partition — messages appended in order
- Consumer reads partition sequentially — offset 1, 2, 3...
- Partition key — same key always goes to same partition (ordering preserved)

**Trade-off:**
- Ordering within partition — strict guarantee
- But: no ordering across partitions — partition A and B independent
- Partition key required — must choose key that preserves ordering needs

**Real-world example:**
Our order system had OrderCreated and OrderUpdated events. Random partitioning — OrderUpdated went to partition A, OrderCreated to partition B. Consumer saw update before create — crash. Fixed: partition by order_id — same order always same partition. Ordering: preserved. But hot key issue — one order had 100 updates, single partition overloaded. Solution: partition by order_id, but monitor hot keys — split hot orders to dedicated topic if needed.

**Rule of thumb:** Partition by business key (order_id, user_id) for ordering. Don't expect ordering across partitions — design accordingly. Monitor hot keys — single partition overload.
