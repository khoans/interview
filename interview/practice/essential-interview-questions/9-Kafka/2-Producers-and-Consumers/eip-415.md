## 415. What is the difference between a topic and a partition?

**A topic is a logical category for messages. A partition is a physical subdivision of a topic for parallelism.**

Topic = table name. Partition = table shard.

**Topic:**
- Logical grouping — all order events in "orders" topic
- Multi-partition — topic split into N partitions
- Subscription unit — consumers subscribe to topics

**Partition:**
- Physical storage — each partition is separate log file
- Parallelism unit — consumers read different partitions
- Ordering unit — ordering guaranteed within partition only

**Trade-off:**
- Topic: logical abstraction — easy to reason about
- Partition: physical reality — determines parallelism, ordering
- Too few partitions — can't scale consumers
- Too many partitions — broker overhead, slower leadership election

**Real-world example:**
Our "events" topic had 1 partition — all events ordered, but single consumer bottleneck. Split to 10 partitions — 10 parallel consumers. Throughput: 100 → 1000 msg/s. But events from same user went to different partitions — user session events out of order. Fixed: new topic "user-events" partitioned by user_id. User ordering: preserved. Throughput: maintained. Lesson: choose partitioning strategy based on ordering requirements.

**Rule of thumb:** One topic per event type. Partitions for parallelism — match partition count to expected consumer parallelism. Partition by key for ordering within that key.
