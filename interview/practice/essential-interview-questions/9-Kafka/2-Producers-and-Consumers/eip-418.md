## 418. What is idempotent producer?

**Idempotent producer ensures exactly-once message delivery per partition — retries don't cause duplicates.**

It's producer-side deduplication — each message has unique sequence number.

**How it works:**
- Producer assigns sequence number to each message
- Broker tracks highest sequence number per partition
- Duplicate (retry) has same sequence number — broker deduplicates
- Result: exactly-once per partition, even with retries

**Trade-off:**
- Exactly-once per partition — no duplicates from producer retries
- But: only per partition — not end-to-end (consumer side still needs idempotence)
- Minimal overhead — sequence numbers are lightweight

**Real-world example:**
Our order processor had duplicate orders — producer retry on transient failure sent same order twice. Customer charged twice, support tickets flooded. Enabled idempotent producer (enable.idempotence=true). Duplicates: eliminated. But idempotent producer only prevents producer-side duplicates — consumer still reprocessed on rebalance. Added consumer-side idempotence (track processed order IDs). End-to-end exactly-once: achieved.

**Rule of thumb:** Enable idempotent producer for all producers — prevents retry duplicates. But not end-to-end exactly-once — consumer must also be idempotent. Combine with transactional consumer for full exactly-once.
