## 416. What is acks configuration in Kafka producer?

**Acks configuration controls how many replicas must acknowledge a message before producer considers it successful.**

It's the primary durability vs. latency trade-off in Kafka.

**Options:**
- **acks=0** — no acknowledgment, fire-and-forget
- **acks=1** — leader acknowledges (leader written to disk)
- **acks=all** — all in-sync replicas acknowledge (full durability)

**Trade-off:**
- acks=0: fastest, but message loss on broker failure
- acks=1: balanced — leader durability, but follower lag risk
- acks=all: safest — survives leader failure, but highest latency

**Real-world example:**
Our analytics pipeline used acks=0 — 1M events/hour, 0.1% loss acceptable. Worked until broker crash — 10 minutes of data lost (100K events). Business impact: analytics gap, incorrect dashboards. Changed to acks=1 — leader acknowledges. Data loss: 0.1% → 0%. Latency: 1ms → 5ms. Acceptable. For financial transactions, use acks=all — no data loss tolerated. Latency: 5ms → 15ms. Worth it for financial integrity.

**Rule of thumb:** acks=1 for most use cases — balanced durability/latency. acks=all for financial/critical data — no loss tolerated. acks=0 only for truly disposable data (metrics, debug logs).
