## 419. What is transactional producer?

**Transactional producer enables exactly-once semantics across partitions and topics — atomic multi-partition, multi-topic writes.**

It's distributed transactions for Kafka — all-or-nothing across multiple partitions.

**How it works:**
- Producer begins transaction
- Writes to multiple partitions/topics
- Commits transaction — all writes visible atomically
- Or aborts — all writes discarded

**Trade-off:**
- Exactly-once across partitions — atomic multi-partition writes
- But: performance overhead — transaction coordination adds latency
- Complexity — transaction management in producer code

**Real-world example:**
Our order system wrote to two topics — orders-created and inventory-reserved. Non-transactional: order created, inventory update failed — inconsistent state. Implemented transactional producer — both writes in single transaction. Consistency: guaranteed. But transaction overhead added 20ms latency — acceptable for order integrity. Also, transaction coordinator became bottleneck — single point for all transactions. Scaled to multiple transaction coordinators — load distributed.

**Rule of thumb:** Transactional producer for multi-topic/partition atomicity — order + inventory, debit + credit. Not needed for single-partition writes — idempotent producer suffices. Monitor transaction coordinator — potential bottleneck.
