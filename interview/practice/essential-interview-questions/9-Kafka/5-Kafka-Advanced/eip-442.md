## 442. What is Avro? Why use it with Kafka?

**Avro is a binary serialization format with schema — compact, fast, schema-evolution friendly.**

It's the default choice for Kafka — Schema Registry integration, efficient serialization.

**Benefits:**
- **Compact** — binary format, no field names in message
- **Fast** — efficient serialization/deserialization
- **Schema evolution** — add/remove fields with compatibility
- **Schema Registry** — built-in integration

**Trade-off:**
- Compact — 50% smaller than JSON
- But: not human-readable — need schema to decode
- Schema dependency — consumer needs schema to deserialize

**Real-world example:**
Our Kafka topics used JSON — 500 bytes per message, 100K msg/s = 50MB/s network. Switched to Avro — 200 bytes per message, 20MB/s network. Network bandwidth: -60%. Also, JSON had no schema — field name typos caused silent data corruption. Avro + Schema Registry: schema enforced, typos caught at compile time. Data corruption: eliminated. But debugging harder — Avro not human-readable. Added Kafka UI with Schema Registry integration — messages decoded with schema, readable in UI.

**Rule of thumb:** Avro for production Kafka — compact, schema-enforced. JSON for debugging/development — human-readable. Schema Registry — enforce schema compatibility. Kafka UI with schema support — debugging support.
