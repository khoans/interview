## 448. How do you tune Kafka performance?

**Kafka performance tuning involves: batch size, compression, partition count, broker resources, and consumer parallelism.**

It's about balancing throughput, latency, and resource utilization.

**Tuning knobs:**
- **Producer batch** — larger batches = higher throughput, higher latency
- **Compression** — lz4/snappy = smaller messages, CPU overhead
- **Partitions** — more partitions = more parallelism, more broker overhead
- **Broker resources** — disk I/O, network, memory
- **Consumer parallelism** — consumers = partitions for max throughput

**Trade-off:**
- Throughput vs. latency — batch size, acks configuration
- Parallelism vs. overhead — partition count
- Compression vs. CPU — smaller messages, more CPU

**Real-world example:**
Our Kafka throughput was 10K msg/s — needed 100K msg/s. Tuning: batch.size 16KB → 1MB, linger.ms 0 → 100ms, compression none → lz4. Throughput: 10K → 100K msg/s. Latency: 5ms → 105ms (batch wait). Acceptable for use case. Also increased partitions 10 → 100 — consumers scaled 10 → 100. Broker resources: disk upgraded to SSD, network 1Gbps → 10Gbps. End-to-end: 10K → 100K msg/s, stable.

**Rule of thumb:** Tune batch size and linger.ms for throughput — accept latency trade-off. Compression for large messages — lz4 good balance. Partitions = expected consumer parallelism. SSD for high-throughput brokers.
