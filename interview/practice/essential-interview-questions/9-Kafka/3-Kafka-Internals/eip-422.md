## 422. What is log compaction in Kafka?

**Log compaction retains only the latest value per key — older values with same key are removed.**

It's for changelog use cases — current state, not full history.

**How it works:**
- Messages have keys — user_id, product_id
- Compactor scans log — keeps latest value per key
- Older values for same key deleted — only latest remains
- Result: compacted log with current state per key

**Trade-off:**
- Storage efficiency — only current state, not full history
- But: not for all use cases — event sourcing needs full history
- Compaction overhead — background process consumes resources

**Real-world example:**
Our user profile updates went to Kafka — 1M users, 100 updates/user/day. Full history: 100M messages/day, storage grew unbounded. Enabled log compaction — only latest profile per user retained. Storage: 100M messages → 1M messages (99% reduction). But audit team needed full history — compaction deleted audit trail. Solution: two topics — "user-profiles" (compacted, current state) and "user-profile-changes" (full history, 7-day retention). Both needs met.

**Rule of thumb:** Log compaction for current state — user profiles, product catalog. Not for event sourcing — use retention policy instead. Two topics if you need both current state and history.
