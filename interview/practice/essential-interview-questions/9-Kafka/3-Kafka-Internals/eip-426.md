## 426. What is ISR (In-Sync Replicas)?

**ISR is the set of replicas that are fully caught up with leader — within configured lag threshold.**

Only ISR members can become leader — ensures no data loss on failover.

**How it works:**
- Leader tracks follower lag — messages behind leader
- Follower within lag threshold (10 seconds default) — in ISR
- Follower exceeds threshold — removed from ISR
- Leader election — only ISR members eligible

**Trade-off:**
- ISR ensures durability — new leader has all committed messages
- But: small ISR reduces availability — can't write if ISR too small
- min.insync.replicas — enforce minimum ISR size

**Real-world example:**
Our topic had replication factor 3 — 1 leader, 2 followers. Network partition — follower isolated, removed from ISR. ISR: 3 → 2. Continued operating. Then leader failed — remaining ISR member became leader. Zero data loss. But min.insync.replicas=1 (default) — accepted writes with ISR=1. That single replica failed — data loss. Changed to min.insync.replicas=2 — writes rejected if ISR < 2. Writes rejected: 0.1% of time. Data loss: eliminated.

**Rule of thumb:** min.insync.replicas = replication factor - 1. For RF=3, min.insync.replicas=2. Writes rejected when ISR too small — better than data loss. Monitor ISR size — alert when below threshold.
