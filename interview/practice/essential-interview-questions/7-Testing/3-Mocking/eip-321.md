## 321. What is mocking?

**Mocking creates fake objects that simulate real dependencies — controlled behavior for testing without real dependencies.**

It's about isolating the unit under test — replace database, API, file system with predictable fakes.

**When to use:**
- External dependencies — databases, APIs, file systems
- Slow operations — network calls, complex calculations
- Unreliable dependencies — flaky services, rate-limited APIs

**Trade-off:**
- Fast, isolated tests — no external dependencies
- But: mocks can drift from reality — mock passes, integration fails
- Over-mocking — tests verify mock interactions, not actual behavior

**Real-world example:**
Our payment service tests called real payment gateway — slow (5 seconds per test), rate-limited (100 calls/hour), flaky (network issues). Mocked payment gateway: 5ms per test, unlimited calls, deterministic. Test suite: 30 minutes → 2 minutes. But mock drifted from real API — tests passed, integration failed. Added contract tests to verify mock matches real API behavior.

**Rule of thumb:** Mock external dependencies — databases, APIs, file systems. Don't mock business logic — test real code. Verify mocks match real behavior with contract tests.
