## 317. What is the red-green-refactor cycle?

**Red-green-refactor is TDD's core cycle — red (write failing test), green (make it pass), refactor (improve code).**

It's about incremental development — small steps, constant feedback, safe refactoring.

**Three phases:**
- **Red** — write test for feature that doesn't exist. Test fails (red).
- **Green** — write minimal code to pass test. Don't worry about quality. Test passes (green).
- **Refactor** — improve code quality. Tests ensure no regression.

**Trade-off:**
- Small steps — easy to debug, reversible
- Constant feedback — know immediately if broken
- But: feels slow — write test, write code, repeat
- Discipline required — resist skipping phases

**Real-world example:**
Our developer skipped green phase — wrote test (red), then wrote "proper" code with error handling, logging, validation. Test still red. 2 hours debugging — over-engineered solution. Coach intervened: write minimal code to pass test. 5 lines, test green. Then refactor — add error handling, logging. Total: 30 minutes. Lesson: green phase is "make it work," refactor phase is "make it right."

**Rule of thumb:** Red-green-refactor in order. Green phase: minimal code, even if ugly. Refactor phase: improve with test safety net. Cycle time: 5-10 minutes per cycle. Longer cycles mean too big steps.
