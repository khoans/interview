## 320. How should you write code that is unit testable?

**Testable code has loose coupling, clear responsibilities, dependency injection, and minimal side effects — designed for testing from start.**

It's not about making code testable — it's about writing good code that happens to be testable.

**Key practices:**
- **Dependency injection** — pass dependencies, don't create them
- **Single responsibility** — one class, one job
- **Pure functions** — same input, same output, no side effects
- **Interfaces** — depend on abstractions, not concretions

**Trade-off:**
- Testable code — better design, loosely coupled
- But: more interfaces, more classes — perceived complexity
- Initial overhead — design before coding

**Real-world example:**
Our legacy code: new DatabaseConnection() inside business logic. Untestable — test required real database. Refactored: constructor injection — DatabaseConnection passed in constructor. Test: pass mock connection. But refactoring took 2 weeks — 50 classes, all had hardcoded dependencies. After: test coverage 10% → 80%. Bug rate: 20/month → 2/month. Refactoring paid for itself in 2 months — less debugging, fewer production incidents.

**Rule of thumb:** Dependency injection for all external dependencies. Single responsibility — classes under 200 lines. Pure functions where possible — no side effects. Design for testing — write test first, code second.
