## 319. What should you NOT unit test?

**Don't unit test trivial code, external dependencies, or implementation details — waste of time, false confidence, or maintenance burden.**

Focus testing effort where bugs hide — business logic, edge cases, complex algorithms.

**What NOT to test:**
- **Trivial code** — getters, setters, constants
- **External dependencies** — frameworks, libraries (they have their own tests)
- **Implementation details** — private methods, internal state
- **Third-party code** — Spring, Hibernate, Apache Commons

**Trade-off:**
- Focus on value — test business logic, not plumbing
- But: judgment required — what's trivial vs. critical
- Risk: external library bug affects you, but you didn't test integration

**Real-world example:**
Our team tested getters/setters — 100 tests for 100 getters. Zero bugs found. Deleted all 100 tests. Also tested Hibernate annotations — test passed, but integration failed (wrong column mapping). Should have tested integration, not annotations. But didn't test payment calculation (third-party library) — library had bug, charged wrong amount. Lesson: don't test trivial, do test integration with third-party.

**Rule of thumb:** Test business logic — calculations, validations, workflows. Don't test getters/setters. Test integration with external libraries — not the library, but your usage of it.
