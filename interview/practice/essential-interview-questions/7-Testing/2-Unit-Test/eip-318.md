## 318. What makes a good unit test?

**Good unit tests are fast, isolated, repeatable, self-validating, and timely — FIRST principles.**

They provide confidence — tests pass, code works. Tests fail, bug found.

**FIRST principles:**
- **Fast** — milliseconds per test, don't wait for tests
- **Isolated** — no dependencies, no shared state
- **Repeatable** — same result every time, no randomness
- **Self-validating** — pass/fail clear, no manual checking
- **Timely** — written before or during development, not after

**Trade-off:**
- Strict isolation — tests are robust, but more mocks
- Fast execution — in-memory, no I/O, but less realistic
- Readable — test name describes scenario, assertions clear

**Real-world example:**
Our bad test: connected to real database, depended on system time, random data, manual verification. Test time: 5 seconds. Flakiness: 20% failure rate. Refactored: in-memory database, fixed time, deterministic data, assertions. Test time: 50ms. Flakiness: 0%. But isolation revealed coupling — code depended on system clock. Refactored code to accept Clock parameter — testable, better design.

**Rule of thumb:** FIRST principles for all unit tests. Test name: shouldExpectedBehavior_whenCondition. Assertions: clear failure message. Setup: under 10 lines. Execution: under 10ms.
