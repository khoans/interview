## 316. What is test-driven development (TDD)?

**TDD is development practice: write test first (fails), write code (passes), refactor — repeat.**

It's about designing through testing — tests drive code design, not afterthought.

**TDD cycle:**
1. **Red** — write failing test (feature doesn't exist yet)
2. **Green** — write minimal code to pass test
3. **Refactor** — improve code, tests ensure no regression

**Trade-off:**
- Better design — testable code, loose coupling
- Confidence — tests prove code works
- But: slower initially — write tests first, more code
- Learning curve — discipline required, not intuitive

**Real-world example:**
Our team tried TDD — first sprint, velocity dropped 50%. Tests first felt unnatural. But bug rate: 20/week → 2/week. Second sprint, velocity recovered — less debugging, less rework. Third sprint, velocity exceeded pre-TDD — confident refactoring, no regression testing. But TDD not for everything — UI, exploratory features, prototypes. Used TDD for business logic, traditional development for UI.

**Rule of thumb:** TDD for business logic, algorithms, complex calculations. Not for UI, prototypes, exploratory features. Start small — one feature, learn cycle. Expect velocity drop first sprint, recovery second, improvement third.
