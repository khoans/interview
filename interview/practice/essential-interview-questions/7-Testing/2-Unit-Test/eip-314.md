## 314. What is @BeforeAll and @AfterAll?

**@BeforeAll runs once before all tests in class — expensive setup. @AfterAll runs once after all tests — expensive cleanup.**

They're for expensive operations — database connections, starting servers — that can be shared across tests.

**When to use:**
- @BeforeAll — start embedded database, connect to test services, load test data
- @AfterAll — close database connections, stop servers, delete test data

**Trade-off:**
- Performance — expensive setup once, not per test
- But: shared state — tests can interfere if they modify shared resources
- Order dependency — tests must not depend on execution order

**Real-world example:**
Our integration tests created fresh database per test — 5 seconds per test, 100 tests = 500 seconds. Moved database creation to @BeforeAll — 5 seconds total, 100 tests = 10 seconds. But tests modified shared database — test A created user, test B expected no users. Fixed: @BeforeEach rolls back database to clean state, @BeforeAll creates database once. Best of both: fast setup, isolated tests.

**Rule of thumb:** @BeforeAll for expensive shared resources — databases, servers. @BeforeEach for test isolation — rollback, reset. Never share mutable state between tests without reset.
