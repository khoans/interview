## 339. How do you test private methods?

**Don't test private methods directly — test through public API. Private methods are implementation details, not contract.**

If you feel need to test private method, it's a code smell — extract to public class.

**Approaches:**
- **Test through public** — call public method, private method exercised indirectly
- **Extract to class** — complex private logic → new class with public interface
- **Reflection** — technically possible, but don't — fragile, implementation coupling

**Trade-off:**
- Encapsulation preserved — private stays private
- Refactoring safe — change private method, tests don't break
- But: indirect testing — can't verify private method specifically

**Real-world example:**
Our class had complex private calculateDiscount() method. Wanted to test directly — 10 edge cases. Refactored: extracted DiscountCalculator class with public calculate() method. Original class uses DiscountCalculator. Tests: 10 edge cases for DiscountCalculator, simple test for original class. Both classes testable, maintainable. But extraction took 2 hours — worth it for testability and single responsibility.

**Rule of thumb:** Test through public API — private methods tested indirectly. Complex private logic: extract to testable class. Never use reflection — implementation coupling, fragile tests.
