## 345. What is flaky test?

**Flaky test passes sometimes, fails other times — same code, different result. Non-deterministic, untrustworthy.**

It's the enemy of test confidence — developers ignore failures.

**Common causes:**
- Timing issues — race conditions, async operations
- External dependencies — network, database, shared state
- Test order dependency — test A modifies state, test B fails
- Randomness — random data, system time

**Trade-off:**
- Quick fix: retry — test passes eventually
- But: doesn't fix root cause — flakiness remains
- Proper fix: identify root cause, fix — time-consuming

**Real-world example:**
Our test suite had 20 flaky tests — 30% failure rate. Developers ignored failures — "probably flaky". Production bug: real failure ignored as flaky, 1000 users affected. Fixed flaky tests: identified causes (timing: 10, shared state: 5, network: 5). Fixed: async waits, test isolation, mocks. Flaky tests: 20 → 0. Took 2 weeks. But trust restored — failures taken seriously.

**Rule of thumb:** Flaky test = highest priority fix. Don't retry — fix root cause. Common fixes: async waits (Awaitility), test isolation (@BeforeEach cleanup), mock external dependencies.
