## 344. What is mutation testing?

**Mutation testing measures test quality — introduces small bugs (mutations), verifies tests catch them.**

It's about testing your tests — do tests actually catch bugs?

**How it works:**
- Mutant created — change == to !=, remove method call, change constant
- Tests run — if test fails, mutant killed (good test)
- If test passes, mutant survived (test gap)
- Mutation score — percentage of mutants killed

**Trade-off:**
- Finds test gaps — tests pass but don't catch bugs
- Confidence — high mutation score = good test quality
- But: slow — run tests for each mutant (100s of mutants)
- False positives — some mutants equivalent (behavior unchanged)

**Real-world example:**
Our code had 80% code coverage — looked good. Mutation testing (PIT): 80% coverage, 40% mutation score. 60% of mutants survived — tests didn't catch bugs. Example: condition removed, tests passed (didn't verify outcome). Added assertions — mutation score: 40% → 85%. But mutation testing added 30 minutes to CI — too slow for every commit. Ran nightly, not on every commit.

**Rule of thumb:** Mutation testing for critical code — business logic, calculations. Run nightly — too slow for every commit. Target: 80%+ mutation score for critical paths.
