## 342. What is parameterized testing?

**Parameterized testing runs same test with different inputs — one test method, multiple test cases.**

It's about DRY tests — avoid copy-paste test for each input variation.

**When to use:**
- Boundary testing — min, max, zero, negative values
- Multiple inputs — 10 different inputs, same expected behavior
- Edge cases — null, empty, special characters

**Trade-off:**
- DRY — one test method, many cases
- Coverage — test all edge cases systematically
- But: harder to debug — which parameter failed?
- Overuse — 100 parameters, test does too much

**Real-world example:**
Our validation had 10 test methods — testValidEmail, testInvalidEmail, testNullEmail... Parameterized test: @ValueSource(strings = {"valid@example.com", "invalid", "null"}). One test method, 10 inputs. Test count: 50 → 5. Coverage: same. But test failed — which input? Added display name — @ParameterizedTest(name = "{index}: {0}"). Failure message: "test 3: null". Clear which input failed.

**Rule of thumb:** Parameterized tests for input variations — boundaries, edge cases. Use display names — clear which parameter failed. Don't mix unrelated cases — one test per behavior.
