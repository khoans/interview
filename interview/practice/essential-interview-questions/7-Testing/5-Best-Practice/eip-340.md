## 340. How do you test static methods?

**Static methods are hard to test — can't mock, tightly coupled. Refactor to instance methods or use PowerMock (last resort).**

Static methods violate testability — dependency you can't replace.

**Approaches:**
- **Refactor to instance** — static → instance method, inject dependency
- **Wrapper class** — static method in wrapper, mock wrapper
- **PowerMock** — mock static methods — but complex, fragile

**Trade-off:**
- Instance methods: testable, mockable — but requires refactoring
- PowerMock: works without refactoring — but complex, slow, breaks with Java updates
- Static: convenient — but untestable without special tools

**Real-world example:**
Our utility class had 20 static methods — StringUtils.isEmpty(), DateUtils.format(). Untestable in isolation. Refactored: StringUtils → StringService (injectable), DateUtils → DateService. Tests: mock services, test in isolation. But 200 call sites to update — static to instance. Automated refactoring (IDE): 2 hours. Test coverage: 0% → 80%. PowerMock considered — rejected due to complexity and Java upgrade issues.

**Rule of thumb:** Avoid static methods — instance methods with dependency injection. Legacy static: wrapper class or extract to service. PowerMock: last resort, plan refactoring.
