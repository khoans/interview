## 347. What is test fixture?

**Test fixture is fixed state for testing — known data, known environment, repeatable results.**

It's about consistent test conditions — same input, same output, every time.

**Types:**
- **Data fixture** — known database state before test
- **Environment fixture** — known configuration, mocks
- **Object fixture** — known object state

**Trade-off:**
- Repeatable — same test, same result
- Debuggable — known state, easy to reproduce failures
- But: setup cost — create fixture before each test
- Maintenance — fixture must evolve with code

**Real-world example:**
Our tests had ad-hoc setup — each test created data differently. Same test, different results (different data). Implemented test fixtures — createTestUser(), createTestOrder(), createTestPayment(). All tests use fixtures. Results: repeatable, debuggable. But fixtures grew — 20 helper methods. Organized: UserFixture, OrderFixture, PaymentFixture. Each fixture class: 5-10 methods. Maintainable, reusable.

**Rule of thumb:** Test fixtures for repeatable tests — helper methods, fixture classes. Keep fixtures simple — 5-10 methods per fixture class. Update fixtures when domain changes.
