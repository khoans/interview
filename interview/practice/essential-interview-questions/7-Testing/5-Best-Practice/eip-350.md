## 350. What is shift-left testing?

**Shift-left testing moves testing earlier in development — test during development, not after. Find bugs when cheapest to fix.**

It's about cost efficiency — bug found in development costs 1 hour. Bug found in production costs 100 hours.

**How to shift left:**
- TDD — write test before code
- Code review — review tests with code
- CI — tests on every commit, not before release

**Trade-off:**
- Early bug detection — fix cost 10x cheaper
- But: requires discipline — write tests first, not after
- Cultural change — developers own testing, not QA only

**Real-world example:**
Our process: develop 2 weeks, QA test 1 week, fix bugs 1 week. Bugs found in QA — 2-week-old code, context lost, fix took 2 days. Shifted left: TDD, tests on every commit, QA involved in sprint. Bugs found: same day as code written. Fix time: 30 minutes vs. 2 days. Release cycle: 4 weeks → 1 week. But cultural change — developers resisted "more work". Showed data: bug fix time 2 days → 30 minutes. Adoption: 3 months.

**Rule of thumb:** Shift-left for all development — TDD, CI, early QA involvement. Measure bug fix time — before/after shift-left. Cultural change takes time — show data, celebrate wins.
