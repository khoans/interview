## 341. How do you test exceptions?

**Test exceptions with assertThrows (JUnit 5) or @Test(expected = ...) — verify exception type, message, and context.**

It's about testing error paths — not just happy path.

**Approaches:**
- **assertThrows** — JUnit 5, captures exception, verify message
- **@Test(expected)** — JUnit 4, less flexible
- **Try-catch in test** — verbose, avoid

**Trade-off:**
- assertThrows: flexible — verify type, message, properties
- But: testing messages is fragile — message changes, test breaks
- Better: test behavior — exception thrown, state unchanged

**Real-world example:**
Our service threw IllegalArgumentException for invalid input. Test: assertThrows(IllegalArgumentException.class, () -> service.method(null)). But message changed ("null input" → "input required"), test broke. Refactored: verify exception type and state — assert user not created, assert database not modified. Message changes didn't break tests. But missed testing exception properties — error code, field name. Added verification: exception.getErrorCode() == "INVALID_INPUT".

**Rule of thumb:** assertThrows for exception type. Verify side effects — state unchanged, rollback occurred. Don't rely solely on exception message — verify error codes, properties.
