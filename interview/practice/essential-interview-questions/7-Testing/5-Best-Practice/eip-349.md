## 349. What is continuous testing?

**Continuous testing runs tests automatically on every code change — commit, build, deploy. Tests are gate, not afterthought.**

It's about fast feedback — know immediately if change broke something.

**How it works:**
- Pre-commit — tests run before commit (local)
- CI pipeline — tests run on every push (automated)
- Quality gate — tests must pass before merge/deploy

**Trade-off:**
- Fast feedback — catch bugs within minutes
- But: test suite must be fast — under 10 minutes
- Flaky tests block deployment — must fix immediately

**Real-world example:**
Our team ran tests manually — before release, 2 days testing. Bugs found late, expensive fixes. Implemented continuous testing: pre-commit (5 minutes), CI (15 minutes), quality gate (must pass). Bugs caught: day of commit vs. 2 weeks later. Fix cost: 1 hour vs. 2 days. But test suite was 1 hour — too slow. Optimized: parallel execution, test slicing. 1 hour → 15 minutes. Developers happy — fast feedback, not blocked.

**Rule of thumb:** Continuous testing for every commit — pre-commit + CI. Test suite under 15 minutes — parallelize, slice. Flaky tests = highest priority — block deployment.
