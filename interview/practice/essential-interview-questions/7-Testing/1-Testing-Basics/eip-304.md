## 304. What is integration testing?

**Integration testing verifies multiple components work together — database connections, external APIs, service-to-service communication.**

It catches bugs unit tests miss — integration points, contracts, data flow.

**When to use:**
- Database operations — queries, transactions, migrations
- External APIs — payment gateways, email services, third-party integrations
- Service boundaries — microservices communication, message queues

**Trade-off:**
- Catches integration bugs — unit tests can't find these
- But: slower — database setup, network calls
- Flakier — external services can be unavailable

**Real-world example:**
Our order service had perfect unit tests — integration failed in production. Database transaction isolation caused deadlocks under load. Unit tests used in-memory database (no transactions). Added integration tests with real PostgreSQL — caught deadlock in staging. Fixed with proper transaction isolation levels. But integration tests were slow — 30 seconds per test. Added test containers (ephemeral databases) — 30 seconds → 5 seconds.

**Rule of thumb:** Integration test all external dependencies — databases, APIs, message queues. Use test containers or dedicated test environments — not production. Keep tests fast — under 10 seconds each.
