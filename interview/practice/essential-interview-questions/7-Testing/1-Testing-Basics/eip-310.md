## 310. What is exploratory testing?

**Exploratory testing is unscripted testing — tester explores application, learns, designs tests on the fly.**

It's about finding bugs automated tests miss — edge cases, usability issues, unexpected interactions.

**When to use:**
- New features — before writing automated tests
- Complex workflows — automated tests too expensive
- Before release — fresh eyes, human intuition

**Trade-off:**
- Finds unexpected bugs — automation follows script, humans explore
- Usability feedback — humans notice UX issues
- But: not repeatable — can't automate, manual effort each time
- Coverage unknown — what was tested, what wasn't

**Real-world example:**
Our checkout flow had 100 automated tests — all passed. Exploratory tester tried: add item, go back, change quantity, proceed. Bug: cart showed old quantity, charged new quantity. $50 discrepancy per order. Automated tests didn't cover this flow — too complex to script. Exploratory testing found it in 30 minutes. But exploratory testing found 5 bugs in first session, 0 in second session. Diminishing returns — automate stable flows, explore new features.

**Rule of thumb:** Exploratory testing for new features, complex workflows, pre-release. Not replacement for automated tests — complementary. Time-box sessions — 1-2 hours, diminishing returns after.
