## 305. What is end-to-end testing?

**End-to-end testing validates complete user flows — from UI to database, all components working together.**

It's the highest confidence testing — if E2E passes, user can complete their task.

**When to use:**
- Critical user journeys — checkout, login, signup
- Cross-service workflows — order placement spans 5 microservices
- Release validation — smoke test before production deployment

**Trade-off:**
- Highest confidence — tests real user experience
- But: slowest — full stack, multiple services
- Flakiest — UI changes, timing issues, network failures
- Most expensive — maintenance, infrastructure

**Real-world example:**
Our e-commerce had 200 E2E tests — full checkout flow for every product category. Test suite: 4 hours. Flaky tests: 30%. Developers ignored failures. Reduced to 20 E2E tests — critical paths only (guest checkout, registered checkout, express checkout). Test suite: 30 minutes. Flaky tests: 5%. Failures taken seriously. But missed bugs in edge cases — gift card checkout broken for 2 weeks. Added targeted integration tests for edge cases.

**Rule of thumb:** E2E tests for critical user journeys only — 10-20 tests max. Not for every feature — too slow, too flaky. Supplement with integration tests for edge cases.
