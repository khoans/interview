## 303. What is unit testing?

**Unit testing verifies single unit of code in isolation — one class or method, no external dependencies.**

It's the foundation of testing pyramid — fast, cheap, numerous.

**Key characteristics:**
- Isolated — mocks/stubs for all dependencies
- Fast — milliseconds per test
- Deterministic — same input, same output, no randomness
- Automated — runs on every commit

**Trade-off:**
- Fast feedback — catch bugs before commit
- Documentation — tests show how to use code
- But: doesn't catch integration issues — unit works, system fails
- Maintenance — refactor code, update tests

**Real-world example:**
Our payment service had no unit tests — only manual testing. Bug: rounding error charged customers $0.01 extra. 10,000 customers affected, $100 refund per customer, $1M total. Added unit tests for all calculations — edge cases, rounding, boundary conditions. Rounding bugs: 5/year → 0. But false confidence — unit tests passed, integration failed (payment gateway timeout). Added integration tests for external calls.

**Rule of thumb:** Unit test all business logic — calculations, validations, transformations. Don't unit test trivial code (getters, setters). Mock external dependencies — databases, APIs, file systems.
