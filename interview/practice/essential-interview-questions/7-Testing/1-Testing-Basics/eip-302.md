## 302. What is the testing pyramid?

**Testing pyramid is a testing strategy — many fast unit tests at base, fewer integration tests in middle, fewest slow E2E tests at top.**

It's about maximizing test coverage while minimizing execution time and maintenance cost.

**Pyramid structure:**
- **Base (70%)** — unit tests — fast, cheap, isolated
- **Middle (20%)** — integration tests — service boundaries, database, external APIs
- **Top (10%)** — E2E tests — critical user journeys only

**Trade-off:**
- Pyramid: fast feedback, low maintenance — most bugs caught by unit tests
- Inverted pyramid (ice cream cone): slow, flaky, expensive — too many E2E tests

**Real-world example:**
Our team had inverted pyramid — 100 unit tests, 200 integration, 500 E2E tests. Test suite: 2 hours. Flaky tests: 50. Developers skipped tests. Rebalanced to pyramid: 1000 unit tests, 200 integration, 50 E2E. Test suite: 15 minutes. Flaky tests: 5. Developers ran tests locally. Production bugs: 15/month → 3/month. But required refactoring — extracted business logic from controllers for unit testing. 2 weeks refactoring, 6 months payoff.

**Rule of thumb:** Pyramid ratio: 70% unit, 20% integration, 10% E2E. Measure test suite time — should be under 10 minutes for local, 30 minutes for CI. Delete flaky tests — they erode trust.
