## 307. What is regression testing?

**Regression testing ensures new changes don't break existing functionality — re-run tests after every change.**

It's about preventing "two steps forward, one step back" — new features shouldn't break old features.

**When to use:**
- Every code change — automated regression suite on every commit
- Before releases — comprehensive regression testing
- After bug fixes — verify fix didn't introduce new bugs

**Trade-off:**
- Catches regressions early — before production
- But: suite grows over time — 1000+ tests, slow execution
- Maintenance — tests for deprecated features still run

**Real-world example:**
Our SaaS platform had manual regression testing — 2 days before each release. Missed regression: password reset broken after email template update. Customer complaints: 500 in first hour. Automated regression suite: 500 tests, runs on every commit. Next regression (search broken after UI update): caught in CI, zero customer impact. But suite grew to 2000 tests — 1 hour execution time. Added test parallelization — 1 hour → 10 minutes.

**Rule of thumb:** Automated regression suite for every commit. Prune tests annually — remove tests for deprecated features. Parallelize test execution — keep feedback under 15 minutes.
