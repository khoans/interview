## 301. What are the different types of testing?

**Testing types span from fast, isolated unit tests to slow, comprehensive end-to-end tests — each catches different bugs at different cost.**

Choose testing mix based on risk, speed requirements, and maintenance cost.

**Main types:**
- **Unit tests** — single class/method, no dependencies, milliseconds
- **Integration tests** — multiple components working together, databases, external services
- **End-to-end tests** — full system, real user flows, slowest but most confidence
- **Functional tests** — verify business requirements, what system should do
- **Regression tests** — ensure new changes don't break existing functionality

**Trade-off:**
- Unit tests: fast, cheap, isolated — but miss integration issues
- E2E tests: catch real user issues — but slow, flaky, expensive to maintain
- Integration tests: middle ground — catch service interaction bugs

**Real-world example:**
Our e-commerce platform had 80% unit tests, 15% integration, 5% E2E. Unit tests passed, production failed — payment service couldn't connect to database. Integration tests missing. Rebalanced: 50% unit, 35% integration, 15% E2E. Production bugs: 10/month → 2/month. Test suite time: 5 minutes → 20 minutes. Trade-off accepted — fewer production incidents, on-call pages dropped 80%.

**Rule of thumb:** Follow testing pyramid — more unit tests, fewer E2E tests. Integration tests for service boundaries. E2E for critical user journeys only.
