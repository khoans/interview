## 306. What is functional testing?

**Functional testing verifies system meets business requirements — does the feature do what users expect?**

It's about testing behavior, not implementation — what system does, not how it does it.

**Key characteristics:**
- Black-box — doesn't know internal implementation
- Requirements-based — test cases from user stories
- User perspective — inputs, expected outputs

**Trade-off:**
- Validates business value — features work as intended
- But: doesn't test internal quality — code could be unmaintainable
- Often manual — requires human judgment

**Real-world example:**
Our banking app had functional test: "Transfer $100 from savings to checking." Test passed — balance updated. But implementation had race condition — concurrent transfers allowed overdraft. Functional test didn't catch concurrency bug. Added non-functional tests (load testing, concurrency testing). But functional testing caught critical bug: interest calculation wrong for leap year. $500K in incorrect interest payments prevented.

**Rule of thumb:** Functional testing for business requirements validation. Not sufficient alone — combine with unit tests (code quality), integration tests (service boundaries), non-functional tests (performance, security).
