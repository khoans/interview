## 336. What is contract testing?

**Contract testing verifies API compatibility between services — provider honors contract, consumer expectations met.**

It's about preventing breaking changes — service changes don't break dependent services.

**How it works:**
- Consumer defines expectations — what requests sent, what responses expected
- Provider verifies contract — responses match expectations
- Contract stored — versioned, shared between teams

**Trade-off:**
- Prevents breaking changes — catch incompatibilities before deployment
- But: contract maintenance — update contract when API changes
- Tooling overhead — Pact, Spring Cloud Contract learning curve

**Real-world example:**
Our payment service changed API — added required field. Order service broke — didn't send new field. Production incident: 1000 failed payments. Implemented contract testing with Pact. Payment service (provider) defines contract. Order service (consumer) defines expectations. CI verifies contract before merge. Next API change: contract test failed, caught before deployment. Zero breaking changes since 12 months.

**Rule of thumb:** Contract testing for service-to-service APIs — microservices communication. Not for internal methods — unit tests sufficient. Use Pact or Spring Cloud Contract — industry standard tools.
