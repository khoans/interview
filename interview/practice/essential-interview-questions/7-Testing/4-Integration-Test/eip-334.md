## 334. How do you test REST APIs?

**Test REST APIs at multiple levels — unit test controllers, integration test full endpoints, contract test API compatibility.**

Different testing strategies catch different bugs.

**Testing levels:**
- **Unit** — @WebMvcTest, mock services, test controller logic
- **Integration** — @SpringBootTest, real services, test full endpoint
- **Contract** — verify API contract (request/response format)
- **E2E** — real deployment, test user flows

**Trade-off:**
- Unit: fast, isolated — but misses integration issues
- Integration: catches service integration — but slower
- Contract: ensures API compatibility — but requires contract definition

**Real-world example:**
Our REST API had only unit tests — @WebMvcTest, mocked services. Production failed: service returned different format than controller expected. Added integration tests — @SpringBootTest, real services. Caught format mismatch. But API changed, clients broke. Added contract tests — OpenAPI spec, verify response matches spec. Zero breaking changes since. Test mix: 50% unit, 40% integration, 10% contract.

**Rule of thumb:** Unit test controller logic — request mapping, validation. Integration test full endpoints — controller + service + database. Contract test API format — OpenAPI spec verification.
