329. What is @SpringBootTest?

**@SpringBootTest spins up the full Spring application context so you can test how your components work together with real wiring — not just in isolation.**

It's the annotation you use for integration tests where you need the actual dependency injection, configuration, and bean lifecycle that Spring provides.

**When to use it:**
- Testing controller-to-service-to-repository flow end to end
- Verifying that Spring configuration, profiles, and bean wiring work correctly
- Testing with real databases using Testcontainers

**Trade-off:**
- You get real confidence — tests exercise actual wiring, not mocks
- But it's slow — loading the full context takes 5-15 seconds per test class
- Tests become heavier to maintain — they need database setup, test data, cleanup

**Real-world example:**
In our search aggregation service, unit tests with Mockito caught logic bugs fast — 200 tests in 10 seconds. But we missed configuration issues — a bean profile was wrong, Solr client timeout wasn't wired from application.yml. Added SpringBootTest for critical paths — search endpoint to Solr integration. Caught three config bugs in the first run. But test suite went from 10 seconds to 2 minutes. The balance: unit tests for logic, SpringBootTest only for integration-critical paths.

**Rule of thumb:** Don't use SpringBootTest for everything — it's a sledgehammer. Use it for verifying wiring, configuration, and cross-layer flows. Keep the count low — 10-20 integration tests, hundreds of unit tests.
