## 100. How do you implement caching in Spring?

**Spring Cache provides declarative caching with @Cacheable, @CachePut, and @CacheEvict — annotate methods, Spring handles the rest.**

No manual cache lookups or invalidation logic. Works with Redis, EhCache, Caffeine, or any CacheManager.

---

### Step 1: Enable Caching

Add `@EnableCaching` to your configuration class:

```java
@SpringBootApplication
@EnableCaching
public class Application { }
```

This activates Spring's cache interception mechanism. Without it, cache annotations are ignored.

---

### Step 2: Choose a Cache Provider

**Caffeine (in-memory, single instance):**

```java
@Configuration
public class CacheConfig {
    @Bean
    public CacheManager cacheManager() {
        CaffeineCacheManager manager = new CaffeineCacheManager("users", "products", "orders");
        manager.setCaffeine(Caffeine.newBuilder()
            .maximumSize(10000)
            .expireAfterWrite(10, TimeUnit.MINUTES));
        return manager;
    }
}
```

**Redis (distributed, multi-instance):**

```java
@Configuration
public class CacheConfig {
    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        RedisCacheManager manager = RedisCacheManager.builder(factory)
            .cacheDefaults(RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofMinutes(10)))
            .build();
        return manager;
    }
}
```

**Multiple cache managers (different TTLs per cache):**

```java
@Configuration
public class CacheConfig {
    @Bean
    public CacheManager shortTtlCacheManager(RedisConnectionFactory factory) {
        return RedisCacheManager.builder(factory)
            .cacheDefaults(RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofMinutes(1)))
            .build();
    }
    
    @Bean
    public CacheManager longTtlCacheManager(RedisConnectionFactory factory) {
        return RedisCacheManager.builder(factory)
            .cacheDefaults(RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofHours(24)))
            .build();
    }
}
```

### Step 3: Use Cache Annotations

#### @Cacheable — Cache the Result

**Basic usage:**

```java
@Cacheable("users")
public User findById(Long id) {
    return userRepository.findById(id); // Only called on cache miss
}
```

**With key generator:**

```java
@Cacheable(value = "users", key = "#id")
public User findById(Long id) { ... }

@Cacheable(value = "users", key = "#email")
public User findByEmail(String email) { ... }

@Cacheable(value = "users", key = "#userId + ':' + #includeProfile")
public User getUserDetails(Long userId, boolean includeProfile) { ... }
```

**With condition (only cache if condition is true):**

```java
@Cacheable(value = "users", condition = "#id > 100")
public User findById(Long id) { ... } // Only cache IDs > 100

@Cacheable(value = "users", unless = "#result == null")
public User findById(Long id) { ... } // Don't cache null results
```

**With specific cache manager:**

```java
@Cacheable(value = "users", cacheManager = "longTtlCacheManager")
public User findById(Long id) { ... } // Uses 24-hour TTL cache
```

---

#### @CachePut — Update the Cache

**Use when you need to update the cache without skipping the method:**

```java
@CachePut(value = "users", key = "#user.id")
public User update(User user) {
    return userRepository.save(user); // Always executes, updates cache
}
```

**Common mistake:** Don't use `@CachePut` on the same method as `@Cacheable` — they conflict.

---

#### @CacheEvict — Remove from Cache

**Evict after method completes:**

```java
@CacheEvict(value = "users", key = "#id")
public void deleteById(Long id) {
    userRepository.deleteById(id); // Cache cleared after delete
}
```

**Evict before method executes (failsafe):**

```java
@CacheEvict(value = "users", key = "#id", beforeInvocation = true)
public void deleteById(Long id) {
    userRepository.deleteById(id); // Cache cleared before, even if method fails
}
```

**Evict all entries in a cache:**

```java
@CacheEvict(value = "users", allEntries = true)
public void bulkUpdate(List<User> users) {
    // Clears entire 'users' cache after bulk update
}
```

---

#### @Caching — Combine Multiple Cache Operations

```java
@Caching(evict = {
    @CacheEvict("users"),
    @CacheEvict(value = "usersByEmail", key = "#user.email"),
    @CacheEvict(value = "usersByPhone", key = "#user.phone")
})
public User save(User user) {
    return userRepository.save(user);
}
```

### Step 4: Custom Key Generators

When SpEL isn't enough, create a custom key generator:

```java
@Component
public class CustomKeyGenerator implements KeyGenerator {
    @Override
    public Object generate(Object target, Method method, Object... params) {
        return target.getClass().getName() + ":" + method.getName() + ":" + Arrays.toString(params);
    }
}

// Usage:
@Cacheable(value = "users", keyGenerator = "customKeyGenerator")
public User findById(Long id) { ... }
```

---

### Step 5: Custom Cache Error Handler

Handle cache failures gracefully (cache failure shouldn't crash the app):

```java
@Configuration
@EnableCaching
public class CacheConfig implements CachingConfigurer {
    @Override
    public CacheErrorHandler errorHandler() {
        return new CacheErrorHandler() {
            @Override
            public void handleCacheGetError(RuntimeException exception, Cache cache, Object key) {
                log.warn("Cache get failed for key: {}", key, exception);
                // Swallow exception, let method execute normally
            }
        };
    }
}
```

### Real-World Implementation: E-Commerce Product Service

**Problem:** Product catalog API — 50K requests/minute, 200ms database queries, database CPU at 90%.

**Solution:** Multi-layer caching strategy.

```java
@Service
public class ProductService {

    // Hot data: 1-minute TTL, high hit rate
    @Cacheable(value = "products", key = "#id", unless = "#result == null")
    public Product findById(Long id) {
        return productRepository.findById(id);
    }

    // Expensive query: 10-minute TTL
    @Cacheable(value = "productSearch", key = "#category + ':' + #page + ':' + #size")
    public Page<Product> search(String category, int page, int size) {
        return productRepository.findByCategory(category, PageRequest.of(page, size));
    }

    // Update: evict single entry
    @CacheEvict(value = "products", key = "#product.id")
    public Product update(Product product) {
        return productRepository.save(product);
    }

    // Bulk update: evict entire cache
    @CacheEvict(value = "products", allEntries = true)
    public List<Product> bulkUpdate(List<Product> products) {
        return productRepository.saveAll(products);
    }

    // Price change: evict from multiple caches
    @Caching(evict = {
        @CacheEvict("products", key = "#id"),
        @CacheEvict("productSearch", allEntries = true)
    })
    public void updatePrice(Long id, BigDecimal newPrice) {
        productRepository.updatePrice(id, newPrice);
    }
}
```

**Configuration:**

```java
@Configuration
@EnableCaching
public class CacheConfig {

    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        return RedisCacheManager.builder(factory)
            .cacheDefaults(RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofMinutes(10))
                .serializeKeysWith(RedisSerializationContext.SerializationPair
                    .fromSerializer(new StringRedisSerializer()))
                .serializeValuesWith(RedisSerializationContext.SerializationPair
                    .fromSerializer(new GenericJackson2JsonRedisSerializer())))
            .build();
    }
}
```

**Results:**
- Response time: 200ms → 5ms (95th percentile)
- Database load: 90% → 15% CPU
- Cache hit rate: 85% overall, 98% for product details
- Flash sale issue: Fixed with `@CacheEvict` on price updates

### Common Pitfalls and Solutions

**1. Self-invocation trap (same as @Transactional):**

```java
@Service
public class UserService {
    public User getUser(Long id) {
        return findById(id); // @Cacheable IGNORED - no proxy interception
    }
    
    @Cacheable("users")
    public User findById(Long id) { ... }
}
```

**Fix:** Inject the service into itself or move `findById()` to a separate repository class.

---

**2. Caching mutable objects:**

```java
@Cacheable("users")
public User findById(Long id) {
    User user = repository.findById(id);
    user.setName("modified"); // Modifies cached object!
    return user;
}
```

**Problem:** Next caller sees modified data.

**Fix:** Return defensive copies or use immutable DTOs.

---

**3. Cache stampede (thundering herd):**

```java
@Cacheable("products")
public Product findById(Long id) { ... }
```

**Problem:** Cache expires at 12:00:00, 1000 requests arrive at 12:00:01, all hit database.

**Fix 1:** Add jitter to TTL:

```java
.entryTtl(Duration.ofMinutes(10 + ThreadLocalRandom.current().nextInt(0, 120)))
```

**Fix 2:** Use cache-aside with lock:

```java
public Product findById(Long id) {
    Product cached = cache.get(id);
    if (cached != null) return cached;
    
    synchronized (this) {
        cached = cache.get(id);
        if (cached != null) return cached;
        
        Product product = repository.findById(id);
        cache.put(id, product);
        return product;
    }
}
```

---

**4. Null value caching:**

```java
@Cacheable("users")
public User findById(Long id) {
    return repository.findById(id); // Returns null for missing users
}
```

**Problem:** Null is cached, user never found even after creation.

**Fix:** Use `unless` to exclude nulls:

```java
@Cacheable(value = "users", unless = "#result == null")
public User findById(Long id) { ... }
```

---

**5. Wrong key generation:**

```java
@Cacheable(value = "users", key = "#user.id")
public User save(User user) { ... }
```

**Problem:** On save, `#user.id` might be null (auto-generated), causing all saves to use same cache key.

**Fix:** Use `@CachePut` for saves, or generate key from non-null fields.

### Monitoring and Metrics

**Add cache metrics:**

```java
@Configuration
@EnableCaching
@EnableMetrics
public class CacheConfig {
    @Bean
    public CacheMeterBinderProvider cacheMeterBinderProvider() {
        return cache -> new CacheMeterBinder(cache, Collections.emptyList(), Duration.ofSeconds(10));
    }
}
```

**Key metrics to track:**
- Cache hit rate (target: >80%)
- Cache miss rate (target: <20%)
- Average cache latency (target: <1ms for in-memory, <10ms for Redis)
- Cache eviction count (spikes indicate TTL too short)
- Cache memory usage (prevent OOM)

### Decision Tree: Which Cache Provider?

```
Do you have multiple service instances?
├─ YES → Redis (distributed, network overhead)
└─ NO → Is latency critical (<1ms)?
    ├─ YES → Caffeine (in-memory, no serialization)
    └─ NO → EhCache (disk persistence, larger capacity)
```

---

### Rule of Thumb

1. **Start simple:** `@EnableCaching` + `@Cacheable` with default settings
2. **Measure first:** Add metrics before optimizing
3. **Cache reads, not writes:** `@Cacheable` for queries, `@CacheEvict` for mutations
4. **Set TTLs:** Never cache indefinitely without eviction strategy
5. **Handle failures:** Cache errors shouldn't crash the application
6. **Test invalidation:** Verify cache clears when data changes
7. **Monitor hit rates:** Below 50%? Reconsider what you're caching
