## 380. What is @Secured annotation?

**@Secured is Spring Security's older role-based annotation — simpler than @PreAuthorize, but less flexible.**

It's role-checking only — no SpEL expressions, no complex conditions.

**How it works:**
- Annotation on method — @Secured("ROLE_ADMIN")
- Spring Security checks if user has specified role
- If yes — method executes
- If no — AccessDeniedException thrown

**Trade-off:**
- Simple — just role names, no SpEL syntax
- Type-safe — role names are strings, no expression parsing
- But: limited — only role checks, no conditions, no resource-based checks
- Legacy — @PreAuthorize is preferred for new code

**Real-world example:**
Our legacy app used @Secured — @Secured("ROLE_ADMIN"), @Secured("ROLE_USER"). Worked for simple role checks. Needed resource-level security — user can only edit own posts. @Secured couldn't express this — only roles, no conditions. Migrated to @PreAuthorize — @PreAuthorize("#postId == authentication.principal.postId or hasRole('ADMIN')"). Resource-level: working. But @Secured still used for simple role checks — simpler, no SpEL overhead. Hybrid approach: @Secured for roles, @PreAuthorize for conditions.

**Rule of thumb:** @Secured for simple role-based checks — simpler syntax. @PreAuthorize for complex conditions — resource ownership, expressions. New code: prefer @PreAuthorize — more flexible, future-proof.
