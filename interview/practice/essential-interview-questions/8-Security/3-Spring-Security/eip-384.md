## 384. What is CORS configuration in Spring Security?

**CORS configuration in Spring Security controls cross-origin access — which domains can call your API from browser.**

It's browser security — server specifies allowed origins, browser enforces.

**How to configure:**
- @CrossOrigin annotation — per-controller or per-method
- WebSecurityConfigurer — global CORS config
- CorsConfigurationSource — programmatic config

**Trade-off:**
- Fine-grained — different origins for different endpoints
- Global config — one place for all CORS rules
- But: misconfiguration breaks legitimate access or allows malicious origins
- Preflight caching — reduces OPTIONS requests, improves performance

**Real-world example:**
Our API had CORS errors — frontend (app.example.com) blocked from calling API (api.example.com). Added @CrossOrigin on controllers — worked. But 20 controllers — missed one, CORS error. Moved to global config — CorsConfigurationSource, allowed origins: https://app.example.com. All controllers: working. Also, initially allowed all origins (*) — security scan flagged. Changed to specific origins — only production and staging frontends. Security: improved.

**Rule of thumb:** Global CORS config — not per-controller. Specific origins only — never wildcard (*) for authenticated APIs. Test in browser — CORS errors only appear in browser, not Postman.
