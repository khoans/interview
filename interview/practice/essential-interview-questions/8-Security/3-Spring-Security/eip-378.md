## 378. What is UserDetailsService?

**UserDetailsService loads user-specific data for authentication — username to UserDetails mapping.**

It's the bridge between your user store and Spring Security — implement once, Spring handles rest.

**How it works:**
- Spring Security calls loadUserByUsername(username)
- Implementation queries database/LDAP/API
- Returns UserDetails — username, password, roles, enabled status
- Spring Security validates password, sets security context

**Trade-off:**
- Simple interface — one method to implement
- Decoupled — Spring Security doesn't know your user store
- But: only username lookup — can't authenticate by email, phone without wrapper
- Password must be stored — can't use passwordless auth directly

**Real-world example:**
Our app had users table — username, password hash, roles. Implemented UserDetailsService — query by username, map to UserDetails. Authentication: working. But users wanted to login with email OR username. loadUserByUsername only accepts one field. Created wrapper service — try username first, if not found, try email. Login: flexible, users happy. Also, password hash was MD5 (weak). Migrated to BCrypt — UserDetailsService returns BCrypt-encoded hash, Spring Security validates.

**Rule of thumb:** UserDetailsService for database/LDAP authentication. Wrapper for multi-field login (username/email). BCrypt password encoding — never plain text or weak hashes.
