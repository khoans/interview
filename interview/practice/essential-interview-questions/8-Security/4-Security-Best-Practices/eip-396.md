## 396. What is security headers?

**Security headers are HTTP headers that enhance security — XSS protection, clickjacking prevention, HTTPS enforcement.**

They're browser instructions — tell browser how to handle your site securely.

**Key headers:**
- **Content-Security-Policy** — restrict script/style sources
- **X-Frame-Options** — prevent clickjacking (DENY, SAMEORIGIN)
- **Strict-Transport-Security** — enforce HTTPS (HSTS)
- **X-Content-Type-Options** — prevent MIME sniffing (nosniff)
- **X-XSS-Protection** — legacy XSS filter (deprecated, but still useful)

**Trade-off:**
- Defense in depth — browser-enforced security
- Easy to implement — one-time config
- But: misconfiguration breaks functionality — CSP blocks legitimate scripts
- Browser support — not all headers supported by all browsers

**Real-world example:**
Our app had no security headers. Security scan: 10 vulnerabilities (XSS, clickjacking, MIME sniffing). Added headers: CSP, X-Frame-Options, HSTS, X-Content-Type-Options. Vulnerabilities: 10 → 2 (CSP misconfiguration). Fixed CSP — whitelist legitimate script sources. Vulnerabilities: 0. But X-Frame-Options: DENY broke embedded widgets (payment gateway). Changed to SAMEORIGIN — allows same-origin embedding, blocks external. Widgets: working. Clickjacking: blocked.

**Rule of thumb:** All security headers — CSP, X-Frame-Options, HSTS, X-Content-Type-Options. Test in multiple browsers — not all headers universally supported. CSP: start report-only, fix violations, then enforce.
