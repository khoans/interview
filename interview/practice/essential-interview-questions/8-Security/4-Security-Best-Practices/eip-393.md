## 393. What is session-based authen vs token-based authen?

**Session auth stores state server-side — session ID in cookie, user data in server memory. Token auth is stateless — all data in token (JWT), no server storage.**

Different trade-offs for different architectures.

**Session-based:**
- Server stores session — Redis, database
- Cookie has session ID — reference to server data
- Easy logout — delete session, immediately invalid
- But: server affinity — sticky sessions or shared session store

**Token-based:**
- Client stores token — localStorage, mobile app
- Token contains data — JWT with claims
- Stateless — no server storage, scales horizontally
- But: hard to revoke — token valid until expiration

**Trade-off:**
- Session: better security — server controls session, instant logout
- Token: better scalability — no session sync, stateless
- Hybrid: short-lived tokens + server-side refresh token tracking

**Real-world example:**
Our monolith used sessions — worked fine. Microservices: session sync across 10 services was complex. Switched to JWT — services validate independently. But logout didn't work — token still valid. Added token blacklist (Redis, 1-hour TTL). Logout: token added to blacklist, rejected on use. Memory: 1M sessions → 10K blacklisted tokens (active logouts only). Scalability: improved. Logout: 1-hour worst-case delay (acceptable).

**Rule of thumb:** Session for monoliths. Token for microservices. Hybrid for best of both — JWT + refresh token tracking.
