## 392. What is the structure of JWT?

**JWT has three parts: Header (algorithm), Payload (claims), Signature (verification) — encoded as base64url, dot-separated.**

Compact format: header.payload.signature

**Structure:**
- **Header** — {"alg": "RS256", "typ": "JWT"} — algorithm and type
- **Payload** — {"sub": "123", "name": "John", "iat": 1516239022} — claims
- **Signature** — HMACSHA256(header + payload, secret) — prevents tampering

**Trade-off:**
- Compact — small enough for HTTP headers
- Self-contained — all info in token, no database lookup
- But: payload is encoded, not encrypted — anyone can read claims
- Size grows with claims — keep payload minimal

**Real-world example:**
Our JWT stored user_id, roles, exp — 200 bytes. Developer added user profile (email, name, address, phone) — token grew to 2KB. HTTP header limit: 4KB. Adding more claims would exceed limit. Also, stored sensitive data (email, phone) — JWT is encoded, not encrypted, anyone can decode. Security audit: flagged. Removed sensitive data — user_id only, fetch profile from database. Token: 2KB → 200 bytes. Sensitive data: exposed → protected.

**Rule of thumb:** Minimal claims — user_id, roles, expiration only. No sensitive data — JWT is not encrypted. Monitor token size — stay under 4KB (HTTP header limit).
