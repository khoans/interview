## 385. What is the principle of least privilege?

**Principle of least privilege: users and systems have minimum permissions necessary to perform their function — no more, no less.**

It's fundamental security — limit blast radius of compromise.

**Application:**
- **User permissions** — regular users can't access admin functions
- **Database accounts** — app user can SELECT/INSERT/UPDATE, not DROP TABLE
- **Service accounts** — service has access only to required resources

**Trade-off:**
- Reduced risk — compromised account has limited access
- Audit compliance — required for PCI-DSS, SOC2, HIPAA
- But: administrative overhead — more roles to manage
- Potential breakage — legitimate operations may fail without proper permissions

**Real-world example:**
Our app database user had full permissions — SELECT, INSERT, UPDATE, DELETE, DROP, ALTER. SQL injection vulnerability — attacker dropped entire database. Restored from backup — 4 hours downtime. Applied least privilege — new database user with SELECT, INSERT, UPDATE, DELETE only. SQL injection: same vulnerability, but damage limited — can't DROP tables. Also, admin users had full access by default. Changed to opt-in — explicit role assignment. Admin accounts: 50 → 5. Attack surface: reduced 90%.

**Rule of thumb:** Least privilege for all accounts — users, services, database. Default deny — grant permissions explicitly, not by default. Regular permission audits — remove unused permissions.
