## 400. What is security logging and monitoring?

**Security logging and monitoring captures security-relevant events — login attempts, access denials, suspicious activity — and alerts on anomalies.**

It's about detection and response — know when you're under attack.

**What to log:**
- **Authentication** — successful/failed logins, logouts, password changes
- **Authorization** — access denials, privilege escalations
- **Data access** — sensitive data accessed, exported, modified
- **System events** — configuration changes, admin actions

**Trade-off:**
- Detection — know about breaches quickly
- Forensics — investigate what happened, scope of breach
- But: volume — millions of events, need filtering/aggregation
- False positives — too many alerts, team ignores them

**Real-world example:**
Our app had no security logging. Breach: attacker accessed admin panel, exfiltrated data. Discovered 3 months later (customer notification). No logs — couldn't determine scope, timeline. Implemented security logging — all auth events, access denials, admin actions. SIEM (Splunk) for aggregation, alerting. Next breach: detected in 15 minutes (unusual admin access from new IP). Response: blocked IP, reset credentials. Damage: minimal. Also, tuned alerts — reduced false positives 90% → 5%. Team responds to alerts — not ignored.

**Rule of thumb:** Log all security events — auth, authorization, data access, admin actions. Centralized logging — SIEM for aggregation. Alert on anomalies — not every event. Tune alerts — reduce false positives. Regular review — weekly security log review.
