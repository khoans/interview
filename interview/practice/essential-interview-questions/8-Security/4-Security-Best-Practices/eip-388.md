## 388. What is input validation?

**Input validation verifies user input meets expected criteria — format, length, range, characters.**

It's the first line of defense — reject bad input before it reaches business logic.

**Validation types:**
- **Format** — email matches email pattern, phone matches phone pattern
- **Length** — username 3-50 characters, password 8-100 characters
- **Range** — age 0-150, quantity 1-1000
- **Characters** — alphanumeric only, no special characters

**Trade-off:**
- Prevents injection attacks — SQL injection, XSS blocked at entry
- Data quality — invalid data rejected early
- But: over-validation blocks legitimate input — edge cases fail
- Maintenance — validation rules must evolve with requirements

**Real-world example:**
Our app accepted any input — no validation. SQL injection, XSS, buffer overflows. Implemented input validation — whitelist allowed characters, max lengths, format checks. Injection attacks: eliminated. But over-validation — name field allowed only letters, rejected "José García" (accented characters). Fixed: Unicode letters allowed. Also, max length 50 — some legitimate names exceeded. Increased to 100. Validation: strict but reasonable. False positives: 5% → 0.1%.

**Rule of thumb:** Validate all input — format, length, range, characters. Whitelist allowed — don't blacklist (always miss something). Allow Unicode — international users. Test edge cases — long strings, special characters, empty input.
