## 387. What is secure by default?

**Secure by default means systems are secure out-of-the-box — users must explicitly opt-in to insecure configurations.**

It's about safe defaults — most users don't change defaults, so defaults must be secure.

**Examples:**
- **Passwords** — minimum length 12, complexity required (opt-out possible but discouraged)
- **HTTPS** — enforced by default, HTTP redirects to HTTPS
- **Permissions** — default deny, explicit grant required
- **Features** — security features enabled by default (CSRF, XSS protection)

**Trade-off:**
- Safe for most users — don't need security expertise
- Reduces misconfiguration — secure defaults prevent common mistakes
- But: may frustrate advanced users — extra steps for legitimate use cases
- Support burden — users need help configuring exceptions

**Real-world example:**
Our API had optional authentication — endpoints public by default, @Authenticated to require auth. Misconfiguration: 10 endpoints left public accidentally. Data breach: 1000 records exposed. Changed to secure by default — all endpoints require authentication, @Public to explicitly allow anonymous. Misconfiguration: now requires explicit action to make public. Accidental exposure: 10 endpoints → 0. But legitimate public endpoints (health check, docs) needed @Public annotation — documented, added to boilerplate.

**Rule of thumb:** Secure defaults — authentication required, HTTPS enforced, permissions denied by default. Explicit opt-out — @Public, @AllowAnonymous. Document exceptions — make it easy to do the right thing.
