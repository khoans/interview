## 391. What is salt in password hashing?

**Salt is random data added to password before hashing — unique per user, prevents rainbow table attacks.**

It ensures identical passwords have different hashes — no pattern recognition.

**How it works:**
- Generate random salt (16-32 bytes) per user
- Combine: hash(password + salt)
- Store salt with hash — salt is not secret
- Verification: hash(input + stored_salt) == stored_hash

**Trade-off:**
- Prevents rainbow tables — attacker can't precompute hashes
- Unique hashes — same password, different users, different hashes
- But: salt must be unique — reusing salt defeats purpose
- Storage — salt adds 16-32 bytes per user (negligible)

**Real-world example:**
Our app hashed passwords without salt — hash(password). Two users with "password123" had same hash. Attacker saw pattern — identified users with weak passwords. Rainbow table attack: 100K hashes cracked in 1 hour. Added unique salt per user — hash(password + salt). Same passwords: different hashes. Rainbow table: useless — would need separate table per salt. Cracking time: 1 hour → 1000 years. Also, initially used same salt for all users — better than no salt, but still vulnerable. Changed to unique salt per user — full protection.

**Rule of thumb:** Unique salt per password — never reuse salts. Store salt with hash — not secret, just random. Use framework defaults — Spring Security generates unique salts automatically.
