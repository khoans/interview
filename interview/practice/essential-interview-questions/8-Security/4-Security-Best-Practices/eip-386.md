## 386. What is defense in depth?

**Defense in depth is multiple layers of security — if one layer fails, others still protect.**

It's about redundancy — no single point of security failure.

**Layers:**
- **Network** — firewalls, VPC, security groups
- **Application** — authentication, authorization, input validation
- **Data** — encryption at rest, encryption in transit
- **Monitoring** — logging, alerting, incident response

**Trade-off:**
- Resilient — attacker must bypass multiple layers
- Risk reduction — single vulnerability doesn't compromise everything
- But: complexity — more layers to manage, potential conflicts
- Cost — each layer has implementation and maintenance cost

**Real-world example:**
Our app had single security layer — authentication. Attacker bypassed auth (SQL injection), had full access. Implemented defense in depth: 1) Parameterized queries (SQL injection prevention), 2) Input validation (reject malicious input), 3) Method-level authorization (@PreAuthorize), 4) Audit logging (detect suspicious activity). Next attack: SQL injection blocked by parameterized queries. Even if bypassed, authorization blocks unauthorized access. Even if bypassed, logging detects anomaly. Attack success rate: 100% → 0%.

**Rule of thumb:** Multiple independent layers — don't rely on single defense. Each layer should be effective alone — but together provide resilience. Regular testing — verify each layer works.
