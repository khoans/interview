## 390. What is password hashing vs encryption?

**Hashing is one-way transformation — can't reverse. Encryption is two-way — can decrypt with key.**

Passwords should be hashed — never encrypted. You verify passwords, don't need to retrieve them.

**Hashing:**
- One-way — hash(password) = abc123, can't get password from abc123
- Verification — hash(input) == stored_hash means password matches
- Algorithms — BCrypt, Argon2, PBKDF2 (slow, salted)

**Encryption:**
- Two-way — encrypt(data, key) = cipher, decrypt(cipher, key) = data
- Use case — sensitive data you need to retrieve (credit cards, SSN)
- Algorithms — AES, RSA

**Trade-off:**
- Hashing: secure for passwords — even DB breach doesn't reveal passwords
- Encryption: necessary for retrievable data — but key management critical
- Wrong choice: encrypting passwords means DB breach = all passwords compromised

**Real-world example:**
Our legacy app encrypted passwords — "for security". DB breach: attacker got encryption key, decrypted all 100K passwords. Password reuse: users used same password elsewhere, accounts compromised. Migrated to BCrypt hashing — passwords can't be decrypted. Next breach: attacker got hashes, but BCrypt is slow — 100K passwords would take 1000 years to crack. Password compromise: 100% → 0%.

**Rule of thumb:** Hash passwords — BCrypt, Argon2, PBKDF2. Never encrypt passwords — no legitimate reason to retrieve them. Encrypt sensitive data you need to retrieve — credit cards, PII.
