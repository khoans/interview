## 365. What is CORS (Cross-Origin Resource Sharing)?

**CORS is a security mechanism controlling which domains can access your API from browser — prevents unauthorized cross-origin requests.**

It's browser-enforced — servers specify allowed origins, browser blocks unauthorized access.

**How it works:**
- Browser sends preflight request (OPTIONS) — asks server if cross-origin allowed
- Server responds with Access-Control-Allow-Origin header — specific domain or *
- Browser allows/blocks response based on header

**Trade-off:**
- Security — prevents malicious sites from accessing your API
- But: misconfiguration breaks legitimate cross-origin access
- Wildcard (*) allows anyone — convenient but insecure for authenticated APIs

**Real-world example:**
Our API was called from frontend (app.example.com) — different domain than API (api.example.com). Browser blocked requests — CORS error. Added Access-Control-Allow-Origin: https://app.example.com. Requests worked. But staging environment used same API — CORS blocked staging. Added environment-specific CORS config — staging allows staging frontend, production allows production frontend. Also, initially used wildcard (*) — any site could call API. Changed to specific origins — only trusted domains allowed.

**Rule of thumb:** Specific origins only — never wildcard (*) for authenticated APIs. Configure per environment — staging, production have different allowed origins. Test CORS in browser — errors only appear in browser, not curl/Postman.
