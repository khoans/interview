## 372. How do you prevent SQL Injection?

**SQL Injection prevention: parameterized queries, ORM, input validation — never trust user input.**

Parameterized queries are the gold standard — SQL structure fixed, data separate.

**Prevention methods:**
- **Parameterized queries** — placeholders (?), values bound separately
- **ORM (Hibernate, JPA)** — uses parameters by default
- **Input validation** — reject unexpected characters, patterns
- **Least privilege** — database user has minimal permissions

**Trade-off:**
- Parameterized queries: most effective — but requires code changes
- ORM: safer by default — but native queries still vulnerable
- Input validation: additional layer — but not sufficient alone

**Real-world example:**
Our app had 100 SQL queries — 50 parameterized, 50 string concatenation. Security audit: 50 vulnerable queries. Refactored all to parameterized — 2 weeks effort. SQL injection: eliminated. But performance regression — parameterized queries 10% slower (query plan caching issue). Added query plan caching — performance restored. Also, database user had DROP TABLE permission — overprivileged. Reduced to SELECT, INSERT, UPDATE, DELETE only. Damage potential: entire database → limited to data manipulation only.

**Rule of thumb:** Parameterized queries for all SQL — prepared statements, named parameters. ORM preferred — but audit native queries. Database user least privilege — no DROP, no schema changes.
