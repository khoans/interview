## 363. What is HTTPS? How does it work?

**HTTPS is HTTP over TLS — encrypted communication between client and server, preventing eavesdropping and tampering.**

It's the foundation of web security — no sensitive data over plain HTTP.

**How it works:**
1. **TLS Handshake** — client and server agree on encryption algorithm
2. **Certificate exchange** — server presents SSL certificate, proves identity
3. **Key exchange** — generate shared session key
4. **Encrypted communication** — all HTTP data encrypted with session key

**Trade-off:**
- Encryption — prevents eavesdropping, man-in-the-middle attacks
- Authentication — certificate proves server identity
- But: performance overhead — TLS handshake adds 100-200ms latency
- Certificate management — renewal, expiration monitoring

**Real-world example:**
Our API was HTTP-only — internal network, "trusted users". Security audit: critical — credentials sent in plain text. Attacker on network could sniff passwords. Enabled HTTPS with Let's Encrypt certificate. Credential theft: possible → eliminated. Performance: TLS handshake added 150ms latency. Added TLS session resumption — repeat connections skip handshake, 150ms → 10ms. Certificate expiration: forgot to monitor, certificate expired, 4-hour outage. Added expiration alerts (30 days, 7 days, 1 day).

**Rule of thumb:** HTTPS for all traffic — internal and external. Use Let's Encrypt for free certificates. Monitor expiration — automate renewal. Enable session resumption — reduce latency for repeat connections.
