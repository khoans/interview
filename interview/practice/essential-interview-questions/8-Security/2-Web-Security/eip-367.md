## 367. How do you prevent CSRF attacks?

**CSRF prevention: CSRF tokens, SameSite cookies, and origin verification — multiple layers of defense.**

Defense in depth — no single defense is perfect.

**Prevention methods:**
- **CSRF tokens** — unique token per session, required for state changes
- **SameSite cookies** — Strict or Lax, prevents cross-site cookie sending
- **Origin/Referer check** — verify request origin matches expected domain

**Trade-off:**
- CSRF tokens: strong protection — but breaks if not implemented correctly
- SameSite cookies: browser-enforced — but older browsers don't support
- Origin check: simple — but can be bypassed in some scenarios

**Real-world example:**
Our app implemented CSRF tokens — unique per session, stored in session, required in form/headers. CSRF attacks: eliminated. But tokens expired with session — users got "invalid token" errors after 30 minutes. Fixed: CSRF token lifetime = session lifetime. Also, SameSite cookies: set to Strict — blocked legitimate cross-origin flows (payment gateway redirect). Changed to Lax — allows top-level navigation, blocks embedded requests. CSRF protection: maintained. Payment flow: working.

**Rule of thumb:** CSRF tokens for all state-changing operations. SameSite=Lax for cookies — balances security and usability. Origin check as additional layer — don't rely solely on one defense.
