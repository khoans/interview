## 374. What is SameSite cookie attribute?

**SameSite controls when cookies are sent with cross-site requests — prevents CSRF, limits cross-site tracking.**

It's browser-enforced cookie security — server sets attribute, browser enforces.

**Three values:**
- **Strict** — cookies never sent cross-site — maximum security, breaks SSO
- **Lax** — cookies sent with top-level navigation — balances security/usability
- **None** — cookies sent cross-site — required for third-party cookies, requires Secure

**Trade-off:**
- Strict: best CSRF protection — but breaks legitimate cross-site flows
- Lax: good protection — allows navigation, blocks embedded requests
- None: required for third-party — but requires HTTPS (Secure flag)

**Real-world example:**
Our app set session cookies without SameSite — vulnerable to CSRF. Added SameSite=Lax. CSRF attacks: blocked. But SSO flow broke — redirect to OAuth provider, cookie not sent on return. Changed to SameSite=None; Secure for OAuth flow only. Session cookies: SameSite=Lax. OAuth cookies: SameSite=None. SSO: working. CSRF: still protected for main session. Also, older browsers ignore SameSite — added CSRF tokens as backup defense.

**Rule of thumb:** SameSite=Lax for session cookies — balances security and usability. SameSite=None for third-party cookies — requires Secure flag. Don't rely solely on SameSite — add CSRF tokens for older browser support.
