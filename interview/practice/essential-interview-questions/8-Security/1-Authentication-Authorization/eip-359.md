## 359. What is the difference between session and token-based authentication?

**Session auth stores state server-side — session ID in cookie, user data in server memory/database. Token auth is stateless — all data in token, no server storage.**

Different trade-offs for different architectures.

**Session-based:**
- Server stores session — Redis, database, memory
- Cookie contains session ID — reference to server data
- Easy revocation — delete session, immediately logged out
- But: server affinity — sticky sessions or shared session store

**Token-based:**
- Client stores token — browser localStorage, mobile app
- Token contains all data — JWT with claims
- Scales horizontally — no session sync needed
- But: hard to revoke — token valid until expiration

**Trade-off:**
- Session: better security — server controls session, easy logout
- Token: better scalability — no session storage, stateless
- Hybrid approach: short-lived tokens + refresh tokens with server-side tracking

**Real-world example:**
Our monolith used sessions — worked fine. Microservices migration: session sync across 10 services was complex (Redis cluster, sticky sessions). Switched to JWT — services validate token independently, no session sync. But logout didn't work — token still valid. Added token blacklist (Redis) for 1-hour window. Logout: immediate → 1-hour delay worst case. Acceptable trade-off for scalability.

**Rule of thumb:** Session for monoliths, simple apps. Token for microservices, mobile apps. Hybrid for best of both — short-lived JWT + server-side refresh token tracking.
