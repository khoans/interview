## 360. What is SSO (Single Sign-On)?

**SSO allows users to authenticate once, access multiple applications without re-authenticating.**

It's about user experience — one login, many applications.

**How it works:**
- User logs into central identity provider (IdP)
- IdP issues token/assertion — proves authentication
- Applications trust IdP — accept token, grant access
- User navigates to another app — already authenticated

**Trade-off:**
- User experience — one login, no password fatigue
- Centralized management — one place to manage users, passwords
- But: single point of failure — IdP down, all apps inaccessible
- Security risk — compromised IdP = access to all apps

**Real-world example:**
Our enterprise had 20 internal apps — 20 passwords, password reset tickets: 50/week. Implemented SSO with Okta. Password reset tickets: 50/week → 5/week. User complaints: "too many passwords" → eliminated. But IdP outage took down all 20 apps — 4-hour outage, 500 users blocked. Added local fallback — critical apps have backup authentication. IdP outage: 100% blocked → 10% blocked (non-critical only).

**Rule of thumb:** SSO for enterprise with 5+ applications. Choose reliable IdP — 99.99% uptime SLA. Implement fallback for critical apps — don't have single point of failure.
