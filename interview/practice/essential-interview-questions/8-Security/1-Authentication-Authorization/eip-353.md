## 353. Explain the OAuth2 authorization code flow.

**Authorization code flow is the most secure OAuth2 flow — user authenticates, receives authorization code, exchanges code for tokens server-side.**

It's designed for server-side applications — code exchanged behind the scenes, tokens never exposed to browser.

**Flow steps:**
1. User clicks "Login with Provider" — redirected to authorization server
2. User authenticates — enters credentials at provider (Google, GitHub)
3. Provider redirects back with authorization code — not token
4. Server exchanges code for tokens — access token + refresh token
5. Server uses access token — calls provider API on behalf of user

```
  User/Browser              Our Server              Auth Provider (Google)
      │                         │                           │
      │  1. Click "Login"       │                           │
      │────────────────────────►│                           │
      │                         │                           │
      │  2. Redirect to provider (client_id + redirect_uri) │
      │◄────────────────────────│                           │
      │                         │                           │
      │  3. User enters credentials at provider             │
      │────────────────────────────────────────────────────►│
      │                         │                           │
      │  4. Provider redirects back with authorization code │
      │◄────────────────────────────────────────────────────│
      │  (redirect to: ourapp.com/oauth/callback?code=abc)  │
      │                         │                           │
      │  5. Browser follows redirect to our server          │
      │────────────────────────►│                           │
      │                         │                           │
      │                         │  6. Exchange code + client│
      │                         │     secret for tokens     │
      │                         │──────────────────────────►│
      │                         │                           │
      │                         │  7. Returns access_token  │
      │                         │     + refresh_token       │
      │                         │◄──────────────────────────│
      │                         │                           │
      │                         │  8. Use access_token to   │
      │                         │     call provider API     │
      │                         │──────────────────────────►│
      │                         │                           │
      │                         │  9. Returns user data     │
      │                         │◄──────────────────────────│
      │                         │                           │
      │  10. Session created,   │                           │
      │      user logged in     │                           │
      │◄────────────────────────│                           │
      │                         │                           │
```

**Key security point:** Steps 6-9 happen entirely server-side — the browser never sees the tokens. The authorization code (step 4) is useless without the client secret, which only the server knows.

**Trade-off:**
- Secure — tokens never exposed to browser, code is useless without client secret
- But: requires server-side component — can't use in pure SPA/mobile
- Extra round-trip — code exchange adds latency (100-200ms)

**Real-world example:**
Our web app implemented authorization code flow — users login with Google. Redirect URI: https://ourapp.com/oauth/callback. Server receives code, exchanges for tokens. Access token stored in session, not browser. But initially exposed tokens in browser — XSS could steal tokens. Moved token storage server-side — session-only, not accessible to JavaScript. Token theft: possible → eliminated.

**Rule of thumb:** Authorization code flow for server-side web apps. Never expose tokens in browser. Use PKCE extension for additional security — prevents code interception attacks.
