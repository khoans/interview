## 358. What are the parts of JWT?

**JWT has three parts: Header (algorithm info), Payload (claims/data), Signature (verification).**

Encoded as base64url, joined with dots: header.payload.signature

**Three parts:**
- **Header** — algorithm (HS256, RS256) and token type (JWT)
- **Payload** — claims (user_id, roles, expiration, issuer)
- **Signature** — header + payload signed with secret/private key

**Trade-off:**
- Header: tells verifier how to validate — but don't trust blindly
- Payload: contains all data — but don't store sensitive data (not encrypted)
- Signature: prevents tampering — but algorithm must be verified

**Real-world example:**
Our JWT stored user_id, roles, exp in payload. Developer added password_hash to payload — "for convenience". Security audit: critical — JWT is encoded, not encrypted, anyone can decode and read password hash. Removed sensitive data immediately. Also, algorithm confusion attack — attacker changed header from RS256 to HS256, forged token. Fixed: explicitly verify algorithm matches expected (RS256 only).

**Rule of thumb:** Never store sensitive data in JWT payload — encoded, not encrypted. Verify algorithm explicitly — don't accept algorithm from token header. Use short expiration — 1 hour max for access tokens.
