## 355. What is OAuth2 client credentials flow?

**Client credentials flow is for machine-to-machine authentication — no user involved, application authenticates itself.**

It's service-to-service access — backend services accessing other backend services.

**How it works:**
1. Client sends client_id + client_secret to authorization server
2. Authorization server validates credentials
3. Returns access token — no refresh token (short-lived only)
4. Client uses token — calls resource server API

**Trade-off:**
- No user context — service identity only, can't track individual users
- Simple — no redirects, no user interaction
- But: credential management — client_secret must be stored securely
- Token scope — usually broad access, not user-specific

**Real-world example:**
Our payment service needed to call fraud detection API — service-to-service. Client credentials flow: payment service has client_id/secret, exchanges for token, calls fraud API. But client_secret stored in config file — committed to Git, exposed. Moved secrets to environment variables, then to HashiCorp Vault. Secret exposure: high → eliminated. Added token caching — reduced token requests 1000/hour → 10/hour.

**Rule of thumb:** Client credentials for service-to-service authentication. Never hardcode client_secret — use environment variables or secret management (Vault, AWS Secrets Manager). Cache tokens — don't request new token for every call.
