## 107. How many non-clustered indexes can a table have?

**Technically: 999 in SQL Server, unlimited in PostgreSQL/MySQL. Practically: 5-10 is the sweet spot for most workloads.**

More indexes aren't better — each one slows writes and consumes storage.

**Database limits:**
- SQL Server: 999 non-clustered indexes per table
- MySQL/InnoDB: No hard limit, constrained by storage
- PostgreSQL: No hard limit, constrained by memory and performance

**Practical guidance:**
- 3-5 indexes: Typical OLTP table (PK, FKs, 1-2 query columns)
- 5-10 indexes: Heavily queried table with diverse access patterns
- 10+ indexes: Data warehouse or reporting table (read-heavy)
- 20+ indexes: Red flag — audit for unused or redundant indexes

**Trade-off:**
- Each index adds 10-30% overhead to INSERT/UPDATE/DELETE
- Index maintenance during writes can cause lock contention
- Storage cost: indexes often exceed table data size

**Real-world example:**
Our analytics table had 47 non-clustered indexes — DBA team added them over 3 years "for optimization." Bulk inserts took 45 minutes. Storage: 800GB table, 2.4TB indexes. Audit showed 28 indexes unused in 90 days. Dropped them, bulk inserts dropped to 12 minutes (73% faster), freed 1.5TB storage. Query performance unchanged. Now we cap at 10 indexes per table, require justification for more.

**Rule of thumb:** Start with 5 or fewer. Add only when slow query logs demand it. Review quarterly — unused indexes get dropped.
