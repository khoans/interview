## 101. What is a database index? How does it work?

**A database index is a data structure that speeds up data retrieval at the cost of slower writes and extra storage.**

It works like a book's index — instead of scanning every page (full table scan), you look up the key and jump directly to the data.

**How it works:**
- Most databases use B-tree indexes — balanced tree structure with sorted keys
- Query `WHERE email = 'user@example.com'` traverses the tree in O(log n) time instead of O(n) full scan
- Index stores the key column plus a pointer to the actual row (heap or clustered)

**Why it matters:**
- Proper indexing turns minute-long queries into millisecond responses
- Without indexes, every query scans the entire table — doesn't scale past 10K rows
- Indexes are the #1 performance tuning lever in relational databases

**Trade-off:**
- Reads get faster (10-100x), but writes get slower — every INSERT/UPDATE must update all indexes
- Each index costs disk space — large tables can have indexes bigger than the data itself
- Too many indexes hurt write performance — we've seen 20+ indexes cause INSERT timeouts

**Real-world example:**
Our order search endpoint took 8 seconds — `WHERE customer_id = ? AND status = ?` scanned 5 million rows. Added composite index on (customer_id, status), query dropped to 12ms (666x faster). But our order creation API slowed from 50ms to 200ms — now updates two indexes instead of one. Trade-off was worth it: 99% of traffic was reads, 1% writes. We optimized writes later with batch inserts.

**Rule of thumb:** Index columns used in WHERE, JOIN, and ORDER BY. Don't index low-cardinality columns (gender, boolean). Monitor index usage — unused indexes are pure overhead.
