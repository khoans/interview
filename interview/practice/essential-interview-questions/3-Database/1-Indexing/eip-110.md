## 110. Have you encountered a scenario where an index hasn't increased query speed?

**Yes — indexes don't help when the query pattern doesn't match the index, statistics are stale, or the table is small enough that full scan is faster.**

The most common case: query filters on a column that isn't the leftmost in the composite index.

**Common reasons indexes don't help:**
- Wrong column order in composite index (query skips leftmost column)
- Function on indexed column (`WHERE YEAR(created_at) = 2024` ignores index on `created_at`)
- Type mismatch (`WHERE user_id = '123'` with string literal on integer column)
- Stale statistics — optimizer thinks full scan is cheaper
- Low selectivity — index filters out too few rows to justify lookup
- Table is tiny — full scan of 100 rows is faster than index traversal

**Trade-off:**
- Adding more indexes isn't the answer — fixing the query or index structure is
- Index hints force usage but can backfire if data distribution changes
- Query rewrites (removing functions) are safer than index changes

**Real-world example:**
We had `WHERE DATE(created_at) = '2024-01-15'` with index on `created_at`. Query took 2 seconds, index ignored. Root cause: wrapping column in `DATE()` function prevented index usage. Rewrote to `WHERE created_at >= '2024-01-15' AND created_at < '2024-01-16'`. Query dropped to 50ms (40x faster) — same index, different syntax. Second case: composite index (last_name, first_name), query filtered only by `first_name`. Couldn't use index. Added separate index on `first_name` for that specific query pattern.

**Rule of thumb:** Check execution plans before assuming indexes work. Avoid functions on indexed columns. Ensure composite index order matches query patterns.
