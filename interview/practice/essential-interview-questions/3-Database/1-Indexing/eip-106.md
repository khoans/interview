## 106. Can a table have multiple clustered indexes?

**No — a table can have only one clustered index because data can only be physically sorted one way.**

The clustered index determines the actual storage order of rows. You can't store the same data in two different physical orders simultaneously.

**Why the limitation:**
- Data rows are stored in the leaf level of the clustered index
- Physical storage is sequential based on the clustered key
- Like organizing a filing cabinet — you can sort by name OR by date, not both

**Workaround:**
- Create one clustered index on the most common access pattern
- Use non-clustered indexes for other query patterns
- Consider indexed views or materialized views for alternative access paths

**Trade-off:**
- Single clustered index simplifies storage but forces you to choose the "primary" access pattern
- Wrong choice means most queries need key lookups (non-clustered → clustered → data)
- Some databases (DB2, Oracle IOT) have similar restrictions

**Real-world example:**
Our orders table needed fast access by `customer_id` (90% of queries) and `order_date` (reporting). We couldn't cluster on both. Chose `customer_id` as clustered — customer queries were fast (no lookup). Reporting queries slowed 40% due to key lookups. Added non-clustered index on `order_date` with INCLUDE clause to cover reporting queries. Compromise: 90% of queries optimal, 10% acceptable.

**Rule of thumb:** Cluster on the most frequently queried column (usually primary key). Use covering non-clustered indexes for other critical query patterns.
