## 109. What is index selectivity?

**Index selectivity measures how well an index filters rows — the percentage of rows eliminated by the index lookup.**

High selectivity (filters out 90%+ rows) means the index is valuable. Low selectivity (filters out <10%) often isn't worth using.

**How it's calculated:**
- Selectivity = (unique values / total rows) × 100%
- Primary key: 100% selectivity (every value is unique)
- Gender: ~0.02% selectivity (2 values, 50% each)
- Country code: ~5% selectivity (200 countries, uneven distribution)

**Why it matters:**
- Optimizer uses selectivity to decide: index scan vs. full table scan
- Low selectivity indexes are often ignored — full scan is cheaper
- High selectivity indexes are prioritized for query plans

**Trade-off:**
- High selectivity columns make great indexes (email, phone, SSN)
- Low selectivity columns waste storage and slow writes (boolean, status)
- Composite index selectivity depends on column combination, not individual columns

**Real-world example:**
Our orders table had 10 million rows. Index on `status` (5 values: pending/shipped/delivered/cancelled/returned) — selectivity 20%. Optimizer ignored it 90% of the time, preferred full scan. Index on `order_number` (unique) — selectivity 100%, always used. We dropped the `status` index, no query slowdown. Storage freed: 400MB. Write performance improved 15%. Now we calculate selectivity before adding any index — if it's below 10%, we require strong justification.

**Rule of thumb:** Index columns with >10% selectivity. For lower selectivity, consider composite indexes or bitmap indexes (data warehouses only).
