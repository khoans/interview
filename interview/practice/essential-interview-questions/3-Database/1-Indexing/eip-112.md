## 112. Why might an index NOT be used by a query?

**Indexes are ignored when the query pattern doesn't match the index structure, statistics are stale, or the optimizer calculates full scan is cheaper.**

The database isn't broken — it made a cost-based decision that scanning the table is faster than using your index.

**Common reasons:**
- **Function on column** — `WHERE UPPER(email) = 'TEST@EXAMPLE.COM'` ignores index on `email`
- **Type conversion** — `WHERE phone = 123456` (number on varchar column)
- **Leading wildcard** — `WHERE name LIKE '%Smith'` can't use B-tree index
- **Wrong composite order** — index (A, B), query filters only on B
- **Low selectivity** — index filters out <10% of rows, full scan is cheaper
- **Outdated statistics** — optimizer thinks table has 100 rows, actually has 10 million
- **Table is small** — full scan of 50 rows is faster than index traversal
- **Implicit conversion** — column is VARCHAR, query passes INTEGER

**Trade-off:**
- Adding hints forces index usage but can backfire when data changes
- Query rewrites are safer but may require code changes
- Statistics updates are low-risk but need regular scheduling

**Real-world example:**
Our search endpoint had index on `customer_email`, query took 3 seconds — index ignored. Root cause: `WHERE LOWER(customer_email) = 'test@example.com'`. Function prevented index usage. Rewrote to store lowercase email in separate column with index, query dropped to 20ms (150x faster). Second case: statistics weren't updated after bulk import — optimizer thought table had 1K rows, actually 50 million. Manual `UPDATE STATISTICS`, query plan changed instantly, 5 seconds → 100ms.

**Rule of thumb:** Check execution plans, not just index existence. Avoid functions on indexed columns. Schedule regular statistics updates. Test with production data volumes.
