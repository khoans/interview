## 103. What is the difference between clustered and non-clustered index?

**Clustered index determines the physical order of data in the table. Non-clustered index is a separate structure pointing to the data.**

A table can have only one clustered index (data is stored once), but multiple non-clustered indexes.

**Clustered index:**
- Data rows are stored in the order of the index key
- Leaf nodes contain actual data pages, not pointers
- Fastest access — no extra lookup after finding the index
- Typically the primary key (but doesn't have to be)

**Non-clustered index:**
- Separate structure from the data (like a book index vs. the book itself)
- Leaf nodes contain pointers to data rows (heap or clustered key)
- Requires two lookups: index → pointer → data (unless covering index)
- Can have many non-clustered indexes on the same table

**Trade-off:**
- Clustered: Faster reads (no indirection), but changing the key is expensive — rows must be physically reordered
- Non-clustered: Flexible (add/remove without touching data), but extra lookup adds latency
- Wrong clustered key choice fragments the table — inserts happen in random physical locations

**Real-world example:**
Our orders table had clustered index on `order_date` (chronological storage). Range queries for reports were fast — sequential disk reads. But order inserts were 3x slower — new orders always had future dates, causing page splits and fragmentation. Switched clustered index to `order_id` (identity column). Inserts became sequential (100% faster), range queries slowed 20% but still acceptable with non-clustered index on `order_date`. Net win for our write-heavy workload.

**Rule of thumb:** Clustered on primary key (identity/sequential). Non-clustered on frequently queried columns. Don't cluster on columns that change or have random values (UUIDs).
