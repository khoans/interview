## 102. What are the types of indexes in SQL?

**The main index types are B-tree, hash, bitmap, and full-text — each optimized for different query patterns.**

Most databases default to B-tree, which handles most workloads. The others are for specialized cases.

**Common types:**
- **B-tree** — default, good for range queries, equality, sorting (ORDER BY)
- **Hash** — equality only (=), fastest for exact matches, useless for ranges
- **Bitmap** — low-cardinality columns (gender, status), common in data warehouses
- **Full-text** — text search, LIKE '%word%' queries, inverted index
- **Composite** — multiple columns, order matters (leftmost prefix rule)
- **Covering** — includes all columns needed by query, no table lookup

**Trade-off:**
- B-tree: Versatile but larger than hash indexes
- Hash: 2x faster for equality, but breaks on `>`, `<`, `BETWEEN`, `ORDER BY`
- Bitmap: Great for analytics, terrible for OLTP — locks entire bitmap on updates
- Covering: Eliminates table access, but index bloat (more storage)

**Real-world example:**
We had a user search by status (active/inactive/pending) — 5 million rows, B-tree index, 400ms queries. Switched to bitmap index (only 3 distinct values), dropped to 50ms. But our update API went from 20ms to 500ms — bitmap locks the whole segment. Reverted to B-tree for the transactional table, kept bitmap for the reporting replica. Different workloads need different indexes.

**Rule of thumb:** B-tree for 90% of cases. Hash for pure equality lookups (primary keys). Bitmap for read-only analytics. Composite for multi-column WHERE clauses.
