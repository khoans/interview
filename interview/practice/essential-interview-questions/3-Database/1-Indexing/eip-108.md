## 108. What is a composite index?

**A composite index is an index on multiple columns — it optimizes queries filtering or sorting by those columns together.**

The column order matters: queries can use the index efficiently only if they filter by the leftmost columns first.

**How it works:**
- Index on (last_name, first_name) sorts by last_name first, then first_name within each last_name
- Query `WHERE last_name = 'Smith'` uses the index efficiently
- Query `WHERE last_name = 'Smith' AND first_name = 'John'` uses it even better
- Query `WHERE first_name = 'John'` cannot use the index (skips leftmost column)

**Why it matters:**
- Single index replaces multiple single-column indexes
- Covers multi-column queries without separate lookups
- Can include additional columns (INCLUDE/COVERING) to avoid table access

**Trade-off:**
- Order matters — wrong order makes index useless for common queries
- Larger than single-column indexes (more storage)
- Maintenance cost: all columns in index must be updated on row changes

**Real-world example:**
Our user search had separate indexes on `state` and `status`. Query `WHERE state = 'CA' AND status = 'active'` used only one index, then filtered manually — 800ms. Created composite index (state, status), query dropped to 15ms (53x faster). But we had another query filtering only by `status` — couldn't use the composite index. Added separate index on `status` for that case. Lesson: composite indexes are powerful but don't replace all single-column indexes.

**Rule of thumb:** Order columns by selectivity (most selective first) and query patterns (most common filters first). Use covering indexes to eliminate table lookups entirely.
