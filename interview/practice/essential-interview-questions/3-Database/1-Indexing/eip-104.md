## 104. When should you create an index?

**Create indexes on columns used in WHERE clauses, JOIN conditions, ORDER BY, and foreign keys — but only after measuring query performance.**

Don't index preemptively. Index based on actual query patterns, not guesses.

**Good candidates:**
- Primary keys (usually indexed automatically)
- Foreign keys — speeds up JOINs and cascade operations
- Columns in WHERE with high selectivity (filters out >10% of rows)
- Columns used in ORDER BY — avoids expensive sort operations
- Composite indexes for multi-column filters (order matters!)

**Bad candidates:**
- Low-cardinality columns (gender, boolean, status with 2-3 values)
- Columns rarely used in queries (nice-to-have, not critical)
- Small tables (<10K rows) — full scan is often faster
- Heavily updated columns — index maintenance kills write performance

**Trade-off:**
- Every index speeds reads but slows writes — INSERT/UPDATE must maintain all indexes
- Storage cost: indexes can exceed table size (we've seen 2x data size)
- Diminishing returns: 5 indexes give 90% of benefit, 20 indexes add overhead with minimal gain

**Real-world example:**
Our e-commerce database had 50+ indexes — DBA team indexed "just in case." Insert latency spiked to 2 seconds during flash sales. Audit found 30 unused indexes (never appeared in query plans). Dropped them, insert latency dropped to 200ms (10x improvement). Query performance unchanged — those indexes weren't being used anyway. Now we require query execution plans before adding indexes. If it's not in a slow query log, it doesn't get indexed.

**Rule of thumb:** Start with primary key and foreign keys. Add indexes based on slow query logs, not speculation. Review quarterly — drop unused indexes. Monitor index size vs. table size.
