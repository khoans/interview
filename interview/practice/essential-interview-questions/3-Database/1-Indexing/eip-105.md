## 105. When should you NOT create an index?

**Don't create indexes on low-cardinality columns, small tables, heavily updated columns, or columns never used in queries.**

Indexes cost write performance and storage — only create them when read performance justifies the overhead.

**Avoid indexing:**
- Low-cardinality columns (gender, boolean, status with 2-3 values) — optimizer often ignores them
- Small tables (<10K rows) — full scan is faster than index lookup
- Heavily updated columns — index maintenance kills write throughput
- Columns not in WHERE, JOIN, ORDER BY — indexes never used, pure overhead
- Redundant indexes — duplicate of existing index or leftmost prefix of composite index

**Trade-off:**
- Every index slows writes by 10-30% — compound that with 20+ indexes
- Storage bloat: indexes can exceed table size (we've seen 3x data size)
- Maintenance overhead: index rebuilds, statistics updates, backup time

**Real-world example:**
Our logging table had indexes on `log_level` (DEBUG/INFO/ERROR), `is_archived` (boolean), and `created_by` (rarely queried). 15 indexes total. Insert performance: 500 rows/second. Dropped 8 unused indexes, insert performance jumped to 5,000 rows/second (10x). Query performance unchanged — optimizer wasn't using those indexes anyway. Now we require 30-day query history before adding any index.

**Rule of thumb:** Start minimal. Add indexes based on slow query logs, not speculation. Review quarterly — drop indexes with zero hits in execution plans.
