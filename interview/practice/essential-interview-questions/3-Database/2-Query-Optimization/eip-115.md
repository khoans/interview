## 115. How do you read an execution plan?

**Read execution plans right-to-left (data flow) and top-to-bottom (execution order), focusing on the most expensive operators first.**

Start with the operation consuming 80% of the cost — that's your optimization target.

**Key operators to recognize:**
- **Index Seek** — efficient, uses index to find specific rows (good)
- **Index Scan** — reads entire index (okay for small tables, bad for large)
- **Table Scan/Clustered Index Scan** — reads entire table (red flag for large tables)
- **Key Lookup/Bookmark Lookup** — jumps from index to table (expensive, indicates missing covering index)
- **Sort** — explicit sort operation (can often be eliminated with proper indexing)
- **Hash Match/Spool** — temporary storage for joins (memory-intensive)

**What to look for:**
- **Cost percentage** — which operator dominates the query cost
- **Estimated vs. actual rows** — big gaps indicate stale statistics
- **Warnings** — missing statistics, implicit conversions, join spills to disk

**Trade-off:**
- Chasing every warning is wasteful — focus on high-cost operators
- Optimizing one query can hurt others — test changes in isolation
- Plans are context-dependent — same query, different parameters, different plan

**Real-world example:**
Our customer search took 8 seconds. Execution plan: Table Scan (45% cost) → Sort (30%) → Nested Loop (25%). Table Scan on 5 million rows was the bottleneck. Added index on search column (`customer_email`), plan changed to Index Seek. Query dropped to 120ms (66x faster). But we noticed Key Lookup (15% cost) — index didn't cover all selected columns. Added INCLUDE clause for remaining columns, Key Lookup eliminated, query dropped to 80ms. Total improvement: 8 seconds → 80ms (100x faster) by fixing operators in cost order.

**Rule of thumb:** Read right-to-left. Fix the highest-cost operator first. Eliminate table scans on large tables. Cover indexes to remove key lookups.
