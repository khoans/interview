## 120. What is a correlated subquery?

**A correlated subquery is a subquery that references columns from the outer query — it executes once per row in the outer result set.**

Unlike regular subqueries (execute once), correlated subqueries can be performance killers on large datasets.

**How it works:**
- Outer query processes row 1 → subquery executes with row 1's values
- Outer query processes row 2 → subquery executes with row 2's values
- Continues for every row — N rows = N subquery executions

**Why it matters:**
- Simple to write — natural expression of "for each row, find..."
- Terrible performance — O(n²) complexity on large tables
- Often rewritable as JOINs — same result, O(n) complexity

**Trade-off:**
- Correlated subqueries are readable but slow
- JOINs are faster but sometimes harder to write
- EXISTS with correlated subquery is often optimized well by databases

**Real-world example:**
Our order audit query: `SELECT * FROM orders o WHERE total > (SELECT AVG(total) FROM orders WHERE customer_id = o.customer_id)`. Correlated subquery calculated average per customer — 100K orders, query took 45 seconds. Rewrote as JOIN with derived table: `JOIN (SELECT customer_id, AVG(total) as avg FROM orders GROUP BY customer_id)`. Query dropped to 800ms (56x faster). Same logic, different execution. For EXISTS checks (`WHERE EXISTS SELECT 1 FROM...`), correlated subquery was fine — optimizer short-circuits on first match.

**Rule of thumb:** Avoid correlated subqueries in SELECT and WHERE clauses on large tables. Rewrite as JOINs. EXISTS/NOT EXISTS with correlated subquery is usually fine — databases optimize these well.
