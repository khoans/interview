## 113. How do you identify slow queries?

**I use a combination of database slow query logs, APM tools, and query execution statistics — not guesswork.**

Slow queries are symptoms; finding them requires instrumentation, not intuition.

**Methods:**
- **Slow query log** — database-level logging for queries exceeding threshold (e.g., >1 second)
- **APM tools** — New Relic, Datadog, AppDynamics show query timing per request
- **Database DMVs** — `sys.dm_exec_query_stats` (SQL Server), `pg_stat_statements` (PostgreSQL)
- **Application logging** — log query duration at the DAO/repository layer

**Trade-off:**
- Slow query logs add I/O overhead — can slow the database by 5-10% under high load
- APM tools sample data — may miss intermittent slow queries
- DMVs reset on restart — need regular snapshots for trend analysis

**Real-world example:**
Our checkout API had random 5-second spikes — happened 2-3 times per hour, impossible to reproduce locally. Enabled slow query log (threshold: 500ms) for 24 hours. Found the culprit: inventory check query doing full table scan on 2 million rows — missing index on `warehouse_id`. Query time: 4.8 seconds. Added index, spikes disappeared. Zero checkout timeouts since. The slow query log captured 47 slow queries that day — we fixed 12 that affected critical paths.

**Rule of thumb:** Enable slow query log in production (threshold: 1 second for OLTP, 5 seconds for reporting). Review weekly. Prioritize queries on critical user paths over background jobs.
