## 116. What is a full table scan? When is it acceptable?

**A full table scan reads every row in a table — no index is used. It's acceptable for small tables, aggregate queries, or when you need most rows anyway.**

Full table scans get a bad reputation, but they're sometimes the optimal choice.

**When it happens:**
- No index exists on the filtered column
- Query has no WHERE clause (SELECT *)
- Optimizer calculates scan is cheaper than index lookups
- Low-selectivity filters (returns >30% of rows)

**When it's acceptable:**
- Small tables (<10K rows) — scan is faster than index traversal
- Aggregate queries (COUNT, SUM) — need to read everything anyway
- Reporting/analytics — full dataset scans are expected
- When you need 30%+ of rows — index lookups become expensive

**Trade-off:**
- Scans are I/O intensive — 1 million rows can take seconds
- Caching helps — second scan is faster if data is in buffer pool
- Parallel scans can speed up large table reads (4x-8x faster)

**Real-world example:**
Our daily sales report did `SELECT SUM(amount) FROM orders WHERE order_date = ?` — 50 million rows, index on `order_date`. Query took 800ms with index seek + key lookups. Forced full table scan with hint — 400ms (2x faster). Why? We needed 2 million rows for that date — index lookups were more expensive than sequential scan. For single-order queries, index was still 50x faster. Different queries, different optimal plans.

**Rule of thumb:** Don't fear full scans on small tables or when fetching 30%+ of rows. Optimize for queries returning <10% of rows. Let the optimizer decide — it usually knows best.
