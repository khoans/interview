## 122. What are query hints?

**Query hints are directives that override the optimizer's decisions — forcing index usage, join order, or join strategy.**

They're a last resort when the optimizer consistently makes wrong choices.

**Common hints:**
- **Index hints** — `USE INDEX`, `FORCE INDEX`, `IGNORE INDEX`
- **Join hints** — `HASH JOIN`, `LOOP JOIN`, `MERGE JOIN`
- **Parallel hints** — enable parallel execution for large scans

**Why they exist:**
- Optimizer lacks business context (knows statistics, not query patterns)
- Stale statistics cause bad plans
- Edge cases where cost model is wrong

**Trade-off:**
- Hints fix immediate problems but create technical debt
- Data distribution changes — today's optimization is tomorrow's bottleneck
- Hints hide symptoms — fixing statistics or query structure is better

**Real-world example:**
Our reporting query joined 5 tables — optimizer chose nested loop joins, 30 seconds. Statistics were current, but optimizer underestimated result size. Added `HASH JOIN` hint — 800ms (37x faster). But 6 months later, data grew 10x — hash joins spilled to disk, query took 2 minutes. Removed hint, updated statistics, optimizer chose merge joins — 400ms. The hint was a crutch that prevented us from fixing the real issue (statistics maintenance). Now we use hints only for emergency production fixes, then fix root cause within 48 hours.

**Rule of thumb:** Avoid hints in production code. Fix statistics, rewrite queries, or add indexes instead. Use hints only for emergency hotfixes — document them and create tickets to remove within sprint.
