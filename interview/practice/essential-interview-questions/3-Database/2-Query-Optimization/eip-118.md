## 118. What is the difference between UNION and UNION ALL?

**UNION removes duplicates. UNION ALL keeps all rows including duplicates.**

UNION ALL is faster — no sorting or deduplication overhead. Use it when you know results don't overlap.

**Key differences:**
- **UNION** — sorts results, removes duplicates (DISTINCT operation)
- **UNION ALL** — concatenates results, no processing
- **Performance** — UNION ALL is 2x-5x faster on large datasets
- **Use cases** — UNION for deduplication, UNION ALL for combining partitions

**Why it matters:**
- UNION's deduplication requires sorting — O(n log n) complexity
- UNION ALL is O(n) — just appends results
- Most queries don't have duplicates — UNION ALL is safe and faster

**Trade-off:**
- UNION guarantees unique results but pays sorting cost
- UNION ALL is faster but may return duplicates if data overlaps
- Wrong choice causes either performance issues or incorrect results

**Real-world example:**
Our monthly report combined data from three partitioned tables (orders_jan, orders_feb, orders_mar) using UNION — 2 million rows, took 8 seconds. Tables were non-overlapping (different months), no duplicates possible. Switched to UNION ALL, query dropped to 1.5 seconds (5x faster). But we had another query combining customer emails from marketing and support tables — used UNION ALL, got duplicate emails. Changed to UNION, duplicates removed correctly. Know your data before choosing.

**Rule of thumb:** Default to UNION ALL — it's faster. Use UNION only when you expect duplicates and need them removed. If tables are partitioned or mutually exclusive, UNION ALL is safe.
