## 121. What is query caching?

     **Query caching stores the result set of a query in memory — identical queries return cached results without re-executing.**

It's a powerful optimization for read-heavy workloads with repetitive queries.

**How it works:**
- Query executes, result set stored in cache (query text + result)
- Identical query arrives — cache returns result, no database hit
- Cache invalidates when underlying tables change

**Types:**
- **Database query cache** — MySQL query cache (deprecated in 8.0)
- **Application cache** — Redis, Memcached storing query results
- **ORM cache** — Hibernate second-level cache for entities

**Trade-off:**
- Cache hits are 100x faster — no database round-trip
- Cache invalidation is expensive — every INSERT/UPDATE/DELETE invalidates related queries
- High-write tables thrash the cache — constant invalidation, low hit rate

**Real-world example:**
Our product catalog had 100K page views/hour, 500 products. Enabled MySQL query cache — 90% hit rate, response time 50ms → 5ms. But product updates (price changes, inventory) caused cache invalidation — 200 updates/hour, cache rebuilt 200 times. During flash sales, updates spiked to 50/hour — cache thrashing, performance worse than no cache. Disabled query cache, added application-level Redis cache with 5-minute TTL. Hit rate: 85%, updates didn't invalidate cache immediately. Stable performance under load.

**Rule of thumb:** Query cache works for read-heavy, rarely-changing data. Avoid for high-write tables. Application-level caching with TTL is more predictable than database query cache.
