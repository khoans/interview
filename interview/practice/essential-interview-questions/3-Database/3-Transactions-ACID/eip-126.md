## 126. What are transaction isolation levels?

**Isolation levels control how much concurrent transactions can see each other's uncommitted changes — trading consistency for concurrency.**

Four standard levels, from weakest to strongest: READ_UNCOMMITTED, READ_COMMITTED, REPEATABLE_READ, SERIALIZABLE.

**The four levels:**
- **READ_UNCOMMITTED** — see uncommitted data, dirty reads possible (never use)
- **READ_COMMITTED** — see only committed data (default: Oracle, PostgreSQL, SQL Server)
- **REPEATABLE_READ** — same query returns same result (default: MySQL)
- **SERIALIZABLE** — transactions run as if sequential (strongest, slowest)

**Why they matter:**
- Wrong level causes anomalies: dirty reads, non-repeatable reads, phantom reads
- Higher isolation = more consistency, less concurrency
- Default is usually correct — change only with specific requirements

**Trade-off:**
- READ_COMMITTED: 10x more concurrent than SERIALIZABLE, but may see changing data
- SERIALIZABLE: perfect consistency, but 5-10x slower under high contention
- Most apps use READ_COMMITTED and handle edge cases in application logic

**Real-world example:**
Our wallet system used MySQL default (REPEATABLE_READ). Users saw stale balances — payment succeeded but balance didn't update for 3 seconds. Changed to READ_COMMITTED — balances updated immediately. But concurrent transfers caused negative balances (lost update). Added optimistic locking (`@Version`). Zero negative balances since, 99.9% first-try success.

**Rule of thumb:** Use database default (usually READ_COMMITTED). Add optimistic locking for concurrent writes. Only use SERIALIZABLE for financial reconciliations.
