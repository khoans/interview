## 129. What is REPEATABLE READ isolation level?

**REPEATABLE READ ensures the same query returns the same result throughout a transaction — non-repeatable reads are prevented, but phantom reads are possible.**

It's MySQL's default and is ideal for reports needing internal consistency.

**What it guarantees:**
- No dirty reads — only committed data
- No non-repeatable reads — same query returns same result
- Phantom reads possible — new rows can appear in range queries

**How it works:**
- Snapshot isolation — transaction sees database as of start time
- Consistent read — all queries use same snapshot
- Write locks still acquired for modifications

**Trade-off:**
- Consistent view — reports don't change mid-execution
- Stale data — won't see commits that happen after transaction starts
- Longer transactions = older snapshots = more stale

**Real-world example:**
Our monthly financial report took 30 seconds to run — 20 queries aggregating millions of rows. With READ_COMMITTED, totals didn't match — data changed between queries. Switched to REPEATABLE_READ — all queries used same snapshot, totals reconciled perfectly. But real-time dashboards showed different numbers (fresh data). We documented: reports are consistent-as-of-start-time, dashboards are live.

**Rule of thumb:** REPEATABLE_READ for reports, analytics, batch jobs needing consistency. READ_COMMITTED for user-facing transactions. Keep transactions short — long transactions hold old snapshots.
