## 125. What is the difference between Atomicity and Consistency?

**Atomicity ensures all operations in a transaction succeed or all fail. Consistency ensures the database moves from one valid state to another.**

Atomicity is about completeness. Consistency is about correctness.

**Atomicity:**
- All-or-nothing guarantee — no partial commits
- If step 3 of 5 fails, steps 1-2 are rolled back
- Implemented via rollback logs and two-phase commit

**Consistency:**
- Constraints enforced — foreign keys, unique, check constraints
- Business rules preserved — balances, invariants
- Application logic + database constraints together

**Why both matter:**
- Atomicity without consistency: all operations commit, but data is invalid
- Consistency without atomicity: data is valid but incomplete (half-updated)

**Trade-off:**
- Atomicity is cheap — databases handle it automatically
- Consistency requires effort — constraints, validation, application logic
- Over-constraining hurts performance — every insert validates all constraints

**Real-world example:**
Our fund transfer had atomicity — debit and credit in one transaction. But we lacked consistency — no check for negative balances. Bug: transferred $10K from $5K balance, account went -$5K. Added check constraint (`balance >= 0`), transfers from insufficient funds now fail atomically. Both properties together prevent corruption.

**Rule of thumb:** Atomicity is automatic with transactions. Consistency requires explicit constraints and validation. Design for both — atomicity protects the process, consistency protects the data.
