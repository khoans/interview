## 127. What is READ UNCOMMITTED isolation level?

**READ UNCOMMITTED allows transactions to see uncommitted changes from other transactions — dirty reads are possible.**

It's the weakest isolation level. Never use it in production unless you can tolerate completely stale or rolled-back data.

**What it allows:**
- Dirty reads — see data that may be rolled back
- Non-repeatable reads — same query returns different results
- Phantom reads — new rows appear mid-transaction

**Why it exists:**
- Maximum concurrency — no read locks, no blocking
- Useful for analytics — approximate aggregates on large datasets
- Reporting where exact accuracy isn't critical

**Trade-off:**
- Fastest reads — no lock contention
- Data may be invalid — transactions can roll back after you read
- Rarely appropriate — even analytics usually need committed data

**Real-world example:**
Our analytics dashboard used READ UNCOMMITTED for "live order count" — 10x faster than READ_COMMITTED. Problem: during flash sales, count spiked 50% higher than actual (uncommitted orders). Marketing made decisions based on inflated numbers. Changed to READ_COMMITTED with 5-second cache — slightly stale but accurate. Zero misleading metrics since.

**Rule of thumb:** Avoid READ UNCOMMITTED for business logic. Only consider for approximate analytics where speed matters more than accuracy. Even then, cached READ_COMMITTED is usually better.
