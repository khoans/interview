## 130. What is SERIALIZABLE isolation level?

**SERIALIZABLE is the strongest isolation level — transactions run as if executed one at a time, preventing all concurrency anomalies.**

It's perfect consistency at the cost of significant performance degradation under concurrent load.

**What it guarantees:**
- No dirty reads — only committed data
- No non-repeatable reads — same query, same result
- No phantom reads — no new rows in range queries
- Full isolation — transactions are completely independent

**How it works:**
- Range locks — entire queried ranges are locked
- Prevents concurrent modifications to affected data
- Transactions effectively serialized (queued)

**Trade-off:**
- Perfect consistency — no anomalies possible
- 5-10x slower under high contention — lock wait times dominate
- Deadlock risk increases — more locks held longer

**Real-world example:**
Our banking reconciliation used SERIALIZABLE — matching debits and credits across accounts. Zero tolerance for errors. Performance: 200ms per transaction (vs 20ms with READ_COMMITTED). But correctness was non-negotiable — $10M+ daily volume. For regular transfers, we used READ_COMMITTED + optimistic locking — 20ms, 99.9% success. Different use cases, different isolation levels.

**Rule of thumb:** SERIALIZABLE for financial reconciliations, audits, regulatory reports. READ_COMMITTED or REPEATABLE_READ for everything else. Test under load — SERIALIZABLE performance degrades non-linearly with concurrency.
