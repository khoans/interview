## 128. What is READ COMMITTED isolation level?

**READ COMMITTED ensures transactions only see committed data — dirty reads are prevented, but non-repeatable reads are possible.**

It's the default for most databases (Oracle, PostgreSQL, SQL Server) and the right choice for 90% of applications.

**What it guarantees:**
- No dirty reads — never see uncommitted changes
- Non-repeatable reads possible — same query can return different results if data changes
- Phantom reads possible — new rows can appear

**Why it's the default:**
- Good balance — prevents the most dangerous anomaly (dirty reads)
- High concurrency — read operations don't block writers
- Matches user expectations — you see what's actually saved

**Trade-off:**
- Can see changing data — report running mid-update may have inconsistent numbers
- Lower isolation than REPEATABLE_READ — some anomalies possible
- Most apps handle edge cases in application logic instead of raising isolation

**Real-world example:**
Our e-commerce inventory check used READ COMMITTED. User adds to cart (10 items), stock shows 95. Another user buys 90 items and commits. First user checks out — stock now 5, order fails. This is expected behavior — inventory changed between check and checkout. We handled it with retry logic: "Item reserved, please recheck availability." 95% success on retry.

**Rule of thumb:** READ COMMITTED for 90% of workloads. Use REPEATABLE_READ for reports needing internal consistency. Use SERIALIZABLE only for financial transactions requiring perfect isolation.
