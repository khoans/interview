## 132. What is a phantom read?

**A phantom read occurs when a transaction re-runs a query and sees new rows that were inserted by another committed transaction — rows that "appeared" like ghosts.**

Unlike dirty reads, phantoms are committed data — but they break the expectation that your transaction sees a stable dataset.

**How it happens:**
- Transaction A queries `WHERE status = 'PENDING'` — returns 100 rows
- Transaction B inserts a new row with `status = 'PENDING'` and commits
- Transaction A re-runs the same query — returns 101 rows
- A new row "appeared" out of nowhere

**Why it matters:**
- Aggregations change mid-transaction — reports don't reconcile
- Batch processing misses rows — new rows inserted after initial query
- Business logic assumes stable dataset — invariants violated

**Trade-off:**
- Preventing phantoms requires range locks or SERIALIZABLE — expensive
- Most apps tolerate phantoms — REPEATABLE_READ is sufficient
- Critical for batch jobs and reconciliations — need SERIALIZABLE

**Real-world example:**
Our batch processor claimed "processing all pending orders." Queried pending orders (50 rows), started processing. Mid-batch, another service inserted 10 new pending orders. Batch finished — 10 orders left unprocessed. Customer orders delayed 24 hours. Fix: SERIALIZABLE for batch job — range lock on `status = 'PENDING'`, no inserts allowed. Zero missed orders since.

**Rule of thumb:** REPEATABLE_READ for most workloads. SERIALIZABLE for batch jobs, reconciliations, or when processing "all rows matching X." Test concurrent inserts — phantoms only appear under load.
