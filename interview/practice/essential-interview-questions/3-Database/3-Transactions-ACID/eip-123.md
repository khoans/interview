## 123. What is a database transaction?

**A database transaction is a sequence of operations executed as a single unit — either all succeed or all fail.**

It's the foundation of reliable data processing. Without transactions, partial updates corrupt data.

**Key properties:**
- **Atomic** — all operations commit or none do (no partial commits)
- **Consistent** — database moves from one valid state to another
- **Isolated** — concurrent transactions don't interfere with each other
- **Durable** — once committed, data survives crashes (written to disk)

**Why it matters:**
- Prevents data corruption from partial failures (power loss, network blips)
- Enables safe concurrent access — two users updating same record
- Business logic depends on it — money transfers, order processing, inventory updates

**Trade-off:**
- Transactions add overhead — 10-30% slower than autocommit
- Long transactions hold locks — block other queries, cause timeouts
- Distributed transactions (2PC) are even slower — avoid when possible

**Real-world example:**
Our payment system had a bug: debit account A, then credit account B — separate queries, no transaction. Network failure between debits and credits happened 3 times per week. $150K in limbo over 6 months, manual reconciliation required. Wrapped both operations in `@Transactional` — zero data corruption incidents in 18 months. Performance cost: 15ms per transaction, acceptable for financial integrity.

**Rule of thumb:** Use transactions for any operation spanning multiple tables or rows. Keep them short — commit within 100ms. Avoid user input or external API calls inside transactions.
