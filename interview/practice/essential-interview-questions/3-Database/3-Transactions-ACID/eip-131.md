## 131. What is a dirty read?

**A dirty read occurs when a transaction reads uncommitted data from another transaction — data that may be rolled back and never actually exist.**

It's the most dangerous anomaly because your application makes decisions based on data that was never real.

**How it happens:**
- Transaction A updates a row (not committed)
- Transaction B reads the updated row
- Transaction A rolls back
- Transaction B now has data that never existed

**Why it matters:**
- Business logic acts on phantom data — approvals, calculations, decisions
- Cascading errors — dirty data propagates through the system
- Hard to debug — data appears correct initially, then vanishes

**Trade-off:**
- Preventing dirty reads costs 5-10% performance (read locks or MVCC)
- Allowing dirty reads is rarely worth the risk — even for analytics
- READ_UNCOMMITTED is the only level allowing dirty reads

**Real-world example:**
Our credit approval system checked account balance before approving loans. Transaction A withdrew $5,000 (uncommitted), Transaction B checked balance ($0), denied $500 loan. Transaction A rolled back — withdrawal never happened. Customer complained, we lost a good client. Fix: READ_COMMITTED — Transaction B only sees committed data. Zero dirty read incidents since.

**Rule of thumb:** Never use READ_UNCOMMITTED in production. READ_COMMITTED prevents dirty reads with minimal performance cost. The 5-10% overhead is insurance against data corruption.
