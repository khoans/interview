## 136. What is 2NF (Second Normal Form)?

**2NF requires 1NF plus no partial dependencies — all non-key columns must depend on the entire primary key, not just part of it.**

It only applies to tables with composite primary keys. Single-column PKs are automatically 2NF.

**Requirements:**
- Already in 1NF
- Composite primary key (multiple columns)
- All columns depend on the full key, not just one column

**Why it matters:**
- Partial dependencies cause redundancy — data repeated for each key combination
- Update anomalies — change data in one row, forget others
- Wasted storage — same data stored multiple times

**Trade-off:**
- 2NF splits tables — more tables, more JOINs
- But eliminates redundancy — update once, not dozens of times
- Query complexity vs. data integrity

**Real-world example:**
Our order_items table had PK (order_id, product_id) + columns: quantity, product_name, product_price. Product name/price depended only on product_id, not order_id. 10K orders × 100 products = 1M rows, product name stored 10K times each. Product price change — update 10K rows. Split: order_items (order_id, product_id, quantity) + products (product_id, name, price). Update: one row. Query: added JOIN, 20ms → 35ms. Storage: 1M rows → 100K products.

**Rule of thumb:** If you have a composite PK, check for partial dependencies. Move columns depending on only part of the key to a separate table. The JOIN cost is worth eliminating redundancy.
