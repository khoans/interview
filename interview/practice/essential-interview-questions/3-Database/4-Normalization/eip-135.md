## 135. What is 1NF (First Normal Form)?

**1NF requires atomic values — each column contains indivisible values, no repeating groups or arrays in a single cell.**

It's the most basic normalization rule. If you're violating 1NF, your schema is fundamentally broken.

**Requirements:**
- Atomic values — no comma-separated lists, no arrays
- No repeating groups — no `phone1`, `phone2`, `phone3` columns
- Each row-column intersection holds exactly one value
- All rows have same columns (no sparse columns)

**Why it matters:**
- Non-atomic values are unqueryable — can't index or filter efficiently
- Repeating groups limit flexibility — what if customer has 4 phones?
- Violations cause application complexity — parse strings, handle nulls

**Trade-off:**
- 1NF increases row count — one row per phone, not one row with 3 phones
- Queries become more complex — JOINs instead of simple SELECT
- But data is actually usable — indexing, filtering, constraints work

**Real-world example:**
Our user table had `skills` column: "Java,Python,SQL". Couldn't find users with "Python" — required LIKE '%Python%', full table scan, 8 seconds. Couldn't enforce valid skills. Normalized: user_skills junction table. Query: `WHERE skill = 'Python'` — index seek, 50ms. Added foreign key to skills table — invalid skills rejected at insert. Zero data quality issues since.

**Rule of thumb:** 1NF is non-negotiable. If you're storing lists in columns, fix it immediately. The query complexity is worth having queryable, constraint-able data.
