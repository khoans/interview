## 134. What are the benefits of normalization?

**Normalization reduces data redundancy, prevents anomalies, and ensures data integrity — at the cost of query complexity.**

The benefits are mostly about data quality and write performance.

**Key benefits:**
- **No duplicates** — customer address stored once, not 500 times in orders
- **Consistency** — update in one place, everywhere reflects the change
- **Storage efficiency** — no redundant data (though indexes often exceed savings)
- **Prevents anomalies** — insert/update/delete don't corrupt data
- **Cleaner schema** — logical grouping, easier to understand relationships

**Trade-off:**
- Writes are faster — update one row, not hundreds
- Reads are slower — more JOINs to reconstruct full data
- Query complexity increases — developers write more complex SQL
- Over-normalization hurts — 10-table JOINs for simple queries

**Real-world example:**
Our product catalog had supplier info embedded in products table — 10K products, 50 suppliers. Supplier changed phone number — update 200 products (multiple warehouses), 3 seconds. Some products missed (pagination bug). Normalized: suppliers table + foreign key. Update: one row, 5ms. Product query added JOIN — 15ms → 25ms. Trade-off accepted: data consistency worth 10ms.

**Rule of thumb:** Normalize for data integrity. Denormalize for read performance — but only after measuring. Start normalized, denormalize strategically based on query patterns.
