## 133. What is normalization?

**Normalization is the process of organizing data to minimize redundancy — each fact stored in one place, one time.**

It's about designing tables so inserts, updates, and deletes don't cause anomalies or inconsistencies.

---

### The Problem Normalization Solves

Imagine an e-commerce system where customer data is embedded directly in every order:

**Before normalization (denormalized):**

| order_id | customer_name | customer_email | customer_address | customer_city | customer_zip | product | amount |
|----------|---------------|----------------|------------------|---------------|--------------|---------|--------|
| 1001 | John Smith | john@email.com | 123 Main St | Boston | 02101 | Laptop | $999 |
| 1002 | John Smith | john@email.com | 123 Main St | Boston | 02101 | Mouse | $29 |
| 1003 | John Smith | john@email.com | 123 Main St | Boston | 02101 | Keyboard | $79 |
| 1004 | Jane Doe | jane@email.com | 456 Oak Ave | Chicago | 60601 | Monitor | $299 |

John's address is stored 3 times. If John moves, we must update 3 rows. Miss one, and the database is inconsistent.

---

### The Three Anomalies Normalization Prevents

**1. Insert Anomaly**

Can't add a new customer until they place an order. Why? Because customer data only exists in the orders table. No order = no place to store customer info.

**Real impact:** Our sales team wanted to pre-register leads in the system. Couldn't do it — database required an order_id (primary key can't be null). We built a workaround: fake $0 orders for leads. Data was polluted with non-orders. Reporting became a nightmare — had to filter out fake orders everywhere.

**2. Update Anomaly**

Customer moves from Boston to Cambridge. They have 500 historical orders. We must update all 500 rows.

**Real impact:** Our order system had this exact problem. Customer moved — we updated their profile, but didn't backfill the change to historical orders. 6 months later, we ran a marketing campaign by city. Sent "Boston customer" emails to customers who hadn't lived there for years. 15% bounce rate, marketing complained. Root cause: 500 order rows with old city, only 1 customer row with new city.

**3. Delete Anomaly**

Delete all of John's orders to clean up test data. Oops — now John's customer record is gone too. We lost information we needed to keep.

**Real impact:** QA asked us to purge test orders from staging. We ran `DELETE FROM orders WHERE customer_name = 'Test User'`. Accidentally deleted the only record of a real customer who happened to have similar naming. Customer called — no order history, no account. Took 3 hours to restore from backup.

---

### After Normalization

**Customers table:**

| customer_id | customer_name | customer_email | customer_address | customer_city | customer_zip |
|-------------|---------------|----------------|------------------|---------------|--------------|
| 501 | John Smith | john@email.com | 123 Main St | Boston | 02101 |
| 502 | Jane Doe | jane@email.com | 456 Oak Ave | Chicago | 60601 |

**Orders table:**

| order_id | customer_id | product | amount |
|----------|-------------|---------|--------|
| 1001 | 501 | Laptop | $999 |
| 1002 | 501 | Mouse | $29 |
| 1003 | 501 | Keyboard | $79 |
| 1004 | 502 | Monitor | $299 |

Now:
- Add customers without orders — insert into customers table, no order required
- Update address — one row in customers table, instantly reflected everywhere
- Delete orders — customers table untouched, no data loss

### Normal Forms (in order)

**1NF — Atomic values, no repeating groups**

No comma-separated lists, no arrays in a single cell. Each cell holds exactly one value.

**2NF — 1NF + no partial dependencies**

All columns depend on the full primary key (applies to composite keys only).

**3NF — 2NF + no transitive dependencies**

Non-key columns depend only on the primary key, not on other non-key columns.

**BCNF, 4NF, 5NF — Specialized cases**

Rarely needed in practice — 3NF handles 95% of real-world scenarios.

---

### Trade-off: Normalized vs. Denormalized

| Aspect | Normalized | Denormalized |
|--------|------------|--------------|
| Data consistency | Excellent — single source of truth | Poor — duplicates can diverge |
| Write performance | Fast — update one row | Slow — update many rows |
| Read performance | Slower — requires JOINs | Faster — data ready in one table |
| Query complexity | Higher — multiple tables | Lower — simple SELECT |
| Storage | Efficient — no redundancy | Wasteful — same data repeated |
| Use case | OLTP (transactions) | Analytics, reporting, caching |

### Our Production Journey: Full Story

**The setup:**
E-commerce platform, 2018. Orders table had embedded customer data: name, email, shipping address, billing address, phone. 2 million orders, 500K customers. Average customer had 4 orders.

**The incident:**
Customer service got a call: "I moved 3 months ago, why are my packages going to my old address?"

Investigation:
- Customer profile (customers table): correct new address
- Last 3 orders (orders table): old address (embedded at order time)
- Warehouse ships to address on order, not customer profile

Root cause: When order is placed, we copied customer address into orders table. Customer updated profile later — orders kept old address.

**The fix:**
Option 1: Backfill all 2 million orders — 4-hour maintenance window, risky
Option 2: Change shipping logic to use customer table address — breaks historical accuracy (orders should reflect address at time of purchase)

We chose Option 2 with a twist: orders table has `customer_id` FK, shipping logic joins to customers table for current address, but stores `shipping_address_snapshot` for historical records.

**The result:**
- Address updates: 2 seconds (500 rows) → 5ms (1 row)
- Marketing campaigns: zero bounce rate from stale addresses
- Customer service: no more "wrong address" complaints
- Query cost: added one JOIN to order history page, 150ms → 180ms (20% slower, but still under 200ms target)

**Lesson learned:**
Normalization isn't academic — it prevents real production incidents. The 20% query slowdown was acceptable for data integrity. We denormalized selectively (address snapshot on order) for historical accuracy, but kept current data normalized.

---

### Rule of Thumb

1. **Normalize to 3NF** for transactional systems — prevents anomalies, ensures consistency
2. **Denormalize selectively** for read-heavy reports — measure first, document why
3. **Hybrid approach** — normalized source of truth, denormalized cache or snapshots for history
4. **Let query performance guide denormalization** — not laziness, not "might be faster someday"
