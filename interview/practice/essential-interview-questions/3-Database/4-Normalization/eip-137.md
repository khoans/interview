## 137. What is 3NF (Third Normal Form)?

**3NF requires 2NF plus no transitive dependencies — non-key columns must depend only on the primary key, not on other non-key columns.**

If a column can be derived from another non-key column, it's transitive and violates 3NF.

**Requirements:**
- Already in 2NF
- No transitive dependencies — non-key columns don't depend on each other
- Every non-key column depends only on the primary key

**Why it matters:**
- Transitive dependencies cause redundancy — derived data stored multiple times
- Update anomalies — change city, forget to update zip code (or vice versa)
- Inconsistency — same data, different values in different rows

**Trade-off:**
- 3NF splits tables — more tables, more foreign keys
- Queries require more JOINs — 5-10% performance cost
- But data is consistent — single source of truth

**Real-world example:**
Our orders table had: order_id, customer_id, customer_name, customer_city, customer_zip. City and zip depended on customer_id, not order_id. Customer moved — update 50 orders, some missed. Orders showed different cities for same customer. Normalized: customers table (customer_id, name, city, zip) + orders (order_id, customer_id, ...). Customer update: one row. Zero inconsistencies since. Query added one JOIN — negligible with proper indexing.

**Rule of thumb:** Ask: "Can this column be derived from another non-key column?" If yes, move it. 3NF is the sweet spot for most OLTP systems.
