# Project Overview

The project is a vehicle marketplace that connects private sellers, car dealerships, and customers. It is one of the largest automotive marketplaces in the United States, operating across multiple brands including Autotrader, Kelley Blue Book (KBB), and Ford Blue Advantage, with around 30 million user visits per month. The system is very large with approximately 200 developers across multiple teams, and my scope is tied to the customer-facing sites, specifically the Vehicle Search Results Page on both frontend and backend.

On the backend, we use Java with Spring Boot in a microservices architecture. The main search service is the LSC (Listing Search Composition) — an aggregation service that interacts directly with the Apache Solr search engine and satellite services for listings, dealers, pricing, and personalization. The system is heavily integrated with AWS — almost all major services are deployed in two regions (us-east-1 and us-west-2) to prevent region outages. Many components use serverless platforms such as AWS Lambda, AWS SQS, and AWS SNS. We also built a Natural Language Search feature using AWS Bedrock, allowing users to search with queries like "red SUV under $30k near Atlanta."

On the frontend, we use a microsite architecture with different technologies. The main search pages use Next.js with a custom Node.js framework wrapper (bonnet-next) for middleware and SSR. Other pages use pure Next.js, WordPress (homepage), and Sitecore CMS (blog/editorial). All frontend applications share a unified UI component library with accessibility support for users with visual impairments.

For data storage, we use PostgreSQL for relational data in satellite services and DynamoDB for high-throughput, low-latency access patterns. ElastiCache (Redis) handles caching at the application layer, and CloudFront serves as the CDN for static assets and cacheable API responses.

For deployment, all major services run on AWS Elastic Beanstalk with Docker on EC2. The LSC and main frontend auto-scale from 20 to 100 instances based on traffic. CI/CD is handled by GitHub Actions with three environments: PR environments (auto-provisioned per pull request), staging, and production with rolling deployments across both AWS regions.
